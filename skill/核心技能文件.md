# 核心技能文件 · AI 漏洞挖掘的边界与报告标准

> 本文件只装**边界**和**报告标准**，不装方法论。具体怎么测、用什么 payload、走什么路径，
> 由你自主决定，并按需检索深度 playbook（见「决策树」末尾的 SKILL 路由）。
> 每次切换测试方向前，回头重读一遍下方「速查卡」。

---

## 1. 垃圾洞清单（置顶 · 第一眼就要看到）

下列发现**绝对不报**，除非你能给出"被利用后造成实际后果"的完整 PoC：

- CORS 跨域配置 —— 除非证明窃取了具体数据
- Sourcemap / .map 泄露 —— 配置问题，不可利用
- HTTP 安全头缺失（X-Frame-Options / CSP / HSTS）—— 理论风险
- 版本号 / 中间件指纹 / 框架特征 —— 信息收集副产品
- Self-XSS —— 只能攻击自己，无危害
- SSL/TLS 配置警告 —— 除非是严重降级攻击
- 单独的开放重定向 —— 无法链式利用时几乎无害
- Rate limiting / 限频缺失 —— 功能建议，不是漏洞
- 目录列举 / 默认页 / 报错堆栈 —— 现象，除非据此拿到结果
- **任何没有可重现 PoC 的发现** —— 不能重现的就不存在

> 出现新的噪音模式 → 立即追加到本清单。

---

## 2. 灵魂金句

> **现象不是漏洞，漏洞是结果。**
> **报的是结果（越权 / 注入 / RCE），不是过程（信息泄露 / 配置问题）。**

写报告前先自问：「我报的是一个现象，还是一个结果？」是现象 → 不报。

| 这是现象（不报） | 这是结果（报） |
|---|---|
| 发现内部 API 路径 | 通过它越权访问了其他用户数据 |
| JS 里暴露 API Key | 用该 Key 成功调用了付费接口 |
| CORS 允许任意来源 | 配合 XSS 实际窃取了用户 token |
| 存在 debug 接口 | 通过 debug 接口执行了任意命令 |
| 源码里有 SQL 拼接 | 构造 payload 成功提取了库内容 |

---

## 3. 速查卡（防遗忘 · 长对话中反复回看）

- CORS ≠ 漏洞　·　无 PoC ≠ 漏洞　·　现象 ≠ 结果　·　Self-XSS = 垃圾
- 状态码 200 ≠ 业务漏洞（逻辑漏洞看业务后果，不看返回码）
- 「可能」不报，只报「已证明」
- 报告必须带 curl / 原始 HTTP 包；P3 以下不写报告
- 物理证据 > 自我声明（说做了 ≠ 真做了，落盘才算数）
- 20 分钟无进展 → 换攻击面，别死磕
- 长链路用 PLAN（前置规划），短链路用 TODO（边走边测）
- 业务逻辑漏洞几乎都是长链路 → 先建模、再拆 TODO、按清单执行
- 越界即停：超出授权范围（含跨子系统）立即停手并标记 NEED_INPUT

---

## 4. 铁律（绝对不做的危险操作）

- 不在授权范围外的任何资产上发包、扫描或建立连接
- 不执行破坏性操作：删库 / 删文件 / 改数据 / 关服务 / DoS / 压测式爆破
- 不做横向扩散：不主动攻击发现的第三方系统、跨子系统不越界
- 不外传数据：不把目标数据上传到外部服务，不留持久化后门
- 不为绕过限制而伪造授权、不从截图里读凭据自行登录（遇登录墙 → 停，标 NEED_INPUT）
- 写文件只写指定临时目录，禁止写入自身读取路径的子目录（防自循环）
- 验证危害到"足以证明"即止，不做超出取证所需的进一步利用

## 5. 七问验证门（报告前逐条自答，全过才写）

| # | 问题 | 不通过则 |
|---|---|---|
| 1 | 在授权范围内吗？ | 停止，不报 |
| 2 | 有完整可重现的 PoC（curl / HTTP 包）吗？ | 补 PoC，没有就不报 |
| 3 | 解释危害需要假设或推测吗？ | 需要假设 → 危害不够直接，不报 |
| 4 | 影响是已证明的还是「可能」的？ | 「可能」不报 |
| 5 | 这是现象还是结果？ | 现象不报 |
| 6 | 不懂安全的开发者能看懂危害吗？ | 写更清楚，或危害不明显 → 不报 |
| 7 | 发到漏洞平台会被接受还是关闭？ | 会被关闭 → 不报 |

第 7 问是终极检验：你自己都觉得会被关掉的，别浪费时间写。

## 6. 决策树（动态分支，不是固定流程）

先收集流量、理解业务，再按目标特征选下一步；不穷举，给的是"怎么选"的框架。

- **按登录态分支**
  - 有登录态 → 优先越权 / IDOR / 业务逻辑　→ `idor-broken-object-authorization`、`business-logic-vulnerabilities`、`api-authorization-and-bola`
  - 无登录态 → 优先未授权访问 / 认证绕过 / 注入　→ `authbypass-authentication-flaws`、`sqli-sql-injection`
- **按技术栈分支**
  - Java 中间件 → 反序列化 / JNDI / EL　→ `deserialization-insecure`、`jndi-injection`
  - PHP → 文件包含 / 类型混淆 / SQL　→ `path-traversal-lfi`、`type-juggling`
  - Node/前端重 → 原型链污染 / DOM XSS　→ `prototype-pollution`、`xss-cross-site-scripting`
- **按功能分支（业务逻辑优先，单系统具体分析）**
  - 支付 / 充值 / 套餐 → 金额篡改 / 精度 / 负数 / 0 元购 / 回调伪造 / 越权 capture
  - 优惠券 / 积分 / 邀请 → 叠券 / 无限链 / 关单后仍可用
  - 下单 / 库存 / 秒杀 → 竞态 / 双花 / 超卖　→ `race-condition`
  - 第三方 / 社交登录 → 回调绑定 / state　→ `oauth-oidc-misconfiguration`
  - 上传 / 导出 / 下载 → 文件类、路径穿越　→ `file-access-vuln` 路由
- **按 ROI 分支**：认证 / 支付 / 数据导出 等高价值功能优先
- **时间约束**：任一方向 **20 分钟无进展 → 必须换攻击面**

> 不确定该进哪条分支时，先走 P0 主路由 `hack`，再到 P1 分类路由
> （`auth-sec` / `api-sec` / `injection-checking` / `recon-for-sec` / `business-logic-vuln` / `file-access-vuln`）。
> SKILL 的作用是**兜底与激活攻击面灵感**，不是限制——始终保留动态调整的自由度。

## 7. 报告格式要求（仅 P1/P2/P3 才写；正文 ≥ 200 字）

```
---
severity: P1 | P2 | P3
title: 简明漏洞标题
target: 目标 URL / 资产
type: 漏洞类型（越权 / 注入 / 逻辑 / RCE …）
date: YYYY-MM-DD
---
## 漏洞描述（是什么结果，不是什么现象）
## 影响范围（已证明的实际后果）
## 复现步骤（含可直接执行的 curl / 原始 HTTP 包）
## PoC（请求 + 响应关键证据）
## 修复建议
```

## 8. 终止协议（状态标记 · 只在最后一条消息的独立行输出一个）

- `VULN_FOUND` —— 发现有 PoC 的真实漏洞，且已写出有效报告
- `LOW_ROI` —— 已充分测试，无有价值发现
- `NEED_INPUT` —— 需要人工输入（登录二次核身 / 验证码 / 越界确认）
- `ERROR` —— 系统错误，需排查

> 标记必须与磁盘证据一致：声明 VULN_FOUND 却无有效报告 → 视为 LOW_ROI。

---

## 9. 防遗忘指令

- 每次决定切换测试方向前，先回顾「速查卡」，确认没有遗忘任何规则。
- 连续测试超过 30 分钟，暂停并重读「速查卡」与「垃圾洞清单」。
- 写任何报告前，逐条跑完「七问验证门」。
