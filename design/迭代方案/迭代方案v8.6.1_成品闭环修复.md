# v8.6.1 成品闭环修复方案

## 目标

v8.6.1 不再增加测试技巧，只修复 v8.6 运行时合同，使每轮授权测试都能稳定产生三种可持久结果之一：已证明 Finding、可信 Negative、结构化 Pending/Deferred Intent。

## 闭环合同

### 1. Surface 身份合同

- 唯一键为 `METHOD /path`。
- 覆盖格唯一键为 `METHOD /path × vuln_class`。
- BusinessGraph、Scheduler、CognitiveState、CoverageLedger、Blackboard 必须使用同一归一化函数。
- 仅在历史数据缺少 method 且当前路径只有一个 method 时，允许兼容匹配。

### 2. 预算合同

- `surface_budget=N` 表示本轮最多授权 N 个 `METHOD /path` Surface。
- 入选 Surface 的所有 vuln class 格都可以闭合。
- `surface_budget=0` 表示不限制，不得阻断动态新发现的 Surface。
- `intent_budget=0` 表示不限制继承 Intent。

### 3. Intent 生命周期合同

- 有效路径：`pending -> claimed -> pending/claimed -> completed|deferred|blocked|abandoned|superseded`。
- 每次 claim 增加 attempts。
- 无新 Fact 且 attempts < 3 时释放回 pending，不得悬空在 claimed。
- attempts >= 3 时必须以结构化 reason deferred。
- 本轮结束时不得持久化 claimed Intent。
- 新 Fact 必须记录 `source_intent_id`，避免把同轮偶然发现误归因给 Intent。

### 4. Blackboard 权威记忆合同

- 任意 SID 格式都能写回，不假设 `run_NNN`。
- Fact 重编号时同步重写 Intent 的 `source_fact_id`。
- Intent 更新必须保存 attempts、reason、resolved_at、spawned_facts 和 outcome_summary。
- 深阴性按 method/endpoint/param/vuln 匹配；浅阴性始终保持开放。
- 动态发现 Surface 入矩阵后，必须再次应用历史深阴性。
- 写回失败必须进入运行结果的 persistence error，不得静默吞掉。

### 5. 业务域与终态合同

- target domain 只影响排序，不删除 BusinessGraph 的其他域标签。
- `domains_covered` 为跨 Run 累计值，不得在新 Run 中回退。
- `surface_index.value` 只能是 high/medium/low。
- LOW_ROI 必须基于 canonical Surface 覆盖状态判断。

### 6. 发布与兼容合同

- Engine Mode 加载 `核心技能文件.v3.md`。
- SKILL 版本升级为 8.6.x。
- `--resume --sid <legacy>` 在未显式指定 project 时能读取旧 `runs/<sid>` 会话。
- project slug 必须清洗，不允许路径穿越。

## 验收门

1. 全量 pytest 通过，且 D3-D6 具备行为测试，不只搜索源码字符串。
2. `python3 run.py --self-check` 全通过。
3. POST inventory + surface budget 可正常闭格。
4. 使用默认 `sess-*` SID 连续两轮，Blackboard 正常累积。
5. 深阴性在第二轮动态发现后仍能立即闭格。
6. 没有任何 claimed Intent 残留到 Blackboard。
7. summary、run_summary、business_graph、run_scope、blackboard 的键和统计一致。
