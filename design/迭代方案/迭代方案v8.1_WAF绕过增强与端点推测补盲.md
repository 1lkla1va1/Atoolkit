# 迭代方案 v8.1 · WAF 绕过增强与端点推测补盲

> **版本**: v8.0 → v8.1 (补丁级迭代)
> **日期**: 2026-07-05
> **驱动数据**: 天积商城靶场 v8.0 实测评分 3190/4800 (66.5%)
> **设计目标**: 恢复可改善的 650 分丢失 (WAF 绕过 400 + 端点推测 250)，预期提升至 3600-3840 分

---

## 0. 设计哲学（延续 v8.0）

v8.0 的核心原则不变：**别教模型怎么打洞，告诉它哪里可能有洞，然后让它自己想办法。**

v8.1 是对 v8.0 的**补丁级迭代**，不改变架构、不新增约束类别、不回退到 v7.x 的处方模式。只做两件事：

1. **给 WAF 绕过提供更丰富的思路分类**（不是给固定 payload，而是给绕过维度）
2. **给端点盲测增加"变体推测"规则**（不是给固定路径列表，而是给命名模式推理）

v8.1 新增约束数：0。新增启发性参考：2 处。认知带宽影响：< 5%。

## 2. Fix-1: WAF 绕过进阶策略增强 (预期 +200~300 分)

### 2.1 问题诊断

v8.0 在 product-delete.php SQLi 测试中尝试了 7 种变体（UNION/OR/AND/hex/注释/大小写/无空格），全部被 WAF 拦截，返回"检测到非法关键字"。upload.php 的 filename 被 sanitize（`../../../tmp/evil.php` → `test.jpg`）。

**根因分析**：

v8.0 的 WAF 对抗协议（§7 决策树）和 reference.md 的 WAF 绕过技术参考覆盖了**传统绕过技巧**（编码、混淆、注释、空白替换），但缺少四个现代 WAF 绕过维度：

1. **传输层绕过**：HTTP 协议特性（分块编码、参数数组化、multipart 解析差异、Content-Type 混淆）
2. **解析层绕过**：后端框架/DB 与 WAF 的解析差异（参数名污染 `id[]=1`、JSON 嵌套、Unicode 规范化差异）
3. **逻辑层绕过**：利用业务逻辑绕开 WAF 规则（合法前缀+恶意后缀、参数拆分重组、多步骤注入）
4. **等价表达式替换**：同一 SQL 语义的多种等价写法（`OR 1=1` → `OR 2>1` → `OR NOT col IS NULL` → `OR TRUE`）

v8.0 的 reference.md WAF 章节只列了"关键字匹配型→大小写混淆→注释穿插→空白替换"这类**字符串层面**的绕过，没有深入到协议层和解析层。

### 2.2 修改方案

**修改文件**: `skill/skillmode-reference.md` §WAF 绕过技术参考（第 135-153 行）

**修改策略**: 保留现有"按拦截类型"分类，在其后追加"进阶绕过维度"章节。不给固定 payload，给**绕过维度分类和思路提示**。

**具体 diff**:

在现有 `**绕过成功判定**` 段落后（第 152 行后），追加以下内容：

```markdown
**进阶绕过维度（传统方法失效时）**

当编码变体、大小写混淆、注释穿插等传统方法全部失败时，说明 WAF 在字符串层面做了充分覆盖。此时需要从更深的维度寻找绕过空间：

- **传输层绕过**：利用 HTTP 协议特性制造 WAF 与后端的解析差异
  - 参数数组化：`id=1' OR 1=1--` → `id[]=1' OR 1=1--`（PHP 接收数组，WAF 可能只检字符串）
  - 参数名污染 (HPP)：`id=1&id=' OR 1=1--`（不同中间件对重复参数的取值规则不同）
  - multipart 解析差异：将 payload 放在 multipart body 的非标准位置
  - Content-Type 切换：同一请求体以 `application/json` / `application/x-www-form-urlencoded` / `multipart/form-data` 分别发送，WAF 可能只解析其中一种

- **解析层绕过**：利用后端引擎与 WAF 的 Unicode/编码规范化时序差
  - Unicode 规范化差异：WAF 看到 `%u0027` (') 放行，后端规范化后得到 `'`
  - 宽字节注入：GBK 环境下 `%df%27` → 后端视为合法宽字符 + `'`，WAF 视为独立字节
  - JSON 嵌套/类型混淆：`{"id": "1 OR 1=1"}` vs `{"id": 1}` vs `{"id": [1, "OR 1=1"]}`
  - 注释嵌套：`/**/UNION/**%0a/SELECT` — WAF 和 DB 对注释边界的理解可能不同

- **逻辑层绕过**：用 WAF 不认识的等价表达式替代已知关键字
  - 算术等价：`OR 1=1` → `OR 2>1` → `OR 3-2=1` → `OR NOT 0`
  - 函数等价：`UNION SELECT` → `UNION ALL SELECT` → `GROUP BY` + `HAVING` 泄露
  - 字符串构造：`'admin'` → `CONCAT('ad','min')` → `CHAR(97,100,109,105,101)`
  - 条件表达：`IF(1=1,a,b)` → `CASE WHEN 1=1 THEN a ELSE b END` → `ELT(1,a,b)`

- **盲注特化策略**：当显注 (UNION/OR/AND) 全部被拦但盲注可能存活时
  - 优先使用布尔盲注：不注入关键字，只观察响应差异（行数/内容/长度变化）
  - 时间盲注作为后备：`SLEEP(5)` 被拦 → `BENCHMARK(10000000,SHA1('a'))` → 条件延迟
  - 基于错误的注入：`extractvalue(1,concat(0x7e,(SELECT ...)))` — 不走 UNION/OR 关键字

**WAF 绕过优先级建议**：传统字符串绕过 → 等价表达式替换 → 传输层/解析层 → 盲注特化。每升一级需要的请求数更多，但绕过的概率也更高。
```

**不做的事**:
- ❌ 不恢复 v7.3 的"5 族 × 2 变体"硬约束 — 会消耗认知带宽

- ✅ 建议优先级顺序 — 避免模型在传统方法上死磕后不知道还有什么可以试

## 3. Fix-2: 端点变体推测协议 (预期 +250 分)

### 3.1 问题诊断

v8.0 未发现 create-order.php，只发现了 create-batch-order.php。两次测试（v7.3 + v8.0）都只找到了 batch 版本。

**根因分析**：

v8.0 的"未知端点盲测协议"（§7 决策树第 241 行）要求对 JS/HTML/API 响应中发现的路径发 probe 请求。这覆盖了**显式出现**的端点。但 create-order.php 没有在任何前端代码中出现——它是一个"约定俗成"的 REST 端点。

模型需要一种**从已知端点推测未知端点**的能力：当发现 `xxx-batch` 时，自动推测 `xxx` 是否存在；当发现 `xxx-list` 时，推测 `xxx-detail` 是否存在。这不是给一个固定路径字典让模型逐个扫描（那会回到 v7.3 的处方模式），而是给**命名模式推理规则**——让模型根据已发现端点的命名规律主动推测。

### 3.2 修改方案

**修改文件**: `skill/核心技能文件.v3.md` §7 决策树 — 未知端点盲测协议（第 241 行）

**当前文本**:
```
- **未知端点盲测协议**：在 JS / HTML / API 响应 / 错误页面中发现的每一个 URL 和路径，不管是否已在攻击面清单中，都必须发送至少一次探测请求（GET + POST），观察其功能和响应结构。特别关注：JS 中动态拼接的路径（模板字符串、变量拼接）、HTML 表单 action 属性（尤其非 API 路径）、API 响应中的 `_links` / `next_url` / `redirect` 字段、错误页面泄露的路径信息、不同角色加载的不同 JS 文件中的端点差异。发现新功能端点 → 立即加入攻击面清单并测试。
```

**修改后文本**:
```
- **未知端点盲测协议**：在 JS / HTML / API 响应 / 错误页面中发现的每一个 URL 和路径，不管是否已在攻击面清单中，都必须发送至少一次探测请求（GET + POST），观察其功能和响应结构。特别关注：JS 中动态拼接的路径（模板字符串、变量拼接）、HTML 表单 action 属性（尤其非 API 路径）、API 响应中的 `_links` / `next_url` / `redirect` 字段、错误页面泄露的路径信息、不同角色加载的不同 JS 文件中的端点差异。发现新功能端点 → 立即加入攻击面清单并测试。
  - **端点变体推测**：对已发现的每个端点，根据命名规律推测可能存在的"兄弟端点"并发送 probe 请求。常见命名模式（启发性提示）：`xxx-batch` / `xxx-list` → 推测 `xxx`（单条操作）；`xxx-detail` → 推测 `xxx-list` / `xxx-edit` / `xxx-delete`；`xxx-create` → 推测 `xxx-update` / `xxx-delete` / `xxx-read`；带版本号的端点（v1/v2）→ 检查其他版本是否存在。推测出的端点发送 GET + POST probe，有新发现则加入攻击面清单。这不是字典扫描——只推测与已知端点有明确命名关系的变体。
```

### 3.3 为什么不加更多端点推测模式

**不做的事**:
- ❌ 不给一个 50 条的"常见 API 路径字典"让模型逐个扫 — 回到 v7.3 处方模式
- ❌ 不要求"每个端点必须推测 N 个变体" — 增加硬性约束
- ❌ 不在 Phase 0 侦察中加"REST API 约定扫描" — Phase 0 已经够重了

**做的事**:
- ✅ 给 4 个命名模式示例 — 让模型知道"可以从命名规律推测"
- ✅ 强调"只推测有明确命名关系的变体" — 避免变成字典扫描
- ✅ 放在"未知端点盲测协议"下 — 与现有协议无缝衔接，不新增独立章节

## 4. Fix-3: 直觉探索阶段强化 (预期间接提升)
