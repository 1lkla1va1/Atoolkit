### 1.4 核心矛盾

```
│           │                     │                         │
│           ▼                     ▼                         │

│    + blackboard.json       + blackboard.json              │
└─────────────────────────────────────────────────────────┘
                        │
                        ▼

                       │
```

### 2.2 与v8.2.1的关键区别

| 维度 | v8.2.1 | v8.3 |
|------|--------|------|
| 并行方式 | 4个agent全程并行 | 2个agent(Phase 1) + 1个agent(Phase 2+3) |
| 输入验证 | idor_input agent兼做(预算不足) | 专属input_precision agent(完整预算) |
| 上下文传递 | 各agent独立，无跨agent上下文 | 黑板机制(blackboard.json)传递发现+阴性+payload响应模式 |
| 测试账号 | 4角色(缺merchant1) | 强制5角色(含merchant1) |
| 缝隙检查 | 只在聚合阶段 | Phase 1→2过渡时 + 最终聚合时(双重) |
| §8.5直觉探索 | 有(发现了SMS模拟器等) | 保留，作为Phase 3独立阶段 |

### 2.3 黑板协议（Blackboard Protocol）

Phase 1每个agent在测试过程中维护`state/blackboard.json`：

```json
{
  "phase": 1,
  "agent": "txn_idor",
  "findings": [
    {"endpoint": "api/user/refund.php", "param": "refund_amount", "vuln_type": "amount-tamper", "severity": "P1"}
  ],
  "negatives_with_context": [
    {
      "endpoint": "api/user/search.php",
      "param": "keyword",
      "payloads_tried": ["'", "OR 1=1-- -", "UNION SELECT", "SLEEP(3)"],
      "response_pattern": "empty products array for all payloads",
      "suspected_filter": "parameterized queries or input sanitization"
    }
  ],
  "request_format_notes": [
    {"endpoint": "api/user/create-batch-order.php", "format": "JSON", "note": "form-urlencoded returns '参数不能为空'"}
  ],
  "discovered_endpoints": [
    {"path": "api/user/balance-records.php", "found_in": "JS analysis", "params": ["user_hash"]}
  ],
  "session_state": {
    "accounts_available": ["admin", "user1", "user2", "merchant1", "merchant2"]
  }
}
```

Phase 2的input_precision agent启动时读取所有Phase 1黑板，获得：
- 已确认的漏洞（避免重复测试）
- 已判阴性的端点+具体测过什么payload+响应模式（针对性重测）
- 请求格式笔记（避免踩"参数不能为空"的坑）
- 新发现的端点（扩展测试范围）

**这就是v8.2单agent渐进式上下文的"人工恢复"——通过黑板机制让Phase 2拥有等价的信息密度。**

```markdown
## Phased Precision Protocol

Testing is organized into three sequential phases. Phase 1 uses parallel agents
for breadth; Phase 2 uses a single agent for precision; Phase 3 is unstructured
exploration.

### Pre-flight checklist (mandatory before Phase 1)

Before starting any testing, verify:
- [ ] All required test accounts are available (minimum: 2 users, 2 merchants
      of same tier, 1 admin — 5 accounts total for e-commerce targets)
- [ ] Session cookies are fresh and validated
- [ ] `state/` directory is initialized with `session_state.md`

### Phase 1: Breadth Parallel (2 agents)

Two agents run in parallel, each covering high-value surfaces that benefit
from independent exploration:

| Agent | Scope | State directory |
|-------|-------|----------------|
| txn_idor | Transaction/payment + IDOR/privilege escalation | `state/txn_idor/` |
| auth_admin | Auth/verification + admin/merchant permissions | `state/auth_admin/` |

Each agent writes findings, negatives, and a `blackboard.json` to its state
directory. The blackboard records not just what was tested, but the response
patterns observed — enabling Phase 2 to test with full context.

**Input validation surfaces are explicitly NOT tested in Phase 1.** SQLi, XSS,
path traversal, file upload, SSRF, and redirect testing is deferred to Phase 2
to ensure full attention and payload diversity.

### Phase Transition: Seam Check + Blackboard Aggregation

After Phase 1 completes:

1. Aggregate all discovered endpoints from both agents (de-duplicated)
2. For each endpoint, classify parameters by type:
   - State params → auth_admin agent
   - Text/search params → deferred to Phase 2
   - Amount params → txn_idor agent
   - ID params → txn_idor agent
3. Check for `seam_gap`: endpoint × parameter type combinations without
   test records from the responsible agent
4. Merge both `blackboard.json` files into `state/blackboard_aggregated.json`
5. Re-test Phase 1 seam gaps before starting Phase 2

### Phase 2: Precision Single Agent (input_precision)

A single agent performs deep input validation testing with full context from
Phase 1:

1. **Read blackboard**: Understand what Phase 1 found, what was negative,
   and what response patterns were observed
2. **Payload diversity**: For each input validation surface, use multiple
   encoding formats (raw, URL-encoded, double-encoded, Unicode) — a payload
   that fails in one format may succeed in another
3. **Negative replay**: Re-test Phase 1 negative endpoints where the response
   pattern suggests filtering rather than parameterized queries
4. **Payment flow tracing**: Trace the complete payment lifecycle
   (create-order → pay → callback → refund/cancel), testing every endpoint
   in the chain — not just the ones visible in JS
5. **Page-level coverage**: Visit non-API pages (pay/index, merchant/index,
   admin dashboard) with every role cookie, check for authorization bypass

State goes to `state/input_precision/`.

### Phase 3: Intuition Exploration (§8.5)

Unstructured exploration phase per core skill file §8.5. Single agent, no
checklist constraints. May revisit any surface from any phase.

### Final Aggregation

After all phases complete:
1. Merge findings from all phases
2. Final seam check (endpoint × parameter × phase coverage)
3. De-duplicate by root cause: `endpoint + root_cause + affected_role`
4. Renumber to `finding_001`, `finding_002`, ...
5. Generate `summary.json`
```

```markdown
## 12. Payload 多样性与编码感知

### 为什么同一 payload 在不同格式下结果不同

Web 应用通常有多层输入处理：Web 服务器解码 → 框架预处理 → 应用层验证 → 数据库查询。
每一层可能对特殊字符做不同的处理（解码、转义、截断、过滤）。

常见场景：
- PHP `$_GET`/`$_POST` 自动 URL 解码一次，但 `php://input`（JSON body）不解码
- Apache mod_rewrite 对 URL 路径做一次解码，query string 不解码
- WAF 在解码后的字符串上匹配关键字，但应用层在原始字符串上查询
- JSON body 中的 `\u0027`（单引号 Unicode）可能绕过基于字节匹配的 WAF

### 实战案例

| 场景 | 原始格式 | 结果 | 编码格式 | 结果 |
|------|---------|------|---------|------|
| search.php keyword | `' OR '1'='1` | 空结果（被预处理拦截） | `'+OR+'1'%3D'1` (URL编码) | 返回全部商品 |
| product-delete product_no | `' OR 1=1--` | WAF拦截 | `%27%20OR%201%3D1--` | WAF放行 |
| upload category | `../../../etc` | 字面存储 | `..%2f..%2f..%2fetc` | 路径穿越 |

### 编码变体测试顺序（启发性）

当 payload 无效果时，按以下顺序尝试编码变体，而不是立即切换 payload：
1. 原始格式（直接发送）
2. URL 编码（`%xx`）
3. 双重 URL 编码（`%25xx`）
4. Unicode 转义（`\u00xx`，适用于 JSON body）
5. 混合编码（关键字 URL 编码，非关键字原始）
```

```markdown
## 13. 支付流程端点追踪

### 为什么容易遗漏支付端点

电商/商城类应用的支付流程通常由多个端点组成，但前端 JS 可能只暴露部分端点
（如 batch 版本隐藏了单条版本，前端跳转隐藏了 API 端点）。

### 完整支付生命周期

```
create-order ──→ pay/index ──→ payment-gateway ──→ callback
    │               │                                │
    │               └── redirect（支付后跳转）         │
    │                                                 │
    └── create-batch-order（批量版本）                 │
                                                      ▼
                              recharge-create ──→ recharge-callback
                                   │
                                   └── balance-pay ──→ single-balance-pay
                                                           │
                                                           ▼
                                                     refund / cancel
```

### 必须检查的端点

每个支付相关端点都必须：
1. 检查是否有单条/批量两个版本（`create-order` vs `create-batch-order`）
2. 检查支付页面（`pay/index.php`）中的 redirect/return_url/callback_url 参数
3. 检查回调接口是否验证签名来源（不能只依赖前端传来的 sign）
4. 检查充值和支付是否共享余额通道（充值漏洞可能影响支付）
5. 检查取消/退款是否在支付完成后仍可执行

### 常见遗漏模式

- `pay/index.php` 的 GET 参数（redirect, sign, order_no）经常被忽略，
  因为前端 JS 可能只使用 POST body
- 单条操作端点（`create-order.php`）在只有批量版本（`create-batch-order.php`）
  的 JS 中不可见，但服务端可能两个都存在
- 支付回调的 `callback_sign` 可能在 create 接口的响应中泄露
```
