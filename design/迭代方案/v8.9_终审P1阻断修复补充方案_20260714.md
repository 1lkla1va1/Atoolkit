# Atoolkit v8.9 终审 P1 阻断修复补充方案（2026-07-14）

## 1. 背景与决策

v8.9 首版实现通过 340 项回归后，两路独立终审仍复现了 authority 交付链和跨 Run 真值链的 P1 阻断。因此暂停提交/推送，本文作为已冻结主方案的定点补充；不修改主方案原文和 SHA-256。

核心决策：

1. 不再接受单个 `authority_trusted=True` 布尔值作为跨 Run 提交授权。有效信任必须同时由首个 agent 动作前冻结的 authority manifest 证明“全后代进程容器化且已静默”和“模型无法直接写入或硬链接 authority/project inode”。
2. 当前内置 Codex adapter 与 POSIX process-group wrapper 无法提供上述证明，必须 fail closed 为 diagnostic，不改写 ProjectState。这是显式功能边界，不再声称当前 live 后端已可信地实现自动跨 Run 累积。
3. 只有 validator 产出的逐项 `validated_project_truth` 可进入 finalizer。Finalizer 不再直接把 session `inventory.json` / `negative_findings.json` / `dead_ends.json` / `intents.json` 当作真值。
4. commit/receipt 是完整的从 HEAD 到 genesis 可达链，不是“找到当前 target 即成功”。发布前先验证链与 live ProjectState 连续性，再做任何项目真值变更。

## 2. 阻断项与修复合同

### 2.1 Runtime attestation 与默认信任

已复现：`finalize_run()` 默认 `authority_trusted=True`，省略参数即可使 dry-run fixture 提交 revision。`nlink == 1` 也无法检出“建链→改 inode→删 alias”的瞬时硬链接攻击。

修复：

- `finalize_run` 与 receipt API 的默认信任值改为 `False`。
- manifest 新增宿主冻结的 runtime attestation，至少含 `process_containment_verified`、`filesystem_isolation_verified`、`provider`和对应冻结 digest。
- effective trust = 调用方请求信任 ∧ authority manifest 自哈希/身份/计划验证通过 ∧ 进程容器证明 ∧ 文件系统写隔离证明。任一缺失均是 untrusted。
- 内置 Codex/wrapper 冻结两项证明为 `false`；仅确定性测试后端可使用显式的 test attestation。Direct CLI 不提供绕过开关。
- 硬链接叶节检测仍保留作为篡改信号，但文档不再把它描述为可独立防御瞬时硬链接。

验收：省略 trust、仅传 `True`、或 manifest 任一 attestation 为 false 时，均必须 `authority_trusted=false`、`delivery_complete=false`、ProjectState revision 不变。

### 2.2 Commit 链完整性与发布前连续性

已复现：最新 target 一命中 verifier 就退出，未验证 target 自身的 previous 边；ProjectState 被改后仍可发布下一 commit，再使历史 receipt 失效。

修复：

- 建立单一 full-chain verifier，从 HEAD 遍历到 genesis，逐节校验自哈希、唯一 digest、previous 存在、revision 连续、`state_before == previous.state_after`、snapshot digest 与项目/session/authority 绑定。命中 target 只记录 reachable，不提前退出。
- 新 commit 准备之前，先验证现有全链；再要求 live ProjectState revision/digest 精确等于 HEAD `revision_after/state_after_sha256`。genesis 必须从 revision 0 和空 previous 开始。
- 连续性失败必须在 ProjectState、commit HEAD 或 run receipt 任何变更前终止。

验收：篡改最新 previous、删除中间节点、更改 live state 或 snapshot 后，新 Run 不得增加 revision；历史和最新 receipt 均报明确 chain reason。

### 2.3 Receipt anchor 事务、恢复与 diagnostic 命名空间

已复现：anchor 已 create 但 HEAD 未更新时崩溃会留下孤儿；其他 session 可超车；diagnostic receipt 会占用同一 immutable trusted anchor，毒化正式 finalize。

修复：

- untrusted/diagnostic receipt 只写 run-local 产物，`receipt_anchor_path` 为空，不创建或推进 authority trusted anchor/HEAD。
- trusted anchor 发布全部在全局 receipt lock 内完成。每次进入先扫描并验证全链：若恰有一个内容有效、`previous == HEAD` 的孤儿，先恢复 HEAD；多个可竞争孤儿、不可达节点或断链一律 fail closed。
- existing same-session anchor 在内容一致且已可达时按幂等继续；已发布的不同内容仍拒绝。
- receipt verifier 与 commit verifier 一样走到 genesis，验证最新 target 的 previous 边。

验收：覆盖 create-anchor 后崩溃、跨 session 超车、diagnostic→trusted 同 session、最新 target 断 previous 四类回归。

### 2.4 Create-only 与 schema-1 backup 崩溃恢复

已复现：临时 inode 通过 hardlink 发布到最终名后、删除临时名前崩溃，留下 `nlink=2`并使重试永久失败；schema-1 backup 直接 O_EXCL 写最终名，崩溃留下 partial final。

修复：

- create-only primitive 使用可识别的私有临时命名、完整 write+fsync 后再 no-clobber publish。重试时只恢复“同一最终 leaf、同 inode、同期望 payload digest、合法 mode/nlink”的唯一临时项；多义性或不同内容不自愈。
- 恢复或删除临时链接后 fsync 父目录。
- schema-1 backup 改用同一 raw-bytes create-only primitive；不再对最终名分块直写。

验收：在 temp-write、publish-link、temp-unlink 各 crash point 重试均只产生一份完整结果；已存在异内容继续拒绝。

### 2.5 路由模板化与 unknown-method 身份

已复现：`/reports/2024` 和 `/reports/2025` 被无条件折叠，导致一条 NEG 关闭另一固定路由，unknown-method 也共用预算 key。

修复：

- runtime path 默认保留实际规范化路径，不再仅凭数字/UUID/hex 形态模板化。
- 仅当 inventory 显式携带 `path_template` 或已验证 path-parameter schema 时才共享模板身份；否则实例路径独立。
- `_find_cell` 候选唯一性加入实际 path/template identity；method-resolution key 同步使用保守路径合同。

验收：固定年份/数字路由不互相闭格或消耗预算；显式 `{id}` 模板仍可按 schema 继承。

### 2.6 Validator-only canonical commit projection

已复现：finalizer 会把未经逐项验证的阴性、dead-end、inventory 附加项直接写入 ProjectState，包括证据属于 A 但声称 B 的 negative，以及没有 authority discovery event 的 phantom surface。

修复：

- validator 在 proof/closure 验证后输出 `validated_project_truth`，含 `inventory`、`coverage_cells`、`findings`、`negatives`、`dead_ends`、`intents`及逐项 source/hash/binding。该对象纳入 validation self-hash。
- inventory 只接受冻结 run plan 初始项、合法 scope amendment 或单调 authority discovery event。额外 session row 不仅不提交，还使 closure 失败。
- negative/dead-end 必须精确绑定 ledger cell、run plan/amendment、evidence envelope、request/response hash 和物理文件。只有 validator 已验证的行进入 projection。
- finalizer 删除对 session truth files 的直接提交读取；ProjectState 只消费冻结 validation projection。模型自写 intent 默认不进真值，仅保留 validator 或 ProjectState 确定性派生的 intent。

验收：证据错绑 negative、out-of-run phantom inventory、无 event discovery、伪 dead-end 均不改 ProjectState；合法精确行可幂等提交。

### 2.7 同名参数的 location 精确保留

已复现：`param_locations: name -> string` 会让同一 `q` 的 query/body 后写覆盖先写，下一 Run 丢失一个 Cell。

修复：

- ProjectState inventory 新增权威 `param_specs: [{name, location, schema?}]`，按 `(name, location)` 去重；`params`/`param_locations` 仅作旧版派生视图。
- recon、planner、orchestrator seed/restore、CellIdentity 全链优先消费 `param_specs`；空 location 不是 query/body 通配符。
- schema 2 旧数据在读取时保守转成 specs，不猜测第二个 location。

验收：`q@query` 与 `q@body` 在提交、重启、下一 Run 恢复后始终是两个独立 Cell。

### 2.8 Engine negative 产出合同

已复现：Engine prompt 只要求普通 `negative_*.md` 和响应片段，validator 却要求精确 Cell、请求/响应 hash 和机器断言的 JSON/`<machine_evidence>` envelope，导致正常零漏洞 Run 稳定失败。

修复：

- Engine prompt、AGENTS、SKILL 和参考文档共用一份 negative/dead-end machine-evidence schema 和最小示例。
- parser 只把指向合法 envelope 的 evidence ref 作为可闭格候选；普通 Markdown 保留为观察/调度提示，不伪装深阴性。
- 提供确定性 helper，可由 exact cell + raw request/response 文件 + assertions 生成并验证 envelope，避免模型手算 digest。

验收：按 prompt 示例生成的阴性可通过 validator；普通 Markdown 仍保持 open；错 Cell/hash/assertion 失败。

### 2.9 `project_complete` 的可计算语义

已复现：当前 canonical summary 把 `project_complete` 恒写为 `False`，与主方案中 run/project 双层完成语义不一致。

修复：

- validator projection 向 ProjectState 提交已验证的全部 exact `coverage_cells`，包括 open/out-of-run 分母；不只提交已闭的 finding/negative/dead-end。
- ProjectState 幂等合并 Cell；较弱/open 状态不覆盖已验证 terminal，证据失效或 target stale 可显式重开。
- `project_complete` 仅在下列全满足时为 true：存在已解析 inventory；unresolved 为空；每个已知 exact cell 为 proof-confirmed/not-vulnerable/evidence-backed not-applicable 终态；无 pending/high-priority Intent；无已验证 pending candidate/backlog；当前 target fingerprint 不处于 stale。
- summary 同时输出可重算计数和 false reason；diagnostic/untrusted Run 可计算但不得改写 authoritative project completion。

验收：与 run scope 无关的一个 open cell、一个 unresolved、pending Intent/candidate 或 stale 任一存在时为 false；全部清空时可为 true，且不依赖自由文本。

## 3. 实施顺序

1. 先修 runtime attestation 默认与 commit/receipt full-chain 验证，确保不再发生新的非授权真值变更。
2. 修 receipt/create-only/schema backup 崩溃恢复，再跑并发与全 crash-point 测试。
3. 修保守 path identity 和 param specs，先稳定精确分母。
4. 增加 validator-only projection，切断 finalizer 对 session truth files 的直接信任。
5. 补齐 Engine negative 生产 helper/schema 和 `project_complete` 计算。
6. 重跑定向对抗、全量回归、self-check、doctor、CLI、真实旧 Run 迁移复算，再做双路独立终审。

## 4. 发布门禁

- 上述每个复现必须有失败先行的回归测试，修复后通过。
- 全量测试、self-check、doctor、compileall、四个直接 CLI、installer 和生成文件一致性全绿。
- 冻结主方案 SHA-256 保持 `1b2fe0eeced092c0e871441c0f80aa5ab03ef142c0d35a56634c9c50b7c22243`。
- 当前内置 live backend 必须在 README/SKILL/CHANGELOG/验收报告中明确标为 diagnostic；在没有真实 containment/filesystem attestation 之前，不把“可信自动跨 Run 累积”列为已交付能力。
- 任一独立终审仍有可复现 P0/P1，禁止 commit/push。
