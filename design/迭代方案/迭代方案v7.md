## 2. 支柱 A · Skill Mode 一等化

### 2.1 设计原则

**核心思想：把 engine 的"外壳强制"翻译成 Skill Mode 的"模型自检协议"。**

engine 模式下，规则由 Python 代码强制执行（⚙）。Skill Mode 下，没有外壳，但有三个替代执行层：

| 执行层 | engine 模式 | Skill Mode 替代 |
|---|---|---|
| 运行时调度 | `orchestrator.py` 主循环 | **自检清单**：模型每完成一个 surface 后自查 |
| 知识卡注入 | `_knowledge_hint_for_state()` | **内联摘要**：关键规则直接写在核心文件里 |
| 覆盖矩阵 | `CognitiveState.seed_matrix()` | **轻量覆盖表**：markdown 表格，边测边填 |
| 终态裁定 | `session_gate.py` 12 条规则 | **终态自检**：5 条必检项，模型自评 |
| 计时/熔断 | 外部计时器 + turn 计数 | **进度声明**：每 5 个 surface 声明一次时间和进度 |

### 2.2 Skill Mode 运行时框架（写入 SKILL.md）

```markdown
## Skill Mode 运行时框架

当不使用 run.py engine 驱动时，按以下框架运行：

### 启动序列
1. 读 authz.md 确认授权范围
2. 读 hint.md 确认测试策略
3. 执行 Phase 0 侦察（§5），输出 attack_surface_list.md
4. 按 attack_surface_list 初始化覆盖表（§7 格式）
5. 按 hint.md 优先级排序测试队列

### 每 surface 循环
对队列中每个 surface：
1. 声明当前 surface: endpoint/method/param/role/risk_tags
2. 按 risk_tags 查阅对应知识卡摘要（§3 内联附录）
3. 执行测试，对每个 risk 维应答：CANDIDATE / NONE:<reason>
4. 写证据（finding 包 或 negative 记录）
5. 更新覆盖表状态
6. 自检清单（§9）：是否满足 depth floor？是否需要换方向？

### 终态自检
测试结束时，执行 §9.2 "终态自检" 中的全部检查项（不在此重复，避免多源定义）。
```

**落地方式**：修改 `SKILL.md`，新增 "Skill Mode Runtime" 章节，上述内容直接写入。同时在 `核心技能文件.v3.md` 的 §10 新增"Skill Mode 自检协议"节。

## 3. 支柱 A 续 · 知识卡内联

### 3.1 问题回顾

13 张知识卡（`knowledge/cards/*.json`）包含关键的测试阈值（IDOR ≥2 角色×2 对象、注入 ≥3 家族、SSRF 5 向量），但只在 `orchestrator._knowledge_hint_for_state()` 里被加载。Skill Mode 下模型从未读取它们。

### 3.2 方案：三级内联

**L1 · 核心规则内联（进核心技能文件 §6）**

把知识卡中"可抬不可降"的阈值直接写进核心文件的覆盖完整性节：

```markdown
### 阴性 depth floor（从知识卡提取，硬阈值）

| 漏洞类 | 最低向量数 | 最低身份数 | 最低对象数 | 来源卡 |
|---|---|---|---|---|
| SQLi/命令注入 | ≥3 payload 家族（classic/UNION/blind/error） | ≥1 | ≥3 参数 | input-validation |
| SSRF | ≥5 向量（127.0.0.1/0.0.0.0/DNS-rebinding/@-prefix/file://） | ≥1 | ≥3 协议 | 待验证（v2 §6 原有规则，无对应知识卡；第二批实测校准） |
| IDOR/越权 | ≥1 payload | ≥2 角色 | ≥2 对象 + 响应差异 | idor-multi-identity |
| 业务逻辑篡改 | ≥3 金额向量（负/零/超大/精度/参数名变体） | ≥1 | ≥1 订单 | business-logic-tamper |
| 认证流绕过 | ≥3 绕过向量（空值/缺失/复用/替换/类型篡改） | ≥2 角色 | ≥1 | auth-flow-abuse |
| 文件上传 | ≥3 绕过技术（双扩展/替代扩展/null byte/content-type） | ≥1 | ≥3 扩展 | file-upload-risk |

未达阈值的阴性一律标 `shallow_negative`，不得标 `not_vulnerable`。
```

**L2 · 假阴性规则内联（进核心技能文件 §3 速查卡扩展）**

```markdown
### 速查卡扩展 · 假阴性陷阱（从知识卡提取）

- 替换 ID 只换一个就下结论 → 不够，至少 3-5 个（同根因去重）
- A 接口响应字段 → 试着喂给 B 接口（越权高产路径）
- sort/orderBy/filter → 被低估的注入点
- 已测端点无参数 → 不代表安全，检查路径本身（信息泄露/未授权访问）
- 单对象单角色测通 → 不代表安全，换角色换对象再测
- 金额字段只测了正值 → 负数/零/精度/单位换算
- 时间字段只看了返回 → 试着传 order_time/create_time/timestamp
- 认证通过后就不再回头测认证面 → 错，注册/找回/重置/captcha 都是独立面
```

**L3 · 完整知识卡保留原位（engine 模式继续用）**

`knowledge/cards/*.json` 不动，engine 模式下继续由 `_knowledge_hint_for_state()` 按需注入。Skill Mode 下靠 L1/L2 内联摘要覆盖 80% 的关键规则。

### 3.3 具体文件改动

| 文件 | 改动类型 | 改动内容 |
|---|---|---|
| `skill/核心技能文件.v3.md` §6 | 修改 | 新增"阴性 depth floor 硬阈值表"（L1 内容） |
| `skill/核心技能文件.v3.md` §3 | 修改 | 扩展速查卡，新增"假阴性陷阱"节（L2 内容） |

## 4. 支柱 A 续 · 轻量阴性格式

### 4.1 问题回顾

v6.1 要求每条阴性 = 独立 `negative_<surface_id>.md`，带 frontmatter。这个格式是给 engine 的 ingest pipeline 设计的（机器解析 frontmatter → 更新覆盖矩阵）。Skill Mode 下没有 ingest pipeline，写 20+ 个独立文件的 token 开销巨大且无人消费。

### 4.2 方案：双格式并存

| 模式 | 阴性格式 | 消费方 |
|---|---|---|
| Engine Mode（不变） | `negative_<surface_id>.md` + frontmatter | `engine/` ingest pipeline |
| Skill Mode（新增） | `negative_findings.md` 合并文件 + 结构化段落 | 人读 + 后续 session resume |

**Skill Mode 阴性格式模板：**

```markdown
# Negative Findings · <session_id>

## NF-001: search.php SQLi
- **endpoint:** GET /api/user/search.php
- **param:** keyword
- **vuln:** sql-injection
- **roles:** anonymous, user
- **vectors tested:** classic (3), UNION (2), blind-time (2), blind-bool (2), error (1) = 10 vectors
- **result:** all returned normal results or error pages, no data extraction
- **evidence:** response contained 0 products for injection payloads vs 1 for normal
- **depth_floor_met:** yes (≥3 families, ≥10 vectors)
- **status:** not_vulnerable
- **next_actions:** none

## NF-002: product-delete.php SQLi
- **endpoint:** POST /api/merchant/product-delete.php
- **param:** product_no
- ...
```

**关键约束：**
- 每条阴性必须有 `vectors tested` 数量明细（满足 depth floor 可审计性）
- 每条阴性必须有 `depth_floor_met: yes/no`（显式声明是否达到阈值）
- 未达阈值的必须标 `status: shallow_negative` + `next_actions`

**与 v2 §6 的兼容性处理：**
v2 §6 中"禁止单篇合并散文阴性"的规则在 v3 中需修改为"**Engine Mode 下**禁止单篇合并散文阴性（因 ingest pipeline 只认逐格 frontmatter）"。Skill Mode 下允许使用上述合并格式——消费方是人或后续 session 的 resume，不需要 frontmatter 解析。v3 核心技能文件中需同时写明两种模式的格式要求，并标注各自的适用范围。

### 4.3 具体文件改动

| 文件 | 改动类型 | 改动内容 |
|---|---|---|
| `skill/核心技能文件.v3.md` §6 | 修改 | 新增"Skill Mode 阴性格式"节，提供模板；注明"Engine Mode 下仍用逐格独立文件" |

## 5. 支柱 B · Phase 0 侦察协议

### 5.1 问题回顾

决策树第一句"先收集流量、理解业务"，但没有标准流程。本次测试中 4 个 subagent 各自重复发现端点，浪费时间且遗漏了一些方向（V19 审核描述 XSS 就是因为侦察阶段没有发现 submit-audit.php 而完全漏测）。

### 5.2 Phase 0 标准流程

```markdown
## Phase 0 · 侦察与攻击面建模

### 0.1 页面与 JS 分析（5 分钟）
- 获取首页、登录页、用户/商户/管理首页的 HTML
- 提取所有 <script src> 引用的 JS 文件
- 对每个 JS 文件：grep 所有 API 路径（fetch/axios/ajax/\.php 调用）
- 输出：endpoint_inventory.md（端点清单）

### 0.2 API 端点枚举（3 分钟）
- 从 JS 分析提取的所有路径，逐一 curl 检查可达性
- 记录：HTTP 状态码、是否需要认证、响应格式
- 区分：公开端点 / 需认证端点 / 不存在端点
- 输出：更新 endpoint_inventory.md

### 0.3 业务流建模（5 分钟）
- 走一遍核心业务流：注册 → 登录 → 浏览 → 下单 → 支付 → 退款
- 走一遍商户流：注册 → 审核 → 添加商品 → 发货
- 走一遍管理流：登录 → 用户列表 → 商户审核
- 记录每个流涉及的 API 端点和参数
- 输出：business_flow.md（业务流图）

### 0.4 攻击面清单生成（2 分钟）
- 从 endpoint_inventory + business_flow 生成攻击面清单
- 每个 surface = endpoint / method / param / role / risk_tags
- 按 hint.md 优先级排序
- 输出：attack_surface_list.md（攻击面清单）

### 0.5 完整性检查
- 对照 authz.md 中的测试策略，确认每个优先级方向都有 surface
- 对照核心技能文件的决策树分支，确认没有空白方向
- 如果有方向无 surface → 回 0.1 补充 JS 分析或手动探测
```

## 6. 支柱 C · 弹性 depth floor

### 6.1 问题回顾

v6.1 的 depth floor 是硬阈值：注入 ≥3 家族、IDOR ≥2×2、业务逻辑 ≥3 向量。对于"一击即中"的漏洞（如 search.php SQLi 第一个 payload 就成功），强制凑数只浪费 token。

### 6.2 方案：两级 depth floor

| 级别 | 条件 | 要求 | 说明 |
|---|---|---|---|
| **快速确认** | 首个 payload 即成功 + 影响明确 | 1 个成功 + 2 个失败对照 | 证明"确实可注入"而非巧合 |
| **完整确认** | 需要排除误报 或 评估严重度 | 达到 v6.1 原阈值 | 需要证明深度（如 UNION 能提多少列、IDOR 能跨多少角色） |

**判断规则：**

```markdown
### 弹性 depth floor

**快速确认条件（满足全部即可跳过完整确认）：**
1. 首个测试 payload 即产生明确的、可区分于正常响应的结果
2. 影响可以直接证明（如：返回了其他用户数据、执行了注入、修改了他人订单）
3. 不是"可能"或"疑似"，而是确定性的结果

**快速确认仍需的最小证据：**
- 1 个成功请求 + 响应
- 1 个对照请求（正常输入）+ 响应（证明差异是 payload 导致的）
- 如果涉及越权：1 个 owner 请求 + 1 个 attacker 请求 + 响应差异

**何时升级到完整确认：**
- 首个 payload 结果模糊（状态码 200 但内容不确定）
- 需要评估严重度（能提多少数据？能越多少角色？）
- 需要排除 WAF/过滤的误判
- 准备写 P1/P2 报告前
```

### 6.3 具体文件改动

| 文件 | 改动类型 | 改动内容 |
|---|---|---|
| `skill/核心技能文件.v3.md` §6 | 修改 | 将现有 depth floor 规则替换为"两级 depth floor"，增加快速确认条件和升级规则 |

### 6.4 开放重定向链利用判定规则（治 P2-4）

> **归属说明**：此规则概念上属于 §1 垃圾洞清单的补充，在 v3 核心技能文件中应放在 §1 "垃圾洞清单" 末尾，而非 §6 "覆盖完整性" 中。放在此节仅为与评估问题 P2-4 对应。

v2 垃圾洞清单写"单独的开放重定向不报"，但未明确什么情况下构成"链式利用"。导致 V22（支付重定向）测试不充分——agent 不确定该测多深。

```markdown
### 开放重定向的"结果"判定（补充 §1 垃圾洞清单）

**单独存在 = 现象（不报）：**
- redirect 参数指向外部 URL，但无其他漏洞可链

**构成结果（报）的条件——满足任一即可：**
1. 重定向目标页能窃取 token/session（如：redirect 到钓鱼页 + 页面包含登录表单）
2. 重定向可与 OAuth/SSO 回调链式利用，劫持授权码
3. 重定向可绕过 referer 检查（如：支付回调的 redirect 被服务端 302 到外站，泄露了 sign/token）
4. 重定向可与 SSRF 链式利用（如：redirect URL 被服务端 fetch 而非浏览器跳转）

**测试 checklist（每个 redirect 参数）：**
- 替换为外部 URL → 是否 302/303 跳转？
- 替换为含 token 的 URL → 跳转后 URL 是否泄露敏感参数？
- 与 OAuth/支付回调组合 → 能否劫持回调？
- 服务端是否 fetch 了 redirect URL → SSRF 链？
```

## 7. 支柱 C 续 · 轻量覆盖清单

### 7.1 问题回顾

v6.1 要求每个 surface 维护 `endpoint/method/param/role/risk_tag/status/evidence_ref/blocker/next_actions` 9 元组。Engine 模式下由 `CognitiveState` 自动维护，Skill Mode 下靠手工写 JSON，实际执行时变成了"测完再补"。

### 7.2 方案：Markdown 覆盖表

**用 markdown 表格替代 JSON，在 Phase 0 生成后边走边更新状态列。**

```markdown
# Coverage Ledger · <session_id>

| # | surface | status | evidence | depth | note |
|---|---|---|---|---|---|
| S01 | POST /register captcha | not_vulnerable | NF-001 | ✓ 5 vectors | captcha 服务端校验有效 |
| S02 | POST /login enum | not_vulnerable | NF-002 | ✓ 3 vectors | 错误信息统一 |
| S03 | POST /refund amount | confirmed | finding_001 | ✓ 4 vectors | 退款金额无上限校验 |
| S04 | GET /order-detail idor | confirmed | finding_004 | ✓ 2×2 | 跨用户读取 |
| S05 | GET /balance-records | shallow_negative | NF-005 | ✗ 数据为空 | 需创建余额记录后重测 |
```

**关键约束：**
- `status` 列只允许：`not_tested / confirmed / not_vulnerable / shallow_negative / blocked`
- `depth` 列只允许：`✓`（达到 depth floor）或 `✗ <原因>`
- 终态时 `not_tested` 的高价值 surface → 终态标 `incomplete`

**并行模式下的覆盖表协调：**
多 agent 并行时，每个 agent 各写一份带方向前缀的覆盖表（如 `coverage_auth.md`、`coverage_txn.md`、`coverage_idor.md`、`coverage_input.md`）。Phase 0 生成的 `attack_surface_list.md` 是共享的初始清单，各 agent 只更新自己负责的 surface 行。聚合阶段（§8.2）合并为统一的 `coverage_ledger.md`，去重后重新编号。

### 7.3 具体文件改动

| 文件 | 改动类型 | 改动内容 |
|---|---|---|
| `skill/核心技能文件.v3.md` §6 | 修改 | 新增"Skill Mode 轻量覆盖表"格式；注明"Engine Mode 下继续用 coverage-ledger.json" |

## 9. 支柱 C 续 · 自检查清单

### 9.1 问题回顾

⚙ 标记的规则（20 分钟切向、50 轮熔断、风险维展开）依赖外部执行器。Skill Mode 下没有执行器，规则形同虚设。

### 9.2 方案：自检点 + 进度声明

```markdown
## §11 Skill Mode 自检查清单

### 每 5 个 surface 自检（或每 15 分钟，先到为准）
□ 过去 5 个 surface 中，是否有连续 3 个 shallow_negative？→ 换攻击面方向
□ 是否有 surface 耗时超过 10 分钟？→ 评估是否该放弃并标 blocked
□ 当前方向是否已经有 confirmed finding？→ 继续覆盖剩余 surface，不要停在首洞
□ 是否遗漏了某个 risk 维的应答？→ 回看决策树分支

### 每 15 个 surface 自检
□ 回顾速查卡，确认没有遗忘规则
□ 回顾垃圾洞清单，确认没有误报
□ 检查覆盖表：高价值 surface 是否全部已测试？
□ 检查 negative 文件：depth floor 是否全部达标？

### 终态自检（测试结束时）
□ 所有高价值 surface 已测试（不是 not_tested）？
□ 所有 confirmed 漏洞有 finding 包？
□ 所有 not_vulnerable 的 depth_floor_met = yes？
□ 所有 shallow_negative 有 next_actions？
□ 覆盖表与实际 evidence 文件一致？

### 终态判定规则
- 有 confirmed finding + 所有高价值 surface 已闭合 → VULN_FOUND (complete)
- 有 confirmed finding + 存在未闭合高价值 surface → VULN_FOUND (incomplete)
- 无 confirmed + 所有 surface 已充分测试 → LOW_ROI
- 有 blocker 无法绕过 → NEED_INPUT
```

| 文件 | 改动类型 | 改动内容 |
|---|---|---|
| `skill/核心技能文件.v3.md` | 新增 | §11 "Skill Mode 自检查清单" |

### 10.4 Skill Mode Finding Schema 精简（治 P2-5）

`engine/reporting/schema.py` 定义的 finding.json 包含 20+ 字段（含条件必填的 `source_proof`、`crypto_chain`、`manual_burp_replay`、`apis`）。Skill Mode 下模型难以记住所有字段，导致频繁遗漏。

**Skill Mode 精简 schema（必填 8 字段 + 条件必填 3 字段）：**

```json
{
  "schema_version": "1.0",
  "id": "finding_auth_001",
  "title": "漏洞标题（一句话结果描述，非现象）",
  "severity": "P1|P2|P3",
  "vuln_type": "idor|sqli|xss|ssrf|auth-bypass|amount-tamper|...",
  "target": "POST /api/user/refund.php",
  "risk": {
    "proven_impact": "已证明的业务影响（不能写可能/疑似）"
  },
  "poc": {
    "file": "poc.sh",
    "steps": ["步骤1 curl 命令", "步骤2 curl 命令"]
  },
  "proof_packets": [
    {"request_file": "request_1.http", "response_file": "response_1.http"}
  ]
}
```

**条件必填（满足条件时必须提供）：**
- `source_proof`：从 JS/前端代码构造请求时必填，写明文件和行号
- `crypto_chain`：涉及加密/签名链路时必填，写明算法和 key 来源
- `manual_burp_replay`：P1/P2 finding 选填（Skill Mode 下 agent 无法运行 Burp Suite；如人工需要复现，由人补写）

**与 engine schema 的关系：** Skill Mode 精简 schema 是 engine schema 的子集。engine 模式下仍用完整 schema（`engine/reporting/schema.py`）。
