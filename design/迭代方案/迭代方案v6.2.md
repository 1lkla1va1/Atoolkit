# Atoolkit v6.2 修复设计：复盘驱动的闭环编排

来源：`runs/shop-blackbox-20260703-002610/atoolkit_postmortem_assessment.md`

日期：2026-07-03

## 1. 结论

本次 740/4800 的主因不是 Atoolkit 没有覆盖理念，而是执行链路没有把覆盖台账、候选台账、完成门和赛后评分串成硬闭环。

对“有没有设计 loop 编排循环”的回答：

- 有：`run_session` 已经不是首洞即停，存在 `max_turns` 外层循环、`state.next_untested()` 队列、`candidate-ledger.json` 和 `session_gate`。
- 不够：`loop_mode`、`proof_budget_floor`、`lens` 等参数之前没有真正参与每轮调度，loop 只是“继续跑下一轮”，不是“recall -> triage -> proof -> coverage -> close”的确定性编排。
- 修复方向：保留模型自主判断漏洞手法，但由外壳决定本轮相位和焦点，减少全矩阵散步、重复 recall、proof_ready 漏进报告、高价值面未闭合等浪费。

## 2. 失败模式映射

| 复盘问题 | 应落地的工程控制 |
|---|---|
| 入口看到了但没有推进验证 | endpoint 一旦进入 inventory，planner 生成 surface，loop phase 强制 recall/triage |
| 发现 4 个漏洞后没有继续覆盖 | VULN_FOUND 不终止，session_gate 未 pass 时终态只能 incomplete |
| 输入验证 0 分 | input-validation surface 必须 depth floor，浅阴性不能闭格 |
| 认证面被登录态掩盖 | auth endpoint 自动增加认证绕过/枚举列 |
| 多身份素材没有矩阵消费 | object/role surface 带 roles/tasks，coverage-first 模式优先高价值 open surface |
| proof-ready 没进报告 | proof phase 永远优先，预算末段进入 proof 保底窗口 |
| 赛后评分没有反馈闭环 | scorecard 只在 run 后读 summary/ledger/findings，不进入攻击 prompt |

## 3. Loop 编排策略

v6.2 把每轮执行划成 5 个 phase：

| phase | 进入条件 | 本轮目标 |
|---|---|---|
| `recall` | 没有候选 | 对 Top surface 做 DIM/CANDIDATE/NONE 逐维应答 |
| `triage` | 有 proposed/triaging 候选 | 推进候选到 proof_ready/refuted/blocked |
| `proof` | 有 proof_ready，或预算进入 proof 保底窗口 | 先写 finding 包，避免已证漏洞漏报 |
| `coverage` | coverage-first 且高价值 surface 未测 | 补认证、交易、越权、输入验证等 open surface |
| `close` | shallow_negative/needs/gaps 未清 | 补阴性证据、标阻塞、生成 coverage_gaps |

关键原则：

- 每轮只推进 Top 队列中的一个候选或一个 surface。
- phase 只控制注意力预算，不教 payload，不替代模型推理。
- proof_ready 永远比继续挖新面优先。
- high_value_not_tested 和 shallow_negative 是完成门，不是提示。

## 4. P0 修复

1. 每轮注入 `Loop 编排器` 块，明确 phase、原因、open pressure、candidate counts、Top 队列。
2. `events.jsonl` 记录 loop 配置和每轮 phase，用于复盘“时间花在哪里”。
3. `proof_budget_floor` 生效：预算尾段停止扩张，优先 proof/成报/收口。
4. `loop_mode=coverage-first` 生效：高价值 surface 未闭合时优先覆盖缺口。

## 5. 下一步 P1

1. 让 `coverage-first` 在候选非空时也能追加 Top open surface 的风险维应答表，而不仅是候选队列。
2. 为交易状态机新增结构化 task：create_object、tamper、transition、replay、verify_accounting。
3. 为认证流新增 block 类型：captcha/sms/manual-needed，不能静默跳过。
4. 新增 `run.py score --sid <sid> --oracle <oracle>`，只做赛后离线评分。

## 6. 验收指标

- 本次这类无 `coverage-ledger.json` / `inventory.json` / `coverage_gaps.md` 的正式 run 必须被标为 `incomplete`。
- 有 `proof_ready` 候选但无 finding 时，终态不得是 complete。
- 每轮 `events.jsonl` 必须能看到 `loop_phase`，复盘可以统计 recall/triage/proof/coverage/close 占比。
- 下一轮 shop 黑盒回归中，认证面、输入验证面、商户/后台角色面至少必须留下 confirmed/not_vulnerable/blocked 之一。
