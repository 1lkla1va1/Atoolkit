## P0 · 必做，直接决定能否稳定过 3000

### P0-1　recon → surface 自动 bootstrap（唯一新模块）+ run.py 硬门

**新增 `engine/surface.py`**：解析 recon 产物，输出 `endpoint/method/param` 清单，复用 `planner` 直出 ledger schema，不另造粒度。

- **JS**：抓 `fetch(/axios/get/post/XMLHttpRequest` 调用 + 字符串字面量里的 `/api/*`、`*.php` 路径；从调用实参与 query 串抽 param。
- **HTML**：`<form action method>`、`<a href>`、`<script src>`、`<img/data-* src>`；`<input name>` 抽 param。
- **HTTP 响应 JSON**：嵌套对象里出现的路径片段（`"/api/..."` 字符串）。
- **`*.har`**（若有）：请求行 `method+path+query`。
- 输出每条带 `source`（`js|html|traffic|har|manual`）与 `last_seen`，供 P1-3 inventory 用。
- 入口函数：`bootstrap(recon_dir: Path) -> list[dict]`，每项含 `endpoint/method/params/source/last_seen`，直接喂 `CoverageLedger.from_endpoints`。

**`run.py` 改造**：

- 增 `--recon-dir <path>`：自动跑 `surface.bootstrap` → `planner.plan_surfaces` → `CoverageLedger.from_endpoints`。
- 增 `--ad-hoc`：显式声明退化单点验证；**无 `--endpoints` 且无 `--recon-dir` 且无 `--ad-hoc` 时，拒绝启动正式覆盖跑**（当前「空→退化为首洞即结」放行，要改成报错退出）。
- 硬门：会话结束若无 `coverage-ledger.json`，终态只能 `incomplete`，不得 `complete`。`_conclude` 已在 `run.py` 路径下总会写 ledger；要补的是「ledger 为空（0 surface）且非 ad-hoc」→ 强制 `incomplete` 并打印原因。

**通用性红线**：不内置任何靶场路径模式、不内置 payload 列表；风险标签只按参数名语义推断（`planner.PARAM_RISK_RULES` 已是此范式）。surface.py 的路径抽取只用通用正则，不出现「天积」「商城」等字样。

**验收**：
- `python3 -m engine.surface --recon-dir fixtures/recon_sample` 输出 surface 列表，每条带 source。
- `run.py --dry-run --recon-dir fixtures/recon_sample` 产出 `coverage-ledger.json`，surface 数 = recon 可发现端点数。
- `run.py --dry-run --target … --authz …`（无 endpoints/recon/ad-hoc）报错退出。
- 代码全文 grep 不出靶场名。

### P0-2　session_gate 显式 `shallow_negative_open` 谓词

**现状**：`shallow_negative` 只在两种情况间接拦——高价值格经 `normalize_status→not_tested` 命中 `high_value_not_tested`；有 `next_actions` 的经 `open_next_actions`。**一个无 next_actions、非高价值的 shallow_negative 格会从 session_gate 漏过**。

**改 `engine/session_gate.py`**：新增显式谓词 `shallow_negative_open`——遍历 surface，任一 `status == shallow_negative`，或其阴性证据未过 `knowledge.negative_sufficient`（向量数/响应数不足）→ `result=INCOMPLETE`，`reason.predicate="shallow_negative_open"`，`action="retest_with_more_vectors_or_mark_not_vulnerable"`。

- 谓词顺序：放在 `high_value_not_tested` 之后、`recoverable_blocked` 之前，保证高价值优先报、shallow 不被吞。
- `knowledge.negative_sufficient` 已存在，直接复用判「向量数/响应数不足」。

**验收**：
- 构造一个非高价值、无 next_actions 的 `shallow_negative` 格，`evaluate_session_gate` 返回 `incomplete` 且 reasons 含 `shallow_negative_open`。
- 全 `not_vulnerable`（达阈值）的 ledger 仍返回 `pass`。

## P1 · 阴性质量与覆盖诚实

### P1-1　skill 文件强制逐格 `negative_*.md`（修复方案阴性通道收口）

**现状**：`_parse_negative` / `harvest_evidence` / `knowledge.negative_sufficient` 通道已建，但 skill 文件不约束格式，实跑用单篇合并 `negative_findings.md`（无 `endpoint:/vuln:` 头）→ 被 ingest 贡献为空。

**改 `skill/核心技能文件.v2.md` §6 或新增小节**：

- **每条阴性 = 独立 `negative_<surface_id>.md`**，frontmatter 含 `endpoint / method / param / vuln / roles / identities / vectors / evidence_types / next_actions`，正文带关键请求/响应片段。
- 证据不足时只能写 `shallow_negative`、留 `next_actions`，**严禁** `not_vulnerable`。
- 经 knowledge cards 抬阈值（已支持「可抬不可降」）：注入类 ≥3 payload 家族；SSRF ≥ `{0.0.0.0, 127.0.0.1, DNS-rebinding, @-前缀, 多协议}`；IDOR/越权 ≥ 2 角色 × 2 对象 + 响应差异。
- 改完运行 `bash codex/regen_agents.sh` 同步 `codex/AGENTS.md`。

**验收**：
- skill 文件 grep 到 `negative_<surface_id>.md` 与 frontmatter 字段清单。
- `codex/AGENTS.md` 含同等约束（regen 后 diff 校验）。

### P1-2　accepted / hit / coverage 分开汇报

**改 `engine/orchestrator.py` `_conclude` 与 `run.py` 打印**：终端摘要同时打印 `accepted_count`、（有 oracle 时）`hit_count`、ledger `{total/closed/high_value_open}`、`shallow_negative_count`、`open_next_actions_count`、未闭合高价值面清单（endpoint+role+risk_tag）。

- `_conclude` 已有 `coverage_ledger / session_gate / shallow_negative_cells`，补 `hit_count`（当传入 oracle 时由 `benchmark_eval` 算）与 `open_next_actions_count`。
- `run.py` main 收尾打印补这几行；现状只打 `closed/total` 与 `session_gate`，要加 `accepted=N hit=N/M high_value_open=N shallow=N next_actions=N` 与未闭合清单。

**验收**：`run.py --dry-run` 收尾输出含上述字段；有 accepted 但 ledger 未闭合时不输出「完整完成」。

### P1-3　discovery 饱和检查 + inventory.json（修复方案 P2-1，v4 未覆盖）

**基于 P0-1 的 `surface.py`**：

- `surface.bootstrap` 产出后落 `runs/<sid>/inventory.json`：每条 endpoint 带 `source / last_seen / discovered_during_testing`。
- 饱和规则：连续 N（默认 3）个来源不再新增 endpoint，才进入深测；深测中新发现 endpoint 必须追加进 ledger 并回 `planner` 补风险格。
- `api_inventory.json` 不再是草稿：能解释每条 endpoint 的 source 与 last_seen。
- session-gate 联动：报告中用到的 endpoint 不在 inventory 且无 `discovered_during_testing` 记录 → 标 `incomplete`（新增谓词 `inventory_evidence_inconsistent` 或复用 `confirmed_evidence_mismatch` 思路）。

**验收**：
- `inventory.json` 每条有 source/last_seen。
- 深测新增 endpoint 出现在 ledger 与 inventory。
- 报告引用未登记 endpoint 时 session-gate 标 incomplete。

## P2 · 复利项，防复发

### P2-1　误判复盘库 → knowledge cards（7 类）

**改 `knowledge/cards/`**：把 7 类误判模式沉淀成 cards（可抬 `negative_sufficient` 门槛）：
单参数 / 单角色 / 单对象 / 单 payload 家族 / 单位误读 / 无回显≠无结果 / 测过端点≠测过参数。

- 现有 5 张（file-upload-risk / input-validation / idor-multi-identity / business-logic-tamper / auth-flow-abuse）保留；新增/补全到覆盖 7 类。

### P2-2　`run.py --self-check` 自检门

**改 `run.py`**：增 `--self-check`，在固定 fixture（recon 目录 + oracle）上断言：
- `surface.bootstrap` 覆盖 oracle 全部端点；
- 故意浅阴性触发 `session_gate=INCOMPLETE`；
- 合格报告被 Guardian `accepted`。

- 复用 `engine/benchmark_eval.py`（离线评估，运行时不导入 oracle 到 prompt）。
- 需要一份最小 fixture：`fixtures/recon_sample/`（JS/HTML/HAR 各一）+ `fixtures/oracle_sample.json`。

**验收**：`python3 run.py --self-check` 退出码 0；故意破坏 fixture 后退出码非 0。

### P2-3　finding 按根因聚合（修复方案 P2-2 引擎落地）

**改 `engine/orchestrator.py` 摘要 / 或新增 `engine/dedupe.py`**：skill §6 已有规则，引擎补实现。

- 聚合键 `finding_key = endpoint + root_cause + affected_role`；多表现写 `facets`（如 `unpaid_recharge / race_replay`），`primary_impact` 取最高。
- 价值排序只影响测试建议，不影响成立性：认证绕过 → 支付/余额 → 对象级授权 → 输入验证/文件/跳转 → 低价值信息暴露。
- 同一充值回调根因不拆成两个独立 critical 计数；summary 同时保留 facet 不丢信息。

**验收**：构造同根因两 facet 的 reports，summary `accepted` 计数为 1，`facets` 列两条。
