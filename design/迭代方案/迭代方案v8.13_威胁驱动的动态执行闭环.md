# Atoolkit v8.13 迭代方案：威胁驱动的动态执行闭环

日期：2026-07-17

## 1. 版本目标

v8.12 已回答“为什么测”：Feature、业务资产、安全不变量和 Threat 决定冻结 coverage 分母。
v8.13 回答“下一步做什么、为什么还不能停”：把每个 Threat 编译成可执行 Experiment Contract，
并依据每轮物理观察动态推进前置条件、测试深度、证明修复和下一动作。

目标链路为：

```text
Feature → Threat → exact Cell
  → Experiment Contract（前置条件 + 证据义务）
  → 真实请求/响应
  → evidence-bound Execution Event
  → Host reducer
  → 动态 Queue / blocker recovery / proof repair
  → Finding 或深阴性
  → Threat closure
```

本版不把通用漏洞清单重新乘到所有 endpoint，也不允许模型在同一 Run 删除或重写 v8.12 的冻结计划。

## 2. 为什么 v8.12 还不够

Threat cell 目前带有 `security_invariant`、`observable_violation`、`preconditions` 和
`evidence_required`，但运行时主要追踪 cell 的最终七态。中间实验过程仍是模型自由文本和文件数量：

- 一个无效对象请求也可能被写成 full negative；
- 已知高价值端点可以一直不执行；
- barrier 出现后没有确定性的恢复阶段；
- Finding 被拒绝只在最终收口时暴露，模型没有及时返工；
- “新增了文件”会刷新进展计时，即使文件没有绑定 exact cell 或可验证观察。

v8.13 增加的是状态机，不是打法数量。

## 3. Experiment Contract

每个 in-scope exact cell 都生成一个合同：

```json
{
  "surface_id": "tm-...",
  "feature_id": "refund",
  "threat_id": "T-refund-overpay",
  "required_obligations": [{
    "obligation_id": "threat-evidence-01-...",
    "source": "threat_model",
    "description": "退款前后余额与订单实付状态"
  }],
  "depth_obligations": ["valid-baseline", "state-before", "state-after"],
  "status": "ready"
}
```

义务来源分两层：

1. 模型威胁推理层：逐项保留 `evidence_required`，用稳定 hash 生成 ID，禁止 Host 改写语义；
2. 确定性最低深度层：根据 identity mode、risk tags、vuln family 和写入/交易语义增加最小实验条件。

最低深度只约束“怎样才有资格下阴性”，不生成新的漏洞 Finding，不扩大 threat denominator。示例：

- `peer_pair` 至少需要 owner baseline、peer attempt、ownership marker；
- 认证枚举至少需要消息、时间、行为三通道；
- 认证绕过至少覆盖字段省略、空/null、类型/结构、流程顺序、绑定/重用等独立族；
- 交易完整性至少有合法基线、边界族、状态前后差异；
- 持久化输入至少有 submit、read-back/render 和跨身份观察；
- SSRF/URL sink 至少执行目的地标记和服务端获取结果检查；安全拒绝与成功回连都可形成实验结果，
  只有后者才能进入 Finding 证明；
- 文件面至少有合法基线、独立绕过族、retrieval/location 结果检查和非文件参数实验；阴性不以
  “上传必须成功”为前提；
- IDOR/BOLA 即使模型错误声明为单身份，也由 Host 强制 owner/alternate identity/ownership marker
  三项最低义务。

这些是实验有效性门，不是“每个 endpoint 都测所有漏洞”的漏洞清单。

## 4. Execution Event

模型每完成一组真实实验，输出单行结构化事件：

```text
EXECUTION_EVENT: {"schema_version":1,"event_id":"...","surface_id":"...",
"completed_obligations":["..."],"evidence_refs":["evidence/..."],
"barrier_signals":[],"outcome":"observed"}
```

Host 只接受：

- `surface_id` 存在且在本轮 frozen denominator；
- obligation ID 属于该 cell 合同；
- evidence ref 为 Run 内普通文件、不可越界且真实存在，Host 在接受时固化每个文件的 SHA-256；
- 完成义务的事件至少有物理证据；
- event ID 首次 create-only；相同重放幂等，异内容冲突拒绝；
- barrier 只能使用已知枚举；未知 signal 保留为观察但不能关闭 cell。

事件自身不证明漏洞或安全。阳性仍必须通过 Finding proof contract，阴性仍必须通过 canonical
negative/depth gate。Event 只证明实验被执行并驱动下一状态，不能替代结果验证。

## 5. 动态状态机

每个合同使用以下状态：

```text
ready → executing → needs_followup → closed
                 ↘ blocked_recoverable
                 ↘ proof_repair
```

- `ready`：前置条件已满足，尚无 evidence-bound event；
- `executing`：已完成部分义务；
- `needs_followup`：有信号但仍缺义务，或已有浅阴性；
- `blocked_recoverable`：空数据、对象不存在、会话失效、格式未解析、角色/挑战缺失；
- `proof_repair`：模型声明阳性或已有 Finding，但 validator 拒绝；
- `closed`：accepted Finding，或深阴性/结构化不适用且实验义务全部满足。

确定性转移：

| 观察 | 下一动作 |
|---|---|
| `empty_dataset/object_absent` | 创建或枚举真实测试对象，随后重做 exact cell |
| `session_expired/auth_required` | 刷新授权会话，重新采基线 |
| `format_unresolved` | 解析 method/content-type/参数位置后重测 |

Barrier 事件保持 append-only，不能由模型文字删除；当且仅当该 barrier 对应的全部 Host recovery
obligation 均有 evidence-bound event 时，reducer 才把它从 active barrier 集合中移除。这样恢复可达，
同时不存在“声称已恢复”旁路。
| `waf_blocked` | 保持 shallow，切换传输/解析/逻辑/盲测族 |
| 写入 200/空响应 | 增加 read-back/render/跨身份闭环 |
| rejected Finding | 按 validator reason 生成 proof repair，不跳到下一 surface |
| 新 endpoint/param | 进入 next-run discovery backlog；同 Run 不直接发包测试 |

## 6. 动态 Queue

Host 每轮只注入有界 Top Queue，每项含 exact identity、当前状态、缺少义务、恢复动作、匹配知识卡和
证据要求。排序：

1. proof repair（避免真洞因包不合格丢失）；
2. 可恢复 blocker；
3. shallow negative 的缺口；
4. 未执行高价值 threat；
5. 其他 open cell。

文件数量和模型文字不再单独构成“有进展”。只有以下事件刷新执行进展：

- 新的 evidence-bound execution event 被 Host 接受；
- accepted Finding 或 canonical negative 使 exact cell 合法推进；
- blocker 被实际恢复并有新的基线证据；
- 经验证的 discovery backlog 新增。

## 7. Frozen plan 与动态发现

v8.13 不在同一 Run 扩大 Threat denominator。原因：若模型看到结果后能添加/删除 threat，规划完整性、
预算和终态都不可复验。

运行中新 endpoint/param 必须绑定来源证据，写入 `execution-backlog.json` 和 authority discovery event；
当前 Run 标记 `next_run_required`。下一 Run 重新执行 Planning/Attack sibling 流程后，才可成为正式
Feature/Threat/Cell。已冻结但尚未执行的 cell 不能因发现新方向被跳过。

## 8. Engine 与 Direct 边界

### Engine Threat Mode

- 在 Attack 首轮前生成 `execution-contracts.json`、`execution-progress.json`、`execution-queue.json`；
- 每轮 Host 解析事件、归并 ledger、执行 gate、重算 queue；
- 阴性 cell 若缺实验义务，回退为 `shallow_negative`；
- finalizer 必须验证 execution projection 与 ledger 一致。

### Direct/Skill Diagnostic

- `skill_runtime init` 自动生成同一合同和队列；
- `observe/checkpoint` 接受 completed obligations 并生成下一队列；
- 缺 runtime artifacts 或 checkpoint 的手工 Run 固定 `runtime_incomplete=true`；
- Direct 继续 `authority_trusted=false/delivery_eligible=false`，不能因为多了状态机伪装为可信 Engine。

## 9. 实施范围

新增 `engine/dynamic_execution.py`，负责合同编译、事件校验、状态归并、队列渲染和 backlog。

修改：

- `engine/orchestrator.py`：Threat Mode 初始化、每轮事件归并、动态 queue 注入、阴性回退和进展定义；
- `engine/skill_runtime.py`：Direct queue/observation/checkpoint 接入同一合同；
- `engine/reporting/validate.py`：Threat Run 的 execution projection/closure 一致性门；
- `skill/runtime-hot-path.md`、`SKILL.md`、AGENTS：Direct preflight 和每实验事件协议；
- version/changelog/design/验收文档。

## 10. 失败先行验收

1. 用户枚举只完成消息通道时不能闭阴性，queue 仍列 timing/behavior；
2. reset token 只测假值/空值时不能闭阴性；
3. 空 balance 数据转 recoverable data-preparation，而非 not_vulnerable；
4. `use_points` 只测布尔值时仍缺交易边界与 state delta；
5. 持久化 XSS 缺 GET/render/peer 任一项均不闭合；
6. SSRF 无 destination/server-fetch proof 不闭合；
7. Finding validator 拒绝后进入 proof repair；
8. evidence 越界、不存在、空文件、未知 obligation、重复 ID 异内容均拒绝；
9. 新 endpoint 只进 backlog，不改变本轮 run plan/cell 数；
10. accepted Finding 可直接关闭对应 cell，但不能关闭 sibling threat；
11. Direct 无 runtime/checkpoint 不得 report-ready；
12. Engine resume 重算同一 execution projection，事件重放幂等；
13. v8.12 authority/lineage/finalizer 全部回归通过。

## 11. 完全体边界

v8.13 完成后，Atoolkit 的完全体链条推进到：

```text
Feature → Threat → Cell → Experiment → Observation → Proof → Report
```

其中前六段都有结构化状态和 fail-closed 门。它仍不保证模型提出所有业务 Threat，也不把未知新功能
强塞进当前 Run。后续完全体需要把动态 discovery 触发的局部重规划变成受监督的新 sibling Run，并在
可证明的网络/进程 containment 下执行，而不是牺牲 frozen-plan 真实性换取表面“动态”。
