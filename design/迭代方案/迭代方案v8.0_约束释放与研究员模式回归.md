# 迭代方案 v8.0 · 约束释放与研究员模式回归

> 一句话结论：v7.3 靶场实测 2190/4800（45.6%），比 v7.1 的 4060 下跌 46%。评估报告归因清晰——不是模型变蠢了，是 37 条约束 + 50+ 条过程性规则消耗了 30-40% 认知带宽，把安全研究员降级成了流水线工人。v8.0 的核心哲学是**翻转框架角色**：从「约束模型行为」翻转为「辅助模型工作」，删除精确处方和数值阈值，保留质量门槛和状态管理，让模型回到 v7.1 时代的研究员状态，同时保留 v7.2/v7.3 的知识深度和状态落盘改进。预期得分恢复到 3800-4200 区间。

---

## 0. 设计哲学：从操作手册到战略指南

### 0.1 v7.1→v7.3 到底做错了什么

v7.1 得分 4060 时的核心文件约 180 行，以纪律和报告标准为主，payload 选择和测试深度由模型自主决定。v7.2 引入 WooYun 知识蒸馏后膨胀到 537 行，v7.3 瘦身到 324 行但仍保留了 v7.2 的核心约束架构。问题不在于行数，在于**约束的性质**发生了根本变化：

| 维度 | v7.1（轻量约束） | v7.2/v7.3（重度约束） |
|---|---|---|
| payload 选择 | 模型从训练知识中自主选取 | 4 条精确处方（`' OR 1=1--` 等） |
| 测试深度 | 模型自行判断何时够 | 硬阈值表（≥3 family, ≥2 role...） |
| 测试终止 | 模型感知到无信号后切换 | 20 分钟强制断路器 |
| 遇验证码 | 尝试绕过 | 标 NEED_INPUT 停止 |
| 端点发现 | 主动探索所有 URL | 等覆盖表驱动 |
| 创造力 | 鼓励 | 被数值配额替代 |

### 0.2 v8.0 的核心原则

**别教模型怎么打洞，告诉它哪里可能有洞，然后让它自己想办法。**

具体落地为三条设计准则：

1. **框架做记录员**：覆盖表、状态文件、报告格式化——这些是机械性工作，框架承担
2. **框架做提醒者**：在模型遗漏大方向时轻轻提醒（「你还没看过认证端点」），而不是精确指挥每个步骤
3. **模型做研究员**：自主选择 payload、自主判断深度、自主决定何时切换——框架信任模型的判断力

### 0.3 v8.0 不做的事

| 不做的事 | 原因 |
|---|---|
| 不删除所有规则 | 质量门槛（七问门、垃圾洞、铁律）保护下限，必须保留 |
| 不删除状态落盘 | v7.3 的压缩锚点协议是对抗遗忘的辅助工具，不是行为约束 |
| 不删除知识卡体系 | 知识卡本身是好的，问题在于被硬编码成了操作手册而非参考材料 |
| 不改 engine/*.py | v8.0 聚焦 Skill Mode 行为重设计，Engine Mode 另行处理 |
| 不做 Fact-Intent Graph | 原路线图的 v8 架构升级需要引擎重构，与本方案正交，推迟到 v9 |

## 1. 丢失的 1870 分：归因与修复映射

评估报告将 v7.1→v7.3 丢失的 1870 分归因为 5 个类别。v8.0 为每个类别设计精确的修复方案，并预估可回收分值。

### 1.1 归因总表与修复映射

| 丢失类别 | 分值 | 占比 | 根因 | v8.0 修复方案 | 预期回收 | 回收率 |
|---|---:|---:|---|---|---:|---:|
| 规则阻止深入探索 | 640 | 34% | NEED_INPUT 过早触发 + captcha 不尝试绕过 | Fix-1: 二次验证绕过优先协议 | 480 | 75% |
| 处方 payload 不够用 | 580 | 31% | 4 条固定 SQLi 处方 + WAF 5家族限定 | Fix-2: 删除 payload 处方，改为目标导向 | 460 | 79% |
| 端点发现被动化 | 670 | 36% | 模型等覆盖表驱动，不主动探索 | Fix-3: 新增未知端点盲测协议 + 直觉探索阶段 | 500 | 75% |
| 跨角色测试遗漏 | 160 | 9% | 覆盖表驱动测试导致忽略直觉 | Fix-3: 同上（直觉探索阶段覆盖） | 120 | 75% |
| 部分命中扣分 | 320 | 17% | 创造性路径被规则收窄 | Fix-2 + Fix-4: 约束放松整体提升创造性 | 200 | 63% |
| **合计** | **1870** | | | | **1760** | **94%** |

**预期得分**：2190 + 1760 = **3950**（区间 3800-4200）

### 1.2 为什么不是 100% 回收

- 部分丢失（约 110 分）属于模型随机性——即使完全自由发挥，单次运行也不可能每次都命中所有漏洞
- v7.1 的 4060 本身包含了运气成分（同一配置的多次运行方差约 ±200 分）
- 保留的质量门槛（七问门等）会阻止一些低置信度的尝试，这是有意为之

## 2. Fix-1：二次验证绕过优先协议（回收 480 分）

### 2.1 问题诊断

当前规则 `§6 line 159`：
> `captcha_required`、短信/二次核身 → 输出 `NEED_INPUT`

这条规则告诉模型：遇到验证码就停。实测中模型对 register.php、admin login、reset-password 三处验证码全部输出 NEED_INPUT 后停止，丢失 640 分中至少 480 分直接归因于此。

### 2.2 修改方案

**删除**当前 `§6` 中关于 `captcha_required → NEED_INPUT` 的规则。

**替换为**以下文本，写入 `§7` 决策树的认证面分支：

```markdown
- **二次验证绕过优先协议**
  遇到 captcha / SMS / 2FA / 核身 时，**绕过测试本身是一个独立的攻击面**，不是阻塞条件。
  在标 NEED_INPUT 之前，必须尝试以下方向（不限于此列表）：

  - 空值 / null / 空字符串提交
  - 通用码（0000, 1234, 1111, admin, test）
  - 验证码重用（同一验证码多次使用、跨会话重用）
  - 字段省略（不发送 captcha 字段，观察后端是否校验）
  - 逻辑跳步（直接请求验证后的接口，跳过验证步骤）
  - 响应篡改（修改前端响应中的验证状态字段）
  - 验证码回显检查（响应 body / headers / cookie / Set-Cookie 中是否包含验证码值）
  - 验证码绑用户检查（A 的验证码能否用于 B 的流程）
  - 类型/通道混淆（短信验证码用在语音验证通道，或反之）
  
  全部失败（至少尝试 5 种不同方向）→ 才能标 NEED_INPUT。
  任一成功 → 绕过本身就是一个 finding（认证绕过）。
```

### 2.3 对应受影响项

| 靶场项 | v7.3 得分 | v7.3 状态 | v8.0 预期 |
|---|---:|---|---|
| V02 register captcha (240) | 0 | NEED_INPUT | 模型会尝试空值/重用/跳步 |
| V03 reset-password (250) | 0 | NEED_INPUT | 模型会尝试步骤跳过/验证码回显 |
| V04 admin brute-force (100) | 100 | 部分命中 | 模型会先绕过 captcha 再爆破 |

### 2.4 同步修改：§6 阻塞分类

将 `§6` 中阻塞分类的 `captcha_required → NEED_INPUT` 改为：

```markdown
`blocked` 必须写明可恢复或不可恢复。`quota_exhausted`、`object_absent`、
可申请 `missing_role`、`captcha_bypass_exhausted`（至少 5 种方向全部失败）属可恢复阻塞 → 转 `next_actions`；
`out_of_scope` 立即停止。
**captcha/SMS/2FA 不是阻塞条件**——先执行二次验证绕过优先协议（§7），全部失败后才标 NEED_INPUT。
```

## 3. Fix-2：删除 payload 处方，改为目标导向（回收 460 分）

### 3.1 问题诊断

当前规则 `§7 line 207` 给了 4 条精确 SQLi 处方：

```
WHERE 等值 → ' OR 1=1--
WHERE LIKE → ' OR '1'='1'
ORDER BY  → 1 DESC--
INSERT/UPDATE → test', field='injected
```

实测中模型对 search.php（大概率 LIKE 查询）只发了这 4 个处方 payload + UNION SELECT + 时间盲注，全部返回 0 条结果，判「无注入」。但模型如果在自由状态下，会尝试 `') OR ('1'='1`、`" OR ""="`、`%' OR 1=1--`、数字型省略引号、注释符变体（`/**/`、`#`、`;--`）、stacked queries、二次注入等 10+ 种变体。

同样的问题也出现在 WAF 协议（限定了 5 个绕过家族的搜索空间）和密码重置 4 模式检测（把灵活的认证攻击变成了 4 格 checklist）。

### 3.2 修改方案 A：SQLi 处方 → 目标导向

**删除** `§7` 中的「SQLi 查询类型适配」段落。

**替换为**：

```markdown
- **SQLi 注入测试**
  对每个可能存在注入的参数，根据参数语义和响应特征自适应选择注入策略。
  你需要覆盖的 SQL 上下文包括但不限于：等值查询、LIKE 模糊查询、ORDER BY 排序、
  INSERT/UPDATE 写入、数字型（无引号）和字符型（需闭合引号）。
  
  对每种上下文，根据你的判断选择合适的 payload 和闭合方式。
  观察响应的多维度差异：返回行数、内容变化、状态码、响应长度、响应时间、错误信息。
  不同 DBMS（MySQL / PostgreSQL / SQLite / MSSQL / Oracle）有不同的语法方言和特征函数，
  根据技术栈线索选择对应的 payload 变体。
```

### 3.3 修改方案 B：WAF 5 家族 → 开放绕过

**删除** `§7` 中「WAF 对抗协议」的「至少尝试 5 个不同家族 × 2 种 payload 变体」硬约束。

**替换为**：

```markdown
- **WAF 对抗协议**
  payload 被拦截时（响应含"非法关键字"/"请求被拦截"/403/payload 被去除），
  分析拦截机制并创造性地尝试绕过。不限制绕过技术的选择范围。
  
  参考方向（不是穷举清单，是启发性提示）：编码变体、混淆、注释穿插、空白字符替换、
  函数等价替换、分块传输、参数污染、Content-Type 切换、大小写变换、嵌套编码等。
  详细的绕过技术参考见 skillmode-reference.md §WAF。
  
  全部失败 → 标 `shallow_negative` + `waf_bypass_exhausted`。
```

### 3.4 修改方案 C：密码重置 4 模式 → 启发性描述

**删除** `§7` 中「密码重置 4 模式检测」的 4 种模式 checklist（A/B/C/D）和「至少测 2 种」硬约束。

**替换为**：

```markdown
- **密码重置/找回流程测试**
  密码重置是高价值认证面。分析其完整请求参数和业务流程，自行设计攻击策略。
  参考方向包括但不限于：验证码泄露检查（响应 body/headers/cookie 中是否包含验证码值）、
  验证码绑定检查（跨用户重用）、流程步骤跳过、目标用户参数篡改、
  token 可预测性/可重用性。
  
  详细的认证攻击模式参考见 skillmode-reference.md §密码重置。
```

### 3.5 修改方案 D：阴性 depth floor 硬阈值 → 弹性描述

**删除** `§6` 中的「阴性 depth floor 硬阈值表」（lines 144-151，6 行阈值表）。

**替换为**：

```markdown
### 阴性 depth floor（弹性判断）

阴性结论需要足够的测试深度支撑。深度由你根据端点复杂度、参数语义、
响应特征和技术栈自行判断，不设硬性数值阈值。

判断原则：
- 参数语义越复杂（多义/多类型/多上下文），需要的测试向量越多样
- 响应差异越微妙（行数变化 vs 状态码变化 vs 内容变化），需要的对照实验越充分
- 涉及越权/多角色的漏洞，必须有跨角色对比证据
- 仅凭单一 payload 类型不足以排除注入类漏洞

未充分测试的阴性一律标 `shallow_negative`，不得标 `not_vulnerable`。
```

### 3.6 对应受影响项

| 靶场项 | v7.3 得分 | v7.3 状态 | v8.0 预期 |
|---|---:|---|---|
| V18 search SQLi (200) | 0 | 处方 payload 无效 | 模型会尝试 10+ 种变体 |
| V17 product-delete SQLi (200) | 0 | 处方 payload 无效 | 模型会适配上下文 |
| V21 upload 路径穿越 (200) | 100 | 部分命中 | 模型会尝试更多编码变体 |
| V13 recharge-callback (150) | 150 | 部分命中 | 更多签名伪造策略 |

### 3.7 保留在 skillmode-reference.md 中的内容

被删除的处方和阈值**不丢失**——它们作为「启发性参考材料」保留在 `skillmode-reference.md` 中，供模型在需要灵感时按需查阅。区别是：

| 之前（操作手册） | 之后（参考材料） |
|---|---|
| 「用这个 payload」 | 「历史上这类场景常见的 payload 包括...」 |
| 「至少测 3 个家族」 | 「如果需要灵感，以下是常见绕过方向...」 |
| 核心文件中强制加载 | reference 文件中按需读取 |

## 4. Fix-3：未知端点盲测 + 直觉探索阶段（回收 620 分）

### 4.1 问题诊断

丢失 1870 分中有 670 分归因于「端点发现被动化」——模型在等覆盖表告诉它测什么，而不是主动发现端点并探索。典型案例：

- `create-order.php`（250 分）：JS 中有这个端点但模型没有纳入覆盖表
- `change-audit-status.php`（240 分）：JS 中有这个端点但模型没有去测
- `pay redirect`（180 分）：支付跳转逻辑在 HTML 表单中但模型没有跟踪

另有 160 分归因于「跨角色测试遗漏」——覆盖表驱动测试导致只测表内项，忽略了 merchant/index 等非标准角色路径。

### 4.2 修改方案 A：新增未知端点盲测协议

在 `§7` 决策树末尾（「什么攻击面都不明显时」之前）新增：

```markdown
- **未知端点盲测协议**
  在 JS / HTML / API 响应 / 错误页面中发现的每一个 URL 和路径，
  不管是否已在攻击面清单中，都必须发送至少一次探测请求（GET + POST），
  观察其功能和响应结构。
  
  特别关注：
  - JS 中动态拼接的路径（模板字符串、变量拼接）
  - HTML 表单的 action 属性（尤其是非 API 路径）
  - API 响应中的 `_links`、`next_url`、`redirect` 字段
  - 错误页面泄露的路径信息
  - 不同角色加载的不同 JS 文件中的端点差异
  
  发现新功能端点 → 立即加入攻击面清单并测试，不等下一轮。
```

### 4.3 修改方案 B：新增直觉驱动探索阶段

在 `§9` 终止协议之前新增 `§8.5`：

```markdown
## 8.5 直觉驱动探索阶段

在完成结构化覆盖（所有覆盖表 surface 已闭合）后、终态自检之前，
预留最后 20% 的测试时间做完全自由的探索。

这个阶段不受任何 checklist 约束，不要求按覆盖表执行。
你可以：
- 重新审视任何引起你直觉注意的端点或响应
- 尝试非标准的攻击技术（不在任何 playbook 中的方法）
- 组合多个已发现的低危问题形成攻击链
- 探索之前因为"不在覆盖表内"而跳过的方向
- 用完全不同的角色/参数组合重测已判阴性的 surface

唯一约束：每个探索方向必须有实际请求作为证据（不能只在推理中想象）。
```

### 4.4 修改方案 C：响应异常嗅探

在 `§3` 速查卡中新增一条：

```markdown
- **响应异常嗅探**：对每个 API 响应，检查是否有意外字段
  （如 callback_sign、内部路径、调试信息、其他用户的敏感数据），
  即使主流程正常。意外字段本身可能是独立的攻击面入口。
```

### 4.5 对应受影响项

| 靶场项 | v7.3 得分 | v7.3 状态 | v8.0 预期 |
|---|---:|---|---|
| V08 create-order (250) | 0 | 未发现端点 | 盲测协议会发现并测试 |
| V12 change-audit-status (240) | 0 | 未发现端点 | 盲测协议 + JS 分析 |
| V11 pay redirect (180) | 0 | 未跟踪表单 | HTML 表单扫描 + 盲测 |
| V16 merchant/index (160) | 0 | 未覆盖角色 | 直觉探索阶段 |

## 5. Fix-4：约束放松与认知带宽释放（回收 200 分）

### 5.1 问题诊断

评估报告量化了当前规则对认知带宽的消耗：

| 消耗源 | 数量 | 认知负担 |
|---|---:|---|
| 约束指令（必须/不得/禁止/只能） | 37 条 | 每条约 0.5% 注意力 |
| 过程性规则（精确的「做 X 然后 Y」） | 50+ 条 | 每条约 0.3% 注意力 |
| 状态文件维护（8 种固定格式） | 8 个 | 机械性工作但需记忆时机 |
| 自检清单（3 级） | 3 级 | 周期性中断测试流 |
| 时间断路器（4 种） | 4 种 | 外部强制中断 |

保守估计 30-40% 的认知资源被用于遵守流程规则。Fix-1 到 Fix-3 删除了最大的几个约束源，Fix-4 进一步放松剩余的过度约束。

### 5.2 修改方案 A：时间断路器放松

**删除** `§7 line 227` 的「20 分钟无进展 → 必须换攻击面」硬约束。

**替换为**：

```markdown
- **时间管理**：如果连续 20 分钟无任何进展信号——包括响应变化、
  新发现的端点/参数、partial success、有趣的信息泄露——考虑换方向。
  但如果有任何微弱信号（哪怕只是"响应长度有 2 字节差异"），继续追。
  模型的直觉比计时器更可靠。
```

### 5.3 修改方案 B：覆盖状态新增 `exploring`

在 `§6` 的覆盖格状态列表中，新增 `exploring`：

```markdown
覆盖格状态：`not_tested`、`confirmed`、`not_vulnerable`、`shallow_negative`、
`blocked`、`not_applicable`、`exploring`。

`exploring` 表示模型正在对该 surface 进行非结构化的深度测试，
不受 depth floor 或 checklist 约束。当探索结束有明确结论后，
转为 `confirmed` 或 `not_vulnerable` / `shallow_negative`。
```

### 5.4 修改方案 C：自检频率进一步放松

将 `§11` 中「每 10 个 surface 自检」改为「每 15 个 surface 自检」，减少周期性中断。
「每 15 个 surface 自检」改为「每 25 个 surface 自检」。
终态自检保持不变。

```markdown
### 阴性 depth floor（弹性判断）

阴性结论需要足够的测试深度支撑。深度由你根据端点复杂度、参数语义、
响应特征和技术栈自行判断，不设硬性数值阈值。

判断原则：
- 参数语义越复杂，需要的测试向量越多样
- 响应差异越微妙，需要的对照实验越充分
- 涉及越权/多角色的漏洞，必须有跨角色对比证据
- 仅凭单一 payload 类型不足以排除注入类漏洞

未充分测试的阴性一律标 `shallow_negative`，不得标 `not_vulnerable`。
```

### B. §7 SQLi 查询类型适配（Before / After）

**Before（v7.3）**：
```markdown
- **SQLi 查询类型适配**：WHERE 等值 → `' OR 1=1--`；WHERE LIKE → `' OR '1'='1'`
```

**After（v8.0）**：
```markdown
- **SQLi 注入测试**
  对每个可能存在注入的参数，根据参数语义和响应特征自适应选择注入策略。
  你需要覆盖的 SQL 上下文包括但不限于：等值查询、LIKE 模糊查询、ORDER BY 排序、
  INSERT/UPDATE 写入、数字型（无引号）和字符型（需闭合引号）。
  
  对每种上下文，根据你的判断选择合适的 payload 和闭合方式。
  观察响应的多维度差异：返回行数、内容变化、状态码、响应长度、响应时间、错误信息。
  不同 DBMS 有不同的语法方言和特征函数，根据技术栈线索选择对应的 payload 变体。
```
