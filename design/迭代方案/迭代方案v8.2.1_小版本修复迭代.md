# 迭代方案 v8.2.1 · 小版本修复迭代（基于 v8.2 审核反馈）

> **性质**：v8.2 设计的补丁版本（不落地 v8.2，直接以 v8.2.1 落地），只修审核报告指出的 P0/P1 问题，不做增量功能。
>
> **核心原则重申（v8.0 金句）**：「别教模型怎么打洞，告诉它哪里可能有洞，然后让它自己想办法」。v8.2.1 的每一处改动都按此原则检验。
>
> **预期得分修正**：v8.2 乐观场景 3700+ 存在方差双重计算，v8.2.1 重新建模后的合理区间为 **3000-3850**（能力基线 3300-3550 ± 纯随机方差 σ≈300），悲观场景 2400-2700（v8.2 未提及）。

## 1. 修复 1 · HackerOne 参数二次过滤

### 1.1 过滤原则

对 v8.2 §2.3 清单按 4 条硬过滤规则清洗：

| 规则 | 剔除案例 | 原因 |
|------|---------|------|
| **A. 框架/CMS 专有** | blobcol/blobkey/blobtable/blobwhere (Keystone.js)、sha/external_status_check_id (GitLab)、fontpath/addondirname/textdomain (WordPress/Nextcloud) | 靶场是天积商城（PHP+Apache+MySQL 电商），这些参数在目标中不存在，注入后会浪费请求 |
| **B. 行业垂直专有** | studentname/phoneormail（教育）、scn（特定屏幕标识） | 靶场无对应业务面 |
| **C. 技术栈不匹配** | variables/operationname（GraphQL）、consumer_key（Twitter OAuth 1.0） | 靶场是 REST + 自有鉴权 |
| **D. 低频 + 弱语义** | subtask_id（Twitter 多步认证，仅 28 报告）、organization_id/group_id（B2B 多租户，靶场是 B2C） | 出现次数虽≥6，但语义过窄 |

### 1.2 过滤后参数清单（89 → 38）

| 漏洞类型 | 原始 | 过滤后 | 保留参数 |
|---------|------|--------|---------|
| **SQLi** | 15 | **5** | hash, acctid, regid, organization_id, post_type |
| **IDOR** | 12 | **6** | target_id, target_type, accounts, group_id, selectedaddressid, time_range |
| **SSRF** | 9 | **5** | import_url, proxy, callback_url, svg, protocol |
| **PathTraversal** | 10 | **3** | filepath, file_path, filepathdownload |
| **OpenRedirect** | 12 | **12** | redirect_uri, redir, return_url, next, return_to, redirect_to, goto, target, callback_url, response_type, scope, state（全部通用，无过滤） |
| **AuthBypass** | 10 | **6** | authenticity_token, auth_type, client_id, redirect_uri, _method, auth_code |
| **BusinessLogic** | 7 | **1** | uuid（plan_id/deliverytype/draftid 等偏 SaaS/电商特有，靶场无对应面） |
| **合计** | **89** | **38** | 缩减率 57% |

### 1.3 Reference 速查表（只放跨类型 Top-3~5）

v8.2 把全量参数塞进 reference 速查表，v8.2.1 改为"每个漏洞类型只放 3-5 个最通用、最容易被模型忽略的参数"，其余下沉到知识卡 secondary_params：

```markdown
## HackerOne 次级参数速查表（按需检索）

> 仅当主参数（id/user_id/keyword/url 等）测试阴性时查阅。
> 数据来源：HackerOne 3319 报告，已过滤框架/行业/技术栈专有参数。

### SQLi 次级（当 keyword/search/id 均阴性时）
hash（哈希查询）, acctid（账户 ID）, regid（注册 ID）, organization_id, post_type

### IDOR 次级（当 id/user_id/order_id 均阴性时）
target_id, target_type, accounts, group_id, selectedaddressid, time_range

### SSRF 次级（当 url/image_url 均阴性时）
import_url, callback_url, svg（SVG 文件 SSRF）, protocol（协议切换）

### PathTraversal 次级（当 filename/category 均阴性时）
filepath, file_path, filepathdownload

### OpenRedirect 次级（当 redirect/return_url 均阴性时）
redirect_uri（OAuth）, next, return_to, goto, target, response_type, scope, state

### AuthBypass 次级（当 username/password 均阴性时）
authenticity_token, auth_type, client_id, redirect_uri, _method（HTTP 方法覆盖）, auth_code
```

**过滤收益**：
- reference 体积从 +30 行降到 +20 行，模型查阅时不被无关参数分散注意力
- 知识卡 match.params 保持精简，secondary_params 按 risk_tag 激活，不会跨方向扩散噪声
- 靶场适配性显著提升（剔除 SaaS/GraphQL/CMS/Git 参数）

### 1.4 知识卡分层激活机制

为避免 input-validation.json 从 12 → 57 参数的膨胀挤压其他方向的测试时间，采用**分层激活**：

- **Tier 1（主参数）**：现有 12 个参数，作为首轮测试目标
- **Tier 2（次级参数）**：HackerOne 新增参数，仅在主参数全部阴性时激活

知识卡 JSON 新增 `match.secondary_params` 字段（与 `match.params` 区分），引擎层在"主参数阴性 → 是否升级测试深度"判断时注入次级参数。Skill Mode 下由模型在"主参数阴性"时主动查阅 reference 速查表获取（不改动 knowledge card JSON）。

## 3. 修复 3 · B1/B2 纪律措辞重写

### 3.1 B1 参数格式自适应协议

**v8.2 原文**（问题：固定 3 种格式顺序 + "必须逐一测试"的硬约束感）：
```
- 参数返回"不能为空/格式错误/缺少字段" → 依次尝试 3 种格式：
  ① application/x-www-form-urlencoded（默认）
  ② application/json（JSON body）
  ③ multipart/form-data（FormData）
  三种格式均报"不能为空" → 检查参数名是否正确（snake_case / camelCase / PascalCase 变体）
```

**v8.2.1 重写**（提醒型，不教具体做法）：
```
- 参数返回"不能为空/格式错误/缺少字段" → 通常意味着参数名或编码方式不对，
  而不是参数真的缺失。考虑尝试不同的 Content-Type、参数命名约定（snake_case / camelCase / PascalCase）、
  或请求结构（form / JSON / multipart）。先解决格式问题再继续测试，不要直接标 negative。
```

**位置**：核心技能文件 §3 速查卡（追加 3 行，比 v8.2 少 1 行）

### 3.2 B2 交易端点参数穷举

**v8.2 原文**（问题：数值阈值"≥4 个维度"违反 v8.0 去阈值原则）：
```
- **交易端点参数穷举纪律**：每个交易端点必须逐一测试 ≥4 个可控参数维度。
  如果端点有 N 个用户可控参数，每个参数至少测 3 个边界值（零/负/超大）。
  只测主流程参数（如 prize_id）而不测次级参数（如 probabilities）→ 阴性结论标 shallow_negative。
  具体方法：先 GET/POST 端点获取完整参数列表（从请求 body/HTML form/JS 代码），
  然后对每个参数逐一测试，记录到覆盖表。
```

**v8.2.1 重写**（去掉数值阈值，保留核心提醒）：
```
- **交易端点参数穷举**：交易端点（支付/充值/退款/积分/抽奖）通常有多个用户可控参数
  （金额/积分/数量/概率/状态/时间等），不要只测主流程参数而放过次级参数。
  对每个可控参数，考虑边界值（零/负/超大/精度）。只测主参数得出的阴性结论不可靠。
```

**位置**：核心技能文件 §7 决策树"交易端点参数穷举"项（修改现有项，+2 行，比 v8.2 少 1 行）

### 3.3 B3/B4/B5 保持 v8.2 原文

B3（页面级越权覆盖）、B4（SQLi/IDOR 独立测试面）、B5（存储型闭环触发条件）已符合"提醒者"定位，措辞无需调整。

## 5. 修复 5 · 约束密度重新计算

### 5.1 v8.2 的分类问题

v8.2 把 5 条执行纪律归类为"过程规则"（18→23），但约束密度公式仍用 18 计算，得 9.3%。这不诚实。

### 5.2 v8.2.1 的诚实计算

| 指标 | v8.0 | v8.1 | v8.2（方案声称） | v8.2.1（诚实） |
|------|------|------|-----------------|---------------|
| 核心文件行数 | 347 | 372 | ~386 | ~371 |
| 硬约束数 | ~18 | ~18 | ~18 | ~18 |
| 过程规则数 | ~15 | ~16 | ~18（应 23） | ~23（诚实） |

## 6. 修复 6 · 知识卡参数分层与注意力再分配评估

### 6.1 问题诊断

v8.2 把 45 个新参数注入 input-validation.json，该卡对应 SQLi/XSS/SSRF/OpenRedirect/RCE 五个方向，参数总数从 12 膨胀到 57。模型看到这种"重卡"时会不自觉地投入更多时间测试对应方向，挤压其他方向（认证/交易/IDOR）的测试预算。

### 6.2 v8.2.1 的分层方案

**JSON 结构扩展**：

```json
{
  "match": {
    "params": ["keyword", "search", "q", ...],           // Tier 1: 主参数（首轮测试）
    "secondary_params": ["hash", "acctid", "regid", ...] // Tier 2: 次级参数（主参数阴性时激活）
  },
  "activation": {
    "primary": "risk_tag 匹配时注入 Tier 1",
    "secondary": "Tier 1 全部阴性 + 端点含可控参数时注入 Tier 2"
  }
}
```
