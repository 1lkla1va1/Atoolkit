# 迭代方案 v6.1 · 榨干—深测—进报告 三支柱

> 一句话结论：v6 把"候选召回"这层补上了，但仍停在"软请求模型自觉发散"。v6.1 把"榨干"从模型自觉改成外壳强制的**确定性骨架**，把"已有的点没深测"改成 **depth floor + 再探测触发**，把"没进报告"改成 **proof 预算保底 + 四类缺口附录强制渲染**。v6 解决"有无"，v6.1 解决"深不深、进不进报告"。

---

## 0. 现状定位（先说清楚 v6 落没落地）

- **v6 是纯设计稿**：`engine/candidate.py` 不存在；`run.py` 无 `--loop-mode`；`assemble_prompt` / `run_session` / `_conclude` 未接 candidate 通道。git 新增的是 v4.1/v5.1 的 `dedupe.py` / `reporting/` / `surface.py`。
- **640 低分根因（R1–R9）大多已被 v4–v5.1 治掉**：forced run.py 路径（R1）、`surface.is_saturated` + `inventory.json`（R4）、`blocker.py`（R3）、`planner.PARAM_RISK_RULES` + 知识卡（R5）、`dedupe.py`（R7）、`session_gate.py`（R8）、`planner.plan_object_pair_tasks/plan_role_diff_tasks`（R9）、`AUTH_FLOW_CLASS` 列 + `auth-flow-abuse` 卡（R6，部分）。
- **v6.1 的前提**：v4–v5.1 已落地的覆盖/阴性/报告链不动主轴，只在 candidate 层之上做三支柱。**v6.1 不重写 orchestrator 主循环，但必须动 `assemble_prompt` / `CognitiveState.update` / `_conclude` 三个钩子**——v6 §5"只增薄层不改 orchestrator"是不诚实的，recall/triage/proof/closure 四相行为差异不可能靠一个不接 loop 的薄层驱动。

## 1. v6 设计审视：10 处缺陷

| # | 缺陷 | 后果 | v6.1 对策 |
|---|---|---|---|
| D1 | "榨干"是软请求：`对每个 surface 提出 1-3 个候选`，模型出 1 个就能过 | 召回回到首洞即停的老路 | §3 确定性候选骨架 + 强制逐维应答 |

| D10 | 单 pass 单视角：一个模型一次发散有盲区 | 高价值维度系统性漏 | §3 多视角/对抗式"你漏了什么" |

## 3. 支柱 A · 召回榨干（确定性骨架替代软请求）

### 3.1 候选骨架：风险维 × surface 强制应答

v6 让模型"自觉提 1-3 个候选"。v6.1 反过来：**外壳按 surface 的 risk_tags 展开成"风险维清单"，模型必须对每一维应答**——有假设就写候选，没有就写 `NONE:<reason>`，不许跳过。

```
surface: POST /api/order/refund  refund_amount  [user]  risk_tags=[amount-tamper,object-ownership]
外壳展开应答表（注入 prompt）：
  [amount-tamper]   ? 候选 / NONE:<为何不是>
  [object-ownership]? 候选 / NONE:<为何不是>
  [idor]            ? 候选 / NONE:<为何不是>     ← 由 risk_tag 派生
  [auth-flow-abuse] ? 候选 / NONE:<为何不是>     ← 认证面强插
模型输出（每维必答）：
  DIM: amount-tamper | CANDIDATE: refund_amount 篡改 0/负数/超额 | need:对比退款后余额 | P2 | probe:改 amount 重发
  DIM: object-ownership | NONE: 无对象 id 参数
  DIM: idor | CANDIDATE: 换 attacker 退款他人订单 | need:owner 建单+attacker 改单号 | P1 | probe:三步越权
  DIM: auth-flow-abuse | NONE: 非认证端点
```

风险维来源（payload-free，复用既有）：
- `knowledge/cards/*.json` 的 `match.vuln_classes` + `false_negative_rules` → 维清单
- `planner.PARAM_RISK_RULES` 的参数语义 → 维派生
- `ledger.VULN_RISK_MAP` → 类→维映射
- 强插维：**每个 surface 必答 `auth-flow-abuse` 维**（治 640 的 R6：有登录态就不再回头测认证绕过）。

### 3.2 强制多样性 + 语义饱和（治 D7）

候选去重键从 v6 的 `endpoint+vuln_class+role+root_cause` 升级为**多维差异键**：

```python
def candidate_diversity_key(c):
    return frozenset({
        _norm(c["endpoint"]), c["vuln_class"], c.get("param",""),
        c.get("role",""), c.get("object",""), c.get("root_cause",""),
        c.get("state_transition",""),  # 新增：状态变化维
    })
```

- 新候选若 diversity_key ⊆ 已有候选并集 → 标 `duplicate`，不计新增。
- recall 退出条件从"连续 K 轮无新增"改为"连续 K 轮所有新候选全 duplicate"——换皮重述过不了。

### 3.3 对抗式"你漏了什么"（治 D10，预算允许时）

recall 末尾插一轮**对抗 pass**（可选，受 `--adversarial-pass` 开关与剩余预算控制）：

```
注入：上一轮共得 N 个候选，覆盖维度 {list}。
任务：列出第 N+1..2N 个**它最可能漏掉**的候选，聚焦 {未应答 NONE 的维度 / 高价值未触维 / 跨流组合}。
禁止：复述已有候选。
```

外壳把对抗 pass 产出的候选按 §3.2 去重后并入 `candidate-ledger.json`，标 `source=adversarial`。单模型盲区由此补一块。

### 3.4 多视角（治 D10，重资产，默认关）

`--lens` 开关注入不同"帽子"分别 recall 同一 surface：`attacker-A / attacker-B / business-abuser / crypto-forger / state-racer`。默认关；只有高价值面 + 预算充足时对 Top M surface 触发。**不写死视角清单的 payload**，只声明视角意图，由模型在该视角下自发产候选。

## 4. 数据结构扩展（candidate schema v1.1 + surface 扩列）

### 4.1 `engine/candidate.py`（v6 提出、v6.1 落地，schema 升 v1.1）

```json
{
  "candidate_id": "cand_001",
  "surface_id": "POST /api/order/refund refund_amount [user]",
  "endpoint": "/api/order/refund", "method": "POST", "param": "refund_amount",
  "role": "user", "object": null,
  "vuln_class": "业务逻辑", "risk_dimension": "amount-tamper",
  "hypothesis": "refund_amount 篡改 0/负数/超额",
  "evidence_need": "对比退款前后余额/订单状态",
  "next_probe": "owner 建单→改 amount 重发→查余额",
  "priority": "P2",
  "status": "proposed",
  "depth_score": 0,            // §6：0=未探 1=单向量 2=多向量 3=多角色/对象 4=根因扩散
  "depth_floor": 3,            // §6：该候选进 proof_ready 前的最低 depth_score
  "evidence_refs": [],
  "blocker": null,
  "reprobe_triggers": [],      // §6：满足任一即从 refuted→re-probe
  "root_cause": null,          // §7：confirmed 后填，供扩散
  "source": "recall|adversarial|lens:attacker-A",
  "created_turn": 0, "updated_turn": 0
}
```

状态机（v6 六态 + 深度）：
```
proposed → triaging → proof_ready(达 depth_floor) → confirmed(出 finding) → root_cause_spread(§7)
                                          ↘ refuted(浅:depth<floor 不算终态) → re-probe(§6 触发)
                                          ↘ blocked(可恢复→next_actions / 不可恢复→NEED_INPUT)
                                          ↘ duplicate
```

### 4.2 `Surface` 扩列（`ledger.py`，治 D2）

`CoverageLedger.Surface` 增（向后兼容，旧字段不动）：
```python
candidate_count: int = 0          # 该格已提候选数
deepest_status: str = ""          # proposed/triaging/proof_ready/confirmed/refuted
depth_score: int = 0              # 取该格候选的 max(depth_score)
negative_depth_checked: bool = False  # §6：阴性 depth floor 是否已闭环校验
```
`session_gate` 据此新增谓词（§6）：`surface_has_candidates_all_shallow_refuted`、`negative_depth_not_checked`。

## 5. 支柱 C 之预算 · 合并优先级工作队列（治 D3）

砍掉 v6 的 recall/triage/proof/closure 四相固定 40/35/20/5。改为**单一优先级队列**，相别只是软提示：

```python
def work_queue_priority(item):
    # item = candidate 或 surface
    if item.status == "proof_ready":           # ★ 保底：已达 depth_floor 的候选永远最前
        return (0, -value_tier(item), -item.depth_score)
    if item.status == "triaging" and item.depth_score < item.depth_floor:
        return (1, -value_tier(item))          # 还没到 depth floor 的探测
    if item.status == "proposed" and item.is_high_value:
        return (2, -value_tier(item))
    if item.status == "proposed":
        return (3, -value_tier(item))
    if item.status == "refuted" and item.reprobe_triggers:
        return (4,)                            # §6 再探测
    return (9,)
```

**proof 保底预算**（治 D6"没进报告"头号原因）：每轮预算检查，若 `proof_ready` 候选数 > 剩余轮数，**强制把剩余预算全分给 proof_ready**，recall/triage 让位。即"已经挖到能证的，先证完再挖"。

recall 退出不再是固定 40%，而是：**风险维应答饱和**（每个高价值 surface 的每个强插维都有候选或 NONE 应答）+ **对抗 pass 已跑**（若启用）+ **连续 K 轮全 duplicate**。

## 6. 支柱 B · 深度 floor + 再探测（治"已有的点没深测"）

### 6.1 depth_score 与 depth_floor

`depth_score` 由外壳按**已落盘证据**计分（不信任模型自报）：
- +1：≥1 个独立向量有响应证据
- +1：≥3 个独立向量（payload 家族/对象 id/角色）有响应证据
- +1：≥2 角色 或 ≥2 对象 的对比响应证据
- +1：根因扩散已做（同根因在兄弟端点/参数/角色验证过）

`depth_floor` 由知识卡派生（复用 `negative_sufficiency`，对称定义）：
```python
# knowledge/cards/idor-multi-identity.json 已有:
#   negative_sufficiency: {min_vectors:3, min_response_count:2, min_identities:2}
# v6.1 对称加 positive_depth_floor: {min_vectors:3, min_roles:2, min_objects:2}
```
- IDOR/越权：floor=3（2 角色×2 对象）
- 业务逻辑篡改：floor=2（≥3 金额向量：0/负/超额/精度）
- 注入：floor=2（≥3 payload 家族：语法/布尔/时间/错误）
- SSRF：floor=2（≥外跳/协议变体/相对路径）

**候选不到 depth_floor 不许标 proof_ready**（防"第一下就中、立刻写报告"的浅阳性，丢更高严重度变体）。

### 6.2 阴性 depth floor 闭环校验（治浅阴性伪装成 not_vulnerable）

现状：`shallow_negative` 由 `resolve_negative_state` 判，但模型写一份像样的 `negative_*.md` 就可能被判 `not_vulnerable`。v6.1 在 **`_conclude` 闭环时**对每个 `not_vulnerable` surface 复核：

```python
if surface.status == "not_vulnerable" and not surface.negative_depth_checked:
    # 重新按卡算 min_vectors/min_response_count，与磁盘证据比对
    if not meets_negative_sufficient(surface, cards):
        surface.status = "not_tested"  # 降级回开
        surface.next_actions.append("阴性证据不达 depth floor，补向量")
```
新增 session_gate 谓词 `negative_depth_not_checked` 命中即 `incomplete`。

### 6.3 再探测触发器（治 D4：闭了就不回头）

候选 refuted 时写 `reprobe_triggers`，后续任一触发即 `refuted→proposed` 重开：

| 触发 | 例 |
|---|---|
| 新角色到账 | 原只有 user，后补 admin → 重测该候选 |
| 新对象类 | 原无订单对象，owner 建单后 → 重测 IDOR 候选 |
| 同流新端点 | 发现 `/api/order/cancel` → 重测 `/api/order/refund` 的状态依赖候选 |
| 同根因新确认 | 兄弟端点确认同根因 → 重测本端点同根因候选 |
| depth_floor 上调 | 知识卡更新提高 floor → 旧"达 floor"候选重开 |

外壳每轮扫描 `reprobe_triggers` 与新增 inventory/role/object/confirmed 比对，命中即重开并注入提示。

## 7. 链式/严重度升级（治 D5，榨干也榨深）

proof_ready 候选确认后，**不立即进报告**，先触发 `root_cause_spread`：

```
新 confirmed 候选 root_cause = "回调信任客户端签名"
外壳派生扩散候选（自动入 candidate-ledger，status=proposed）：
  - 同根因 × 兄弟端点：/api/pay/callback、/api/recharge/callback
  - 同根因 × 兄弟参数：sign/signature/notify_url
  - 同根因 × 升级路径：单次重放→并发重放（P3→P1）
扩散候选走正常队列；确认后合并为同一 finding 的 facets（复用 dedupe.py 的 root_cause 聚合）。
```

这把"找到一个就深挖成多报告"的偏科（640 R7）反过来用：**找到的不是终点，是扩散的起点**。

## 8. 支柱 C · 报告保底 + 四类缺口附录（治"没进报告"）

### 8.1 proof_ready 保底（§5 已述）
proof_ready 永远队列最前 + 剩余预算保底。即"挖到能证的，必证"。

### 8.2 四类缺口附录强制渲染（治 D6，把 v6 §6 的"运行后复盘"拉到当轮）

`_conclude` 末尾算四类清单，渲染进 `final_report.md` 附录 + 独立 `coverage_gaps.md`：

| 类 | 来源 | 渲染 |
|---|---|---|
| ① 发现了但没测 | `proposed`/`triaging` 候选，无 finding | "未深测候选清单" + next_probe |
| ② 测了但没深入 | `confirmed` 但 root_cause_spread 未做；`not_vulnerable` 但 depth 未校验 | "浅确认/浅阴性清单" |
| ③ 阻塞未恢复 | `blocked` 且 blocker 可恢复但 next_actions 未清 | "可恢复阻塞清单" |
| ④ 漏进报告 | `proof_ready` 无 finding；`demoted`；`verify=不确定` | "待复验/未证清单" + 证据路径 |

**铁律**：四类任一非空，终态不得 `complete`，须 `incomplete_with_findings` 或 `incomplete`，且 `final_report_status=draft_incomplete`。这把"已有漏洞但没进报告"从静默丢失变成**必须列在报告里的可见缺口**。

### 8.3 demoted 不丢（既有行为保留 + 可见化）
`enforce.guardian_check` 的 demoted 既有留档，v6.1 把 demoted + verify 不确定**列进附录④**，给复验路径，不再只存内存。

## 9. flow surface 一等公民（治 D8）

`surface.py` 现只抽数端点。v6.1 在 `planner` 加 `plan_flow_surfaces`：识别**端点序列成流**（同 feature + CRUD 动词序列 + 状态依赖），生成 `surface_type=flow` 的候选格：

```
flow: order(create→detail→pay→refund→cancel)  risk_tags=[business-logic,state-transition]
候选：取消已退款单 / 退款已取消单 / 支付后改 amount / 并发支付+取消
```

flow surface 不进 endpoint×vuln 矩阵（避免污染），进 `candidate-ledger.json` 独立段，status 同候选机。知识卡新增 `business-flow-abuse.json`（payload-free，只列状态依赖证据要求）。

### 10.2 orchestrator 三钩子诚实改动（治 D9）

**`assemble_prompt`**：在 `state.to_prompt_block()` 后插 `candidate_block`：
- recall 相：注入"本 surface 风险维应答表"（§3.1）+ Top N 未应答维
- 非 recall 相：注入 Top M 候选工作队列（§5 优先级）
- proof 保底：注入"达 depth_floor 待证候选"清单 + finding schema 提醒

**`CognitiveState.update`**：解析新增协议行：
```
DIM: <dimension> | CANDIDATE: ... | NONE: ...
TRIAGE: <cand_id> | proof_ready|refuted|blocked|needs_more | reason | evidence_ref | next_action
REPROBE: <cand_id> | trigger | reason
SPREAD: <cand_id> | root_cause | sibling_endpoint|sibling_param|escalation
```
调用 `CandidateLedger.apply()` 落盘 + 回填 surface 的 candidate_count/depth_score。

**`_conclude`**：

## 11. 可能漏掉的建议（你点名要的）

这些是 v6.1 自身的设计风险 / v6 没考虑的点，**先记不先做**，避免 v6.1 又变臃肿：

1. **depth_score 仍靠磁盘证据计分，模型可不写证据绕过**。终极解是 verify.py 对阳性候选做确定性重放计数（已有 verify_idor，扩成通用 `verify_depth`）。v6.1 先靠"闭环校验 + session_gate"兜，verify_depth 留 v6.2。
2. **对抗 pass / 多视角成本高**（多轮模型调用）。若无预算控制，recall 阶段就能把 token 烧光。`--adversarial-pass` 必须受 `max_turns` 子配额约束，且只对高价值面触发。
3. **flow surface 识别是启发式**，可能误把无关端点串成流，产生噪声候选。需一个"流置信度"门槛，低置信度的流候选只进候选台账不进工作队列。
4. **root_cause_spread 可能爆炸**：一个 confirmed 派生 N 个扩散候选，每个再确认又派生。必须加扩散深度上限（默认 2 层）+ 去重（§3.2 diversity_key 已防同根因重复）。
5. **四类缺口附录可能成为新"伪完成"出口**：模型学会把啥都塞进附录④"待复验"来逃避深测。须 session_gate 限附录④条数 + 高价值候选不许进附录④（必须证或显式 NEED_INPUT）。
6. **模式 A（纯 skill 直跑）仍接不上 candidate 层**：640 的 R1 教训是"改 engine 不让 mode A 变好"。candidate-ledger 在 engine 里，mode A 又会拿不到。v6.1 须把"风险维逐维应答"**直接下沉进 `核心技能文件.v2.md` 决策树**（不靠外壳注入），否则 mode A 重演 640。
7. **多模型分工（v6 §6 列为 v6.1+）刻意没做**：v6.1 仍是单模型多相。原因：多模型分工需要稳定 candidate schema 做模型间契约，v6.1 先把 schema（v1.1）和 depth_score 跑稳，v6.2 再上多模型。
8. **知识卡 payload-free 红线 vs depth_floor 具体向量**：depth_floor 说"≥3 payload 家族"但不说哪 3 个，这是对的（守 payload-free）。但模型可能不知"3 个家族"指啥。解：卡的 `positive_depth_floor` 只给数量门槛，向量家族名由模型按维度自选，外壳只数证据文件里的独立向量数。
9. **candidate-ledger.json 膨胀**：长会话候选数百条，每轮全量回灌 prompt 会爆 token。须像 coverage matrix 一样只回灌 Top N + 未应答维，不全量回灌（`assemble_prompt` 已按 Top N 设计，但要焊死 candidate 段也走 Top N）。
10. **"榨干"与"够证明即止"红线冲突**：核心技能文件 §4"够证明即止，不做超出取证的进一步利用"。depth floor / root_cause_spread 看似违反"即止"。须明确：depth floor 是**取证完整性**（证明这真是漏洞、且严重度没低估），不是"进一步利用"——前者必做，后者红线禁止。这条边界要写进 skill 文件，否则模型会用"够证明即止"逃避 depth floor。

## 12. 完成定义

v6.1 完成时，至少满足：

1. `engine/candidate.py` 存在，候选落盘 `candidate-ledger.json`，不全在对话里。
2. `assemble_prompt` 注入风险维应答表，模型对每维必答（CANDIDATE 或 NONE）。
3. 候选不到 `depth_floor` 不许 `proof_ready`；阴性闭环校验不达 floor 降级回 `not_tested`。
4. `reprobe_triggers` 命中时 refuted 候选重开。
5. proof_ready 候选保底预算；proof_ready 无 finding → 终态 `incomplete`。
6. `_conclude` 渲染四类缺口附录（`coverage_gaps.md` + final_report 附录），任一非空不得 `complete`。
7. `session_gate` 含 4 条新谓词（§10.1）。
8. `核心技能文件.v2.md` 下沉"风险维逐维应答" + depth floor 纪律（治 mode A），已 regen AGENTS.md。
9. `--self-check` 加 6 条断言全绿。
10. 用既有 fixture（`fixtures/oracle_sample.json` + `reporting/basic_idor`）跑 recall-first dry-run：候选数 > coverage 格数、proof_ready 有 finding、四类附录可渲染。
