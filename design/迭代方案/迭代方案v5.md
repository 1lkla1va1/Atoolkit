# 迭代方案 v5 · 项目记忆层 + 证据链驱动的综合报告

> 一句话结论：**v4/v4.1 已经把覆盖台账、recon bootstrap、session-gate、自检门、根因聚合等主轴基本接上；v5 不再重复修覆盖，而是补两块当前最影响体验与迭代效率的能力：项目级外部记忆，以及从漏洞确立开始编排证据链、最终稳定输出综合报告。**

### 0.2 v5 不再重复解决的问题

- 不重写 coverage-ledger。
- 不重写 session-gate。
- 不重写 surface bootstrap。
- 不把 oracle 放进模型 prompt。
- 不把某个靶场路径、账号、分值写入核心规则。
- 不用“更复杂的 prompt”替代外部结构化状态。

### 0.3 v5 真实缺口

1. **项目记忆缺口**：每次新会话仍要重新读项目，缺少“代码项目图谱/项目轮廓”。现有 `knowledge-base/graphify-out/` 更偏方法论知识图谱，不是代码结构图谱。
2. **报告链路缺口**：现在模型确立漏洞时仍主要写 `report_*.md`，报告是末端文本，不是从结构化证据链渲染出来。
3. **最终交付缺口**：当前产物是若干 `report_*.md`、`summary.json`、`coverage-ledger.json`，缺少稳定的 `final_report.md` 综合报告。
4. **加密场景缺口**：加密链路、算法发现位置、key/iv 来源、加解密辅助文件没有成为报告必填结构。
5. **非功能点证据缺口**：从 JS/源码/配置构造数据包时，缺少文件、行号、函数、构造逻辑到报告的标准映射。

## 1. v5 总目标

### 1.1 项目记忆层目标

让新会话不再从零考古。新会话先读 `project-memory/PROJECT_BRIEF.md` 与少数索引文件，即可知道：

- 项目是什么。
- 入口命令是什么。
- 核心模块各自负责什么。
- 数据流怎么走。
- 某类需求通常改哪些文件。
- 最近设计决策是什么。

### 1.2 报告链路目标

把“报告写作”从模型自由作文改成：

```text
候选漏洞
  -> 确立漏洞证据链 finding_<id>/
  -> 结构化校验 validate_finding()
  -> Guardian / Verify / Session Gate
  -> final_report.md 综合报告
```

核心原则：

> **漏洞确立阶段强制沉淀证据链，最终报告阶段只做稳定、清晰、统一的综合呈现。**

## 2. 需求清单

### R1 项目记忆层

`R1.1` 新增 `project-memory/`，保存项目级外部记忆。

`R1.2` 新会话只需优先阅读项目记忆层，即可获取项目轮廓。

`R1.3` 项目记忆必须分“人读摘要”和“机器读图谱”两类。

`R1.4` 每次重要迭代后，需要更新项目记忆层，记录模块变化、数据流变化和设计决策。

`R1.5` 项目记忆层不能替代源码事实；它是导航索引，具体实现仍以代码为准。

### R2 漏洞确立证据链

`R2.1` 模型确立漏洞时，不能只写 `report_*.md`，必须落结构化 finding 包。

`R2.2` finding 包必须能回答：功能点、API、参数、证明数据包、响应证据、Burp 手工复现、PoC、源码/前端证据、加密链路。

`R2.3` finding 证据链不完整时，不能进入最终综合报告。

`R2.4` 允许保留旧 `report_*.md` 兼容路径，但 v5 新链路以 `finding.json` 为权威。

### R3 最终综合报告

`R3.1` 最终输出统一为 `final_report.md`。

`R3.2` 一级标题固定：

```md
# xxx 授权安全测试报告
```

`R3.3` 每个漏洞使用二级标题：

```md
## 1. 漏洞名称：xxx
```

`R3.4` 每个漏洞必须包含三个三级标题，顺序固定：

```md
### 安全风险
### 安全建议
### 漏洞证明
```

`R3.5` `漏洞证明` 下可以有四级标题，用于展开功能点、API、参数、数据包、Burp 复现、PoC、加密链路、源码证据。

### R4 功能点与 API 展示

`R4.1` 报告必须写清楚：

```text
xx 模块 -> xx 功能，触发 xx 接口，xx 参数存在 xx 漏洞。
```

`R4.2` 每个漏洞必须列出 API 表格：

```md
| 方法 | 路由 | 功能说明 | 风险参数 |
```

`R4.3` 每个 API 要说明“用来干什么”，不能只列路径。

### R5 非功能点构造数据包

`R5.1` 如果漏洞不是从页面功能点触发，而是从 JS/源码/配置构造数据包，报告必须写明来源。

`R5.2` 来源字段必须包含：文件、行号或函数、构造逻辑、可构造的数据包、风险参数。

### R6 加密链路

`R6.1` 有加密时，报告必须呈现“原理链条”，但不要求粘完整加解密代码。

`R6.2` 加密链路必须包含：加密范围、算法、发现位置、key/iv/密钥来源、解密/重加密方式、脚本或 Burp 插件路径。

`R6.3` 报告必须标注加解密辅助文件，例如 `decrypt.py`、`encrypt.py`、`burp_crypto_helper.py`。

## 3. 总体设计

### 3.1 v5 架构分层

| 层 | 职责 | 落点 |
|---|---|---|
| 项目记忆层 | 新会话快速理解项目与改动范围 | `project-memory/`、`tools/project_memory.py` |
| 漏洞确立层 | 模型在 PASS 前沉淀结构化证据链 | `runs/<sid>/findings/finding_<id>/finding.json` |
| 报告校验层 | 判断 finding 是否足以进入综合报告 | `engine/reporting/validate.py` |
| 报告渲染层 | 从结构化 finding 渲染 `final_report.md` | `engine/reporting/render_md.py` |
| 现有裁决层 | Guardian、Verify、Session Gate、coverage-ledger | `engine/enforce.py`、`engine/verify.py`、`engine/session_gate.py` |

### 3.2 新链路与现有链路关系

当前链路：

```text
模型写 report_*.md
  -> harvest_evidence()
  -> guardian_check()
  -> coverage-ledger
  -> summary.json
```

v5 目标链路：

```text
模型写 finding_<id>/finding.json + 证据文件
  -> collect_findings()
  -> validate_finding()
  -> guardian_check(rendered finding excerpt 或 legacy report)
  -> verify_fn()
  -> coverage-ledger
  -> render_final_report()
  -> final_report.md
```

兼容策略：

- v5 阶段一仍采集 `report_*.md`，避免破坏现有流程。
- 新增 `finding.json` 后，优先使用结构化 finding。
- `report_*.md` 逐步退化为兼容输入，不再是最终交付物。

### 4.2 文件职责

| 文件 | 职责 |
|---|---|
| `PROJECT_BRIEF.md` | 3 分钟项目轮廓：项目目标、入口、核心链路、当前迭代重点 |
| `MODULE_MAP.md` | 模块职责表：`run.py`、`engine/orchestrator.py`、`ledger.py` 等各自负责什么 |
| `DATA_FLOW.md` | 数据流：recon -> surface -> planner -> ledger -> model -> evidence -> gate -> summary/report |
| `ARTIFACTS.md` | 运行产物说明：`runs/<sid>/authz.md`、`coverage-ledger.json`、`summary.json`、`events.jsonl`、future `final_report.md` |
| `CHANGE_IMPACT.md` | 改某能力通常要动哪些文件，例如“报告”要动 skill、orchestrator、reporting、run.py |
| `CODE_GRAPH.json` | 机器可读图谱：模块、函数、输入、输出、依赖、产物 |
| `decisions/*.md` | 架构决策记录，记录为什么这么设计 |

### 4.3 工具设计

新增：

```text
tools/project_memory.py
```

命令：

```bash
python3 tools/project_memory.py refresh
python3 tools/project_memory.py check
python3 tools/project_memory.py query reporting
```

功能：

- 扫描 Python AST，提取模块、函数、导入依赖、CLI 参数。
- 扫描 Markdown 标题，提取设计文档索引。
- 扫描 `run.py --help`，更新入口命令信息。
- 输出 `CODE_GRAPH.json`。
- 检查项目记忆是否过期，例如代码中新增 `engine/reporting/` 但 `MODULE_MAP.md` 未记录。

### 4.4 项目记忆更新纪律

每次完成影响架构的改动后，必须同步更新：

- 新模块：更新 `MODULE_MAP.md`。
- 新数据流：更新 `DATA_FLOW.md`。
- 新产物：更新 `ARTIFACTS.md`。
- 新决策：新增 ADR。
- 新入口参数：更新 `PROJECT_BRIEF.md` 和 `CODE_GRAPH.json`。

## 6. 最终综合报告设计

### 6.1 输出路径

```text
runs/<sid>/final_report.md
```

`summary.json` 增加：

```json
{
  "final_report_path": "/abs/path/runs/<sid>/final_report.md"
}
```

`run.py` 收尾打印：

```text
综合报告: runs/<sid>/final_report.md
```

### 6.2 模板

```md
# {{ target_name }} 授权安全测试报告

## {{ index }}. 漏洞名称：{{ title }}

### 安全风险

{{ risk.proven_impact }}

### 安全建议

{{ recommendation }}

### 漏洞证明

#### 功能点说明

{{ feature_point.statement }}

#### 涉及 API 路由

| 方法 | 路由 | 功能说明 | 风险参数 |
|---|---|---|---|
| {{ method }} | {{ path }} | {{ purpose }} | {{ risk_params }} |

#### 参数漏洞说明

{{ param_risk_description }}

#### 证明数据包

{{ proof_packets }}

#### 手工 Burp Suite 复现步骤

{{ manual_burp_replay }}

#### PoC

{{ poc }}

#### 加密链路说明

{{ crypto_chain_or_omit }}

#### 源码或前端证据

{{ source_proof_or_omit }}
```

### 6.3 渲染规则

- 没有加密时，不输出“加密链路说明”标题。
- 没有源码/前端证据时，不输出“源码或前端证据”标题。
- 有多条 API 时，全部写入 API 表格。
- 数据包过长时，正文摘关键片段并引用证据文件路径。
- PoC 可以引用文件，但必须说明如何执行或如何在 Burp 中复测。

## 7. 新模块落点

### 7.1 新增目录

```text
engine/reporting/
  __init__.py
  schema.py
  collect.py
  validate.py
  render_md.py
  templates/
    final_report.md.j2
```

### 7.2 schema.py

职责：

- 定义 finding 数据结构。
- 提供 `load_finding(path)`。
- 保持 schema 轻量，优先使用标准库 dataclass / dict，不引入重依赖。

### 7.3 collect.py

职责：

- 从 `runs/<sid>/findings/finding_*/finding.json` 采集结构化 finding。
- 兼容旧 `report_*.md`。
- 输出统一 finding 列表。

### 7.4 validate.py

职责：

检查 finding 是否可进入 `final_report.md`。

硬规则：

- 必须有 `title/severity/vuln_type/target`。
- 必须有 `risk.proven_impact`。
- 必须有 `recommendation`。
- 必须有 `feature_point` 或 `source_proof`。
- 必须至少有一条 API。
- 必须至少有一份证明数据包。
- 必须有 Burp 手工复现步骤。
- 必须有 PoC。
- 如果有 `crypto_chain`，必须包含算法/发现位置/key 来源/iv 来源/辅助文件路径。

输出：

```json
{
  "accepted": [],
  "rejected": [
    {
      "finding_id": "finding_001",
      "reason": "missing proof_packets"
    }
  ]
}
```

### 7.5 render_md.py

职责：

- 将 accepted finding 渲染为 `final_report.md`。
- 保证标题结构固定。
- 保证综合报告中每个漏洞顺序稳定。

排序建议：

1. severity：P1 -> P2 -> P3。
2. 价值：认证/支付/越权/数据导出/文件/RCE/输入验证。
3. endpoint 字典序。

## 8. 与现有代码的接线方案

### 8.1 orchestrator.py

当前 `harvest_evidence()` 只采集 `report_*.md`、`negative_*.md`、其它文件。

v5 改造：

- 增加 finding 包采集：

```text
runs/<sid>/findings/finding_*/finding.json
```

- `harvest_evidence()` 返回增加：

```json
{
  "finding_objs": []
}
```

- `_conclude()` 中：
  - 先收集 structured findings。
  - 调 `validate_findings()`。
  - 对 accepted findings 渲染 `final_report.md`。
  - 输出 `final_report_path`。

### 8.2 enforce.py

短期：

- Guardian 继续检查 legacy Markdown。
- 对 structured finding，新增 `guardian_check_finding(finding)`，先复用字段判断，不急着替换旧逻辑。

中期：

- Markdown Guardian 退化为兼容路径。
- finding Guardian 成为主路径。

### 8.3 verify.py

短期：

- `extract_poc(report_md)` 保留。
- 新增从 finding `poc.file` 或 `proof_packets.request_file` 抽请求的 helper。

中期：

- `verify_fn` 优先使用 structured finding 的 PoC 文件。

### 8.4 run.py

收尾逻辑增加：

- `summary.json.final_report_path`。
- 终端打印综合报告路径。
- `--self-check` 增加报告链路断言。

### 8.5 skill/核心技能文件.v2.md

报告格式要求从“直接写 Markdown 报告”改为：

- 确立漏洞时必须落 `findings/finding_<id>/finding.json`。
- 证明数据包、PoC、加密脚本、源码证据必须放在同目录。
- 最终综合报告由系统生成。

保留一句兼容：

```text
若当前运行环境尚未接入 reporting 模块，可临时写 report_*.md，但必须按同等字段完整呈现证据链。
```


## 11. 验收标准

### 11.1 项目记忆层验收

- `project-memory/PROJECT_BRIEF.md` 存在，能说明项目目标、入口、主链路。
- `project-memory/MODULE_MAP.md` 存在，能说明 `engine/` 下各模块职责。
- `project-memory/DATA_FLOW.md` 存在，能说明数据从 recon 到 report 的流向。
- `project-memory/CHANGE_IMPACT.md` 存在，能定位“报告改造”涉及的文件。

### 11.2 报告链路验收

- 给定结构化 finding，可以生成 `final_report.md`。
- 最终报告一级标题固定为 `# xxx 授权安全测试报告`。
- 每个漏洞二级标题固定为 `## N. 漏洞名称：xxx`。
- 每个漏洞包含 `### 安全风险`、`### 安全建议`、`### 漏洞证明`。
- 漏洞证明里包含功能点、API 路由、参数说明、证明数据包、Burp 复现、PoC。
- 加密 finding 能输出加密链路说明，并标注辅助文件路径。
- 源码/JS 构造 finding 能输出文件、行号或函数、构造逻辑。

### 11.3 回归验收

- `python3 run.py --self-check` 仍通过。
- 新增 reporting fixture 后，自检包含 final report 断言。
- 旧 `report_*.md` 路径不被破坏。
- 无 finding 或 finding 校验失败时，不生成误导性的综合报告。

## 12. 非目标

- 不把最终综合报告做成 Word/PDF。v5 只先做 Markdown。
- 不引入复杂前端 UI。
- 不让模型直接排版最终报告。
- 不把完整加解密代码塞进报告正文。
- 不让 reporting 模块事后脑补缺失证据。
- 不改变 Guardian 的基本价值判断：无 PoC、无响应证据、投机措辞仍拒收/降级。

## 13. 风险与控制

| 风险 | 表现 | 控制 |
|---|---|---|
| schema 太重，模型难落盘 | finding.json 字段太多导致执行阻力大 | MVP 字段最小化，复杂字段可选，但 final report 必需字段硬校验 |
| 旧链路被破坏 | report_*.md 不再被采集 | v5 保留 legacy 兼容，structured finding 优先 |
| 报告模块脑补证据 | final_report.md 看起来完整但证据不存在 | validate.py 只引用已存在文件，缺文件直接 rejected |
| 项目记忆过期 | 新会话读到旧地图 | `project_memory.py check` 检测模块变更未记录 |
| 加密内容过度展示 | 报告被脚本淹没 | 报告只写链路、原理、文件路径，不贴长代码 |

## 15. 完成定义

v5 完成时，应该满足：

1. 新会话能通过 `project-memory/` 快速理解项目，不必全量重读。
2. 模型确立漏洞时有结构化 finding 协议。
3. 结构化 finding 可校验、可拒绝、可渲染。
4. `runs/<sid>/final_report.md` 是最终综合报告。
5. 综合报告标题结构稳定，符合：
   - H1：`xxx 授权安全测试报告`
   - H2：`漏洞名称：xxx`
   - H3：`安全风险 / 安全建议 / 漏洞证明`
6. 漏洞证明清晰覆盖功能点、API、参数、数据包、Burp 复现、PoC。
7. 加密场景呈现原理链条和辅助文件路径。
8. JS/源码构造场景呈现文件、行号或函数、构造逻辑。
9. `python3 run.py --self-check` 包含 reporting 断言并通过。
