# 迭代方案 v8.2 · HackerOne 参数集注入 + 执行纪律修复

> 一句话结论：v8.1 的 2020 分退步不是"改坏了"而是"改错了方向"——修的是 WAF 绕过和端点推测（v8.0 已命中的领域），放过了参数格式自适应、交易参数穷举、页面级覆盖等执行层面的系统短板。v8.2 双管齐下：①从 HackerOne 12,618 份报告中蒸馏 3,319 份高质量报告的参数模式，注入知识卡和 reference，解决"测什么参数"的数据盲区；②补 5 条执行纪律，解决"该做的没做"的稳定性问题。预期从 v8.1 的 2020 分恢复到 3200-3800 区间。

## 0. v8.1 退步归因（精简版）

### 0.1 数据总览

| 指标 | v7.1 | v7.3 | v8.0 | v8.1 |
|------|------|------|------|------|
| 得分 | 4060 | 2190 | 3190 | 2020 |
| 命中率 | 84.6% | 45.6% | 66.5% | 42.1% |
| 命中数 | 20/24 | 11.5/24 | 16/24 | 10/24 |

v7.1 是最高分基线（4060），后续所有迭代的得分都没有超过 v7.1。

### 0.2 v8.0→v8.1 退步根因

v8.1 代码变更是纯增量的（+52/-3 行，0 新增约束），没有"改坏"任何东西。退步来自两层叠加：

1. **LLM agent 固有方差**（~20%）：v8.0 吃到正向方差红利（基线 2600 + 红利 590 = 3190），v8.1 吃到负向方差（基线 2600 - 波动 580 = 2020）
2. **注意力稀释**：v8.1 新增 ~60 行 WAF/端点推测指令，模型在这些方向投入更多时间，挤压了基础覆盖的执行

**核心问题**：v8.1 的 3 个 Fix 对应的是 v8.0 已命中的领域，6 个退步项（1170 分）中没有任何一个能被 v8.1 的修改防止。

### 0.3 v8.2 策略转向

## 3. 支柱 B · 执行纪律修复

从 v8.1 退步诊断中提取的 5 条执行纪律，全部采用**条件触发式**（遇到 X 场景时必须做 Y），不是通用约束。

### 3.1 参数格式自适应协议（治 V06/V19，+360 分）

**问题**：product-image-delete 返回"商品编号不能为空"，submit-audit 返回"请填写审核描述"——参数存在但格式不对，模型没有自动切换格式重试。

**修复**：在 §3 速查卡新增：

```
- 参数返回"不能为空/格式错误/缺少字段" → 依次尝试 3 种格式：
  ① application/x-www-form-urlencoded（默认）
  ② application/json（JSON body）
  ③ multipart/form-data（FormData）
  三种格式均报"不能为空" → 检查参数名是否正确（snake_case / camelCase / PascalCase 变体）
```

**位置**：核心技能文件 §3 速查卡（追加 4 行）

### 3.2 交易端点参数穷举强制（治 V12/V15，+380 分）

**问题**：lottery.php 只测了 prize_id 操控（服务端忽略），未测 POST probabilities 参数篡改；create-batch-order 创建了订单但未测 use_points 参数。

**修复**：在 §7 决策树的"交易端点参数穷举"项追加：

```
- **交易端点穷举纪律**：每个交易端点必须逐一测试 ≥4 个可控参数维度。
  如果端点有 N 个用户可控参数，每个参数至少测 3 个边界值（零/负/超大）。
  只测主流程参数（如 prize_id）而不测次级参数（如 probabilities）→ 阴性结论标 shallow_negative。
  具体方法：先 GET/POST 端点获取完整参数列表（从请求 body/HTML form/JS 代码），
  然后对每个参数逐一测试，记录到覆盖表。
```

**位置**：核心技能文件 §7 决策树（修改现有"交易端点参数穷举"项，+3 行）

### 3.3 页面级越权测试（治 V10/V22，+340 分）

**问题**：merchant/index 页面未从越权角度测试（用 user cookie 访问），pay/index.php 的 redirect 参数完全未测试。

**修复**：在 §7 决策树的 role/object-pair 分支追加：

```
- **页面级越权覆盖**：Phase 0 识别的非 API 页面（merchant/index, pay/index, admin/dashboard 等）
  必须用每个已获取的角色 cookie 逐一访问。
  普通用户 cookie 访问 merchant/admin 页面 → 检查是否返回完整内容（非 403/重定向）。
  页面中如有表单或链接，其 action/href 中的参数也纳入测试范围。
```

**位置**：核心技能文件 §7 决策树 role/object-pair 分支（+3 行）

### 3.4 已知端点 SQLi 全覆盖（治 V18，+200 分）

**问题**：product-delete.php 只测了 IDOR（换 product_no 看是否跨商户），完全未做 SQLi 测试。

**修复**：在 §7 决策树的 SQLi 注入测试分支追加：

```
- **SQLi 与 IDOR 是独立测试面**：对同一个含 ID 参数的端点，
  IDOR 测试（替换 ID 看越权）和 SQLi 测试（注入 SQL payload）是两个独立维度。
  已测 IDOR 阴性 ≠ SQLi 阴性。每个含用户可控 ID/数值/字符串参数的端点都必须做 SQLi 测试。
```

**位置**：核心技能文件 §7 决策树 SQLi 分支（+2 行）

### 3.5 存储型 XSS 闭环强制（治 V19 部分，+180 分）

**问题**：submit-audit XSS 提交了 payload 但 POST 返回"请填写审核描述"（参数传递问题），未解决参数格式后闭环验证。

v8.0 核心文件 §6 已有"存储型漏洞闭环验证"规则（POST→GET 渲染验证→跨用户验证），但 v8.1 run 中模型未执行。问题不是缺少规则，而是规则的触发条件不够明确。

**修复**：在 §6 存储型漏洞闭环验证中追加触发条件：

```
- **触发条件**：任何 POST 请求返回 200/空响应 + 端点名称暗示数据持久化
  （submit/create/update/save/add/post/review/audit/comment/feedback）→ 必须执行 GET 闭环。
  POST 返回"参数不能为空" → 先修复参数格式（§3 格式自适应），修复成功后重新 POST，再 GET 闭环。
```

**位置**：核心技能文件 §6 存储型漏洞闭环验证（+2 行）

## 9. HackerOne 数据管线技术细节

### 9.1 管线脚本

```
hackerone_pipeline_v2.py (工作目录)
├── 读取 hackerone_train.parquet (14.2MB, 10,094 报告)
├── 过滤: resolved + bounty + full + 200chars → 3,319 报告
├── 6 种正则提取参数名 (url_query/json_key/php_array/form_data/html_form/php_var)
├── 3 种正则提取端点路径 (http_method/api_path/php_endpoint)
├── 按 weakness.name 归一化聚类 (20 个标准类型)
├── 输出: hackerone_param_patterns.json + .md
└── 输出: knowledge_card_updates.json (含与现有卡的 diff)
```

### 9.2 噪音过滤策略

以下参数类型被排除：
- **Tracking 参数**：`_ga`, `__utma/b/c/z`, `utm_*`, `_shopify_*`（GA/UTM 追踪）
- **HTML 属性**：`onerror`, `onload`, `onmouseover`（XSS payload 组成部分，不是参数名）
- **通用框架参数**：`data`, `value`, `key`, `items`, `result`（过于泛化）
- **HTTP 标准参数**：`method`, `body`, `header`, `status`（协议层，非应用层）
- **短参数**：长度 < 3 字符（如 `id` 已覆盖，但 `q`, `x` 等噪音过滤）

### 9.3 数据集许可

Hacker0x01/hackerone_disclosed_reports 使用公开数据集，允许研究用途。提取的参数模式是统计聚合结果，不包含任何原始报告内容。
