# Atoolkit v8.12 迭代方案：Engine 两阶段威胁计划与 SRC 运行真实性

日期：2026-07-17

## 1. 结论与版本目标

v8.11 已经建立 Feature → Threat → Cell → Finding → Canonical Report 合同，但 Engine
仍在模型启动前把 endpoint 与通用漏洞类组合为 coverage 分母。真实 ByteDance 历史 Run 又暴露出
四个不能靠提示词解决的问题：

1. Run 只有叙述型 `inventory.json` 和模型手写报告，没有 manifest、frozen plan、精确 ledger、
   原始请求响应或 Finding 包，无法证明“测了什么”和“为什么能下结论”；
2. 两个测试账号共用浏览器会话，登录后身份互相覆盖，却仍把跨账号端点写成“安全”，
   把阻塞伪装成阴性；
3. 报告包含原始认证材料、API Key 与个人信息，运行产物和对外交付没有秘密边界；
4. “缺少 CSRF / CSRF token 暴露”被写成 P3，但 PoC 没有证明跨站可观察的状态变化，
   本质仍是现象或假设链。

v8.12 的目标不是增加漏洞清单或 payload，而是让 Engine 真正执行两阶段合同：

```text
Host 收集/规范化 Recon
  → 离线 Planning Session（无目标网络）
  → 模型生成 Feature/Threat
  → Host 校验、冻结、建立 lineage
  → Attack Session 只消费编译后的 Threat Cells
  → 身份/证据/结论门
  → Finalizer 脱敏生成唯一报告
```

完成后，任何“安全”“未发现”或漏洞声明都必须能回到一个冻结 threat、精确 cell、有效实验身份和
物理证据；否则终态只能是 `incomplete/NEED_INPUT`，不能写成安全结论。

## 2. 为什么这样设计

### 2.1 规划与攻击必须是两个不可变 Run

如果同一个模型会话先写计划、再测试、最后修改计划，那么 coverage 分母可以随着结果变化：
漏测的威胁可以被删掉，误报可以被补写成计划，manifest 也无法证明首个动作前的真实意图。

因此 v8.12 使用两个 sibling session：

- `<sid>.planning`：只读脱敏 Recon，无目标网络；模型只能产出 feature/threat 候选；
- `<sid>`：Host 校验、编译、冻结后才启动攻击模型；本轮不得扩大 denominator。

两者各自在首个模型动作前拥有 authority-owned run plan 与 immutable manifest。Attack manifest
绑定 Planning manifest 的 session、authority path 和 SHA-256，形成不可替换的 phase lineage。

### 2.2 模型负责业务推理，Host 负责准入

模型适合从页面、JS、流量和 API 语义中判断业务资产、安全不变量、攻击者能力和可观察突破；
但模型不能决定自己的输出是否完整、是否在授权路径内、是否泄露秘密或是否具备测试身份。

Host 只做确定性工作：

- inventory/feature/threat 的 schema 与引用校验；
- discovery 证据充足度与 scope path 校验；
- threat 编译为 exact cells；
- 身份上下文是否互相独立；
- 计划和父 manifest hash 是否匹配；
- Finding 因果证据与 Canonical 报告脱敏。

### 2.3 阻塞不是阴性

对象越权需要 owner/peer，角色越权需要低权/高权，交易需要可回查的测试数据。缺少前置条件时，
请求返回 NotFound、空数组或当前账号自己的数据不能证明安全。v8.12 把身份要求放进 threat cell，
并由 host-owned `identity-readiness.json` 判定。身份不满足的 cell 保持 open/blocked；任何模型文本都
不能把它关闭为 `not_vulnerable`。

### 2.4 报告不应复制运行秘密

原始 HTTP 证据必须保留才能复验，但它和可提交报告不是同一信任域：

- 原始 Cookie/Authorization/API Key 只允许存在于权限收紧的凭据文件或 Finding/evidence 包；
- Host 只把稳定占位符作为 Planning 的显式输入，不注入原值；
- Planning inventory、feature、threat 与 Canonical report 不允许出现高置信秘密；
- Finalizer 渲染 HTTP/PoC 片段时自动替换为 `<redacted:kind:hash>`，保留同值可关联性。

当前 Codex workspace sandbox 不能证明“同一 OS 用户下全局只读不可见”。因此实现还必须让本轮
`cookies.txt` 延迟到 Planning 完成后才落盘、Planning 网络固定关闭、输出再次做秘密扫描。这能
阻止本轮凭据注入和外传，但不宣称拥有容器级 read isolation；完全硬隔离仍依赖后端监督器。

## 3. Planning Session 合同

### 3.1 启动模式

Engine CLI 新增：

```text
--planning-mode auto|threat|legacy
--feature-graph <prebuilt.json>
--threat-model <prebuilt.json>
--identity-role label:role
--identity-marker label:non-secret-marker
```

规则：

- live + `auto` + `--recon-dir`：默认两阶段 threat；
- live + `auto` 但无 Recon：拒绝启动，要求补 Recon 或显式选择 legacy；
- live legacy 必须显式 `--planning-mode legacy`，manifest 固定 degraded；
- dry-run 的 `auto` 保持 legacy，避免自检隐式调用真实模型；
- feature/threat 必须成对提供；提供时走相同校验和 freeze，只跳过规划模型调用；
- `--ad-hoc` 只能显式 legacy，不得伪装为完整 threat run。

### 3.2 脱敏 Recon Snapshot

Host 从 `--recon-dir` 复制有界文本证据到 Planning Session：

- 只收普通文件，拒绝 symlink/hardlink；
- 限制单文件、文件数与总字节数；
- 保留原相对路径与原文件 SHA-256；
- Cookie、Authorization、API Key、Token、密码、手机号、邮箱替换为稳定 hash 占位符；
- 输出 `discovery-evidence.json`，记录 copied/omitted/redacted，不记录外部绝对路径；
- 规范化 `inventory.json` 同样经过秘密扫描。

Planning manifest 的 `planning_artifacts` 在模型启动前绑定 inventory、discovery index 与每个
snapshot 文件。模型篡改输入后，即使产出了合法 JSON，也不得进入攻击阶段。

### 3.3 Discovery 充足度

v8.11 只要求六个 channel 均有状态；v8.12 再增加 adequacy gate：

- 六个 channel 仍必须逐一应答；
- 至少两个 channel 为 `covered`；
- 至少一个来自代码/资源组：`js_ref|inline_script|asset_ref`；
- 至少一个来自导航/运行组：`page_link|path_inference|response_body`；
- 每个 covered channel 必须引用 Planning Snapshot 内真实文件；
- 全部 blocked/N/A 不得启动 Attack。

这不保证零遗漏，但能阻止“只有一份接口清单也声称完成威胁建模”。

### 3.4 Path Scope

Planning prompt 和 validator 同时获得 frozen `base_path/allow_paths/deny_paths`：

- feature API 只能引用授权 path；
- 同 origin 但跨模块的端点必须进入 `unassigned_endpoints`，reason 为 scope amendment；
- 不得因为同域名就自动扩展当前 Run；
- 新发现只进入下一 Run discovery/Intent，不动态扩大 Attack denominator。

## 4. Threat 与身份合同

### 4.1 Threat 新增身份要求

每个 threat 规范化一个 `identity_requirement`：

```json
{
  "mode": "peer_pair",
  "roles": ["user"],
  "minimum_distinct_credentials": 2,
  "reason": "需要 owner 与同级 peer 比较对象归属"
}
```

允许模式：

- `single`：单一请求上下文足够；可为匿名，需登录时显式把 minimum 设为 1；
- `anonymous_plus_authenticated`：匿名基线加一个登录身份；
- `peer_pair`：两个不同凭据、同级角色；
- `role_pair`：两个不同凭据、不同要求角色；
- `stateful_owner`：一个身份加可创建、可回查的自有测试对象。

授权/IDOR threat 不允许默认为 `single`。Planner 未声明时按 vuln family 保守提升要求。

### 4.2 `identity-readiness.json`

Host 从 CLI 凭据只计算指纹，不复制原值：

```json
{
  "schema_version": 1,
  "identities": [{
    "label": "owner",
    "role": "user",
    "credential_sha256": "...",
    "marker_sha256": "..."
  }],
  "threats": [{
    "feature_id": "agent-plan",
    "threat_id": "T-object-read",
    "ready": false,
    "reason_code": "distinct_identity_missing"
  }]
}
```

同一 credential hash 被多个 label 引用时只算一个身份。缺少 marker 不一定阻止攻击探索，但涉及
跨账号阴性闭格时必须具备可证明身份/对象差异。identity readiness 由 manifest 绑定，模型不能改写。
原始多身份 header 只在 Planning 完成后写入权限收紧的 Attack `identities.json`，并按 label 隔离；
它不进入 Planning artifact 或 Canonical report。这样 Host 的“ready”与 Attack 模型实际可用身份一致。
Finalizer 尚未开始的崩溃恢复必须复算并匹配 frozen 指纹；身份条件变化必须新开 sid。
一旦存在 finalization journal/receipt，该 sid 已成为不可变交付，不允许继续模型执行；后续测试新开 sid。

### 4.3 Threat Cell 唯一分母

`CognitiveState` 新增 `seed_threat_cells()`，直接按 compiler 输出创建一格，不再调用
`_classes_for_endpoint()`。Threat 模式禁止 endpoint × DEFAULT_VULN_CLASSES 回退。

Run plan 的 admitted cell 同时冻结：

- exact cell identity；
- `surface_id`；
- `feature_id/threat_id`；
- invariant/observable violation；
- identity requirement/readiness。

`threat-coverage.json` 在每次 ledger sync 时重新派生。缺身份的 threat 始终 open。

现有 runtime cell identity 以 vuln class 为最后一维。若两个不同 threat 在
asset/method/path/param/location/role/vuln_class 上完全相同，compiler 必须拒绝计划并要求使用不同的
业务 vuln class 或更精确 target 维度，不能让它们在 Engine state 中静默碰撞。

## 5. Finding 因果门与结论一致性

### 5.1 CSRF 专用证明合同

CSRF 不再走 generic response differential。正式 Finding 必须：

- `verification.evidence_type=cross_site_state_change`；
- 有 `state_before/cross_site_request/state_after` 三阶段物理证据；
- cross-site 请求携带与目标不同的 `Origin` 或等价浏览器来源证据；
- 引用物理 `cross_site_initiator_file`，其中包含可执行的 form/img/fetch/XHR/navigation 发起原语；
- 提供可复算 `state_delta`；
- 证明受害者会话被浏览器带上；
- 影响是状态变化，而不是“响应中有 token”“缺少 token”或攻击者理论上可读响应。

这会拒绝 ByteDance 历史报告中的两类 P3，但不会误杀真正的转账、绑定、改密等 CSRF。

### 5.2 Identity-aware Closure

Validator 重算 threat plan 与 identity readiness：

- unready threat 的 cell 若被写成 confirmed/not_vulnerable/not_applicable，closure fail；
- 只有 blocked/not_tested/shallow_negative 可保留；
- blocked/high-value-open 会使 Canonical 状态 incomplete/NEED_INPUT；
- 报告不得使用“安全”“严格隔离”等措辞替代未完成结论。

### 5.3 报告秘密边界

Canonical renderer 对 Finding 字段、HTTP 包和 PoC 片段统一脱敏。原文件路径仍保留，复验人员可在
本地受限 evidence 包查看原值。稳定占位符让 owner/attacker 两份 Cookie 是否相同仍可比较。

## 6. Manifest Schema 4 与 Phase Lineage

新增冻结字段：

```json
{
  "schema_version": 4,
  "run_phase": "attack",
  "phase_parent": {
    "session_id": "sess-001.planning",
    "manifest_path": "/authority/manifests/sess-001.planning.json",
    "manifest_sha256": "..."
  }
}
```

约束：

- Planning manifest：`run_phase=planning`、`planning_mode=threat_discovery`、无 parent；
- Attack threat manifest：`run_phase=attack`、`planning_mode=threat_model`、必须有 parent；
- parent 必须位于同一 authority `manifests/`、project_id 相同、phase=planning；
- parent session 必须为 `<attack sid>.planning`；
- hash、authority id、target、scope、base path 与模型 provenance 不一致均 fail closed；
- Attack 逐个绑定晋升后的 Recon snapshot，并在终态递归校验 Planning parent artifact binding；
- legacy attack 可无 parent，但必须 `planning_degraded=true`。

- `engine/data_hygiene.py`：高置信秘密检测、稳定脱敏、公开产物扫描；

- `engine/runtime_manifest.py`：schema 4 phase lineage；
- `engine/reporting/validate.py`：parent/identity/threat closure 与 CSRF 因果门；
- `engine/reporting/render_md.py`：Canonical 输出脱敏；

## 8. 失败先行测试

1. live auto + Recon 进入 planning；无 Recon 时拒绝隐式 legacy；
2. planning manifest/run plan 在 adapter 首次调用前存在；
3. planning adapter 网络关闭，Attack adapter 才使用用户接受的网络模式；
4. model 修改 planning input 后 phase transition 失败；
5. discovery 只有代码组或只有导航组时拒绝；
6. path 越界 API 不能进入 feature/threat；
7. threat cell 数等于 compiler 输出，不出现默认漏洞类笛卡尔积；
8. parent manifest 缺失、错 session、错 hash、错 project 均拒绝；
9. owner/peer 使用相同 credential 时 identity readiness=false；
10. unready IDOR cell 不能被阴性关闭；
11. 独立 owner/peer 可进入测试，但仍需物理差异证据才能闭格；
12. planning snapshot 与 final report 不出现原始 Cookie/API Key/Token/手机号/邮箱；
13. 原始 Finding evidence 不被破坏，报告中的同值占位符稳定；
14. 缺 CSRF token、token header 暴露、仅 response differential 的 CSRF Finding 被拒绝；
15. 有 before/cross-site/after 与 state delta 的真实 CSRF 通过；
16. legacy dry-run/旧 manifest 兼容，且始终 degraded；
17. 全量 v8.8-v8.11 authority/finalizer 测试保持通过。

## 9. 完全体边界

v8.12 达成的是“Engine 可执行威胁计划完全体”，不是“自动挖洞终局”：

- 同一证据输入下，规划、分母、身份阻塞和最终结论可复验；
- 不再用漏洞清单替代业务威胁；
- 不再把没条件测写成安全；
- 不再把运行秘密复制到 SRC 报告；
- 不再让现象型 CSRF 进入正式 Finding。

仍不保证：模型一定提出所有威胁、黑盒 Recon 一定发现全部功能、当前 Codex backend 具备可证明
pre-exec 出站白名单或进程完全静默。后两项仍必须在 manifest/delivery 中诚实降级。

v9 的终局标准仍是：在可证明的网络/进程监督器下，Feature → Threat → Cell → Experiment →
Proof → Report 全链唯一真值，并能把每个未命中归因到 discovery、plan、identity、execution 或 proof，
而不是归因到“模型今天状态不好”。
