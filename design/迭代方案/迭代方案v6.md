# 迭代方案 v6 · 有无优先的模型榨干 Loop

> 一句话结论：当前外壳已经能防“伪完成”，但仍偏向“拿一个 surface 就尝试闭格”。v6 要在 finding 之前增加一层候选发现循环：先把模型的发散能力榨干，最大化“哪里可能有洞”的召回，再把候选分批证实、淘汰、闭格。先解决有无，后续再优化深度、效率和报告质量。

---

## 0. 目标与非目标

### 0.1 目标

- 先最大化漏洞候选召回：发现端点、参数、角色、对象、流程、状态变化中的“可能存在漏洞”的点。
- 降低早停：不能因为第一个小洞、一个报错、一个验证码、一次无回显就停止探索。
- 把模型能力用在发散：让模型提出多个候选假设和下一步探测，而不是过早写报告。
- 把外壳能力用在收敛：候选必须进入台账，台账决定下一轮优先级、证明/淘汰/阻塞。
- 保持通用：不写入任何测试场路径、答案、分值、payload 清单或特定业务假设。

### 0.2 非目标

- 不绕过授权、验证码、人审、风控或二次验证边界。
- 不把 oracle 或答案注入 prompt。
- 不要求第一版就自动证明所有候选；v6.0 先把候选空间挖出来。
- 不用“更长 prompt”替代外部状态；loop 的记忆仍由文件和 ledger 维护。

---

## 1. 核心思路：双台账

现有 `coverage-ledger.json` 负责“surface 是否闭格”。v6 增加 `candidate-ledger.json`，负责“可能存在漏洞的假设是否被处理”。

| 台账 | 问题 | 粒度 | 状态 |
|---|---|---|---|
| `inventory.json` | 有哪些 endpoint / 来源是什么 | endpoint + source | discovered |
| `coverage-ledger.json` | 哪些 surface 已测完 | endpoint + method + param + role + risk_tag | not_tested / confirmed / not_vulnerable / blocked / not_applicable |
| `candidate-ledger.json` | 哪些漏洞假设被提出、验证、淘汰 | surface_id + hypothesis + evidence_need | proposed / triaging / proof_ready / confirmed / refuted / blocked / duplicate |

关键变化：

- 发现阶段不要求马上写 `finding.json`。
- 模型每轮必须产出候选假设或有效淘汰证据。
- 候选假设必须落盘，不能只留在对话里。
- 后续 proof loop 从候选台账里挑高价值候选做证明。

---

## 2. Loop 总体结构

```text
bootstrap
  -> recall loop（榨干候选）
  -> triage loop（低成本筛选）
  -> proof loop（证据链证明）
  -> closure loop（阴性/阻塞/未测收口）
  -> summary/report
```

### 2.1 Bootstrap：建立疆域

输入：

- 授权范围与凭据。
- `--recon-dir` 或 `--endpoints`。
- 可选多身份账号。

动作：

- `surface.bootstrap()` 生成 endpoint inventory。
- `planner.plan_surfaces()` 展开 surface。
- 初始化 `coverage-ledger.json`。
- 初始化空 `candidate-ledger.json`。

输出：

- 可测 surface 队列。
- 高价值面 Top 队列。
- 已知身份/角色/对象约束。

### 2.2 Recall Loop：只问“可能哪里有洞”

目的：最大化候选召回，不急着证明。

每轮注入：

- Top N 个未覆盖高价值 surface。
- 最近发现但未登记的新 endpoint。
- 最近 refuted 的候选摘要，避免重复。
- knowledge cards 中与当前 surface 匹配的误判提醒。

模型任务：

- 对每个 surface 提出 1-3 个候选假设。
- 对每个候选写明最小验证动作。
- 对阻塞写明 blocker 和恢复动作。
- 不写最终报告，除非已经有完整结果证据。

模型输出协议：

```text
CANDIDATE: <surface_id> | <vuln_class> | <hypothesis> | <evidence_need> | <priority:P1/P2/P3> | <next_probe>
BLOCKER: <surface_id> | <reason> | <recover_action>
DISCOVERED_ENDPOINT: <method> <path> | <source> | <why_relevant>
```

外壳动作：

- 解析 `CANDIDATE`，写入 `candidate-ledger.json`。
- 解析 `DISCOVERED_ENDPOINT`，回灌 `inventory.json` 和 `coverage-ledger.json`。
- 同一候选按 `endpoint + vuln_class + role + root_cause` 去重。
- 如果连续 K 轮没有新增 candidate / endpoint，进入 triage loop。

退出条件：

- 高价值 surface 均至少有候选或充分阴性。
- 连续 2-3 轮无新增候选且 inventory 饱和。
- 预算达到 recall 阈值，例如前 40% turns。

### 2.3 Triage Loop：低成本筛选候选

目的：把“可能有”分成 proof_ready / refuted / blocked，而不是让模型陷在一个点上。

选择策略：

1. 高价值候选优先：auth / role / object / payment / amount / upload / ssrf / admin。
2. 证据成本低优先：只需换对象、换角色、改参数、对比响应即可判断的候选先做。
3. 阻塞可恢复优先：缺对象、缺角色、数据为空，先生成 next_actions。

每轮注入：

- Top M 个 proposed candidate。
- 每个 candidate 的 `evidence_need` 和 `next_probe`。
- 已有请求/响应片段摘要。

模型任务：

- 执行最小探测。
- 输出差异：状态码、关键字段、对象归属、金额/数量/权限变化、异步结果。
- 不足以证明时标 `needs_more_evidence`，并给下一步。

输出协议：

```text
TRIAGE: <candidate_id> | proof_ready|refuted|blocked|needs_more_evidence | <reason> | <evidence_ref> | <next_action>
```

外壳动作：

- `proof_ready` 进入 proof loop。
- `refuted` 记录淘汰理由，避免重复探索。
- `blocked` 交 `blocker.resolve_blocker()`，可恢复则转 next_actions，不可恢复才收口。
- `needs_more_evidence` 限制重试次数，超过阈值降级或转 blocked。

### 2.4 Proof Loop：把候选变成 finding 包

目的：只对 proof_ready 候选投入深测成本，生成可交付证据。

每轮注入：

- Top K 个 proof_ready candidate。
- finding schema v1 要求。
- 证据文件路径规范。
- Guardian 的垃圾洞清单和授权边界。

模型任务：

- 生成 `findings/finding_<id>/finding.json`。
- 写 request/response/poc。
- 对多身份/多对象类漏洞补足响应差异。
- 有加密或前端构造时补 `source_proof` / `crypto_chain`。

外壳动作：

- `validate_finding()` 校验结构。
- `guardian_check_finding()` 校验报告质量。
- 可选 `verify_fn()` 确定性复验。
- accepted 后：
  - candidate 状态 `confirmed`。
  - coverage surface 状态 `confirmed`。
  - summary/final_report 纳入。

### 2.5 Closure Loop：诚实收口

目的：防止“候选很多但报告少”或“没证明就说完成”。

收口规则：

- `confirmed`：有 accepted structured finding。
- `refuted`：有最小探测证据，能解释为什么不是洞。
- `blocked`：阻塞分类明确，恢复动作已尝试或需要人类输入。
- `not_vulnerable`：必须满足 negative_sufficient，且逐格 `negative_<surface_id>.md` 落证据。
- `shallow_negative`：不能闭格，必须留 `next_actions`。

终态：

- 有 confirmed，但仍有高价值 candidate/surface 未处理：`incomplete_with_findings`。
- 无 confirmed，但候选召回和阴性闭格充分：`no_vuln_found` 或 `low_roi`。
- 有 NEED_INPUT：`needs_input`。
- 证据不一致：`error`。

## 3. 调度策略：先有无，后优化

v6.0 不追求最优调度，先用简单比例：

| 阶段 | turns 预算 | 目的 |
|---|---:|---|
| Recall | 40% | 榨干候选和新 endpoint |
| Triage | 35% | 低成本筛候选 |
| Proof | 20% | 证明最高价值候选 |
| Closure | 5% | 阴性/阻塞/状态收口 |

动态调整：

- 如果 recall 连续 3 轮无新增候选，提前进入 triage。
- 如果 proof_ready 候选 ≥ 3，提前插入 proof burst，避免证据窗口丢失。
- 如果 session-gate 报 high_value_not_tested，下一轮强制回 recall/triage。
- 如果模型卡在一个 candidate 超过 2 轮无新证据，外壳切换候选。

## 4. 防止再次变“垃圾”的硬门

### 4.1 不准发现后不测

任何 `DISCOVERED_ENDPOINT` 必须进入 inventory 和 coverage-ledger；未测试前 session-gate 不能 pass。

### 4.2 不准测一点就闭

阴性必须满足 `negative_sufficient`；否则只能 shallow，并生成 next_actions。

### 4.3 不准小阻塞打回后放弃

blocker 必须分类：

- 可恢复：创建对象、换账号、换角色、刷新数据、重新登录。
- 需要输入：验证码、短信、人审、缺授权账号。
- 不可测：越权范围外、危险动作、破坏性动作。

只有后两类能停止当前分支；可恢复 blocker 必须转任务。

### 4.4 不准登录注册口只测通不测风险

auth endpoint 默认展开候选：

- 枚举/差异响应。
- 注册流程滥用。
- 找回/验证码流程缺陷。
- token/session 生命周期。
- 角色创建/升级/绑定。
- 登录后对象归属差异。

这些是通用风险面，不依赖测试场答案。

### 4.5 不准报告漏洞

confirmed candidate 必须对应 finding 包；final_report 只从 accepted finding 渲染。若有 confirmed candidate 无 finding 包，终态 error 或 incomplete。

## 5. 最小落地方案 v6.0

先不重写 orchestrator，只增薄层。

### 5.1 新增 candidate 模块

```text
engine/candidate.py
```

职责：

- 定义 candidate schema。
- 解析 `CANDIDATE/TRIAGE/DISCOVERED_ENDPOINT` 行。
- 读写 `candidate-ledger.json`。
- 去重、排序、统计。

建议字段：

```json
{
  "candidate_id": "cand_001",
  "surface_id": "...",
  "endpoint": "/api/example",
  "method": "POST",
  "param": "amount",
  "role": "user",
  "vuln_class": "业务逻辑",
  "hypothesis": "amount 参数可能存在篡改",
  "evidence_need": "对比正常金额与篡改金额后的订单/余额结果",
  "next_probe": "使用自有账号创建测试对象并修改 amount",
  "priority": "P2",
  "status": "proposed",
  "evidence_refs": [],
  "blocker": null,
  "created_turn": 0,
  "updated_turn": 0
}
```

### 5.2 orchestrator 接线

- `assemble_prompt()` 增加 candidate Top 队列。
- 每轮 `state.update()` 后解析 candidate 输出。
- `_conclude()` 输出 candidate 统计：
  - `proposed`
  - `proof_ready`
  - `confirmed`
  - `refuted`
  - `blocked`
  - `high_value_open_candidates`

### 5.3 run.py 开关

新增：

```bash
--loop-mode coverage|recall-first
--recall-turn-ratio 0.4
--candidate-top-n 8
```

默认可以先保持现有 coverage；测评/正式覆盖跑启用 `recall-first`。

### 5.4 自检

`run.py --self-check` 新增断言：

- MockAdapter 输出 `CANDIDATE` 后生成 `candidate-ledger.json`。
- MockAdapter 输出 `DISCOVERED_ENDPOINT` 后 inventory/ledger 回灌。
- `proof_ready` candidate 无 finding 包时终态不能 complete。
- 连续无新增候选后 loop 进入 triage。

## 6. 后续优化 v6.1+

- 候选价值模型：按影响、证据成本、可恢复性、角色/对象覆盖度排序。
- 多模型分工：一个模型 recall，一个模型 triage，一个模型 proof reviewer。
- 候选聚类：把同一根因的多表现合并，避免重复证明。
- 失败记忆：把 refuted/blocked 模式沉淀为 knowledge cards。
- 自适应预算：根据 hit rate 自动调整 recall/triage/proof 比例。
- 运行后复盘：输出“发现了但没测 / 测了但没深入 / 阻塞未恢复 / 漏进报告”的四类清单。

---

## 7. 完成定义

v6.0 完成时，至少满足：

1. 有 `candidate-ledger.json`，候选不会只存在于模型回复里。
2. 新发现 endpoint 自动进入 inventory 和 coverage-ledger。
3. 模型能在 recall 阶段批量提出候选，而不是首洞即停或直接写报告。
4. proof_ready 候选必须生成 finding 包，否则不能 complete。
5. session-gate 同时看 coverage open surface 与 candidate open hypothesis。
6. 终端摘要分开打印：accepted findings、open candidates、proof_ready、blocked、refuted、high_value_open。
