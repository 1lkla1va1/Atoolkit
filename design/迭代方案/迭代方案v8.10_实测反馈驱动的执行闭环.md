# Atoolkit v8.10 迭代方案：实测反馈驱动的执行闭环

日期：2026-07-16

## 1. 结论与目标

v8.9 还不是“完全体”。它已经把 authority、Finding proof、跨 Run 精确身份、finalizer 和交付事务收紧到 fail-closed，但本轮 Qwen/QoderWork 实测没有真正进入这些工程路径；与此同时，Engine 内仍存在会制造执行假阴性的合同漂移。因此 v8.10 不再增加更多漏洞打法，而是把“风险推导 → 知识激活 → 现场观察 → 状态归并 → 阴性门禁 → 最终投影”接成一条机器可验证链。

本轮目标：

1. Direct/QoderWork 即使只能产出 diagnostic，也能使用确定性的初始化、观察归并和 checkpoint，不再只靠 agent 自觉维护 Markdown。
2. 每个用户可控参数都独立进入输入校验风险维；显式 `risk_tags` / `vuln_classes` 不再被 planner 丢弃。
3. WAF 拦截、会话过期、对象不存在、空数据集、角色缺失和格式未解析不得闭成 `not_vulnerable`。
4. `shallow_negative` 与 `exploring` 在 authoritative coverage ledger 中保真，不再悄悄折叠成 `not_tested`。
5. 多 agent 结论先写独立 observation，再由 reducer 归并；冲突不得由后写覆盖先写，proof-valid Finding 可确定性推翻早期浅阴性。
6. 报告只从 canonical validator/projector 生成；`state/*.md` 继续作为人类视图，不能反向改写机器真值。

## 2. 实测证据与根因

### 2.1 状态与报告投影分叉

`runs/shop_2026-07-15/state/findings_summary.md` 已记录 `balance-records IDOR`，但同轮 `reports/score_comparison.md` 仍写“未命中”。这证明发现摘要和最终报告没有共享同一投影源。

### 2.2 参数风险维丢失

同轮 `inventory.json` 把 `product-delete.php/product_no` 只标为 IDOR。当前 `planner.plan_surfaces` 又忽略 inventory 自带的 `risk_tags` / `vuln_classes`，`product_no` 仅推导 object ownership，不推导独立 input/injection 风险，直接复现“IDOR 测过即误认为参数测过”。

### 2.3 知识卡路由名义接通、实际失配

`knowledge.match_cards` 未把 `risk_tags` 纳入匹配上下文，也漏读单值 `param`。因此 SKILL 所写“按 risk_tags 查卡”在一部分 planner surface 上不会发生。WAF 卡虽然存在，遇“非法关键字”时仍可能不被下一轮 prompt 激活。

### 2.4 阴性门只数向量，不识别前置条件

现有 `negative_sufficient` 主要检查向量、响应、身份和证据类型；它不理解“所有响应都来自 WAF”“session 已过期”“测试的是不存在对象”“列表为空”。因此错误实验也可能积累足够数量，伪装成深阴性。

### 2.5 状态 schema 漂移

核心文件和 reference 允许 `not_tested / confirmed / not_vulnerable / shallow_negative / blocked / not_applicable / exploring`，但 `engine.ledger` 只保存五态，并把 `shallow_negative` 折叠为 `not_tested`。这使调度、统计和报告门必须依赖旁路字段猜测原状态。

### 2.6 Direct 模式缺少可执行工程入口

v8.9 清楚声明 Direct Skill 为 diagnostic，这是正确的安全边界；问题是 Direct 模式只有文字流程，没有一个方便 QoderWork 调用的 init/observe/checkpoint 工具。结果是工程层“零发挥”，而不是 authority 设计本身错误。

### 2.7 指令冲突

核心文件顶部写“遇验证码立即停”，后文又要求先测试至少五类绕过方向。该冲突会诱发过早 `NEED_INPUT`。修复后统一为：不得读取截图凭据或代替真人完成挑战；先在安全范围内测试流程绑定/跳步/空值等绕过，全部失败后再请求人工输入。

## 3. 设计

### 3.1 Planner 风险并集

- `infer_risk_tags` 接收 declared tags/classes，并与参数语义推导做并集。
- 所有非空用户参数至少得到 `input-validation`；除二进制文件体外得到 `injection`，保证对象 ID/编号参数仍有独立注入维。
- 文本持久化语义补 `xss`，URL/回调/文件/金额等保留专用风险。
- inventory 显式声明只能增加维度，不能降低确定性推导得到的维度。

### 3.2 知识卡路由修复

- 匹配上下文纳入 `param`、`risk_tags`、`barrier_signals`。
- 卡片支持 `match.risk_tags`。
- WAF/IDOR 卡扩充通用 `_no` / hash / product/order 等对象参数，不针对具体靶场路径。
- Direct checkpoint 输出每个 open surface 的 `knowledge_card_ids` 和紧凑 hint，避免加载整套 playbook。

### 3.3 实验前置条件与 barrier gate

negative observation 新增：

- `barrier_signals`: `waf_blocked / session_expired / auth_required / object_absent / empty_dataset / missing_role / challenge_unsolved / format_unresolved / waf_bypass_exhausted`
- `preconditions`: `auth_valid / data_ready / object_exists / ownership_known / roles_ready / request_shape_resolved`

规则：

- 任何可恢复 barrier 均不得闭阴性。
- `object_absent/empty_dataset` 转 `blocked(recoverable)`，next action 是创建/枚举真实测试数据后重测。
- `session_expired/auth_required` 转 `blocked(recoverable)`，要求新鲜会话重测。
- `waf_blocked` 或 `waf_bypass_exhausted` 固定为 `shallow_negative`，并激活 WAF 卡；“绕过尝试很多”也不能证明后端无漏洞。
- IDOR 阴性要求 owner、peer、真实对象归属和 access result；不存在对象的拒绝响应不计所有权阴性。

### 3.4 七态 ledger

authoritative status 改为七态并明确：

- 闭合：`confirmed / not_vulnerable / not_applicable`
- 打开：`not_tested / shallow_negative / blocked / exploring`

迁移读取保留旧旁路 `negative_depth=shallow`；新写入直接保存 `shallow_negative`。统计、next surface 和 session gate 均按七态计算。

### 3.5 Direct diagnostic runtime

新增 `engine.skill_runtime`：

```bash
python3 -m engine.skill_runtime init \
  --run-dir <run> --target <url> \
  --inventory <inventory.json> [--recon-dir <recon>]

python3 -m engine.skill_runtime observe \
  --run-dir <run> --agent-id <id> --input <observation.json>

python3 -m engine.skill_runtime checkpoint --run-dir <run>
```

- `init` 合并 inventory 与 recon 产物，生成机器 ledger、queue 和 diagnostic runtime status；不自建可信 authority，不修改 ProjectState。
- `observe` 以 create-only 文件保存单 agent observation，拒绝 surface 不存在、越界 evidence ref、重复 observation ID 异内容。
- `checkpoint` 读取所有 observation，调用同一 negative/barrier/proof 规则归并 ledger，生成 `state/checkpoint.json` 与 `execution-queue.json`。
- 多 agent 阳性/阴性冲突进入 `exploring`；只有 canonical Finding 通过 proof validator 后才可确定性关闭为 `confirmed`。proof-valid 阳性可覆盖此前因对象错误产生的浅阴性，并保留冲突审计记录。
- 所有 Direct 产物显式 `authority_trusted=false / delivery_eligible=false`。

### 3.6 分层指令

- 新增短文件 `skill/runtime-hot-path.md`，只含启动、每格循环、barrier、checkpoint 和终态门。
- `SKILL.md` 将 Direct/QoderWork 首读入口改为 hot path；完整核心和 reference 按需读取。
- 修正验证码冲突，增加每个 phase/每 10 surface 强制 checkpoint。

### 3.7 报告一致性

- checkpoint 只输出机器状态与队列，不自行渲染最终漏洞报告。
- final report/summary 继续由 `reporting.validate + render` 的同源结果生成。
- 若 canonical Finding、ledger 和人类摘要不一致，runtime status 标 `projection_stale=true`，禁止 report-ready。

## 4. 兼容性与非目标

- 不把 Direct/QoderWork 升格为可信跨 Run authority；没有外部 containment attestation 就永远是 diagnostic。
- 不保证真实 SRC 一定挖到漏洞，也不以单个靶场 oracle 分数作为发布真值。
- 不自动执行爆破、OCR、验证码代答、DoS 或超范围探测。
- 不把具体 payload 固化进 planner/checkpoint；知识卡继续只表达维度和证据要求。
- 旧五态 ledger 可读；新写七态。旧 Markdown 阴性仍是观察，不能直接进入 ProjectState。

## 5. 测试与发布门

必须新增失败先行的回归：

1. `product_no` 同时推导 IDOR 与 input/injection；显式 tags 不丢失。
2. risk tag 和单值 param 能命中对应知识卡。
3. WAF、空数据、过期 session 即使向量数达标也不能 `not_vulnerable`。
4. 真实 owner/peer/object 证据可正常闭合 IDOR 阴性。
5. 七态 round-trip、stats、session gate 全一致。
6. Direct init/observe/checkpoint、并发 observation、异内容重复 ID、越界 evidence、冲突归并、proof-valid 阳性翻案。
7. hot path/AGENTS/SKILL 生成一致性、doctor/self-check、compileall、全量 pytest、`git diff --check`。

发布前必须同时通过：定向测试、全量回归、self-check、doctor、regen check、CLI smoke、文档一致性。任一 proof/authority 安全回归失败不得 push。
