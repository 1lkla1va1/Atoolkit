# Atoolkit v8.8.0 迭代方案：跨 Run 事实闭环与交付真值

> 状态：独立审核驳回初稿后已完成重大修订，批准实施
> 版本定位：运行时与数据合同修复，不新增漏洞测试技巧
> 基线：v8.7.0 (`8eb705c`)
> 审核记录：`design/评估方案/v8.8修复方案独立审核_20260713.md`

## 1. 背景与问题定义

v8.7.0 已建立 root finding / impact / chain 证明合同，但更新后的两次实跑暴露出：代码中的合同、实际加载的规则、会话产物和项目级记忆并未形成同一条闭环。

已确认的失效模式：

1. 运行自称 v8.6.1，而仓库已是 v8.7.0；缺少可机械核对的版本和规则指纹。
2. Reporting collector 只扫描 `findings/finding_*/finding.json`；legacy 平铺或 `evidence/finding_*` 产物被完全忽略，并以 `0 accepted / 0 rejected / exit 0` 假成功。
3. Summary/Coverage 可由模型手写，与物理证据、finding 包、negative 和 Fact-Intent 状态不一致。
4. confirmed Fact 只导入 Graph，不会恢复已知阳性 Coverage，也不会阻止下一轮重测。
5. 动态发现的 endpoint 只存在当前 session `inventory.json`，未进入项目级 inventory。
6. 历史深阴性不会因后续 confirmed Fact 失效，可造成真漏洞再次被跳过。
7. Scheduler 会在新 endpoint 之前重新选择已完全闭合的高价值 endpoint。
8. Business Graph 重建时覆盖历史观测元数据；历史 Fact 也没有以可用摘要注入下轮 prompt。
9. 结构化 finding 没有项目级 root-cause 注册表，跨 Run 可重复计数。

v8.8.0 的目标不是“让每轮产出更多报告”，而是保证每轮只能产生三种可复用真值：

- 已确认且去重的 root finding；
- 有深度、可被新证据推翻的 negative；
- 带精确缺口和证据引用的 pending/deferred Intent。

## 2. 非目标

- 不增加 payload、绕过技巧、漏洞类别或自动利用能力。
- 不将旧 Skill Mode 报告自动升级为 accepted；legacy 只能进入迁移/复验队列。
- 不用 endpoint 级布尔值代替 param × vuln class 级 Coverage。
- 不保证多 Run 一定增加漏洞数，只保证新信息可累积、重复可识别、冲突可解决。
- 不在本版本引入数据库、远程服务或外部消息队列。

## 3. 核心不变式

### 3.1 运行指纹不变式

每个 Engine Mode session 必须在发起模型/网络前生成 `run_manifest.json`：

```json
{
  "schema_version": 1,
  "atoolkit_version": "8.8.0",
  "source_revision": "<git sha or unknown>",
  "mode": "engine",
  "project": "t_example",
  "session_id": "sess-...",
  "primary_target": "https://t.example/",
  "authorized_scopes": ["https://t.example/"],
  "authz_sha256": "...",
  "source_dirty": false,
  "source_tree_sha256": "...",
  "instruction_sources": [{"kind": "core_skill", "path": "...", "sha256": "...", "injected": true}],
  "reporting_schema_version": 2,
  "project_state_schema_version": 1,
  "created_at": "..."
}
```

- manifest 生成失败时不得开始运行。
- Summary 必须引用 manifest 路径与版本，不允许模型自由填写工具版本。
- Engine/Skill Mode 都必须生成 manifest；只记录实际读取/注入的 instruction source，不把“文件存在”写成“已加载”。
- 结束时生成 `run_receipt.json`，绑定 validation、summary、Coverage 和 project-state delta 的 SHA-256。
- 每轮动态 prompt 的 hash 记入 `events.jsonl`，不伪造成启动前 manifest 字段。
- provenance 不是密码学签名；manifest/receipt 不存 Cookie/Bearer 或原始 authz。
- 项目根 `AGENTS.md` 是 Atoolkit 的 Codex 权威文件；`codex/AGENTS.md` 是兼容副本。`doctor --json` 只读检测空/漂移 AGENTS 和 foreign `/src`，绝不覆盖其他独立项目。
- Skill Mode 必须先运行 `python3 -m engine.runtime_manifest init-manifest --mode skill --run-dir <run> ...`，并把权威 manifest 写入模型 writable session 之外的项目状态目录，才允许首个网络动作。session 仅保存引用/副本；receipt 必须绑定启动 manifest 的初始 digest。

### 3.2 Reporting fail-closed 不变式

Canonical finding 唯一可接受布局：

```text
<run>/findings/finding_<id>/finding.json
```

Collector 必须同时检测但不信任：

- `<run>/findings/finding_*.json`
- `<run>/evidence/finding_*/finding.json`
- `<run>/findings/` 和 `<run>/evidence/` 下最多两层的 suspicious `finding*.json`（不跟随 symlink）

规则：

1. legacy 文件必须作为 rejected migration candidate 列出，不得静默忽略。
2. 无 canonical finding 默认为 `empty_input`，exit 2；只有显式 `--allow-empty` 且 session gate/Coverage/candidate/ingestion 全部通过时可 exit 0。summary 不能授权空集成功。
3. CLI 退出码固定为 `0=成功 / 1=拒绝或冲突 / 2=空输入或前置缺失 / 3=IO/内部错误`。
4. CLI 输出必须包含 `discovered/canonical/legacy/accepted/rejected/ingestion_errors/warnings`。
5. 扫描只覆盖 canonical 和两种已知 legacy 布局，外加 `findings/`/`evidence/` 有界 suspicious scan；不跟随 symlink，不做无界 `**` 递归。
6. 同 ID 异内容是 `duplicate_id_conflict`；malformed/oversize/越界路径是 ingestion error。
7. 引入 `ValidationContext(primary_target, authorized_scopes, manifest)`：相对 `METHOD /path` 只绑 primary target，`--allow` 绝不作为 base；缺 manifest 时相对 target fail closed。Finding target/apis、proof packet 内 absolute URL/Host header、PoC/curl 中的 scheme/host/port/path-prefix 全部使用同一 context 校验，并校验 method/path 一致性。
8. 库级唯一入口 `validate_run_artifacts()` 由 orchestrator 和 CLI 共用；产出带 manifest/finding/evidence hash 的 `finding_validation.json`。
9. Summary/final report/benchmark 只是 validation artifact 的确定性投影；Coverage 只核对，不为 finding 补字段。

### 3.3 项目级真值不变式

新增单一权威 `project_state.json` schema `1`，由 `ProjectStateStore` 单 writer 以 lock + revision + temp/fsync/replace 原子提交。Blackboard、Business Graph、`surface_index`、`domains_covered` 和 Markdown summary 都是派生兼容视图，不得反向产生真值。

```json
{
  "schema_version": 1,
  "revision": 7,
  "project_scope": ["https://t.example/"],
  "merged_run_ids": [],
  "facts": [],
  "intents": [],
  "negatives": [],
  "dead_ends": [],
  "inventory": {"surfaces": {}, "unresolved": {}},
  "cell_registry": {},
  "finding_registry": {},
  "run_history": {}
}
```

唯一数据流：

```text
accepted structured root finding -> finding_registry -> Fact -> cell_registry
negative/dead-end -> cell_registry（仅在精确证据合同通过时）
inventory + facts + cells -> BusinessGraph/surface_index/domains/summary 派生视图
```

- 损坏 JSON、未知高版 schema、revision 冲突都 fail closed，不用空状态覆盖旧文件。
- 同 SID resume 更新同一 run_history 记录，不重复增加 run 数。
- 证据引用使用 `session:<sid>/<relative-path>` 或 `project:<relative-path>` + SHA-256；绝对机器路径不进去重键。

#### inventory

- key：`asset_id :: METHOD /normalized/path`
- value：`method/path/params/roles/risk_tags/sources/first_seen/last_seen/seen_in_runs`
- `asset_id` 包含 normalized scheme + host + effective port；项目 inventory、本轮 recon 和动态 discovery 按 key union。
- 不得用空字段或启发式默认值覆盖历史物理观测。
- 无 method 的发现进 `unresolved` 并生成 method-resolution Intent，不伪造 GET，不进 Coverage 终态。

#### cell_registry

- key：`asset_id :: METHOD /path :: param @ role_scope × normalized_vuln_class`
- status：`confirmed / not_vulnerable / not_applicable / open / stale`
- 包含 `evidence_refs/source_run/updated_at/depth/roles/objects/fingerprint`。
- `role_scope=unknown` 不是通配符，不得闭合其他角色格。
- confirmed 优先级高于同身份历史 negative；新 confirmed 必须将同 cell negative 标为 `superseded`。
- 后续普通 negative 不得覆盖 confirmed；只有显式 replay/refutation 或 target revision revalidation 可改变 `current_applicability`，并保留历史 finding。
- 下一轮从 registry 恢复精确 cell，不用 endpoint 级 `tested=true` 猜测。

#### finding_registry

- exact key：`asset + normalized vuln family + METHOD + templated path + risk param/location + access boundary/affected role`
- value：`canonical_finding_id/first_seen/last_seen/seen_in_runs/evidence_refs/facets/impact_claims/status`
- 对象 ID 只做 route templating；不同 endpoint/param 默认不自动合并，只有显式 `root_cause_ref` 或人工 merge 才跨位置聚合。
- 只有 accepted root finding 可进 registry；candidate/chain hypothesis 不得入册。
- `claim.profile`、自由文本 invariant/title 只用于 `possible_duplicate`，不作 v8.8.0 确定性主键。未来只有写入 schema 枚举且有独立归一测试后才可升为 discriminator。

### 3.4 调度不变式

启动顺序固定为：`load/validate project state -> union project inventory + current input -> plan role-aware cells -> restore registry truth -> compute run scope -> seed scheduled cells -> invoke model`。项目 inventory 非空时，后续 Run 可不重复提供 recon。

新的调度顺序：

1. 高优先级 pending Intent（其 `target_endpoint/params/roles/vuln_class` 真正进入 Tier 1）
2. 未完成的高价值业务流 cell
3. shallow negative 的精确缺口 cell
4. 新发现的 endpoint/cell
5. 其他未闭合 cell

已完全闭合 endpoint 不能进入 `must_test`；`must_test_cells` 只包含当前计划的 open role-aware cell。已全闭且无 actionable Intent 时直接生成 `no_work` Run，不调用模型。仅以下情形可重开：

- 存在新 param/role/vuln class；
- 有显式 revalidation Intent；
- 经验证的 `target_revision` 变化。

证据 SHA-256 只用于完整性校验，不存在“过期”语义；普通响应内容波动不触发重开。

### 3.5 Business Graph 合并不变式

- Business Graph 是 inventory + accepted facts 的可重建投影，不是真值源。`build_from_inventory()` 从“覆盖”改为“保守 merge”。
- 观测值优先于启发式值；只做集合 union，不丢 `last_fact_summary/observed_roles/objects/state_effects`。
- `confirmed_vuln_classes` 与 `domains` 分存，vuln class 不得污染业务域。本版仅修复现有域分类不一致，不重写完整域体系。
- 历史 confirmed Fact 的摘要、根因、证据引用和未完成 Intent 必须以有界上限注入 prompt，不只显示计数。

### 3.6 Intent 与 dead-end 不变式

- 抽取统一 `finalize_active_intent()`，在 matrix closed、budget、danger、needs-confirm、interrupt 和正常 conclude 所有路径执行。检查本轮新 Fact 的 `source_intent_id`；有新 Fact 即 completed，不得释放回 pending。
- 普通 budget skip/模型选择 skip/缺账号不是 dead-end。只有带精确 asset/method/path/param/role/class、枚举 reason code、refutation、可验 evidence ref 和 source run 的 `not_applicable` 可跨 Run 继承。
- `run_scope.json` 和最新 `run_summary.md` 可保留为 current view，但每轮还必须保存 `run_history/<sid>.json|md`。

Intent 状态迁移冻结为：

| 当前 | 事件 | 结果 |
|---|---|---|
| pending | 调度 | claimed |
| claimed | 产生 `source_intent_id` 匹配的 confirmed Fact | completed |
| claimed | NEED_INPUT/needs-confirm | blocked(`human_verification_needed`) |
| claimed | 有实际尝试但未达重试阈值 | pending（保留 attempts/outcome） |
| claimed | 达到重试阈值且无信号 | deferred(`no_observable_signal`) |
| claimed | danger/interrupt/系统异常且未完成尝试 | pending（记录 interruption，不增加 evidence attempt） |
| blocked | 所需账号/人工确认已提供 | pending |
| deferred | 到达 retry window 或显式人工重开 | pending |
| pending | 被更精确 Intent 替代/人工放弃 | superseded/abandoned |

项目提交后禁止存在 `claimed/in_progress`。resume 只恢复 pending/blocked/deferred 中满足重开条件的 Intent，不重复增加 attempts。

## 4. 实现工作包

- 结束阶段自动生成带 digest 的 `finding_validation.json`，不依赖模型记忆 CLI 命令。

### WP3：原子 Project State 和 inventory/cell registry

- 新建 `engine/project_state.py`，实现 schema 验证、legacy import、lock/revision/原子提交与派生兼容视图。
- 实现 asset/role-aware inventory/cell/dead-end/finding/run-history 合并。
- `_discover_and_register_endpoints()` 返回结构化 record；无 method 进 unresolved inventory，不伪造 GET。
- Run 启动时将项目 inventory 与本轮输入 union，结束时写回。
- 从 cell registry 恢复 confirmed/negative/not_applicable 精确格。

### WP4：Scheduler/Business Graph/Prompt 增量化

- Scheduler 以 open cell 而不是所有 Business Graph endpoint 生成 `must_test`。

- 统一域分类。
- Prompt 注入最多 N 个历史 Fact 和 M 个精确浅阴性缺口，限制字符长度。

### WP5：项目 finding registry 和历史摘要

- 仅从 normalized accepted root findings 构建保守 deterministic fingerprint。
- 路径 ID-like 段模板化，不同对象 ID 不新建 root。
- 保留每轮出现、facet 和 evidence refs。
- `run_history/<sid>.json|md` 从 project-state 中按 SID keyed 记录派生，本轮包含 new_root/known_root/duplicate_candidate/open_cell_delta/inventory_delta。

### WP6：文档、版本和自检

## 5. 迁移与兼容策略

1. 首次导入旧 Blackboard 时保留备份；损坏 JSON 或未知高版本 fail closed，禁止降级覆盖。
2. v1 Skill Mode Fact/negative 保持 `legacy_unvalidated`，不进 confirmed cell/finding registry。
3. v2 confirmed Fact 只在其 project-relative evidence ref 存在，且可绑定 accepted/proof-confirmed root validation 记录时恢复 positive；否则生成 revalidation Intent。
4. v2 `surface_index.tested=true` 不能生成精确 closed cell，只保留为 legacy coverage hint。
5. v2 negative 缺 asset/method/vuln/role/depth 时保持 open；合法空 param 与字段缺失必须区分。
6. v2 dead-end 默认不继承，除非满足 v8.8 精确证据合同。
7. `discovered_endpoints` 迁入 canonical inventory 或 unresolved inventory；旧 Business Graph 只做提示。
8. 旧 `run_summary.md` 不反向填充真值库。
9. 迁移为幂等：稳定 ID 由内容生成，不用 list length 或当前时间。

## 6. 验收测试

### 6.1 Reporting

- canonical valid finding + manifest：accepted=1, exit 0。
- canonical invalid finding：rejected=1, exit 1。
- 平铺 legacy JSON：legacy=1, rejected=1, exit 1。
- `evidence/finding_*/finding.json`：legacy=1, rejected=1, exit 1。
- summary 声称 12 finding 但无 canonical：ingestion_error, exit 1。
- 空目录：默认 exit 2；只有完整 session gate/Coverage 和显式 `--allow-empty` 可 exit 0。
- relative target + manifest primary target 通过；只有 `--allow`/缺 manifest/secondary scope 缺 `scope_ref` 时失败。
- proof packet/PoC 中 disguised host、越权 Host、错端口和 path-prefix escape 全部失败。
- duplicate ID conflict、symlink escape、malformed/oversize 失败。
- validation 后修改 finding/evidence 时 digest 失效。
- benchmark 忽略 legacy/candidate/chain hypothesis/impact facet。
- 手写 summary accepted 不能增加 benchmark 分数。

### 6.2 两轮/三轮项目行为

1. Run 1 动态发现 `POST /api/refund`，Run 2 无 recon 也能从项目 inventory 恢复。
2. 无 method 发现保留在 unresolved，不闭 Coverage；观测 method 后幂等提升。
3. Run 1 confirmed Fact，Run 2 同 asset/method/path/param/vuln/role cell 恢复 confirmed 并不进普通 must_test；另一 role 仍保持 open。
4. Run 1 深阴性、Run 2 confirmed，Run 3 不得再被 negative skip，且 negative 标 superseded；后续普通 negative 不覆盖 historical positive。
5. Run 1 已闭 endpoint + Run 2 新 endpoint，`surface_budget=1` 必须选新 endpoint。
6. 同 asset/method/templated path/param/vuln/role 的对象 ID 变体总数为 1；不同 method/param/role 不误合并。
7. 两个 asset 的同 path 不合并。
8. Business Graph 的 observed metadata 经过三轮 build 不丢失，vuln class 不进 domains。
9. 本轮 Intent 产生 Fact 后 conclude，Intent 为 completed，不是 pending/deferred。
10. generic SKIP 不进 dead-end；合同完整的 not_applicable 可在下轮恢复。
11. 项目已全闭且无 Intent 时 adapter 调用次数为 0。
12. 同 SID resume 不增加 run 计数；并发两轮不丢 inventory/cell/finding observation。
13. `run_history` 有三条独立记录，current summary 只是最新视图。

### 6.3 发布门

- `python3 -m pytest -q` 全通过。
- `python3 run.py --self-check` 全通过。
- `bash codex/regen_agents.sh --check` 通过。
- Engine 和 Skill Mode 均有 manifest，且 fake adapter 首次 `run()`/首个网络动作前 manifest 已存在。
- `engine.__version__ == SKILL.md version == manifest tool_version == CHANGELOG release`。
- dirty source tree 在 manifest 中可见；root AGENTS 漂移与 foreign `/src` collision 由 doctor 稳定报告。
- 新增的两轮/三轮行为测试不得用源码字符搜索替代。
- 不得修改或删除现有安全边界测试来获得通过。
- 工作树只包含 v8.8 有关文件。

## 7. 风险与回退

- **schema 扩展风险**：保留 v2 字段，为 v1/v2 提供幂等 normalize；不会破坏原文件。
- **错误 skip 风险**：仅精确 cell key + confirmed/depth-sufficient 可闭格；任何模糊迁移都保持 open。
- **prompt 膨胀风险**：Fact/Intent/浅阴性块设定数量和字符上限。
- **过度去重风险**：finding key 包含安全不变量和 role，不只依赖 endpoint/vuln 类名。
- **实现范围风险**：若 WP3/WP5 无法在一次迭代中完成行为验收，不得以“已写文档/已有字段”宣布完成；必须降版本范围或保持未发布状态。

## 8. 审核后批准决议

1. 不直接改造旧 Blackboard 为多真值容器；使用单 writer 的 `project_state.json` schema 1，旧 Blackboard 为兼容输入/派生视图。
2. confirmed finding 永久保留历史事实；只有显式 replay/target revision revalidation 改变当前适用性。
3. 自由文本 invariant 不进 exact fingerprint；仅稳定 claim profile/invariant code 可作 discriminator。
4. 未知 method 进 unresolved inventory 和 method-resolution Intent，不进 canonical cell。
5. Skill Mode v8.8 只做 provenance、fail-closed 与迁移提示，不自动改写 legacy finding。
6. v8.8.1 后再考虑普通 response fingerprint 自动过期、全域分类重写与跨端点自动 root merge。
