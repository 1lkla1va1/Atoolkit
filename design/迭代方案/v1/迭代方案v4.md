## P0 · 必做，直接决定能否稳定过 3000

### P0-1　落地未入库模块 + 强制 run.py 路径
- 把 `ledger.py / planner.py / session_gate.py / blocker.py / knowledge.py`（当前未跟踪）连同已改的 `orchestrator.py / run.py` 一起入库。
- `核心技能文件.v2.md` 改为：**正式覆盖跑必须经 `run.py`**（即报告所称 mode B）；`mode A`（skill 直跑）只允许用于单点 ad-hoc 验证，不得宣称项目级完成。
- 硬门：会话结束若无 `coverage-ledger.json`，终端状态只能 `incomplete`，不得输出 `complete`。`session_gate` 在 ledger 存在时已能判；需让 `run.py`/`_conclude` 在「无 ledger」这一支也拒绝 `complete`。
- 堵住的失分：报告根因 B1（形态 A 绕开权威台账）——覆盖漏测 2550 分的大头。

### P0-2　recon → surface 自动 bootstrap（唯一新模块）
- 新增 `engine/surface.py`：解析 recon 产物，输出 `endpoint/method/param` 清单：
  - **JS**：抓 `fetch(/axios/get/post/XMLHttpRequest` 调用 + 字符串字面量里的 `/api/*`、`*.php` 路径。
  - **HTML**：`<form action>`、`<a href>`、`<script src>`、`<img/data-* src>`。
  - **HTTP 响应 JSON**：嵌套对象里出现的路径片段。
  - **`*.har`**（若有）：请求行 method+path+query。
- `run.py` 增 `--recon-dir <path>`：自动跑 `surface.py` → `planner.plan_surfaces` → `CoverageLedger.from_endpoints`。**无 `--endpoints` 且无 `--recon-dir` 时，拒绝启动正式覆盖跑**（退化 ad-hoc 需显式 `--ad-hoc`）。
- 复用 `planner.extract_params / infer_risk_tags / infer_roles`，使 bootstrap 直出 ledger schema，不另造粒度。
- **通用性红线**：不内置任何靶场路径模式、不内置 payload 列表；风险标签只按参数名语义推断（`planner.PARAM_RISK_RULES` 已是此范式）。
- 堵住的失分：报告根因 A1/B2（recon 资产没自动进端点级台账）——即 2550 分中「信息已在手却没转任务」的 72%。

## P1 · 阴性质量与覆盖诚实

### P1-1　session_gate 显式 `shallow_negative` 谓词
- 现状：`shallow_negative` 只在两种情况间接拦——高价值格经 `LEGACY_STATUS_MAP→not_tested` 命中 `high_value_not_tested`；其余靠迁移代码注入 `next_action` 间接触发 `open_next_actions`。**一个无 next_action、非高价值的 shallow_negative 格会从 `session_gate` 漏过**（`orchestrator._conclude` 用 `open_risk_cells` 部分兜底，但纵深不足）。
- 改：新增显式谓词 `shallow_negative_open`——任一 surface 状态为 `shallow_negative`，或其阴性证据未过 `knowledge.negative_sufficient`（向量数/响应数不足）→ `result=INCOMPLETE`，`reason="shallow_negative_open"`。
- 堵住的失分：报告根因 A3（阴性闭格门槛不足）——判错 560 分。

### P1-2　skill 文件强制逐格 `negative_*.md`
- 现状：通道已建（`_parse_negative` 读 frontmatter、`harvest_evidence` 收 `negative_*.md`、`knowledge.negative_sufficient` 判充分性），但本轮实跑用的是单个合并 `negative_findings.md`（自由散文、无 `endpoint:/vuln:` 头）→ 被 ingest 却贡献为空。
- 改 `核心技能文件.v2.md`：**每条阴性 = 独立 `negative_<surface_id>.md`**，frontmatter 含 `endpoint/method/param/vuln/roles/identities/vectors/evidence_types/next_actions`，正文带关键请求/响应片段。
- 证据不足时只能写 `shallow_negative`、留 `next_actions`，**严禁** `not_vulnerable`。
- 经 knowledge cards 抬阈值（已支持「可抬不可降」）：
  - 注入类 ≥3 个 payload 家族；
  - SSRF ≥ `{0.0.0.0, 127.0.0.1, DNS-rebinding, @-前缀, 多协议(file://,gopher://)}`；
  - IDOR/越权 ≥ 2 角色 × 2 对象 + 响应差异。
- 堵住的失分：判错 560 分 + 向量窄 200 分。

### P1-3　accepted 数与覆盖完成度分开汇报
- `_conclude` 已调 `evaluate_session_gate`；确保终端摘要同时打印：`accepted_count`、（有 oracle 时）`hit_count`、ledger `{total/closed/high_value_open}`、`shallow_negative_count`、`open_next_actions_count`、未闭合高价值面清单。
- 堵住的失分：报告根因 B4（Guardian 域错位，质量掩盖覆盖）——成本低，从 P2 提到 P1。

## P2 · 复利项，防复发

### P2-1　误判复盘库 → knowledge cards
- `knowledge.py` 已 `load_cards()` 且 `match_cards/negative_sufficient` 用 cards **抬**判据。把报告点名的 7 类误判模式沉淀成 cards：单参数 / 单角色 / 单对象 / 单 payload 家族 / 单位误读 / 无回显≠无结果 / 测过端点≠测过参数。
- 这些 cards 直接拉高 `negative_sufficient` 门槛 → 自动反哺 P1-1/P1-2，无需额外接线。

### P2-2　端到端回归（把 benchmark_eval 从离线工具变自检门）
- `benchmark_eval.py` 现为「离线回归、运行时不导入」。给 `run.py` 加 `--self-check`：在固定 fixture（recon 目录 + oracle）上断言——surface bootstrap 覆盖 oracle 全部端点、故意浅阴性触发 `session_gate=INCOMPLETE`、合格报告被 Guardian `accepted`。
- 这样「写了但被绕开」的失败无法再静默复现。

## 验证标准（怎么知道迭代生效）
- 同一 target 用 `run.py --recon-dir` 重跑：产出 `coverage-ledger.json` 含 N 个 surface（N = recon 可发现端点数），conclude 时高价值 `not_tested` 为 0，每条阴性要么 `not_vulnerable`（达阈值）要么 `shallow_negative`。
- `benchmark_eval` 覆盖召回 ≥ 本轮漏测的 13 个端点（2550 分桶）。
- 任意「判错」项被浅测时，`session_gate` 返回 `INCOMPLETE`。
