# 640 覆盖归因修复方案

> 来源：`design/评估报告_640分覆盖归因_20260630.md`
> 目标：把技能包从“高价值漏洞追踪助手”修到“可评估、可回归、可量化覆盖的通用 SRC 工作流”。
> 原则：靶场只作为回归集；oracle 只参与离线评估，不进入攻击 prompt；所有模式都必须吃到覆盖纪律与完整性门。

---

## 1. 修复目标

本次修复不追求给某个靶场“抄答案”，而是修掉三类通用失效：

1. **执行路径错位**
   - 现状：`engine/orchestrator.py` 有覆盖矩阵和知识卡注入，但模式 A 只读 `skill/核心技能文件.v2.md`，没有外壳，导致矩阵、知识卡、Session Gate 全缺勤。
   - 目标：模式 A 和模式 B 都必须具备最低覆盖纪律。能接 engine 时默认接 engine；不能接时，skill 文件本身也要有轻量 ledger 和参数/角色纪律。

2. **覆盖粒度过粗**
   - 现状：engine 矩阵是 `endpoint × vuln_class`，缺 `method / param / role`，会漏参数篡改、认证流程、角色对越权等。
   - 目标：升级为 `endpoint × method × param × role × risk_tag` 的 coverage-ledger，并把负向证据、阻塞、待补测任务全部落盘。

3. **缺完整性门与回归量化**
   - 现状：Guardian 只管单篇报告质量，不能阻止整场作战在高价值格未测、可恢复阻塞未处理、证据不一致时强行收口；也没有 benchmark-eval 衡量迭代是否变好。
   - 目标：新增 `session-gate` 和 `benchmark-eval`，分别管收口完整性与离线回归评分。

---

## 2. 总体架构

修复后的链路分三层：

| 层 | 组件 | 责任 | 落点 |
|---|---|---|---|
| 共同约束层 | 核心技能文件 + AGENTS | 模式 A/B 都能读到的覆盖纪律、参数语义、认证绕过、阻塞恢复、去重规则 | `skill/核心技能文件.v2.md`、`codex/AGENTS.md` |
| 作战外壳层 | coverage-ledger / session-gate / blocker-resolver / planners | 维护覆盖台账、注入知识卡、阻止伪完成、生成下一步行动 | `engine/ledger.py`、`engine/session_gate.py`、`engine/blocker.py`、`engine/planner.py`、`engine/orchestrator.py` |
| 评估回归层 | benchmark-eval | 把任意靶场 oracle 与本次 coverage/findings/evidence 对齐，输出得分与漏测归因 | `engine/benchmark_eval.py` |

关键设计取舍：

- `Guardian` 继续只管“单篇发现是否可报”。
- `Session Gate` 管“整场作战是否可以宣称完成”。
- `coverage-ledger` 是权威台账，`summary.json / coverage.json / module_queue.json` 都从它派生，禁止再由临时脚本硬编码。
- 知识卡仍可保留为外壳注入源，但参数语义、认证绕过、role-pair 的最低规则必须下沉到 skill 文件，避免模式 A 再次收不到卡。

## 3. P0 修复

### P0-1. 核心技能文件补“覆盖完整性硬纪律”

修改 `skill/核心技能文件.v2.md`，新增一节“覆盖完整性”，同步重生成 `codex/AGENTS.md`。

必须写入的规则：

- 每个发现的攻击面都要拆成 `endpoint / method / param / role / risk_tag`。
- 高价值格不得停留在 `not_tested`。
- `blocked` 必须区分可恢复和不可恢复；可恢复阻塞不得进入最终盲区。
- `LOW_ROI` 只能用于覆盖充分后的无发现；未闭合高价值格只能输出 `incomplete` 或 `NEED_INPUT`。
- 同一端点同一根因只算一个发现，多个表现写成 facet。
- 无论是否已有登录态，注册、登录、找回、验证码、审核、锁定/解锁、用户枚举都要作为一等认证面覆盖。

验收：

- `bash codex/regen_agents.sh` 后 `codex/AGENTS.md` 包含同等约束。
- 模式 A 不接 engine 时，也能在 prompt 中看到 ledger、参数语义、认证绕过、阻塞恢复规则。

### P0-2. coverage-ledger 升级为权威台账

新增 `engine/ledger.py`，把当前 `CognitiveState.matrix` 中的 endpoint×class 逻辑抽成独立组件，并升级 schema。

建议 schema：

```json
{
  "surfaces": [
    {
      "surface_id": "POST /api/user/refund.php refund_amount [user]",
      "endpoint": "/api/user/refund.php",
      "method": "POST",
      "param": "refund_amount",
      "roles": ["user"],
      "risk_tags": ["payment", "amount-tamper"],
      "feature": "refund",
      "status": "not_tested",
      "evidence_ref": null,
      "blocker": null,
      "next_actions": [],
      "source": "js|html|traffic|manual"
    }
  ]
}
```

状态枚举：

- `not_tested`
- `confirmed`
- `not_vulnerable`
- `blocked`
- `not_applicable`

兼容策略：

- `positive` 迁移为 `confirmed`
- `negative_with_evidence` 迁移为 `not_vulnerable`
- `shallow_negative` 迁移为 `not_tested`，并保留 `next_actions`
- `skipped` 迁移为 `not_applicable`
- 旧 `state.json` 可读，保存时写新 `coverage-ledger.json`

验收：

- `run.py --dry-run --endpoints ...` 生成 `runs/<sid>/coverage-ledger.json`。
- `state.json` 仅保存会话认知快照，coverage 明细以 ledger 为准。
- 后续 `summary.json / coverage.json / module_queue.json` 均可由 ledger 派生。

### P0-3. session-gate 阻止强行收口

新增 `engine/session_gate.py`，在 `_conclude()` 中调用，与 `engine/enforce.py` 的 Guardian 并列。

Gate 谓词：

1. 存在高价值 surface 仍为 `not_tested`。
2. 存在可恢复 `blocked`。
3. 存在 `NEEDS` 或 `next_actions` 未转成明确补测任务。
4. 若传入 oracle/eval hint，存在 oracle 命中项无测试记录。
5. 存在 `confirmed` 声明与证据文件不一致。

输出：

```json
{
  "result": "pass|incomplete|needs_input|error",
  "reasons": [
    {
      "predicate": "recoverable_blocked",
      "surface_id": "POST /api/lottery draw [user]",
      "action": "create_new_object_or_reset_quota"
    }
  ]
}
```

收口规则：

- Guardian accepted 报告存在，只能说明“有真洞”，不能说明“整场完成”。
- Session Gate 不通过时，最终状态必须是 `incomplete` 或 `needs_input`。
- 报告标题和 summary 必须标记“部分覆盖”，并列出未闭格清单。

验收：

- 构造 `module_queue=[]` 但 ledger 仍有 `not_tested` 的用例，结果必须为 `incomplete`。
- 构造 evidence 中 `confirmed:false` 但 summary 写 confirmed 的用例，结果必须为 `error` 或 `incomplete`，并列出不一致文件。

### P0-4. benchmark-eval 离线评估

新增 `engine/benchmark_eval.py`，只用于回归评分，不参与攻击决策。

命令：

```bash
python3 -m engine.benchmark_eval \
  --oracle oracle.json \
  --summary runs/<sid>/summary.json \
  --coverage runs/<sid>/coverage-ledger.json
```

输入适配：

- JSON
- YAML
- CSV
- PHP `return [...]` 或 `array(...)`

统一 oracle 字段：

```json
{
  "id": "case-001",
  "endpoint": "/api/user/order-detail.php",
  "method": "GET",
  "params": ["order_no"],
  "class": "idor",
  "score": 160
}
```

输出：

- 命中项
- 漏测项
- 重复命中项
- 非 oracle 真实发现
- 总分
- 分类覆盖率
- endpoint/param/role 覆盖率

验收：

- 能复算本报告中的 640 分。
- oracle 内容不会出现在 `assemble_prompt()` 或模型输入中。
- 未命中项能映射到 coverage-ledger 中的 `not_tested / blocked / not_vulnerable` 状态。

### P0-5. 模式 A 接上 engine 或具备最小本地纪律

推荐双轨：

- 默认路径：`codex/AGENTS.md` 明确建议用 `python3 run.py` 起会话，让模式 A 通过入口进入 orchestrator。
- 退化路径：当用户明确只用纯 Codex/skill 时，skill 文件要求模型维护 `coverage-ledger` 片段，并按参数语义/认证绕过/role-pair 规则自查。

验收：

- `codex/AGENTS.md` 不再只说“外壳会注入知识卡”，还要说明“若当前会话没有外壳，按本文件内置的轻量 ledger 执行”。
- `run.py --dry-run` 的输出中能看到 coverage-ledger、session-gate 状态。

## 4. P1 修复

### P1-1. 参数语义 playbook 下沉

修改 `skill/核心技能文件.v2.md`，把参数语义路由写进决策树；同时在 `engine/planner.py` 中实现确定性打标。

路由规则：

| 参数语义 | risk_tags |
|---|---|
| `amount/refund_amount/use_points/price/fee/balance/stock` | `amount-tamper`, `accounting` |
| `order_time/timestamp/create_time` | `time-tamper` |
| `order_no/product_no/user_id/user_hash/id` | `object-ownership`, `idor` |
| `redirect/return_url/callback_url` | `redirect-chain`, `callback` |
| `image_url/url/fetch` | `ssrf` |
| `filename/category/file/upload` | `file-upload`, `path-traversal` |
| `keyword/search/sort/orderBy/filter` | `input-validation`, `injection` |
| `status/discount/role/state` | `enum-tamper`, `privilege` |

验收：

- 输入含 `refund_amount` 的 endpoint，ledger 自动生成 `amount-tamper` 格。
- 输入含 `search/sort/orderBy` 的 endpoint，ledger 自动生成 `input-validation` 格。

### P1-2. blocker-resolver

新增 `engine/blocker.py`。

阻塞类型：

| blocker | 是否可恢复 | 默认行动 |
|---|---|---|
| `quota_exhausted` | 是 | 重置数据、换账号、创建新对象 |
| `object_absent` | 是 | owner 账号创建对象 |
| `missing_role` | 是/需人 | 注册新号或请求角色账号 |
| `captcha_required` | 否/需人 | 输出 NEED_INPUT |
| `out_of_scope` | 否 | 停止，不绕过 |

验收：

- `lottery_remaining=0` 这类阻塞不能直接进入最终盲区，必须生成 `next_actions`。
- `captcha_required` 输出 `needs_input`，不尝试绕过人工验证边界。

### P1-3. role-pair / object-pair planner

在 `engine/planner.py` 中新增对象型端点规划器。

识别对象型参数：

- `id`
- `uid`
- `user_id`
- `order_id`
- `product_id`
- `merchant_id`
- `address_id`
- `hash`
- `*_no`

生成三步任务：

1. owner 创建或枚举对象。
2. attacker 读取、修改、删除或操作该对象。
3. owner/victim 回查业务影响。

验收：

- 对 `/api/user/order-detail.php?order_no=...` 生成 `owner:user -> attacker:user` 的读格。
- 对商户商品/订单类端点生成 `merchant_a -> merchant_b` 的读/改格。
- 对 admin/user-list 这类高权限面生成 `anonymous/user/merchant/admin` 角色差异格。

### P1-4. 认证/验证绕过一等分支

改 skill 决策树，认证面不再挂在“无登录态”下。

必须覆盖：

- 注册
- 登录
- 找回/重置
- captcha/sms/verify-code
- 审核状态
- 锁定/解锁
- 用户枚举
- token/session/role switch

验收：

- endpoint 中命中 `register/login/password/captcha/sms/admin/token/session` 时，即使已有登录态，也生成 `auth-flow` 风险格。
- `auth-flow-abuse` 知识卡能被 ledger 命中并注入。

## 5. P2 修复

### P2-1. Discovery 饱和检查点

新增 discovery 阶段输出 `runs/<sid>/inventory.json`，并在 ledger 初始化前冻结第一版攻击面。

来源：

- HTML 页面链接和表单
- JS 静态抽取
- 已登录流量
- 已知入口人工补充
- 后续深测中新发现 endpoint

饱和规则：

- 连续 N 个来源不再新增 endpoint，才进入深测。
- 深测中新发现 endpoint 必须追加进 ledger，并回到 planner 补风险格。
- `api_inventory.json` 不能再只是草稿；它必须能解释每条 endpoint 的 source 和 last_seen。

验收：

- 报告中用到的 endpoint 必须存在于 inventory 或有 `discovered_during_testing` 记录。
- inventory 与实测证据不一致时，session-gate 标 incomplete。

### P2-2. 按根因去重与价值优先

新增发现聚合规则：

```json
{
  "finding_key": "endpoint + root_cause + affected_role",
  "facets": ["unpaid_recharge", "race_replay"],
  "primary_impact": "accounting_tamper"
}
```

价值排序只影响测试建议，不影响漏洞是否成立：

1. 认证/验证绕过
2. 支付/余额/退款/积分
3. 对象级授权
4. 输入验证/文件/跳转
5. 低价值信息暴露

验收：

- 同一充值回调根因不会拆成两个独立 critical 计数。
- session summary 同时保留 facet，避免信息丢失。

## 6. 实施顺序

### 第 1 批：让修复一定生效

1. 修改 `skill/核心技能文件.v2.md`，加入覆盖完整性、参数语义、认证绕过、阻塞恢复、去重规则。
2. 运行 `bash codex/regen_agents.sh`。
3. 新增 `engine/ledger.py`，并让 `engine/orchestrator.py` 从 `CognitiveState.matrix` 迁移到 ledger。
4. 新增 `engine/session_gate.py`，在 `_conclude()` 中调用。

### 第 2 批：让覆盖变宽

1. 新增 `engine/planner.py`，实现参数语义 planner 与 role-pair planner。
2. 新增 `engine/blocker.py`，让可恢复阻塞转为 `next_actions`。
3. 更新 `engine/knowledge.py` 的匹配输入，使知识卡可匹配 `method / param / role / risk_tags`。

### 第 3 批：让效果可量化

1. 新增 `engine/benchmark_eval.py`。
2. 准备 `test_Assessment_score/` 下的 oracle fixture。
3. 增加 dry-run 与离线评估回归命令。

## 7. 测试与验收清单

### 单元测试

- ledger migration：旧 `state.json` 能迁移到新 `coverage-ledger.json`。
- surface expansion：endpoint+params 能生成正确 risk_tags。
- session-gate：5 条谓词分别触发正确状态。
- blocker-resolver：5 类 blocker 输出正确 action/status。
- benchmark-eval：oracle 与 findings/ledger 能正确匹配。

### 集成测试

```bash
python3 run.py --dry-run \
  --target https://t.example \
  --authz demo \
  --endpoints fixtures/endpoints.txt \
  --max-turns 3
```

期望：

- 生成 `coverage-ledger.json`。
- 未闭合高价值格时，终态为 `incomplete`。
- 有 accepted 报告但 ledger 未闭合时，不输出“完整完成”。

### 回归测试

```bash
python3 -m engine.benchmark_eval \
  --oracle test_Assessment_score/localhost-shop/oracle.json \
  --summary pentest-data/localhost-shop/summary.json \
  --coverage pentest-data/localhost-shop/coverage-ledger.json
```

期望：

- 可复算 640。
- 漏测项按 `not_tested / blocked / not_vulnerable / no_surface` 归因。
- 重复命中项能识别同根因多 facet。

---

## 8. 非目标

- 不把天积商城路径、账号、分值写入核心 skill 或 planner。
- 不把 oracle 注入模型 prompt。
- 不用靶场评分规则替代漏洞报告标准。
- 不降低 Guardian 标准来换覆盖分。
- 不把所有知识卡一次性塞进 prompt；仍保持按 surface/risk 匹配注入。

---

## 9. 完成定义

本修复完成时，至少满足：

1. 模式 A 纯 skill 路径不再只依赖“外壳会注入”的假设。
2. 模式 B 运行产物中有权威 `coverage-ledger.json`。
3. `session-gate` 能阻止高价值未测、可恢复阻塞、证据不一致情况下的“完整收口”。
4. 参数语义、认证绕过、role-pair 三类漏测能被 planner 系统生成待测格。
5. `benchmark-eval` 能对任意 oracle 做离线评分，且不影响攻击决策。
6. 天积商城只作为回归集，用来量化覆盖是否提升，而不是成为代码里的特殊分支。
