# 迭代方案 v5.1 · 结构化 finding 闭环与综合报告落地补强

> 一句话结论：**v5 方向正确，但 MVP 需要从“渲染器可演示”升级为“外壳可识别、可校验、可裁决、可交付”的完整闭环。v5.1 不改变 v5 的目标，专门补清递归采集、字段映射、Guardian/Verify 适配、session-gate 关系和自检验收。**

---

## 0. v5.1 修正范围

### 0.1 继承 v5 的不变目标

- 新增 `project-memory/`，减少新会话重复考古。
- 漏洞确立时沉淀 `runs/<sid>/findings/finding_<id>/finding.json`。
- 由结构化 finding 渲染稳定的 `runs/<sid>/final_report.md`。
- 兼容旧 `report_*.md`，但新链路以 `finding.json` 为权威。
- 加密链路、源码/JS 构造数据包、Burp 手工复现进入标准证据结构。

### 0.2 v5 审核发现的必须补清项

| 编号 | 缺口 | 影响 | v5.1 修正 |
|---|---|---|---|
| G1 | `findings/finding_*/` 是子目录，但现有 `harvest_evidence()` / `count_evidence_files()` 只看根目录文件 | finding 包写了也不被外壳识别，进展计数也不刷新 | 增加递归采集与递归证据计数规则 |
| G2 | v5 MVP 只证明 schema/render，不接 `_conclude()` / `run.py` | 不能证明项目真实流程可用 | MVP 升级为 CLI 闭环 |
| G3 | Guardian/Verify 当前只吃 Markdown | structured finding 可能绕过报告质量门或无法复验 | 增加 finding Guardian 和 finding PoC 解析 helper |
| G4 | finding schema 与 `summary.json` / `benchmark_eval` 字段未映射 | oracle 命中、报告聚合、旧评估链路可能断 | 定义 normalized finding 输出字段 |
| G5 | `final_report.md` 与 session-gate 未通过时的关系不清 | 可能把未完成测试误当最终交付 | 引入 final report 状态策略 |
| G6 | `skill/核心技能文件.v2.md` 仍要求直接写 Markdown 报告 | 模型不会稳定产出 finding 包 | 明确协议迁移和 `codex/AGENTS.md` 同步 |

## 1. v5.1 总目标

v5.1 的完成标准不是“能从一份 fixture 渲染 Markdown”，而是：

```text
模型/fixture 产出 finding 包
  -> 外壳递归采集 finding 包和证据文件
  -> validate_finding() 拒收缺证据 finding
  -> guardian_check_finding() 复用质量门
  -> verify helper 能从 finding PoC/request 抽请求
  -> accepted findings 映射到 summary/coverage/benchmark 兼容结构
  -> render_final_report()
  -> run.py 写 summary.json.final_report_path 并打印路径
  -> --self-check 覆盖 reporting 断言
```

核心原则：

> **结构化 finding 不是额外产物，而是 v5.1 后漏洞确立、报告渲染、交付汇总的权威输入。**

## 4. finding schema v1

### 4.1 必填字段

```json
{
  "schema_version": 1,
  "id": "finding_001",
  "title": "订单详情接口存在水平越权漏洞",
  "severity": "P1",
  "vuln_type": "越权/IDOR",
  "target": "https://t.example",
  "risk": {
    "summary": "攻击者可读取其他用户订单详情",
    "proven_impact": "实测 B 账号读取到 A 账号订单收货地址与金额"
  },
  "recommendation": {
    "summary": "服务端按当前登录身份校验订单归属",
    "details": [
      "查询订单详情时绑定 current_user_id 与 order_id"
    ]
  },
  "feature_point": {
    "module": "订单模块",
    "function": "订单详情查询",
    "trigger": "进入订单详情页时触发",
    "trigger_api": "GET /api/order/detail",
    "vulnerable_param": "order_id",
    "statement": "订单模块 -> 订单详情查询，触发 GET /api/order/detail 接口，order_id 参数存在水平越权漏洞。"
  },
  "apis": [
    {
      "method": "GET",
      "path": "/api/order/detail",
      "purpose": "根据订单 ID 查询订单详情",
      "risk_params": ["order_id"],
      "params": [
        {
          "name": "order_id",
          "location": "query",
          "risk": "服务端未校验订单归属"
        }
      ]
    }
  ],
  "proof_packets": [
    {
      "name": "attacker_read_victim_order",
      "request_file": "request_1.http",
      "response_file": "response_1.http",
      "evidence_summary": "响应中返回受害者订单地址与金额"
    }
  ],
  "manual_burp_replay": [
    "登录攻击者账号并进入订单详情页面",
    "使用 Burp Suite 捕获 GET /api/order/detail 请求",
    "将 order_id 修改为受害者订单 ID",
    "重放请求后响应中返回受害者订单详情"
  ],
  "poc": {
    "type": "curl",
    "file": "poc.sh",
    "description": "替换 Cookie 与 order_id 后可复现越权读取"
  },
  "source_proof": null,
  "crypto_chain": null
}
```

### 4.2 非功能点来源字段

当漏洞不是从页面功能点触发，而是从 JS/源码/配置构造数据包时，`feature_point` 可为空，但 `source_proof` 必填：

```json
{
  "type": "frontend_js",
  "file": "static/js/app.xxx.js",
  "line": 1288,
  "function": "submitOrder",
  "evidence": "该函数拼接 /api/order/detail 请求并传入 order_id",
  "constructed_packet_file": "request_1.http",
  "risk_param": "order_id"
}
```

校验规则：

- `file` 必填。
- `line` 或 `function` 至少一个必填。
- `constructed_packet_file` 必须存在。
- `risk_param` 必须能在 `apis[].risk_params` 或 proof packet 中解释。

### 4.3 加密链路字段

有加密时，`crypto_chain` 必填：

```json
{
  "scope": "POST JSON 整体加密",
  "algorithm": "AES-CBC",
  "discovered_at": {
    "file": "static/js/crypto.js",
    "line": 42,
    "function": "encryptPayload"
  },
  "key_source": "登录初始化接口返回字段 cryptoKey",
  "iv_source": "前端常量 DEFAULT_IV",
  "decrypt_method": "使用 decrypt.py 解密 Burp 捕获的请求体",
  "reencrypt_method": "修改明文参数后使用 encrypt.py 重新加密",
  "helper_files": [
    "crypto_notes.md",
    "decrypt.py",
    "encrypt.py",
    "burp_crypto_helper.py"
  ],
  "security_statement": "解密后 JSON 中的 order_id 参数未校验对象归属"
}
```

校验规则：

- 必须包含 `algorithm`、`discovered_at.file`、`key_source`、`iv_source`。
- `helper_files` 中每个文件必须存在。
- 报告正文只写原理链条和文件路径，不粘长代码。

## 5. normalized finding 映射

为兼容现有 `summary.json`、`benchmark_eval` 和根因聚合，`collect.py` 输出统一结构：

```json
{
  "id": "finding_001",
  "title": "订单详情接口存在水平越权漏洞",
  "severity": "P1",
  "class": "越权/IDOR",
  "vuln_class": "越权/IDOR",
  "target": "https://t.example",
  "endpoint": "/api/order/detail",
  "endpoints": ["/api/order/detail"],
  "method": "GET",
  "methods": ["GET"],
  "params": ["order_id"],
  "roles": ["unknown"],
  "evidence_file": "findings/finding_001/finding.json",
  "proof_files": [
    "findings/finding_001/request_1.http",
    "findings/finding_001/response_1.http",
    "findings/finding_001/poc.sh"
  ],
  "root_cause": "越权/IDOR",
  "affected_role": "unknown",
  "primary_impact": "实测 B 账号读取到 A 账号订单收货地址与金额",
  "raw_finding_path": "findings/finding_001/finding.json"
}
```

映射规则：

- `class/vuln_class/root_cause` 来自 `vuln_type`。
- `endpoint` 取第一条 API 的 `path`。
- `endpoints` 来自 `apis[].path`。
- `method/methods` 来自 `apis[].method`。
- `params` 来自 `apis[].risk_params` 和 `apis[].params[].name` 去重。
- `evidence_file` 指向 `finding.json` 相对 run 目录路径。
- `proof_files` 汇总 PoC、request、response、crypto/source helper。
- `roles` 暂无字段时为 `["unknown"]`，后续可加 `affected_roles`。

## 6. reporting 模块职责

### 6.1 schema.py

职责：

- `load_finding(path) -> dict`
- `resolve_finding_file(finding_dir, ref, run_dir) -> Path`
- `normalize_finding(finding, finding_path, run_dir) -> dict`

约束：

- 使用标准库，不引入 pydantic 等重依赖。
- 读 JSON 失败要返回清晰错误。
- 路径解析必须阻止 `../` 逃逸到 `runs/<sid>/` 外。

### 6.2 validate.py

职责：

- `validate_finding(finding, finding_path, run_dir, authorized_hosts=None) -> ValidationResult`
- `validate_findings(items, run_dir, authorized_hosts=None) -> {"accepted": [], "rejected": []}`

硬规则：

- `severity` 必须是 `P1/P2/P3`。
- `title/vuln_type/target` 必填。
- `target` 必须在授权 host 内。
- `risk.proven_impact` 必填，且不得是“可能/疑似/理论上”。
- `recommendation.summary` 或 `recommendation.details` 至少一项存在。
- `feature_point` 或 `source_proof` 至少一项存在。
- 至少一条 API，且每条 API 必须有 `method/path/purpose/risk_params`。
- 至少一份 `proof_packets`，每份必须有存在的 request/response 文件。
- 必须有 `manual_burp_replay`，且至少 2 步。
- 必须有 `poc.file`，文件存在且能抽出 curl 或 raw HTTP。
- 有 `crypto_chain` 时按 4.3 校验。
- 有 `source_proof` 时按 4.2 校验。

输出示例：

```json
{
  "accepted": [
    {
      "id": "finding_001",
      "path": "runs/sid/findings/finding_001/finding.json"
    }
  ],
  "rejected": [
    {
      "id": "finding_002",
      "path": "runs/sid/findings/finding_002/finding.json",
      "reasons": ["missing proof_packets[0].response_file"]
    }
  ]
}
```

### 6.3 collect.py

职责：

- 递归查找 `runs/<sid>/findings/finding_*/finding.json`。
- 加载、校验、归一化 finding。
- 可选兼容 legacy `report_*.md`。

接口：

```python
def collect_structured_findings(run_dir, authorized_hosts=None) -> dict:
    return {
        "accepted": [],
        "rejected": [],
        "normalized": []
    }
```

### 6.4 render_md.py

职责：

- `render_final_report(findings, output_path, target_name) -> Path`
- 固定标题结构：

```md
# xxx 授权安全测试报告

## 1. 漏洞名称：xxx

### 安全风险
### 安全建议
### 漏洞证明
```

渲染规则：

- 没有加密时不输出“加密链路说明”。
- 没有源码/前端证据时不输出“源码或前端证据”。
- 多 API 全部进入 API 表格。
- 数据包过长时只摘关键片段，并引用证据文件路径。
- PoC 必须说明执行方式或 Burp 复测方式。
- 不得输出 rejected finding。

## 7. orchestrator 接线

### 7.1 递归证据计数

现有 `count_evidence_files(workdir)` 只统计 run 根目录普通文件。v5.1 改为：

- 统计根目录证据文件。
- 统计 `findings/finding_*/` 下的证据文件。
- 排除 `state.json/authz.md/cookies.txt/events.jsonl/summary.json/final_report.md/coverage-ledger.json/inventory.json`。
- 大文件不读内容，只计数。

目标：模型写入 finding 包时，本轮必须被识别为有进展。

### 7.2 递归 harvest_evidence

`harvest_evidence(workdir)` 返回结构增加：

```json
{
  "reports": [],
  "report_objs": [],
  "negatives": [],
  "files": [],
  "finding_objs": [],
  "finding_validation": {
    "accepted": [],
    "rejected": []
  },
  "normalized_findings": []
}
```

规则：

- legacy `report_*.md` 仍只从根目录采集。
- `negative_*.md` 仍只从根目录采集，避免误吃 finding 内说明文档。
- structured finding 只从 `findings/finding_*/finding.json` 采集。
- `files` 必须包含 finding 目录下的 request/response/poc/helper 文件，用于状态回灌和审计。

### 7.3 CognitiveState.update

短期策略：

- legacy `report_*.md` 继续按现有 `_report_cell()` 闭格。
- structured finding 进入 `normalized_findings` 后，也要按 `endpoint + vuln_type` 闭格。
- finding 闭格必须要求 validation accepted。
- rejected finding 只记录事件，不闭格。

### 7.4 _conclude

`_conclude()` 调整顺序：

```text
1. triage legacy reports
2. collect/validate structured findings
3. guardian_check_finding accepted structured findings
4. verify structured finding PoC（可选）
5. 合并 accepted legacy reports 与 accepted structured findings
6. aggregate / normalize summary findings
7. session_gate 裁定整场完成度
8. gate 通过或存在 accepted findings 时按策略渲染 final_report.md
9. out 增加 final_report_path / final_report_status
```

`out` 增加：

```json
{
  "final_report_path": "/abs/path/runs/<sid>/final_report.md",
  "final_report_status": "complete | draft_incomplete | not_generated",
  "structured_findings": {
    "accepted": 1,
    "rejected": 0
  }
}
```

## 8. Guardian 与 Verify 适配

### 8.1 guardian_check_finding

新增：

```python
def guardian_check_finding(finding: dict, finding_dir: Path, authorized_hosts=None) -> Verdict:
    ...
```

短期实现方式：

1. 先跑 `validate_finding()`。
2. 将 finding 渲染为临时 Markdown excerpt。
3. 复用 `guardian_check(excerpt, evidence_dir=finding_dir, authorized_hosts=...)`。

临时 Markdown excerpt 必须包含：

- frontmatter：`severity/title/target/type`
- 已证明影响。
- 可执行 PoC 内容或 raw HTTP 内容。
- 响应证据关键片段。
- 结果动词，避免 L7 误降级。

### 8.2 verify.py helper

新增：

```python
def extract_poc_from_file(path: str | Path) -> Request | None
def extract_poc_from_finding(finding: dict, finding_dir: Path) -> Request | None
```

优先级：

1. `poc.file`
2. `proof_packets[].request_file`

规则：

- `poc.type == "curl"` 时按 curl 解析。
- raw HTTP request 按现有 `extract_poc()` 能力等价解析。
- 非幂等请求仍遵守 `verify.replay()` 的默认安全门，不自动重放改删。

## 9. final_report 与 session-gate 策略

### 9.1 状态定义

| 状态 | 条件 | 是否生成 `final_report.md` | 说明 |
|---|---|---|---|
| `complete` | 有 accepted finding，且 session-gate pass | 是 | 可作为正式综合报告 |
| `draft_incomplete` | 有 accepted finding，但 session-gate incomplete/needs_input | 是 | 报告开头必须标注“测试未完成”，并列出未闭合高价值面摘要 |
| `not_generated` | 无 accepted finding | 否 | 不生成误导性空报告 |

### 9.2 draft_incomplete 报告头

当 gate 未通过但已有 accepted finding 时，报告开头在 H1 后增加：

```md
> 状态：测试未完成。本报告仅汇总已确认漏洞；仍存在未闭合高价值攻击面，不能视为完整安全测试结论。
```

并增加一节：

```md
## 测试完整性说明
```

内容来自：

- `session_gate.reasons`
- `open_risk_cells`
- `coverage_ledger.stats.high_value_open`

### 9.3 summary.json

`run.py` 写入：

```json
{
  "final_report_path": "/abs/path/runs/<sid>/final_report.md",
  "final_report_status": "draft_incomplete",
  "structured_findings": {
    "accepted": 1,
    "rejected": 0
  }
}
```

终端打印：

```text
综合报告: /abs/path/runs/<sid>/final_report.md (draft_incomplete)
```

## 12. 回归验收矩阵

| 场景 | 输入 | 期望 |
|---|---|---|
| legacy report | `report_001.md` | 旧 Guardian 路径不破坏 |
| structured basic | `findings/finding_001/finding.json` | accepted，生成 final report |
| missing evidence | 缺 `response_1.http` | rejected，不进 final report |
| source constructed | `source_proof` 有 JS 文件/行号/函数 | final report 输出源码或前端证据 |
| encrypted payload | `crypto_chain` 有算法/key/iv/helper | final report 输出加密链路 |
| gate incomplete | accepted finding + 高价值面未闭合 | 生成 draft_incomplete，不伪装 complete |
| no accepted | 全 rejected 或无 finding | 不生成 final report |
| path escape | `poc.file=../../x` | rejected |

## 13. 完成定义

v5.1 完成时必须同时满足：

1. 新会话能通过 `project-memory/` 快速理解项目。
2. 模型确立漏洞时默认落 structured finding 包。
3. 外壳能递归采集 finding 包和证据文件。
4. finding 能被校验、拒收、Guardian 质检和可选 Verify 复验。
5. accepted structured finding 能映射到现有 summary/benchmark 兼容字段。
6. `runs/<sid>/final_report.md` 由 accepted finding 稳定渲染。
7. session-gate 未通过时，报告状态明确为 `draft_incomplete`。
8. `summary.json` 包含 `final_report_path/final_report_status/structured_findings`。
9. `python3 run.py --self-check` 包含 reporting 断言并通过。
10. legacy `report_*.md` 路径仍可用，不破坏旧流程。
