# Atoolkit v9.0 迭代方案：完全体结果归因与跨 Run 续航

日期：2026-07-18

## 1. 目标与边界

v9.0 的“完全体”不是保证每轮挖出全部漏洞，而是把以下八项变成可验证的工程合同：

1. 没有静默遗漏；
2. 没有无证据 Finding；
3. 没有手工报告绕过真值；
4. 没有虚假覆盖率；
5. 没有垃圾风险进入 SRC 提交物；
6. 每个未命中都能准确归因；
7. 新发现自动驱动下一轮；
8. 更换模型后仍复用同一执行、证据和跨 Run 合同。

本版本不承诺自动发现未知业务威胁，也不把 Direct/Qoder 运行伪装成可信 authority。可信提交仍要求外部父进程提供可证明的进程、文件和网络边界；当前 bundled backend 继续 fail closed 为 diagnostic。

## 2. v8.13 实跑审计结论

仓库基线回归为 `400 passed, 1 skipped`，但最新靶场/SRC 产物暴露的是运行入口与跨轮真值断链，不是单元测试回归：

| 现象 | 物理证据 | 根因 | v9 控制 |
|---|---|---|---|
| 三轮 SRC 均直接生成 `final_report.md` | 对应目录缺 `run_manifest.json`、`run_plan.json`、coverage/candidate ledger 和 delivery receipt | Direct 手工工作流仍允许把 Markdown 当交付物 | verified-submission gate；无 receipt 的报告固定为 orphan/unverified |
| 13 个结构化 Finding 全部不满足现行 proof contract，另一个 Run 只有 `report.md` | v8.13 validator 接受数为 0 | Finding 包格式、物理包、exact cell、claim/impact 与最终报告分叉 | run audit + proof rejection 归因 + proof-repair continuation |
| 报告含原始 Token/Cookie/PII | 多个 report-like 文件命中敏感信息扫描 | Canonical renderer 的脱敏可被手写报告绕过 | submission checker 对报告哈希、receipt、secret scan 三重验证 |
| `message/markRead` 先报 IDOR，后续真实双账号闭环证明隔离有效，但旧报告仍保留 | `findings_summary.md` 与 `negatives_summary.md` 对同一端点冲突 | 多轮写进同一目录；无 root finding 撤销/冲突队列 | 每 Run 独立 session；miss attribution 暴露冲突；ProjectState continuation 触发复验，不直接继承散文 Finding |
| `summary.json` 写 complete，但缺 manifest/ledger/closure | 手工 summary 与 validator 结果冲突 | 终态由模型散文决定 | summary/final report 仅由 exactly-once finalizer 投影；audit 标记 manual terminal claim |
| 500、当前用户自己的 Token、无限调用、未证明 ATO 等进入候选报告 | 报告与 submission summary 自己也承认需补强或不建议提交 | 垃圾风险门只在 Canonical 路径生效，且 generic response differential 仍可能包装 500 | structured submission policy：无安全边界突破的 500/自有凭证/资源消耗不得进入 SRC 提交物 |
| v8.13 `execution-backlog.json` 只写本 Run 文件 | incomplete finalizer 只提交 proof roots，丢弃 inventory/intents/backlog | “next_run_required”是标签，不是跨 Run 可执行任务 | Host 生成 continuation，可信 incomplete Run 也可只提交 continuation，不提交未证漏洞/阴性 |
| 未命中只能看到总数或 closure reason | 无 per-cell miss ledger | discovery/planning/identity/execution/proof 混为一个 incomplete | `miss-attribution.json` 对每个 frozen cell、未分配 endpoint、未知 method、新 discovery、proof reject 唯一归因 |

## 3. v9 唯一链路

```text
Host-owned Manifest/Plan
  → Discovery evidence
  → Feature/Threat/Cell
  → Experiment/Event
  → Finding or Negative
  → deterministic validation
  → per-cell Miss Attribution
  → Next-Run Agenda
  → ProjectState continuation commit
  → exactly-once Finalizer
  → Receipt-bound Canonical Report
  → Verified Submission
```

Markdown summary、`state/*.md`、模型终态字符串和手写 `report.md` 都只是诊断输入，不能进入提交真值。

## 4. 实施设计

### 4.1 Outcome/Miss Attribution 合同

新增确定性归因器，对以下对象逐一输出稳定 ID、cause code、证据引用和下一动作：

- frozen coverage cell；
- inventory 中没有 exact cell 的 endpoint/param/role；
- unresolved method；
- rejected/unbound Finding；
- execution event 发现的 next-run surface；
- identity/prerequisite/barrier；
- shallow negative 与未开始实验。

主要 cause code：`confirmed`、`negative_proven`、`not_applicable`、`execution_not_started`、`experiment_incomplete`、`insufficient_depth`、`identity_missing`、`prerequisite_blocked`、`proof_rejected`、`planning_unassigned`、`method_unresolved`、`discovery_next_run`、`state_unsupported`。

每个对象必须且只能有一个主归因。unsupported/ambiguous 状态计入 `unexplained`，使 v9 closure fail closed。归因器只解释状态，不把 open 变 closed。

### 4.2 Next-Run Agenda 与可信 continuation

归因器从非终态对象生成 bounded、去重、稳定 ID 的 continuation：

- proof reject → `proof_repair`，最高优先级；
- 可恢复 blocker → `prerequisite_recovery`；
- shallow/open cell → `experiment_followup`；
- 新 endpoint/param → `discovery_replan`；
- 未分配/未知 method → `planning_repair`。

外部可信 finalizer 可以在 incomplete Run 中只提交 Host 生成的 continuation；不得提交模型声称的阴性、dead-end 或未验证 Finding。continuation 进入 ProjectState intents，并带 exact endpoint/method/param/role/threat、source run 和 evidence ref。Scheduler 必须优先消费 continuation，即使新 endpoint 尚未进入旧 inventory。

### 4.3 Verified Submission

新增提交检查器，正式提交必须同时满足：

- `delivery_status.delivery_complete=true`；
- receipt 与 authority anchor 验证通过；
- `final_report.md` 哈希等于 receipt 中 canonical report；
- attribution 完整且 closure 通过；
- canonical report 不含原始 credential/PII；
- report 是 finalizer 投影，不是模型手写文件。

不满足时只允许 `draft/unverified`，CLI 返回非零。文档不再指导 Agent “生成 final_report.md”；Agent 只生成 Finding/negative/evidence。

### 4.4 Structured Submission Policy

在 proof schema 之后再执行可提交性门：

- CORS/安全头/版本/限频/目录/栈等默认噪音沿用现有规则；
- 单次类型错误/500/崩溃但无数据、权限、状态或执行后果，拒绝；
- 当前用户自己的 Token/Key 只有在证明跨边界获取或成功用于未授权结果后才可报；
- ATO/RCE/批量数据继续要求对应 nonce、identity marker、有限计数和物理 proof；
- `proof_rejected` 不丢弃，进入下一轮 proof repair，不进入 SRC report。

### 4.5 Legacy/Direct Run Audit

新增只读审计入口，输出机器 JSON，不回填旧真值。检查：

- authority/run plan/ledger/feature/threat/execution/delivery 缺失；
- orphan report、manual complete claim；
- canonical Finding 通过/拒绝数量；
- report-like 文件敏感信息种类（不输出秘密值）；
- 同 endpoint 的 positive/negative 摘要冲突；
- 多轮混写一个目录；
- 八项完全体标准逐项 pass/fail。

旧 Run 只能经保守迁移形成 open inventory/continuation；不能因已有报告升级为 accepted Finding。

### 4.6 Manifest/Receipt 版本门

Manifest 升级并冻结 `outcome_contract_version`、`submission_contract_version`。v9 receipt 强制包含：

- `miss_attribution`；
- `next_run_agenda`；
- 既有 validation/inventory/coverage/candidate/project commit；
- 完整闭合且有 Finding 时的 canonical report。

旧 schema 继续可审计，但不能伪装成 v9 verified submission。

## 5. 兼容与迁移

- ProjectState 继续保持现有 schema 2，continuation 使用受约束 intent 记录，避免无必要的大迁移；
- v8.13 Run 不被原地改写，audit 明确显示 `legacy/unverified`；
- 新 manifest 使用 v9 contract，旧 manifest 不强制生成新衍生产物；
- trusted/untrusted 边界不放宽；Direct 新增状态机仍只是 diagnostic；
- frozen denominator 不变，新 discovery 仍不扩张当前 Run，而是由 agenda 驱动新 sibling Run。

## 6. 失败先行验收

1. 手写 `final_report.md`、无 receipt → submission 拒绝；
2. report 哈希被替换 → submission 拒绝；
3. report 含原始 Cookie/Token/邮箱/手机号 → submission 拒绝；
4. 500/type confusion 只有 response differential → Finding 被 submission policy 拒绝；
5. 当前用户自己的 token 未证明跨边界使用 → 拒绝；
6. 每个 frozen cell 都有且仅有一个 miss attribution；未知状态产生 unexplained 并阻断 closure；
7. 新 discovery 不改当前 denominator，但生成 `discovery_replan` continuation；
8. incomplete trusted Run 不提交阴性/coverage closure，只提交 verified continuation；
9. 下一 Run scheduler 把 continuation endpoint 放入 must-test，即使它不在旧 inventory；
10. rejected Finding 生成 proof-repair continuation；
11. legacy 三轮 SRC audit 能识别 missing authority、orphan report、secret exposure、truth conflict；
12. v8.13 authority/finalizer/project-state/benchmark 全量回归继续通过。

## 7. 发布条件

- 新增 v9 定向测试全部通过；
- 全量 pytest 通过；
- `run.py --self-check`、`run.py --doctor`、regen、compileall、diff-check 通过；
