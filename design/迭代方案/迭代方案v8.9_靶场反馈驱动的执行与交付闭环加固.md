# Atoolkit v8.9.0 迭代方案：靶场反馈驱动的执行与交付闭环加固

> 状态：第一轮双路 REJECT、第二轮一方 APPROVE/一方提出两项阻断后再次修订，待定点复审  
> 基线：v8.8.0 (`023d184`)  
> 实跑输入：`../runs/shop_2026-07-13` 及 v8.8 验收遗留项  
> 版本定位：修复会虚高覆盖率、污染跨 Run 真值、破坏交付可验证性的缺陷；不增加攻击 payload 或漏洞利用能力
> 第一轮审核结论：原方案存在 self-authority、非原子 finalizer、可删减 closure 输入和不可落地的 pre-exec 假设，已按意见重写信任边界与事务协议

## 1. 实跑结论

本次靶场仍是旧版运行产物：有 recon、响应和人工报告，但没有 v8.8 要求的启动 manifest、canonical finding、JSON Coverage/Candidate 台账、项目提交快照和 receipt。把该目录交给 v8.8 validator 时得到 `precondition_missing / exit 2`，因此旧报告可用于定位工程缺陷，不能直接进入 proof-confirmed 项目真值。

从靶场 recon 重新 bootstrap 得到 99 条记录、119 个 planned surface，并暴露以下会直接影响发现率的偏差：

- 低置信字符串和 JSON 路径被默认为 GET，30 个实际 POST 路径出现 phantom GET；
- 三元表达式 URL、`FormData.append()` 和 fetch body 解析不完整，`product-add.php`、`product-edit.php`、`upload.php`、`submit-audit.php` 的 method/param 丢失或错误；
- merchant 登录/注册角色被推成仅 merchant，匿名入口与成功后角色没有表达；
- 当前 runtime cell 没有真正把 asset 和 role 纳入身份，一个角色/一个资产的阴性可以误关闭其他角色/资产；
- 模型纯文本 `SKIP | low ROI` 无物理证据也能把 cell 关闭为 `not_applicable`，最终 `empty_allowed / exit 0`。

因此 v8.8 已建立的跨 Run 数据结构方向正确，但执行期和交付期仍有数个 P0 漏洞，可让“覆盖完成率、空跑成功率、跨 Run 继承质量”高于真实值。v8.9 必须先修真值，再谈多 Run 发现增量。

## 2. 修复优先级与发布门槛

### P0：阻断发布

1. **全 Run closure 漏检**：只要存在一条 canonical finding，缺 inventory/coverage/candidate/session gate 也会 `valid / exit 0`。
2. **文本 SKIP 假闭环**：任意 SKIP reason 可变成 terminal `not_applicable`，零请求、零证据可 `empty_allowed`。
3. **asset/role 聚合闭格**：matrix key 不含 asset/role；一个角色或资产的 finding/negative 会关闭同路径的全部格。
4. **无效 validation 污染项目真值**：manifest/proof 已失败时 orchestrator 仍提交 negative/inventory/finding。
5. **manifest replay**：把 A run 的 manifest 复制到 B run 仍可能通过 authority 校验。
6. **receipt 绑定可变文件**：第一轮 receipt 绑定 live `project_state.json`，第二轮提交后第一轮自然失效。
7. **Skill Mode 假收尾**：文档串联多个可选命令，却没有与 Engine 共用的 deterministic finalizer；receipt 可以绑定一个 exit 2 的 validation 并成功生成。
8. **symlink 任意覆盖**：模型可预建 `finding_validation.json` 等叶子 symlink，父进程直接 `write_text()` 会覆盖 workspace 外可写文件。
9. **执行前授权缺口**：`authorized_hosts` 只在收尾验证，未在已解析命令执行前拦截 URL；自然语言 deny path 没有机器策略表示。
10. **surface method/param 错误**：低置信路径伪造 GET、三元 URL/FormData/fetch options 误解析，产生虚假 cell 并漏掉真实 POST 参数。
11. **项目/资产命名冲突**：默认 project 只按 host，忽略端口和应用 base path；显式同项目多资产时又会回退到第一个 scope。
12. **singular param 丢失**：ProjectState 只合并 `params`，planner 的单个 `param` 不能跨 Run 保留。

### P1：本版本同时修复的正确性项

1. benchmark 信任可自行重算的 validation/summary digest，未重放 canonical validator/receipt。
2. 合法 `low_roi` 完整终态在 `run.py` 被误报 exit 1；只能在收紧 SKIP 后改正。
3. historical Intent 仍可能把未验证自由文本原样注入 prompt，形成跨 Run prompt injection。
4. Codex adapter 实际开放全网却将 allow-hosts 呈现为约束；当前版本没有可证明的 command broker/egress backend，必须默认禁用 live 或显式降级且如实记录。
5. Skill Mode cookie/session/authz 文件可能以 0644 落盘。
6. 旧 Run 没有可执行的迁移入口，真实 shop 产物无法形成待复验队列。

### 明确延期到 v8.9.1 的非阻断项

- target deployment fingerprint/stale：v8.9 只冻结 schema 字段、保存显式 fingerprint 和 `unknown/same/changed-unapplied` 状态，不自动重开 cell；完整精确 stale/revalidation 在 v8.9.1 实现。这样不会继续把“尚未实现”写成已实现。
- workspace AGENTS 安装器与体验优化、复杂 project move/rebind UX。
- 完整 JavaScript AST/跨文件数据流；v8.9 采用保守 extraction，不能确定的 method/param 进入 unresolved queue。
- benchmark legacy 展示体验；canonical 重验、authority anchor 和 run binding 不能延期。

### P2：交付完整性

- 增加根 README，明确 Engine/Skill 两个入口、信任等级、终态命令、退出码和迁移策略。
- 旧运行只作为 migration candidate；不自动升级为 accepted。

发布门槛：下文列出的正确性、安全和故障注入测试全部通过；全量测试、自检、doctor、regen check、compileall、diff check 全通过；两名独立审计者复核实现，无未解决 P0。不能用“测试数量”替代逐项合同验收。

## 3. 审定后的核心不变式

### 3.1 Cell 真身份与规范化版本

运行时、Coverage、Candidate、ProjectState 使用同一个 versioned `CellIdentity v2`：

```text
origin_asset_id + app_namespace + METHOD + endpoint_template + param_location/name
+ actor_role + subject_role + object_kind + vuln_class
```

序列化键：

```text
<origin>|ns=<namespace>|<METHOD> <endpoint>|<location>:<param>
|actor=<role>|subject=<role>|object=<kind>|class=<class>
```

规范冻结如下：

- `origin_asset_id` 只包含小写 scheme、IDNA 小写 host、显式 effective port；去除 userinfo、fragment 和 path。只允许 HTTP/HTTPS。
- `app_namespace` 默认 `/`，不得从 target URL path 或 trailing slash 猜测；只有显式 `--base-path`（由可信父进程写入 authority manifest 的 `base_path_explicit=true`）才能设为其他路径。`/login` 与 `/login/` 都只是 endpoint，不是业务根信号。namespace 独立于 endpoint，避免双前缀；默认 project slug 只纳入显式 namespace。
- 相对 URL 以当前显式 namespace 解析；绝对 URL 重新拆 origin/namespace/endpoint。跨 origin 必须在 authorized project scope 中。
- endpoint 先 percent-decode unreserved 字符，拒绝 encoded slash/backslash/NUL，消解 dot segment，合并重复 slash，保留有意义 trailing slash，query 不进 path；query/body/path/header/cookie 用 `param_location` 区分。
- object-id templating 只替换经 schema/观测确认的对象段，不把任意数字盲目归一；算法版本写入 manifest/project state。
- 一个 surface 展开为多个 actor-role cell。IDOR/越权还必须声明 `subject_role`（资源所有者/受影响主体）和 `object_kind`；同级 A/B 使用 actor=`attacker:user`、subject=`owner:user`，不能折叠为普通 `user`。
- `unknown` 不是通配符，不能关闭任何已知 actor/subject cell。Finding 有多个明确 affected role 时展开多个 exact cells。
- finding/negative/dead-end 只能关闭自身明确声明且证据绑定的 origin、namespace、method、param、actor、subject/object、class。
- endpoint 级视图只是 exact cells 的聚合投影：仅当冻结 run scope 中所有 required cells terminal 才能显示 run closed。

`tests/fixtures/cell_identity_v2.json` 保存 absolute/relative URL、端口、base path、dot segment、percent encode、query、trailing slash、actor/subject 的 golden vectors。identity 算法变化必须升级 schema，不能静默重算旧 key。

### 3.2 Method 真值

- fetch 默认 GET 只适用于确实位于 fetch 调用上下文且 options 未声明其他 method 的情况。
- HTML link/src 等浏览器导航可确定为 GET；HTML form 使用显式或 HTML 默认 GET。
- 脱离调用上下文的 JS string、JSON string、自由文本 path 的 method 为 unresolved `""`，不得进入 Coverage 或调度。
- 同一来源发现了 observed method 后，合并时去除该 path 的低置信 unresolved 重复项，但保留 provenance。
- CLI 裸 endpoint 不再默认 GET；必须写 `METHOD path`，否则进入 unresolved 并生成 method-resolution Intent。

Canonical recon producer 是 `inventory.json` schema 2；人工 `attack_surface_list.md` 只能通过显式 `migrate-recon` 转换，转换结果保留 `source/confidence/observed_roles`，不让 Markdown 静默覆盖物理 traffic/HAR。regex 不是通用 JS 语义分析器：三元变量、FormData 和局部 alias 能确定时提取；跨文件变量、动态 method 和无法绑定的 body 一律 unresolved，不伪造完整性。

### 3.3 冻结 Run Scope、发现队列与两层完成度

启动模型前，由可信父进程在 authority 中写不可变 `run_plan.json`：

```json
{
  "session_id": "...",
  "identity_version": 2,
  "admitted_cells": [],
  "method_resolution_items": [],
  "candidate_baseline": [],
  "budget": {},
  "sha256": "..."
}
```

- `admitted_cells` 是本轮预算选中的 exact cells；`run_closure_gate` 只要求这些 cell 闭合，预算外 cell 保留为 project backlog，不能被标成已测。
- `run_complete` 表示冻结 scope 闭合；`project_complete` 只有项目 inventory、unresolved queue、pending proof-ready candidate 和所有高价值 backlog 都闭合时才为真。二者在 summary/CLI 分开显示。
- 父进程运行中发现的新 surface 追加到 authority-owned monotonic `discovery_events.jsonl`，不能从最终 inventory 删除。它默认进入 project backlog；只有 scheduler 明确 admit 后，追加到 `scope_amendments.jsonl` 并成为本轮 closure obligation。
- unresolved path 进入 `method_resolution_items`，由 HEAD/OPTIONS/traffic/call-context 等非利用型观察处理。每项状态只能是 `open/resolved/blocked/not_applicable`；resolved 必须给 observed method provenance，not_applicable 必须满足 dead-end 合同。
- 本轮预算未选择的 unresolved 是 project backlog，不阻止 `run_complete`；已 admit 的 unresolved 未解决会阻止 `run_complete`。存在 actionable backlog 时 `project_complete=false`，不能返回 `no_work`。
- candidate 第一次进入 parent-maintained ledger 后写 authority monotonic event；删除 session JSON 不能让 proof-ready gap 消失。

### 3.4 Closure、Proof 与提交白名单

Validator 固定计算两个互不替代的 gate：

```text
proof_gate       = canonical ingestion + schema + evidence + manifest binding
run_closure_gate = frozen run_plan + monotonic discoveries/candidates + exact-cell coverage
                   + session + negative/dead-end contracts
```

终态矩阵：

| proof | closure | findings | 状态 | exit |
|---|---|---:|---|---:|
| fail | any | any | `invalid` | 1 |
| pass | fail | >0 | `incomplete_with_findings` | 2 |
| pass | fail | 0 | `incomplete` | 2 |
| pass | pass | >0 | `valid` | 0 |
| pass | pass | 0 且 `--allow-empty` | `empty_allowed` | 0 |
| pass | pass | 0 且未授权空集 | `empty_input` | 2 |
| operational error | any | any | `error` | 3 |

- closure gate 对每个 Run 执行，不能只在 findings=0 时执行。
- 零 canonical finding 且 manifest/ingestion 无错误时，`proof_gate=pass`；空集是否成功只由 closure 与显式 `--allow-empty` 决定。
- proof-fail 时不得修改 `project_state`。失败信息只追加到 authority-owned `failed_run_events.jsonl`，它不是项目真值，也不参与下轮调度。
- benchmark 可读取 proof-pass finding，但必须同时显示 closure 不完整，不得把它说成完整运行。

提交白名单冻结为：

| Outcome | Finding | Inventory | Negative | Dead-end | Intent | Project run history |
|---|---|---|---|---|---|---|
| manifest/proof fail | 否 | 否 | 否 | 否 | 否 | 否；仅 authority failure event |
| proof pass、closure fail | proof-valid root finding | 仅从该 finding 确定性派生的 exact surface | 否 | 否 | 仅从该 finding 派生的 revalidation/spread intent | `incomplete_with_findings` |
| proof pass、closure fail、零 finding | 否 | 否 | 否 | 否 | 否 | 否；仅 authority diagnostic event，不增加 project revision |
| proof+closure pass | proof-valid root finding | 逐项 schema/authority event 校验通过项 | exact identity+evidence 校验通过项 | 完整合同校验通过项 | 结构化校验通过项 | 完整状态 |

Finding 集合采用 all-or-nothing ingestion：同 ID 冲突、malformed canonical 或任一 canonical rejected 会让 proof gate fail；不能从 mixed batch 偷偷部分提交。closure-fail 时不信任 model-writable inventory/negative/dead-end/intent，只保留 proof-valid finding 及其确定性派生事实。

### 3.5 SKIP、negative 与 dead-end

- `CELL ... | SKIP | ...` 只记录调度结果 `deferred/not_tested`，不能成为 `not_applicable`。
- 唯一可关闭 `not_applicable` 的来源是结构化 `dead_ends.json`，每条必须包含 exact cell identity、枚举 `reason_code`、`refutation`、current-run `evidence_ref + sha256`、`source_run`。
- empty closure gate 再次校验每个 `not_applicable` 的合同与证据，不信任 ledger 中已有 status。
- `budget/low_roi/model_choice/missing_optional_account` 均不是 terminal dead-end。
- negative/dead-end evidence schema 必须绑定 request/response pair、target origin+namespace+method+path、param/location、actor identity、subject/object identity、source session、文件 hash 和机器断言。仅有文件名/hash 或散文 reason 不足以关闭 cell。
- evidence 在 finalizer 前由父进程停掉模型及子进程后复制到 authority immutable evidence snapshot；closure 使用 snapshot hash，不使用之后仍可改写的 session 文件。
- proof-ready candidate、needs-reproduction candidate、缺证据 negative 均阻止对应 admitted cell closure；明确 rejected/duplicate/superseded 才不阻止。

### 3.6 Authority 信任边界、Manifest 与项目身份

Threat model 明确区分“模型可写 session”与“可信 host authority”：digest 只能检测意外变化，不能阻止同时能改文件并重算 digest 的主体。只有 authority anchor 位于 agent/model 子进程不可写根目录时，交付才可标记 `authority_trusted=true`。

- Engine Mode：父进程在启动受限模型子进程前创建 host authority，模型 sandbox 只写 session；authority root 不传成 writable root。
- Wrapped Skill Mode：外部人工终端/runner 先创建 authority 和 run plan，再启动 Skill agent，agent 退出后由同一外部 runner finalize。agent 不能调用会修改 authority 的命令。
- Direct Skill Mode：同一 agent 在整个 workspace 内自行 init/finalize 时属于 self-authorized，只能生成 `diagnostic/untrusted` 产物，绝不能标记 accepted delivery 或抗篡改 receipt。README/SKILL 必须醒目标注 Engine/Wrapper 才是可验证交付路径。

authority 下的稳定 `project_identity.json` 包含随机生成后不可变的 `project_id`。Manifest 同时绑定：

- `project_id`、规范化 project scope 和 primary target；
- `session_id == run_dir.name`；
- 稳定 project/session identity、authority root identifier 和 run-plan digest；绝对路径是 locator，不是身份；
- 启动 manifest 初始 digest；
- 可选 target fingerprint 和 network policy。

把 manifest 复制到另一 run、另一 project 或修改 scope 必须失败。项目移动不改变 `project_id/session_id/commit hash chain`，旧 receipt 仍可验证；v8.9 不提供自动 rebind，仅允许 verifier 通过显式 locator 找到同一 authority identity。

默认 project slug 使用 `host + port + normalized base-path` 的短可读部分与 hash，避免同 host 多业务冲突。显式同项目多资产则原子扩展 `project_scope`，所有输入必须带明确 asset。

### 3.7 Exactly-once Finalizer、immutable commit chain 与 receipt

Finalizer 不是“依次写几个文件”，而是 authority-owned、持锁、可恢复的状态机。事务 key 为 `project_id + session_id + manifest_sha256`，全阶段持 session finalizer lock，项目提交阶段再取得 project lock 并执行 expected-revision CAS：

```text
NEW
 -> INPUTS_SNAPSHOTTED
 -> GATES_EVALUATED
 -> PROJECT_PREPARED
 -> PROJECT_COMMITTED
 -> PROJECTIONS_WRITTEN
 -> RECEIPT_ANCHORED
 -> DELIVERY_WRITTEN
```

- 每个阶段完成后原子更新 authority `finalization_journal.json`；journal 记录输入 hash、outcome、expected revision 和下一步。
- 同一事务重复 finalize 从最后完成阶段恢复，不再次增加 revision；两个并发 finalizer 只有一个能通过锁/CAS。
- `PROJECT_PREPARED` 保存 state-before snapshot、确定性 delta、state-after snapshot 和 hashes。CAS 冲突重新基于新 revision 计算同一幂等 delta；run id 已合并时复用旧 commit。
- commit 后崩溃不能盲目回滚 proof-valid 项目真值；恢复流程补齐 projection/receipt/delivery。receipt 失败时 delivery=failed，但 journal 如实保留 project committed。
- 在 `INPUTS_SNAPSHOTTED` 前必须确认 model process group 已终止；否则拒绝 finalize，消除输入与 safe-I/O 的并发 TOCTOU 面。

每轮 ProjectState 提交后在 authority commit log 与 session projection 中写 immutable `project_state_commit.json`：

```json
{
  "session_id": "...",
  "project_id": "...",
  "revision_before": 3,
  "revision_after": 4,
  "state_before_sha256": "...",
  "delta": {},
  "state_after_sha256": "...",
  "previous_commit_sha256": "...",
  "commit_sha256": "...",
  "committed_at": "..."
}
```

authority 保存 immutable state-before/state-after snapshot 或内容寻址副本，commit chain 可独立重放；receipt 不绑定 live `project_state.json`。canonical serialization 使用 UTF-8、sorted keys、无多余空白；计算 `commit_sha256/receipt_sha256` 时排除自身 hash 字段。authority 另存 `receipt_anchor.json`（receipt hash、commit hash、previous anchor hash），session 中重算 receipt 不能覆盖该 anchor。

`verify-receipt` 分开输出：

- `integrity_valid`：authority manifest/run plan/commit chain/receipt anchor/artifact hashes 自洽；
- `delivery_complete`：proof+closure、mandatory artifacts、projection consistency 和 authorization assurance 均满足对应模式。

失败或 incomplete Run 可生成 diagnostic receipt，但 `delivery_complete=false`。mandatory artifact 集按 outcome 固定，不能由调用者任意少传。连续多轮后，第一轮 receipt 仍应通过 authority snapshot/chain 验证。

### 3.8 单一 finalizer 与 Skill Wrapper

Engine 和 Skill Mode 共用 `python3 -m engine.finalize`：

```text
validate proof + closure
  -> 按 gate 提交允许的 project truth
  -> project_state_commit.json
  -> derived blackboard/run history/summary/final report
  -> run_receipt.json
  -> verify receipt
  -> delivery_status.json
```

所有 authoritative 输入使用 JSON：`inventory.json`、`coverage-ledger.json`、`candidate-ledger.json`、可选 `dead_ends.json`/`intents.json`；Markdown 只是派生视图。Skill 文档不得再声称几个松散命令等价于原子收尾。

可信外部调用合同：

```bash
python3 -m engine.skill_wrapper run \
  --project-dir <project> --run-dir <session> --target <url> \
  --authority-dir <agent不可写目录> -- <agent command...>

python3 -m engine.finalize \
  --authority-dir <authority> --project-id <id> --session-id <sid>
```

- `project-dir/run-dir/manifest` 从 authority identity 与 journal 推导，finalizer 不接受 agent 任意拼接的 project path。
- wrapper 强制 `init authority + manifest + run_plan -> agent -> terminate process group -> snapshot -> finalize`；agent 异常退出也会生成 diagnostic delivery。
- exit code 与 validator 一致：`0=integrity valid + delivery complete`，`1=invalid proof/conflict/authorization violation`，`2=incomplete/empty without permission/untrusted direct skill`，`3=operational or recovery error`。
- 旧 `runtime_manifest receipt` 入口降为 diagnostics，只能生成 `delivery_complete=false`，不能代替 finalizer。
- Engine orchestrator 调用同一个 Finalizer library；Skill wrapper 与 Engine 对同一 fixture 的 normalized finding、commit delta、summary、receipt outcome 必须等价。

### 3.9 安全写入与执行授权的现实边界

- 新建 `engine.safe_io`。不能使用“先 resolve 再 write”的 check/use 模式；以已打开可信 root 的 directory fd 为锚，逐级 `openat`/等价相对操作并使用 `O_DIRECTORY|O_NOFOLLOW`，同目录 0600 临时文件、fsync、原子 replace；append 使用 `O_APPEND|O_NOFOLLOW`。不支持所需原语的平台 fail closed，不能静默降级。
- leaf symlink 由原子 replace 覆盖且不跟随；parent symlink/hardlink/跨 root/cross-device replace 被拒绝；finalizer 在模型进程终止后操作 immutable snapshot。
- 权威写入清单：manifest authority/session projection、run plan、scope amendments、discovery/candidate events、runtime state、events、inventory、Coverage、Candidate、Fact-Intent、business graph、run scope、ProjectState、commit/snapshot、validation、summary/final report、blackboard/run history、adapter response、receipt/anchor、delivery。CI 用 `rg` allowlist 防新增父进程裸 `write_text/open(...,'w')`。
- `AuthorizationPolicy` 仍用于验证 manifest/evidence/finding URL 和受控 transport，但不能谎称能拦截嵌套 `codex exec` 内部 shell。
- `--exclude-endpoint` 是调度排除，不等同授权 deny。新增独立 `--allow-path`/`--deny-path`。
- v8.9 没有可证明的 Codex command broker/OS egress backend。因此默认 `--network-policy deny-live`，非 dry-run 拒绝启动；`--allow-unrestricted-egress` 是明确的危险降级，只记录 `authorization_assurance=unrestricted_user_accepted`，绝不写 `preexec_enforced`。该模式可以做授权目标探索，但完整交付状态必须显示 authorization assurance 降级。
- 未来 hard backend 必须让全部 HTTP/DNS 经受控 transport，并用真实第二监听器验证 forbidden host/path、redirect、DNS/IP rebinding、IP literal、端口、脚本/变量/文件间接 URL 均为 **0 packet reached**；Mock policy 单测不算 hard enforcement。该 backend 不在 v8.9 假装实现。

### 3.10 Target fingerprint 占位与跨 Run 注入边界

- 新增 `--target-fingerprint` / `--target-fingerprint-file`，只接受用户或部署系统提供的显式非空 fingerprint，不根据普通响应波动猜测。
- v8.9 只保存并比较 fingerprint，输出 `unknown/same/changed-unapplied`；变化时发出显式诊断但不自动改 cell。v8.9.1 才实现保留历史 finding、精确标记 stale/open 和 revalidation Intent 的状态迁移。
- fingerprint 未知或 changed-unapplied：不自动失效，也不伪称 current。
- 历史 Fact/Intent/negative 只以有界、escaped JSON data block 注入；legacy/unvalidated description 标明 trust level，禁止原样拼接 XML/Markdown 控制文本。

## 6. 成功标准

v8.9 发布成功不要求每轮新增漏洞。固定 oracle fixture 上必须满足：

- `run_scope_reduction = closed_admitted_cells / admitted_cells` 可重算，删行不能提高；
- `project_open_delta = open_before - open_after` 与 commit delta 一致；
- `duplicate_exploration_rate = retested_terminal_cells / attempted_cells` 不因错误继承而虚降；
- `unique_proof_root_delta` 只计 proof-valid registry 新 fingerprint，允许为 0；
- oracle recall/false-positive 只从 canonical validator 重放结果计算；
- summary、final report、benchmark 是同一 validation+commit 的纯投影，claim 数值 marker 必须存在于物理证据；`incomplete_with_findings` 不得写“全覆盖/可交付”。

发布层执行两个层级：固定可重复 oracle/故障注入 fixture 是硬门；新的 v8.9 真实靶场 Run 是效果观察，不把目标天然没有新漏洞误判为产品失败。后续在固定授权、账号、模型、预算和部署 fingerprint 下连续 3–5 Run，知识闭环有效的标准是 inventory/未决队列可解释扩展、open cell 下降或 blocker 明确、重复探索率下降、唯一 proof root 不重复计数；漏洞数可以不增加。
