# Atoolkit v8.11 迭代方案：威胁建模真值与 Canonical 交付

日期：2026-07-17

## 1. 结论与版本目标

v8.10 修复了参数风险并集、barrier-aware negative、七态 ledger 和 Direct
observation/checkpoint，但真实 Run 仍然暴露两个结构性问题：

1. 测试计划仍以 endpoint/param/risk-tag 为起点，risk tag 实际承担了漏洞清单的角色；
2. Direct/Qoder 模型可以绕过 runtime 与 validator，直接写 `final_report.md`。

v8.11 不增加 payload，也不宣称一次完成终局 v9。它交付“完全体”的第一段可执行闭环：

1. 引入结构化 `feature-graph.json` 与 `threat-model.json`；
2. 由 threat compiler 把业务威胁编译为精确 coverage cell；
3. discovery channel、feature、threat 与 cell 建立可追踪关系；
4. 无 threat model 的 Direct 兼容运行固定标为 degraded，永不 report-ready；
5. observation 自动绑定 `feature_id/threat_id`，checkpoint 生成 threat coverage；
6. Engine/Wrapper manifest 记录模型和规划模式；
7. shared finalizer 独占 canonical report 生成权，并把报告纳入 receipt/delivery。

## 2. 核心原则

### 2.1 模型负责提出威胁，机器负责验证结构

引擎不根据漏洞清单替模型生成 threat。模型必须基于业务功能写出：

- 被保护的安全不变量；
- 攻击者和前置条件；
- 滥用动作；
- 安全实现应有的结果；
- 可观察的边界突破；
- 为什么当前 feature 存在该威胁；
- 对应 API、参数、角色与证据要求。

机器只做 schema、引用关系、inventory 一致性、编译、覆盖和证据门控。

### 2.2 risk tag 降级为知识路由

在 threat-model 模式下，risk tag 只用于：

- 选择当前 cell 的知识卡；
- 调整优先级和证据深度；
- 提醒模型存在尚未解释的风险维。

它不再自动决定 coverage 分母。coverage 分母来自模型明确声明且通过 schema 的 threat。

### 2.3 Canonical report 是 finalizer 产物

`final_report.md` 是保留文件名：

- valid/empty-allowed + closure pass：finalizer 从 proof-confirmed root findings 生成；
- proof-valid 但 closure 未通过：只生成 `draft_report.md`；
- proof fail/manifest fail：不保留任何 `final_report.md`；
- receipt 绑定 canonical report，`delivery_complete` 必须验证其 hash；
- agent 手写同名文件会被 finalizer 的 canonical 投影替换或移除。

## 3. 新增数据合同

### 3.1 `feature-graph.json`

最小结构：

```json
{
  "schema_version": 1,
  "target": "https://target.example/",
  "discovery_channels": {
    "js_ref": {"status": "covered", "evidence_refs": ["recon/app.js"]},
    "inline_script": {"status": "covered", "evidence_refs": ["recon/home.html"]},
    "asset_ref": {"status": "covered", "evidence_refs": ["recon/config.js"]},
    "page_link": {"status": "covered", "evidence_refs": ["recon/pages.json"]},
    "path_inference": {"status": "covered", "evidence_refs": ["recon/probes.json"]},
    "response_body": {"status": "covered", "evidence_refs": ["recon/traffic.har"]}
  },
  "features": [{
    "feature_id": "refund",
    "name": "订单退款",
    "actors": ["user"],
    "assets": ["account_balance", "order"],
    "objects": ["order", "refund"],
    "states": ["paid", "refunded"],
    "trust_boundaries": ["browser_to_refund_api"],
    "inputs": ["order_no", "refund_amount"],
    "actions": ["request_refund", "read_balance"],
    "apis": [{
      "endpoint": "/api/refund",
      "method": "POST",
      "params": ["order_no", "refund_amount"],
      "roles": ["user"]
    }]
  }],
  "unassigned_endpoints": []
}
```

约束：

- 六个 discovery channel 必须全部出现；
- `covered` 必须有物理 evidence ref；
- `blocked/not_applicable` 必须有 reason；
- inventory 中每个 resolved endpoint 必须属于 feature，或进入
  `unassigned_endpoints` 并说明原因；
- feature 必须有 actor、asset/object、action 和 API；
- feature API 的 method/param 必须与 inventory 一致。

### 3.2 `threat-model.json`

最小结构：

```json
{
  "schema_version": 1,
  "features": [{
    "feature_id": "refund",
    "coverage_note": {
      "input_surface": "覆盖订单号、退款金额和身份上下文",
      "behavior_surface": "覆盖退款状态转换和余额回查",
      "depth_strategy": "owner/peer + state before/after"
    },
    "threats": [{
      "threat_id": "T-refund-overpay",
      "vuln_class": "refund-amount-invariant-bypass",
      "security_invariant": "累计退款不得超过订单实际支付金额",
      "attacker": "authenticated_user",
      "asset": "account_balance",
      "preconditions": ["owned_paid_order"],
      "abuse_action": "提交超过实付金额的退款值",
      "expected_secure_result": "服务端拒绝超额退款",
      "observable_violation": "余额增加超过订单实付金额",
      "reasoning": "退款金额由客户端传入且会改变余额",
      "targets": [{
        "endpoint": "/api/refund",
        "method": "POST",
        "params": ["refund_amount"],
        "roles": ["user"]
      }],
      "evidence_required": ["state_before", "exploit", "state_after"],
      "unruled_out": ["concurrent_refund"]
    }]
  }]
}
```

约束：

- feature graph 中每个 feature 必须在 threat model 中出现；
- 没有 threat 的 feature 必须给出 `no_threat_reason`；
- `coverage_note` 三字段非空；
- threat 必须包含 invariant、attacker、preconditions、abuse、secure result、
  observable violation、reasoning、target 和 evidence requirement；
- target 必须指向该 feature 的 API；参数和 method 必须与 inventory 一致；
- `vuln_class` 可自由命名，不要求命中固定漏洞分类。

### 3.3 编译后的 coverage cell

每个 cell 增加：

- `feature_id`
- `threat_id`
- `security_invariant`
- `observable_violation`
- `evidence_required`

精确身份仍保留 asset/method/path/param/location/actor/subject/object/vuln_class。
同一 endpoint/param 上的两个业务 threat 不得合并。

Threat-mode Finding 必须额外绑定：

- `feature_point.feature_id`
- `claim.threat_id`（或等价顶层规范化字段）

validator 只允许 Finding 关闭与该 feature/threat 编译结果精确匹配的 cell；自由文本标题、
相似 vuln class 或同路径其他 threat 均不能串证据。

### 3.4 `threat-coverage.json`

checkpoint 派生：

```json
{
  "schema_version": 1,
  "planning_mode": "threat_model",
  "stats": {"features": 1, "threats": 2, "open_threats": 1},
  "features": [],
  "threats": []
}
```

一个 threat 只有全部编译 cell 闭合才闭合；一个 feature 只有全部 threat 闭合才闭合。

## 4. Direct/Qoder Runtime

`engine.skill_runtime init` 新增：

```bash
python3 -m engine.skill_runtime init \
  --run-dir <run> --target <target> --inventory <inventory> \
  --feature-graph <feature-graph.json> \
  --threat-model <threat-model.json>
```

行为：

- 两个 threat 文件都存在：`planning_mode=threat_model`，使用 compiler；
- 只提供一个：hard error；
- 都不提供：保留旧 planner 兼容，但写
  `planning_mode=legacy_risk/planning_degraded=true/report_ready=false`；
- init 把规范化副本和 hash 写入 run；checkpoint 检查副本未漂移；
- observation 自动绑定 surface 的 feature/threat，显式不一致时拒绝；
- queue 展示 feature、threat、invariant 和 violation，不再只展示 vuln class。

Direct 模式继续保持 `authority_trusted=false/delivery_eligible=false`。新增
`skill_runtime finalize` 只生成诊断 validation/delivery 状态，不允许产生可交付
`final_report.md`。

所有 evidence ref 和 planning artifact ref 必须是 run 内相对路径，拒绝绝对路径、
`..`、symlink/hardlink 和缺失文件。Direct 输入位于 run 外时先读取、规范化并写入
run 内副本，后续只使用副本。

## 5. Wrapped Skill 与 Engine

### 5.1 Wrapped Skill

wrapper 可接收 inventory + feature graph + threat model：

- 外部 host 在 agent 启动前验证并规范化三份输入；
- compiler 输出 exact cells；
- authority `run_plan.json` 冻结编译后 cell；
- manifest 绑定 threat artifacts hash；
- 缺 threat model 时只能使用显式 `--legacy-risk-plan`，且 manifest 记录 degraded。

可信 wrapper 启动后发现的新 endpoint/param/threat 不得扩大当前 frozen run plan；它们进入
结构化 `next_run_discoveries` / pending Intent，交给下一次 host-owned plan。Direct
diagnostic 可以展示动态候选，但不得把它们伪装成已授权闭格。

### 5.2 Engine

v8.11 记录 `planning_mode=legacy_risk`，保持现有单阶段 Engine 兼容。Engine 的
两阶段“先 discovery/威胁建模，再由 host freeze threat plan，再 attack”放到 v8.12，
避免在本版破坏 pre-model manifest 和 authority run-plan 语义。

## 6. 运行溯源

manifest 新增冻结字段：

```json
{
  "execution_provenance": {
    "provider": "openai",
    "model": "gpt-5.5",
    "adapter": "codex",
    "settings": {"temperature": null, "seed": null}
  },
  "planning_mode": "threat_model",
  "planning_artifacts": {
    "feature-graph.json": "sha256...",
    "threat-model.json": "sha256..."
  },
  "canonical_report_required": true
}
```

Engine 从 adapter 提取 provider/model/adapter；wrapper 提供对应 CLI 参数或记录
`unknown`，不能从报告文本推断模型。

`execution_provenance` 只允许 provider/model/adapter/temperature/seed/reasoning-effort
等非秘密字段；禁止复制环境变量、命令行凭据、Cookie、Token、代理 URL 或任意 settings。
这些字段属于 manifest replay identity，恢复时必须完全一致。

## 7. Canonical Delivery

shared finalizer 在 `PROJECTIONS_WRITTEN` 阶段：

1. 从 frozen validation 的 proof-confirmed paths 重新读取 canonical finding；
2. full closure：生成 `final_report.md`；
3. valid empty：生成明确的“无可提交漏洞”报告；
4. incomplete-with-findings：生成 `draft_report.md`，移除 stale final report；
5. invalid/precondition-missing：移除 stale final/draft report；
6. summary 记录 report path/status/hash；
7. receipt 在 `canonical_report_required=true` 时强制包含并验证 `final_report`；
8. delivery complete 同时要求 report hash、proof、closure、authority 和 assurance 通过。

`final_report.md`、`draft_report.md` 和 report status 文件必须加入 finalizer 的 derived-file
排除集，绝不能作为 frozen input 或模型证据进入 snapshot。finalizer 渲染时只读取
authority snapshot 中 proof-confirmed Finding 和证据，生成后再投影到 live run。

## 8. Validator 扩展

当 manifest `planning_mode=threat_model` 时，closure gate 额外要求：

- feature graph/threat model 文件存在且 hash 匹配；
- schema 和引用关系有效；
- threat coverage 存在；
- 所有 in-scope threat 闭合；
- feature/threat/cell 计数一致；
- accepted finding 的 feature/threat 必须能映射到已编译 cell。

规划文件的 hash 校验使用 no-follow、安全根路径读取；`covered` channel 的 evidence refs
同样必须存在且位于 run 内。仅填写六个布尔值不能通过 discovery gate。

legacy manifest 不追溯要求 threat 文件，但 summary/delivery 明确记录 degraded。

## 9. 测试策略

先写失败测试：

1. discovery channel 缺失/无 evidence 时拒绝；
2. inventory endpoint 未分配 feature 时拒绝；
3. threat target method/param 不存在时拒绝；
4. 自由命名 threat 编译为 exact cell，且不铺满通用漏洞矩阵；
5. observation 自动绑定 threat，伪造 threat_id 被拒绝；
6. checkpoint 聚合 threat/feature coverage；
7. legacy Direct 永不 report-ready；
8. manifest 保存 provider/model/planning mode/hash；
9. planning artifact 被篡改时 manifest/validator fail closed；
10. finalizer 替换手写 final report并绑定 canonical hash；
11. invalid run 不保留 final report；
12. empty valid run 输出零漏洞 canonical report；
13. receipt 缺必需 canonical report 时不得 complete。
14. threat-mode Finding 缺失/伪造 threat_id 时拒绝；
15. 动态发现不能扩大 trusted frozen plan；
16. report 文件不能进入 finalizer input snapshot；
17. provenance 拒绝秘密或任意 settings 字段。

发布前执行全量 pytest、self-check、doctor、regen check、compileall、git diff check。

## 10. 非目标与后续

v8.11 不做：

- 不让引擎伪造 LLM threat；
- 不自动执行 payload 或危险操作；
- 不宣称 threat model 可以保证零遗漏；
- 不把 Direct 升格为 trusted authority；
- 不在同一版本重写 Engine 为两阶段模型调用；
- 不以 8180 oracle 特判路径或参数。

v8.12：Engine 两阶段 threat-plan freeze、页面/功能触发状态和动态局部重建模。

v9 完全体标准：Feature→Threat→Cell→Observation→Proof→Report 全链唯一真值，
所有未命中都可归因，零垃圾 finding，且不同模型在相同运行合同下可复现实验。
