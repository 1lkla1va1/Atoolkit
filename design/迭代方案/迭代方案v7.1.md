## 2. 支柱 A · SQLi 查询类型适配规则（治 P0-1）

### 2.1 问题回顾

v4 评估确认 search.php 存在 LIKE 注入（`' OR '1'='1` 返回全部 6 商品），但 input-agent 用 `' OR 1=1--` 测试后报 not_vulnerable。根因是 `' OR 1=1--` 在 LIKE 上下文中被解析为 `LIKE '%' OR 1=1--%'`，`--` 注释了闭合引号但 `1=1` 在 LIKE 内部不生效。

### 2.2 方案：查询类型感知 payload 矩阵

在核心技能文件 §7（决策树）的"按参数语义分支"下新增 SQLi 子规则：

```markdown
### SQLi 查询类型适配（v7.1 新增）

SQL 注入的 payload 必须按查询上下文适配。测试每个注入点时，按以下矩阵逐一发送：

| 查询类型 | 典型场景 | 必须包含的 payload | 检测指标 |
|---|---|---|---|
| WHERE 等值 | `WHERE id='$input'` | `' OR 1=1--`, `' OR '1'='1'--` | 返回行数差异 |
| WHERE LIKE | `WHERE name LIKE '%$input%'` | `' OR '1'='1`, `%' OR '1'='1` | 返回行数 > 正常搜索 |
| ORDER BY | `ORDER BY $input` | `1 DESC--`, `(SELECT 1)--`, `IF(1=1,name,id)` | 排序变化 / 时间差异 |
| INSERT/UPDATE | `SET name='$input'` | `test', field='injected` | 字段值被修改 |
| 子查询/HAVING | `GROUP BY ... HAVING $input` | `' OR '1'='1`, `1) OR 1=1--` | 返回数据变化 |

**阴性结论的对比验证（强制）：**
- 每个注入点必须对比"正常请求"和"注入请求"的返回数量/内容差异
- 不能仅看 HTTP 状态码或 JSON success 字段——LIKE 注入成功时状态码仍是 200、success 仍为 true
- 如果正常搜索返回 N 条结果，注入 payload 必须验证是否返回 ≠N 条

**payload 清洁度规则（v7.1 关键补充）：**
- 在 LIKE 上下文中，payload 必须**不带前缀文本**——`test' OR '1'='1` 会变成 `LIKE '%test' OR '1'='1%'`，OR 子句被 LIKE 包裹而失效
- 正确做法：直接用 `' OR '1'='1`（空前缀），让 LIKE 解析为 `LIKE '%' OR '1'='1%'` → `LIKE '%'`（true）OR `'1'='1%'`（true）
- 在 WHERE 等值上下文中前缀不影响结果（`WHERE id='test' OR 1=1--` → OR 在外层生效），但 LIKE 上下文中必须去掉前缀
- 通用规则：测试注入时，payload 应直接以控制字符开头（`'`、`"`、`)`），不要在前面添加测试文本

**payload 编码注意事项：**
- URL 编码特殊字符：`'` → `%27`, 空格 → `%20` 或 `+`
- 在 LIKE 上下文中，`%` 本身就是通配符，payload 中的 `%` 可能干扰查询
- 优先使用 `' OR '1'='1`（不含 `%`）而非 `%' OR '1'='1`
```

## 3. 支柱 B · 聚合缝隙检查协议（治 P1-1, P3-2）

### 3.1 问题回顾

v7 的 4 agent 按方向分工（auth/txn/idor/input），但 submit-audit.php 同时涉及 auth（审核状态变更）和 input（文本字段 XSS），导致两个 agent 都以为对方会测。

### 3.2 方案：聚合阶段三步缝隙检查

在 SKILL.md 的 "Parallel Agent Protocol → Aggregation phase" 中新增缝隙检查步骤：

```markdown
### 聚合阶段（v7.1 扩展）

After all agents complete:

1. 合并所有 findings 到统一 coverage_ledger.md
2. **缝隙检查（v7.1 新增 · 必须在去重前执行）：**
   a. 列出所有发现的端点（endpoint 去重集合）
   b. 对每个端点，列出所有参数及其类型分类：
      - 状态参数（status/audit_status/state） → 归 auth agent
      - 文本参数（name/description/keyword/reason） → 归 input agent
      - 金额参数（amount/price/refund_amount/use_points） → 归 txn agent
      - ID 参数（order_no/product_no/user_id/merchant_id） → 归 idor agent
   c. 检查每个端点的每种参数类型是否都有对应 agent 的测试记录
   d. 未覆盖的「端点×参数类型」组合 → 标为 seam_gap，列入补测队列
3. 对 seam_gap 执行补测（可由聚合 agent 直接执行或分配给对应方向 agent）
4. 按同根因去重规则合并重复项
5. 重新编号为 finding_001, finding_002, ...
6. 生成 summary.json

### 缝隙检查示例

```
端点: POST /api/merchant/submit-audit.php
  参数: audit_status (状态) → auth agent 已测 ✅
  参数: description (文本) → input agent 未测 ❌ → seam_gap
  参数: merchant_id (ID) → idor agent 已测 ✅

端点: POST /api/user/create-batch-order.php
  参数: product_no (ID) → idor agent 已测 ✅
  参数: quantity (金额) → txn agent 已测 ✅
  参数: use_points (金额) → txn agent 未测 ❌ → seam_gap
```
```

```markdown
## 0.1.5 非 API 页面与表单扫描（2 分钟，v7.1 新增）

**目标**：发现不在 JS 中的高价值页面和表单提交路径。

### 操作步骤

1. **扫描 HTML 中的表单 action**：对 0.1 获取的每个页面，提取所有 `<form action="...">` 路径
2. **扫描服务端渲染链接**：提取所有 `<a href="...">` 中的内部路径（排除 # 和 javascript:）
3. **检查常见高价值页面模式**：按路径模式逐一 curl 检查是否存在：

4. **检查 URL 参数**：对每个可达页面，检查 URL 中是否接受以下参数：
   - `redirect`、`return_url`、`callback_url`、`next`、`url`
   - `file`、`page`、`path`、`template`
   - `token`、`auth_token`、`code`
5. **记录到 endpoint_inventory.md**：非 API 端点用 `[HTML]` 前缀标记

- 不要忽略 `<iframe src>` 和 `<meta http-equiv="refresh" content="...">` 中的路径
- 服务端渲染的页面可能不包含 JS 文件，但仍然接受 URL 参数

## 9. 支柱 A 续 · 枚举三通道规则（治 P2-3）

### 9.1 问题回顾

### 9.2 方案：枚举测试三通道

```markdown
### 用户枚举三通道（v7.1 新增）

用户枚举不能仅测错误消息差异。必须覆盖三个通道：

| 通道 | 测试方法 | 检测指标 |
|---|---|---|
| 消息差异 | 有效用户名 + 错误密码 vs 无效用户名 + 错误密码 | 响应 body 差异 |
| 时间差异 | 同上，测量响应时间 | ≥100ms 差异（有效用户名可能多一次 DB 查询）**需至少 3 次测量取中位数，排除网络波动** |
| 行为差异 | 对有效用户名连续发送 5 次错误密码 vs 无效用户名连续 5 次 | 是否触发锁定/captcha/限频（仅对有效用户名触发） |

**时间差异测量注意事项：**
- 单次测量不可靠——网络延迟、服务器负载、缓存命中都会影响响应时间
- 至少 3 次测量取中位数，有效用户名和无效用户名交替发送
- 差异需持续稳定（3 次中至少 2 次有效用户名响应时间 > 无效用户名）才算显著
- 本地靶场网络延迟低，≥50ms 差异即值得记录；远程目标需 ≥100ms

**约束：三个通道全部测完才能下阴性结论。** 仅测消息差异不够。
```

## 10. 支柱 C 续 · 自检落盘格式（治 P3-1）

v7 的自检清单设计了"每 5 surface 自检"和"每 15 surface 自检"，但 v4 评估未找到任何 agent 执行自检的证据。

### 10.2 方案：自检结果写入磁盘

在核心技能文件 §11（Skill Mode 自检查清单）中新增落盘要求：

```markdown
### 自检落盘格式（v7.1 新增）

每次自检执行后，将结果追加写入 `selfcheck_log.md`：

```markdown
## Self-check #1 (after surface S05, ~15min)
- 连续 shallow_negative: 否（2 个 confirmed, 3 个 not_vulnerable）
- surface 超时: 否（最长 6 分钟）
- 当前方向有 confirmed: 是
- risk 维遗漏: 未发现
- 动作: 继续当前方向

## Self-check #2 (after surface S10, ~30min)
- 连续 shallow_negative: 是（S08/S09/S10 连续 3 个）
- 动作: 切换到下一攻击面方向
```

**约束：终态自检时检查 selfcheck_log.md 是否存在且条目数 ≥ 预期值（surface 数 / 5）。**
无 selfcheck_log.md → 终态标 incomplete + coverage_gap "self-check not executed"。
```

### 10.3 具体文件改动

| 文件 | 改动类型 | 改动内容 |
|---|---|---|
| `skill/核心技能文件.v3.md` §11 | 修改 | 自检清单新增落盘格式要求（§10.2 内容） |
