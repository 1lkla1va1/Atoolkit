# Atoolkit v9.1.0 迭代方案：跨 Run 续航、精确格身份与剩余安全门修复

日期：2026-07-20

## 1. 复证结论

增量审核列出的 P0/P1/P2/P3 中，P0-2、P1-3、P1-4、P1-5、P1-7、P2-8、P2-9、P2-14 与 P3 属实。P0-1 需拆成两个问题：当前 backend 无法证明进程 containment，因此 `authority_trusted=false` 是正确的 fail-closed 结果，不能通过改布尔值修复；但 untrusted Run 生成的 Host agenda 没有显式的下一 Run 消费入口，属于可修的产品断链。P1-6“proof 文件无 hash”不成立：`validate_run_artifacts()` 已把 accepted Finding 的全部 `proof_files` 加入 `artifact_hashes`，`verify_validation_artifact()` 会重算并拒绝替换；本轮补回归证明，不增加 Agent 可同时改写的自报 digest。

本机已实际执行修复前基线：`419 passed, 1 skipped`。因此旧 `.pytest_cache` 的失败节点不能作为当前版本失败证据。

## 2. 目标

1. 在不伪造 authority 的前提下，让上一 Run 的 Host agenda 可显式、可校验、可绑定地进入下一 Run 调度。
2. exact cell identity 保留具体漏洞类，stored/reflected/DOM XSS、horizontal/vertical/generic IDOR 不再因语义族归一串格。
3. LOW_ROI 必须有物理直觉探索证据。
4. 后续 Run 不自动继承输入验证类阴性；新阴性必须证明使用了不同编码族和策略族。
5. 敏感数据检测器与脱敏器使用独立规则，避免同一盲区同时漏脱敏和漏检测。
6. captcha/SMS/2FA 在至少五类绕过方向完成前保持 recoverable，而不是直接 NEED_INPUT。
7. library/finalizer 默认全部 fail closed：无 scope 不验证、无 method 不猜 GET、authority 默认 false。
8. 删除确认无调用的 legacy summary 映射函数。

## 3. 修复设计

### 3.1 显式跨 Run 降级续航

新增 `--continue-from-run <prior-run>`：

- 只接受真实目录内的 `miss-attribution.json`、`next-run-agenda.json` 与 `finding_validation.json`；拒绝软链接、路径逃逸、超限文件和结构错误。
- 重算 validation digest、Attribution digest、Agenda projection、count/source hash；任何不一致 fail closed。
- 将通过校验的 continuation 规范化为 `continuation-input.json`，加入下一 Run 的 immutable manifest artifact binding。
- agenda endpoint 必须是相对路径或当前授权 origin；method 必须显式有效，unknown 不得转 GET。
- continuation 只进入本 Run scheduler/inventory，标 `source=v9_host_continuation`、`diagnostic_only=true`、`authority_trusted=false`；它不能直接写 ProjectState、关闭 cell 或使 submission eligible。新的可信证据仍须由当前 Run 产生。
- 若未来存在真实 containment attestation，原 authority commit 路径保持不变；本补丁不设置 `authority_trusted=true`。

### 3.2 Cell identity v3

新增 `exact_vc()`，与知识路由使用的 `norm_vc()` 分离：

- `norm_vc()` 继续返回 `xss/idor/business` 等语义族，只用于路由、排序与统计。
- `exact_vc()` 保留具体安全边界：`stored-xss`、`reflected-xss`、`dom-xss`、`horizontal-idor`、`vertical-idor` 等使用不同 key；同义语言别名可映射到同一 exact token。
- `canonical_project_cell_key()`、runtime `CellIdentity`、ProjectState cell/dead-end/negative/finding 投影全部使用 exact token，同时记录 `vuln_family=norm_vc(raw)`。
- Cell identity 升至 3，ProjectState 升至 schema 3。schema 2 cell 保留旧状态与证据并转为 `stale_requires_retest`，不得自动闭合 v3 cell；Finding registry 与物理证据不删除。

### 3.3 直觉探索合同

新增 `intuition-exploration.json`：

- `status=completed`，至少一个 direction；每个 direction 有稳定 ID、理由、实际请求/响应 evidence refs。
- evidence 必须位于 Run 内、非软链接、非空并进入 validation artifact hashes。
- `_conclude()` 在 LOW_ROI 时缺失/无效则降为 incomplete；shared finalizer 再次验证，防 library 旁路。
- 有 confirmed Finding 的 Run 不以此作为 proof 成立条件；该合同只约束 LOW_ROI。

### 3.4 跨阶段输入阴性重测

- ProjectState 的 input/injection 类 `not_vulnerable` 不再自动继承为终态，而是恢复成 `shallow_negative`，携带 prior vectors/encoding families/strategy families。
- negative evidence packet 派生并记录 `encoding_families` 与 `strategy_families`；可来自显式 packet 字段或稳定 vector label。
- 有 prior negative 时，当前 Run 必须至少出现一个新的编码族和一个新的策略族；旧记录无族元数据时要求当前至少两个编码族与两个策略族。
- final closure gate 重算上述差异；模型文字或 `depth_sufficient=true` 不能绕过。

### 3.5 独立敏感检测器

- `redact_text()` 保持替换职责。
- `sensitive_kinds()` 改用独立 `_DETECT_*` 规则，额外覆盖空白/引号/URL query/JWT 等变体；不调用 redactor。
- 检测到但未被 redactor 覆盖的内容会导致 planning/report/submission fail closed，形成安全冗余。

### 3.6 Blocker 口径

- `captcha_required` 改为 recoverable，返回至少五类绕过检查任务。
- 新增 `captcha_bypass_exhausted`，只有结构化记录至少五个不同方向后才为 NEEDS_INPUT。
- `challenge_unsolved` 默认映射 recoverable；没有 attempt evidence 不得升级。

### 3.7 其余 fail-closed 修复

- `finalize_run(authority_trusted=False)` 默认安全；只有 Engine/Wrapper 的显式调用可传 true。
- `_target_allowed()` 在 context 与 authorized_hosts 都缺失时返回 false；测试/调用方必须提供 scope。
- `canonical_surface_key()` 默认 method 为空；裸路径返回空。`CellIdentity` 拒绝空/非法 method；ledger/scheduler/candidate 不再补 GET，unknown 进入 unresolved。
- 删除 `_map_finding_for_summary()`。
- 增加 proof 文件替换后 `verify_validation_artifact()` 失败的回归，作为 P1-6 的确定性证据。

## 4. 非目标与硬边界

- 不伪造 macOS 上不存在的 cgroup/job/container attestation。
- 不让 diagnostic prior agenda 直接改 ProjectState。
- 不把历史 schema 2 聚合 cell 猜分为具体漏洞类。
- 不用 oracle/靶场特判补 coverage。
- 不因兼容旧裸路径而继续猜 GET；迁移必须显式补 method。

## 5. 发布门

1. P0/P1/P2/P3 对应失败先行回归全部通过。
2. 全量 pytest、self-check、strict doctor、compileall、diff-check 通过。
3. schema 2 → 3 迁移不删除 Finding/证据，但旧聚合 cell 不可闭合 v3 兄弟类。
4. prior agenda tamper、跨 origin、unknown method、计数/hash 不一致全部拒绝。
5. LOW_ROI 无直觉探索证据被降级。
6. 后续 input negative 未使用新 encoding/strategy 时 closure 失败。
7. 无授权 scope 的 standalone Finding 验证失败。
8. 当前 backend 最终仍报告 `authority_trusted=false`；任何测试不得断言本补丁建立了强 authority。
