# Atoolkit v8.4 迭代方案：Cairn 式 Fact-Intent 架构

**基线版本**: v8.3 (分阶段精确打击)
**目标版本**: v8.4.0
**迭代哲学**: v8.3 解决了"广度与精度的分离"，v8.4 解决"发现的串联"——借鉴 Cairn 的 Fact-Intent 架构，让每一个已确认发现自动生成下游探索方向，将链式利用从"模型自觉"升级为"系统驱动"。

---

## 一、回归分析：v8.3 遗留问题

### 1.1 v8.3 解决了什么

| 问题 | v8.3 解法 | 效果 |
|------|----------|------|
| 并行 agent 上下文碎片化 | Phase 1 (2并行) + Phase 2 (1精度) | 输入验证精度恢复 |
| 输入验证被 IDOR 挤占 token | Phase 2 专属 input_precision agent | 深度回归 v8.2 水平 |
| 跨 phase 上下文丢失 | blackboard.json 黑板聚合 | Phase 2 继承 Phase 1 上下文 |
| 缺测试账号 | Pre-flight 强制 5 角色检查 | IDOR 双向覆盖完整 |

### 1.2 v8.3 仍未解决的问题

| 问题 | 根因 | 影响 |
|------|------|------|
| **链式利用不稳定** | chain_assessment 是 finding 的条件字段，模型经常跳过或敷衍 | 单点漏洞无法串联为高价值攻击链 |
| **跨 agent 发现隔离** | Phase 1 的两个 agent 互不知晓对方发现，直到过渡阶段才聚合 | auth_agent 发现的泄露信息无法实时指导 txn_agent 的测试方向 |
| **发现→下一步方向 断裂** | 确认一个漏洞后，"还能挖什么"完全依赖模型记忆和推理 | 模型经常"找到一个洞就停"，不会主动顺藤摸瓜 |
| **Intuition Phase 缺乏方向** | Phase 3 直觉探索完全自由，无结构化引导 | 有时发现高价值链，有时完全浪费 token |

### 1.3 核心矛盾（一句话）

> **v8.3 解决了"怎么测"的阶段分离问题，但没解决"测完一个之后该测什么"的方向推导问题。Cairn 的 Fact→Intent 机制正是填补这个缺口。**

## 二、Cairn 架构回顾与适配

### 2.1 Cairn 核心概念

Cairn（github.com/oritera/Cairn）在腾讯云黑客松 610 队中拿到唯一 AK（全通）。其架构精髓：

| 概念 | Cairn 定义 | Atoolkit v8.3 对应 | v8.4 升级方向 |
|------|-----------|-------------------|-------------|
| **Blackboard** | 共享知识存储，所有 agent 通过它协调 | `blackboard.json` (v8.3已有) | **升级为 Fact-Intent Graph** |
| **Fact** | 已确认的发现（验证过的知识） | confirmed findings | **结构化 + 增加 Intent 生成钩子** |
| **Intent** | 声明的探索方向（未知的步骤） | 不存在 | **新增：从 Fact 动态生成** |
| **Stigmergy** | agent 通过 Blackboard 间接协调 | Phase 过渡时聚合 | **增强：Intent 驱动的实时调度** |
| **OODA Loop** | Observe → Orient → Decide → Act | per-surface loop (隐式) | **显式化：每轮循环后触发 Intent 生成** |

### 2.2 Cairn 的适配原则

**不完全照搬 Cairn，做以下适配**：

| Cairn 原版 | Atoolkit 适配 | 理由 |
|-----------|-------------|------|
| Server 进程管理 Graph | Graph 作为内存+磁盘结构，由 orchestrator 管理 | Atoolkit 是单次会话工具，非持续服务 |
| Worker Container (Docker) | QoderWork agent / Codex agent | 运行环境不同 |
| Intent 由 LLM 自由生成 | Intent 由**规则引擎 + LLM 补充**混合生成 | 纯 LLM 生成不稳定，规则保底 |
| Dispatcher 调度 Worker | Orchestrator 按优先级分发 Intent | 保持现有架构 |
| 多 Worker 并行执行 Intent | Phase 内并行 + Phase 间串行 | 保持 v8.3 的分阶段设计 |

### 2.3 设计哲学

> **"别教模型怎么打洞，告诉它哪里可能有洞"** ——Intent 系统的使命不是规定具体攻击手法，而是确保模型不会遗漏"顺藤摸瓜"的方向。Intent 是**路标**，不是**路线**。

## 三、v8.4 架构设计

### 3.1 全局架构（v8.3 + Intent 层）

```
Phase 1: 广度并行扫描 (2 agents)
  txn_idor agent              auth_admin agent
  每确认一个发现              每确认一个发现
  → 写 Fact                   → 写 Fact
  → 生成 Intent               → 生成 Intent
      │                             │
      ▼                             ▼
  state/txn_idor/              state/auth_admin/
  fact_intent.json             fact_intent.json
                  │
                  ▼
        Phase 过渡:
        · 缝隙检查
        · 黑板聚合
        · Intent 合并去重       ← v8.4 新增
        · Intent 优先级排序     ← v8.4 新增
                  │
                  ▼
Phase 2: 精度单 agent (input_precision)
  工作队列 = 遗留 Intent(Phase 1) + 新 Surface   ← 变化
  每完成一个 Intent → 可能生成新 Intent            ← v8.4
                  │
                  ▼
Phase 3: 直觉探索 (§8.5)
  · 未解决的 Intent 作为直觉探索的起点             ← v8.4
  · 自由探索产生的新发现也进入 Fact→Intent 循环    ← v8.4
```

### 3.2 核心数据模型：Fact-Intent Graph

```python
# engine/graph.py (新增)

from dataclasses import dataclass, field, asdict
from datetime import datetime, timezone
from enum import Enum

    RECON = "recon"                # 侦察扩展

    CONFIRMED_VULN = "confirmed"

    INFO_DISCLOSURE = "info_disclosure"

@dataclass
class Fact:

class Intent:
    intent_id: str

    source: str
    description: str
    vuln_class: str = ""
    target_endpoint: str = ""

    status: str = "pending"
    assigned_phase: int = 2
    assigned_agent: str = ""
    created_at: str = field(default_factory=lambda: datetime.now(timezone.utc).isoformat())
    resolved_at: str = ""
    outcome_summary: str = ""

# 注：Fact/Intent dataclass 定义字段 schema（供文档参考和 IDE 提示），
# FactIntentGraph 内部使用 dict 以兼容 JSON 序列化。
# 字段名和类型与 dataclass 保持一一对应。


class FactIntentGraph:
    """Fact-Intent 图：管理发现与探索方向的有向图。"""

    def __init__(self):
        self.facts: list[dict] = []
        self.intents: list[dict] = []
        self._next_fact_id = 1
        self._next_intent_id = 1

    def add_fact(self, fact_data: dict) -> tuple:
        fact_data.setdefault("fact_id", f"fact_{self._next_fact_id:03d}")
        self._next_fact_id += 1
        self.facts.append(fact_data)
        new_intents = IntentRuleEngine.generate_intents(fact_data, self)
        for intent in new_intents:
            intent["intent_id"] = f"intent_{self._next_intent_id:03d}"
            self._next_intent_id += 1
            self.intents.append(intent)
        return fact_data, new_intents

    def fact_from_candidate(self, candidate: dict, fact_type: str = "confirmed") -> dict:
        chain = candidate.get("chain_assessment") or {}
        return {
            "source_type": fact_type,
            "source_candidate_id": candidate.get("candidate_id", ""),
            "endpoint": candidate.get("endpoint", ""),
            "method": candidate.get("method", "GET"),
            "params": [candidate.get("param", "")] if candidate.get("param") else [],
            "vuln_class": candidate.get("vuln_class", ""),
            "summary": candidate.get("hypothesis", ""),
            "evidence_refs": candidate.get("evidence_refs", []),
            "chain_feasible": chain.get("chain_feasible", False),
            "chain_path": chain.get("chain_path", ""),
            "chain_final_impact": chain.get("final_impact", ""),
        }

    def get_pending_intents(self, *, agent: str = "", phase: int = 0,
                            priority: str = "", limit: int = 10) -> list[dict]:
        pending = [
            i for i in self.intents
            if i.get("status") == "pending"
            and (not agent or i.get("assigned_agent") in (agent, "any", ""))
            and (not phase or i.get("assigned_phase", 0) <= phase)
            and (not priority or i.get("priority") == priority)
        ]
        priority_order = {"high": 0, "medium": 1, "low": 2}
        pending.sort(key=lambda x: priority_order.get(x.get("priority", "low"), 9))
        return pending[:limit]

    def resolve_intent(self, intent_id, status, summary="", spawned_facts=None):
        for intent in self.intents:
            if intent.get("intent_id") == intent_id:
                intent["status"] = status
                intent["outcome_summary"] = summary
                intent["spawned_facts"] = spawned_facts or []
                intent["resolved_at"] = datetime.now(timezone.utc).isoformat()
                return intent
        return None

    def merge_graph(self, other):
        id_map = {}
        for fact in other.facts:
            old_id = fact["fact_id"]
            fact_copy = dict(fact)
            fact_copy["fact_id"] = f"fact_{self._next_fact_id:03d}"
            self._next_fact_id += 1
            id_map[old_id] = fact_copy["fact_id"]
            self.facts.append(fact_copy)
        for intent in other.intents:
            intent_copy = dict(intent)
            intent_copy["intent_id"] = f"intent_{self._next_intent_id:03d}"
            self._next_intent_id += 1
            old_src = intent_copy.get("source_fact_id", "")
            if old_src in id_map:
                intent_copy["source_fact_id"] = id_map[old_src]
            self.intents.append(intent_copy)

    def stats(self) -> dict:
        intent_by_status = {}
        for i in self.intents:
            s = i.get("status", "pending")
            intent_by_status[s] = intent_by_status.get(s, 0) + 1
        return {
            "total_facts": len(self.facts),
            "total_intents": len(self.intents),
            "intents_by_status": intent_by_status,
            "chain_links": sum(1 for f in self.facts if f.get("chain_feasible")),
            "high_priority_pending": sum(
                1 for i in self.intents
                if i.get("status") == "pending" and i.get("priority") == "high"
            ),
        }


def merge_agent_graphs(agent_graphs: list) -> "FactIntentGraph":
    """Phase 过渡时合并多个 agent 的 Graph，Intent 去重。"""
    merged = FactIntentGraph()
    for g in agent_graphs:
        merged.merge_graph(g)
    # Intent 去重：同一 source_fact + target_endpoint + vuln_class 只保留最高优先级
    # 注意：必须包含 source_fact_id，否则不同 Fact 为同一端点生成的 Intent 会被错误合并
    # （例：refund.php 和 coupon.php 都生成 business-logic-chain，target_endpoint="" 会碰撞）
    seen = {}
    deduped = []
    prio = {"high": 0, "medium": 1, "low": 2}
    for intent in merged.intents:
        key = (intent.get("source_fact_id"), intent.get("target_endpoint"),
               intent.get("vuln_class"))
        if key in seen:
            existing = seen[key]
            if prio.get(intent.get("priority"), 9) < prio.get(existing.get("priority"), 9):
                deduped = [i for i in deduped
                           if i.get("intent_id") != existing.get("intent_id")]
                deduped.append(intent)
                seen[key] = intent
        else:
            seen[key] = intent
            deduped.append(intent)
    merged.intents = deduped
    return merged


def intent_work_queue(graph, remaining_surfaces, phase=2):
    """构建统一工作队列：Intent + Surface，按优先级排序。"""
    queue = []
    for intent in graph.get_pending_intents(phase=phase, limit=30):
        tier = 0 if intent.get("priority") == "high" else (
               2 if intent.get("priority") == "medium" else 4)
        queue.append({"type": "intent", "data": intent, "sort_key": (tier,)})
    for surface in remaining_surfaces:
        is_high = any(t in ("auth-flow", "amount-tamper", "idor", "payment")
                      for t in surface.get("risk_tags", []))
        queue.append({"type": "surface", "data": surface,
                      "sort_key": (1 if is_high else 3,)})
    queue.sort(key=lambda x: x["sort_key"])
    return queue
```

### 3.3 Intent 生成规则引擎（核心创新）

Intent 生成的设计原则：**规则引擎负责"方向生成"，模型负责"具体执行"**。规则是确定性的——给定相同的 Fact，任何模型都会生成相同的 Intent 方向。但如何测试这些方向，仍由模型自主决定。

```python
class IntentRuleEngine:
    """确定性 Intent 生成规则引擎。"""

    MAX_INTENTS_PER_FACT = 5  # 每个 Fact 最多衍生 5 个 Intent

    @staticmethod
    def generate_intents(fact, graph):
        intents = []
        for rule in IntentRuleEngine._rules():
            if rule["condition"](fact, graph):
                intent = rule["generate"](fact)
                if intent:
                    intents.append(intent)
        # 去重 + 上限
        seen = set()
        deduped = []
        for i in intents:
            key = (i.get("target_endpoint"), tuple(i.get("target_params", [])),
                   i.get("vuln_class"))
            if key not in seen:
                seen.add(key)
                deduped.append(i)
        return deduped[:IntentRuleEngine.MAX_INTENTS_PER_FACT]

    @staticmethod
    def _rules():
        return [
            # 规则 1：认证组件弱点 → 链式利用
            # 触发：confirmed 认证类 + chain_feasible=true
            # 示例：captcha 不消耗 → 暴力破解 SMS 码 → 密码重置
            {
                "name": "auth_chain",
                "condition": lambda f, g: (
                    f.get("source_type") == "confirmed"
                    and f.get("vuln_class") in (
                        "auth-bypass", "captcha-bypass", "auth-flow-abuse",
                        "验证码绕过", "认证绕过")
                    and f.get("chain_feasible")),
                "generate": lambda f: {
                    "source_fact_id": f["fact_id"],
                    "source": "chain",
                    "description": f"链式利用：{f.get('chain_path', f.get('summary', ''))}",
                    "vuln_class": "auth-bypass-chain",
                    "priority": "high", "assigned_phase": 2,
                    "assigned_agent": "input_precision",
                },
            },
            # 规则 2：信息泄露 → 凭证利用
            # 触发：泄露了 sign/key/token/secret
            # 示例：recharge-create 泄露 callback_sign → 伪造充值回调
            {
                "name": "info_leak_credential",
                "condition": lambda f, g: (
                    f.get("source_type") in ("confirmed", "info_disclosure")
                    and any(kw in (f.get("summary", "") + f.get("vuln_class", "")).lower()
                            for kw in ("泄露", "leak", "disclosure", "sign", "key",
                                       "secret", "token", "credential", "密码", "密钥"))),
                "generate": lambda f: {
                    "source_fact_id": f["fact_id"],
                    "source": "escalation",
                    "description": (
                        f"利用泄露凭证尝试伪造请求或提权"
                        f"（来源: {f.get('endpoint', '?')}，"
                        f"寻找使用该凭证的下游端点）"),
                    "vuln_class": "privilege-escalation",
                    "target_endpoint": "",  # agent 需自行发现使用凭证的端点
                    "priority": "high", "assigned_phase": 2,
                    "assigned_agent": "input_precision",
                },
            },
            # 规则 3：SQLi confirmed → 数据提取升级
            # 示例：search.php SQLi → 提取 users 表密码哈希
            {
                "name": "sqli_extraction",
                "condition": lambda f, g: (
                    f.get("source_type") == "confirmed"
                    and f.get("vuln_class", "").lower() in (
                        "sqli", "sql-injection", "sql注入")),
                "generate": lambda f: {
                    "source_fact_id": f["fact_id"],
                    "source": "escalation",
                    "description": "SQLi 数据提取：尝试读取敏感表/列",
                    "vuln_class": "sqli",
                    "target_endpoint": f.get("endpoint", ""),
                    "target_params": f.get("params", []),
                    "priority": "high", "assigned_phase": 2,
                    "assigned_agent": "input_precision",
                },
            },
            # 规则 4：跨端点交叉测试
            # 触发：端点有 ≥2 种不同类型参数
            # 示例：submit-audit 有 status+description → 交叉测

                "name": "cross_param_type",
                "condition": lambda f, g: (
                    f.get("source_type") == "confirmed"
                    and len(f.get("params", [])) >= 2),
                "generate": lambda f: {

                    "source": "cross",
```

### 3.4 Intent 生命周期

```
  Fact confirmed
       │
       ▼ 自动生成
  Intent (pending)
       │
       ▼ agent 取出执行
  Intent (in_progress)
       │
       ├──→ completed (产生新 Fact → 可能产生新 Intent)
       ├──→ abandoned (无价值，记录原因)
       └──→ blocked (前置条件不满足)
              │
              ▼ 条件满足后
           pending (重新排队)
```

**Intent 上限控制**（防止爆炸）：

```python
def run_session(...):
```

### 4.1 SKILL.md 变更

**替换 Phased Precision Protocol 为 Fact-Intent Protocol**：

```markdown
## Fact-Intent Protocol

Testing follows a discovery-driven loop: every confirmed finding automatically
generates exploration directions (Intents) that guide subsequent testing.

### Intent Generation (after each confirmed finding)

When a finding is confirmed, generate 1-3 Intents based on the finding type:

| Finding type | Auto-generated Intent direction |
|---|---|
| Auth component weakness (chain_feasible) | Chain exploitation: end-to-end attack |
| Info disclosure (keys/signs/tokens) | Credential use: forge requests or escalate |
| SQLi confirmed | Data extraction: read sensitive tables |
| Multi-param endpoint confirmed | Cross-param: test other param types |
| WAF-blocked negative | Bypass retry: encoding variants |
| Business logic (payment/refund/points) | Fund chain: construct full attack chain |
| IDOR confirmed | Impact escalation: batch access / broader scope |
| SSRF confirmed | Internal probe: metadata / internal services |

### Intent Lifecycle

- **pending**: Generated but not yet tested
- **in_progress**: Currently being tested
- **completed**: Testing done, may have spawned new Facts
- **abandoned**: Tested but no value found (record reason)

Limits: max 5 Intents/Fact, max 30 pending globally, max 3 chain depth.

### Phase 2 Work Queue

Phase 2 merges two sources:
1. Inherited Intents from Phase 1 (from `fact_intent.json`)
2. Remaining surfaces not yet tested

Priority: high Intents > untested surfaces > medium Intents > low Intents.

### Recording

Write to `state/<agent>/fact_intent.json`. Update status after testing.
New findings create new Facts that may spawn further Intents.
```

## 九、项目级 Blackboard（跨 run 持久化）

### 9.1 问题场景

v8.4 的 Fact-Intent Graph 解决了单次 run 内的发现串联，但对大厂 SRC（需要 5-10 个 run 才能覆盖完整业务线），每次 run 仍然从零开始——重复侦察、重复排除已知死胡同、重复推导已明确的攻击方向。

**典型浪费场景**：
- Run 1 确认了"退款无上限"，Run 2 不知道这个事实，又花时间重新测试退款端点
- Run 1 排除了 search.php SQLi（5 族 payload 全阴性），Run 2 不知道这个结论，又重新测一遍
- Run 1 发现了充值签名泄露但没来得及测伪造回调，Run 2 不知道这个 pending 方向

### 9.2 目录结构

```
runs/
└── {target_project}/              # 项目级（如 "heasec-shop"）
    ├── blackboard.json             # 累积知识（跨 run 持久化）

    ├── run_scope.json              # 本轮攻击域声明（每次 run 更新）

        │   ├── findings/
```

- `runs/{target}/sessions/run_NNN/` = 现有的 run 目录结构，完全不变

### 9.5 Run 启动协议（Run Startup Protocol）

每次 run 启动时，agent 执行以下序列：

```
Step 1: 读取项目级 Blackboard
  → 如果 runs/{target}/blackboard.json 不存在 → 首次 run，跳过 Step 2-4

Step 2: 构建初始状态
  → facts[] 中 source_type=confirmed → 标记为已知事实，跳过测试
  → negatives[] 中 depth_sufficient=true → 标记为 not_vulnerable，跳过
  → dead_ends[] → 标记为 not_vulnerable + 附排除证据，跳过
  → intents[] 中 status=pending → 加入本轮待执行 Intent 队列

Step 3: 读取本轮域声明 (run_scope.json)
  → 确定本轮聚焦的业务域（如 domain: "txn"）
  → 从 business_graph.json 过滤属于该域的端点
  → 与 Step 2 的跳过列表做差集 → 得到本轮实际测试 surface 清单

Step 4: 初始化 v8.4 Graph
  → 将 blackboard 的 pending intents 导入 FactIntentGraph
  → 将 blackboard 的 facts 作为 pre-existing facts 加载
  → 后续 run 内测试基于此 Graph 继续

Step 5: 进入 Phase 0 侦察（仅补充发现）
  → 对比 business_graph.json 的已知端点和新发现的端点
  → 新端点 → 加入 discovered_endpoints + 分配 domain
```

### 9.6 Run 结束协议（Run Shutdown Protocol）

每次 run 结束时，agent 执行以下序列：

```
Step 1: 收集本轮产出
  → 从 fact_intent.json 提取本轮新增的 facts 和 intents
  → 从 findings/ 提取本轮确认的漏洞
  → 从 negative_findings.md 提取本轮的阴性结论

Step 2: 合并到 Blackboard
  → 新 confirmed facts → 追加到 blackboard.facts[]（source_run = 本次 run ID）
  → 新 negatives → 合并到 blackboard.negatives[]
     · 已存在的 surface_key → 比较 vectors_tried，保留更深的记录
     · 新的 surface_key → 追加
  → 新 dead_ends（充分测试后 refuted）→ 追加到 blackboard.dead_ends[]
  → 新 intents → 合并到 blackboard.intents[]
     · 已完成的 → status=completed
     · 未完成的 → status=pending（留给下次 run）
     · 新产生的 → status=pending
  → 新端点 → 合并到 blackboard.discovered_endpoints[]

Step 3: 更新域覆盖统计
  → 计算每个 domain 的 surfaces_tested / surfaces_total
  → 更新 domains_covered 中各域的 status 和 runs

Step 4: 离线 Intent 推导
  → 对本轮所有新增 confirmed facts 运行 IntentRuleEngine
  → 新生成的 Intent 如果本轮未执行 → 写入 blackboard.intents[] 的 pending 列表
  → 对 blackboard 中所有 pending intents 做优先级重排
     · 优先级公式: vuln_class 价值档 × (1 + 关联 facts 数) × 衰减因子
     · 衰减因子: 每经过一个 run 未执行 × 0.9（防止老 Intent 永远占位）

Step 5: 保存 Blackboard
  → 写入 runs/{target}/blackboard.json
  → 生成 run_summary.md（本轮新增/更新/跳过的统计）
```

```python
        - confirmed facts 作为 pre-existing（跳过测试）
        - pending intents 加入工作队列
        - dead_ends 和 depth_sufficient negatives 作为跳过标记
        """
        # 导入历史 facts（不按域过滤——事实是跨域知识，所有域都应继承）
        # 例如：auth 域发现的 token 泄露可能被 txn 域的 Intent 利用
        for fact in blackboard.get("facts", []):
            self.facts.append({
                **fact,
                "_pre_existing": True,  # 标记为历史，不重复测试
            })

        # 导入 pending intents
        for intent in blackboard.get("intents", []):
            if intent.get("status") == "pending":
                if domain and intent.get("assigned_domain", "") != domain:
                    continue
                self.intents.append(intent)

        # 返回跳过列表（供 CoverageLedger 初始化）
        skip_surfaces = []
        for neg in blackboard.get("negatives", []):
            if neg.get("depth_sufficient"):
                skip_surfaces.append({
                    "endpoint": neg["endpoint"],
                    "param": neg.get("param", ""),
                    "vuln_class": neg.get("vuln_class", ""),
                    "status": "not_vulnerable",
                    "reason": f"排除于 {neg.get('deepest_run', '?')}",
                })
        for de in blackboard.get("dead_ends", []):
            skip_surfaces.append({
                "endpoint": de["endpoint"],
                "param": de.get("param", ""),
                "status": "not_vulnerable",
                "reason": f"死胡同: {de.get('refutation', '')}",

def merge_run_to_blackboard(blackboard_path: str, run_graph: FactIntentGraph,
                            run_id: str, run_negatives: list[dict] = None):
    """Run 结束时合并 run 产出到项目级 blackboard。"""
    import json, pathlib
    bb_path = pathlib.Path(blackboard_path)
    bb = json.loads(bb_path.read_text(encoding="utf-8")) if bb_path.exists() else {
        "schema_version": "1.0", "facts": [], "intents": [],
        "negatives": [], "dead_ends": [], "discovered_endpoints": [],
    }

    # 合并 facts
    existing_fact_keys = {
        (f.get("endpoint"), f.get("vuln_class"))
        for f in bb.get("facts", [])
    }
    for fact in run_graph.facts:
        if fact.get("source_type") != "confirmed":
            continue
        key = (fact.get("endpoint"), fact.get("vuln_class"))
        if key not in existing_fact_keys:
            fact_copy = dict(fact)
            fact_copy["fact_id"] = f"bb_fact_{len(bb['facts']) + 1:03d}"
            fact_copy["source_run"] = run_id
            bb["facts"].append(fact_copy)

    # 合并 intents（更新状态）
    existing_intent_keys = {
        (i.get("source_fact_id"), i.get("description"))
        for i in bb.get("intents", [])
    }
    for intent in run_graph.intents:
        key = (intent.get("source_fact_id"), intent.get("description"))
        if key in existing_intent_keys:
            # 更新已有 intent 的状态
            for existing in bb["intents"]:
                if (existing.get("source_fact_id") == intent.get("source_fact_id")
                        and existing.get("description") == intent.get("description")):
                    existing["status"] = intent.get("status", existing.get("status"))
                    existing["outcome_summary"] = intent.get("outcome_summary", "")
                    break
        else:
            intent_copy = dict(intent)
            intent_copy["intent_id"] = f"bb_intent_{len(bb['intents']) + 1:03d}"
            intent_copy["created_in_run"] = run_id
            bb["intents"].append(intent_copy)

    # 合并 negatives
    for neg in (run_negatives or []):
        surface_key = neg.get("surface_key", "")
        existing_neg = next(
            (n for n in bb.get("negatives", []) if n.get("surface_key") == surface_key),
            None,
        )
        if existing_neg:
            # 保留更深的记录
            if neg.get("vectors_tried", 0) > existing_neg.get("vectors_tried", 0):
                existing_neg.update(neg)
                existing_neg["deepest_run"] = run_id
        else:
            neg_copy = dict(neg)
            neg_copy["deepest_run"] = run_id
            bb.setdefault("negatives", []).append(neg_copy)

    # Intent 衰减：对本轮未执行的 pending intents 应用衰减因子
    # 衰减公式: effective_priority = base_priority × 0.9^(runs_without_execution)
    # 阈值 0.5: low 约 7 run / medium 约 13 run / high 约 17 run 后自动 abandoned
    DECAY_FACTOR = 0.9
    ABANDON_THRESHOLD = 0.5  # priority_score < 此值则 abandoned
    PRIO_BASE = {"high": 3.0, "medium": 2.0, "low": 1.0}
    run_ids_this_session = {run_id}  # 本轮执行过的 intent 不算衰减
    for intent in bb.get("intents", []):
        if intent.get("status") != "pending":
            continue
        if intent.get("created_in_run", "") == run_id:
            continue  # 本轮新产生的 intent 不衰减
        runs_since = bb.get("total_runs", 0) - int(
            intent.get("created_in_run", "run_000").replace("run_", "") or 0)
        if runs_since < 1:
            runs_since = 1
        base_score = PRIO_BASE.get(intent.get("priority", "low"), 1.0)
        effective = base_score * (DECAY_FACTOR ** runs_since)
        if effective < ABANDON_THRESHOLD:
            intent["status"] = "abandoned"
            intent["outcome_summary"] = f"衰减放弃: {runs_since} run 未执行, effective={effective:.2f}"

    # 更新元数据
    bb["total_runs"] = bb.get("total_runs", 0) + 1
    bb["last_run"] = run_id
    bb["last_updated"] = datetime.now(timezone.utc).isoformat()

    bb_path.parent.mkdir(parents=True, exist_ok=True)
    bb_path.write_text(json.dumps(bb, ensure_ascii=False, indent=2), encoding="utf-8")
    return bb
```

## 十、攻击面分域（大规模 SRC 适配）

### 10.1 问题场景

大厂 SRC 目标通常有 200-500+ 个端点，单次 run 的 context window 无法覆盖全部。如果不分域，agent 会面临两个困境：

1. **广度陷阱**：每个端点浅尝辄止，测了 200 个 surface 但每个只用了 2-3 个 payload
2. **上下文耗尽**：token 预算在侦察和低价值 surface 上用尽，高价值端点没有足够深度

v8.4 的 Fact-Intent 解决了"方向推导"问题，但没解决"每轮测试范围"问题。攻击面分域让每次 run 的攻击面有界——"本轮只测认证相关的 30 个端点"而非"测试所有 500 个端点"。

```python
def _has_keyword_match(text: str, keywords: list[str]) -> bool:
    """分隔符边界匹配：防止 'sort' 匹配 'resort'、'order' 匹配 'disorder'。"""
    import re
    for kw in keywords:
        # 检查关键词是否在分隔符边界（-, _, /, ., 字符串首尾）出现
        pattern = r'(?:^|[-_/.])' + re.escape(kw) + r'(?:$|[-_/.])'
        if re.search(pattern, text):
            return True
    return False


def classify_endpoint_domain(endpoint: str, params: list[str] = None,
                             risk_tags: list[str] = None) -> list[str]:
    """将一个端点分类到一个或多个域。

    返回匹配的域 ID 列表（一个端点可能属于多个域，
    如 submit-audit.php 同时属于 admin 和 input）。
    """
    ep_lower = endpoint.lower()
    params_lower = [p.lower() for p in (params or [])]
    tags_lower = [t.lower() for t in (risk_tags or [])]

    matched = []
    for domain_id, rules in DOMAIN_RULES.items():
        score = 0
        # 端点关键词匹配（分隔符边界）
        if _has_keyword_match(ep_lower, rules["endpoint_keywords"]):
            score += 2
        # 参数关键词匹配（精确或前缀）
        for kw in rules["param_keywords"]:
            if any(kw == p or p.startswith(kw + "_") for p in params_lower):
                score += 1
        # 风险标签匹配
        for tag in rules["risk_tags"]:
            if tag.lower() in tags_lower:
                score += 1
        if score > 0:
            matched.append((domain_id, score))

    if not matched:
        return ["info"]  # 默认归入 info 域
    # 返回得分最高的 1-2 个域
    matched.sort(key=lambda x: -x[1])
    return [m[0] for m in matched[:2]]


def filter_surfaces_by_domain(surfaces: list[dict],
                              target_domains: list[str]) -> list[dict]:
    """按域过滤 surface 清单。"""
    if not target_domains:
        return surfaces  # 不指定域 = 全量
    return [
        s for s in surfaces
        if any(
            d in target_domains
            for d in classify_endpoint_domain(
                s.get("endpoint", ""),
                s.get("params", []),
                s.get("risk_tags", []),
            )
        )
    ]
```

### 10.9 域间交叉处理

**问题**：一个端点可能属于多个域（如 `/api/merchant/submit-audit.php` 同时属于 `admin` 和 `input`）。如果 Run 2 聚焦 `input` 域，是否会测到 `admin` 域的端点？

**规则**：
1. **域分类允许多归属**：`classify_endpoint_domain()` 返回得分最高的 1-2 个域
2. **域过滤是"或"逻辑**：`target_domains=["input"]` 会包含所有被分类为 input 的端点，即使它们同时也属于 admin
3. **域边界是建议性的**：SKILL.md 明确说 "Domain scope is advisory, not a hard wall"——如果 Fact-Intent 链式推导跨域，agent 应该跟随
4. **跨域发现记录到 blackboard**：如果 Run 2（input 域）发现了 submit-audit 的管理权限问题，这个 fact 写入 blackboard 时 `assigned_domain` 标记为 "admin"，Run 3（admin 域）会看到

### 10.10 域覆盖统计与 run 规划建议

每次 run 结束后，blackboard 的 `domains_covered` 自动更新。下次 run 规划时：

```python
def suggest_next_domains(blackboard: dict, max_domains: int = 2) -> list[str]:
    """建议下次 run 应该聚焦的域。"""
    domains = blackboard.get("domains_covered", {})
    # 优先选 not_started 的域
    not_started = [d for d, info in domains.items() if info.get("status") == "not_started"]
    if not_started:
        return not_started[:max_domains]
    # 然后选 partial 且覆盖率最低的
    partial = [
        (d, info.get("surfaces_tested", 0) / max(info.get("surfaces_total", 1), 1))
        for d, info in domains.items()
        if info.get("status") == "partial"
    ]
    partial.sort(key=lambda x: x[1])
    return [d for d, _ in partial[:max_domains]]
```

这个函数可以在 run_summary.md 中输出建议：`"建议下次 run 聚焦: admin, file (当前覆盖率 0%)"`
