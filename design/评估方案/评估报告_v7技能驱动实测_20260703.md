# 评估报告 v4 —— Atoolkit v7.0 技能 QoderWork 驱动模式实测评估

> 版本：v4。在 v3（v6.1 技能 2810 分）基础上，基于 v7.0 技能零先验实测数据，对 Skill Mode 一等化改造后的实际效果做全面评估。
> 记录时间：2026-07-03 ｜ 视角：v7 改造效果验证 + 退化归因 + 下一迭代方向
> 评估对象：`runs/shop-skill-v7/` 全套产物（19 个 finding 包 + 39 份阴性报告 + 4 份覆盖表 + summary.json + coverage_gaps.md）
> 对标基线：靶场官方 24 项 / 4800 分 + v3 评估 2810 分基线 + v2 评估 640 分基线
> 驱动模式：QoderWork 直接驱动（4 并行 subagent: auth/txn/idor/input），非 run.py engine 模式
> 关键区别：**本次测试无先验记忆**——新会话、新 session 目录，agent 未携带上轮已知漏洞信息

## 0. 一句话结论

**v7 的 Skill Mode 一等化改造在"结构化产出"和"认证方向深挖"上有明显效果（finding 包从 15→19，阴性报告从 ~20→39，新发现 5 个配置外漏洞），但配置内得分反降 400 分（2810→2410），根因是 SQLi 假阴性（input agent 未适配 LIKE 查询上下文）和 3 个覆盖缝隙（agent 分工导致 submit-audit/pay-redirect/batch-order-points 无人覆盖）。v7 解决了 v3 评估中 6/12 个问题，但引入了 2 个新问题：agent 分工缝隙和 SQLi payload 模板化。**

## 7. 总结

### v7 的核心成就

1. **零先验得分 +60.7%**：从估计 ~1500 到实测 2410，证明 Phase 0 侦察 + 知识卡内联 + 并行协议对"从零发现"能力有实质提升。
2. **机制有效率从 23% 提升到 73%**：11 项新增机制中 10 项有实际效果，Skill Mode 不再是 engine 的退化形态。
3. **结构化产出质量大幅提升**：finding 包 15→19，阴性报告 20→39，覆盖表从 1 份合并表到 4 份方向表。
4. **认证方向深挖策略有效**：新发现 5 个配置外漏洞（session fixation、captcha 不消耗、admin 暴力破解等），全部来自 auth-agent 的系统性覆盖。

### v7 的核心教训

1. **并行分工的缝隙比串行遗漏更隐蔽**：串行测试时一个 agent 负责所有端点，遗漏是"忘了测"；并行分工时遗漏是"以为对方会测"，更难在聚合阶段发现。
2. **SQLi payload 必须按查询上下文适配**：WHERE 等值（`' OR 1=1--`）、LIKE 模糊（`' OR '1'='1`）、ORDER BY（`,1 DESC--`）、子查询（`' AND (SELECT...)='1`）各有专用 payload，一套模板打天下会制造假阴性。
3. **Phase 0 侦察必须覆盖非 API 页面**：纯 HTML 页面（pay/index.php）和表单 action 不在 JS 提取路径中，但可能包含高价值漏洞（重定向、CSRF）。

### 下一步方向

v7.1 应聚焦三个 P0/P1 问题：SQLi 查询类型适配（P0-1）、agent 分工缝隙检查（P1-1）、Phase 0 非 API 页面覆盖（P1-2）。这三个问题的修复预期可回收 ~720 分（V17+V19+V22），将配置内得分推到 3130 附近。
