# Atoolkit v8.8.0 修复方案独立审核

> 审核日期：2026-07-13
> 审核对象：`design/迭代方案/迭代方案v8.8_跨Run事实闭环与交付真值.md` 初稿
> 审核方式：Reporting/运行指纹与 Cross-Run/数据合同两条独立审核

## 1. 审核结论

**初稿不通过，重大修订后可实施。**

初稿对问题的定位正确，但存在以下阻断风险：

- 空 finding 仍可被 summary 间接授权为成功；
- manifest 只证明文件存在，不证明实际规则被注入；
- Finding `target` 强制绝对 URL 会破坏已有 `METHOD /path` 合同；
- summary/benchmark 尚未强制绑定 validator 产物与证据哈希；
- 把 inventory/cell/finding 与 Fact-Intent 全部塞入 Blackboard 3.0，存在多写入者、更新丢失与并发覆盖风险；
- cell 身份未纳入 role，可将某一角色的阴性错用到其他角色；
- `UNKNOWN` method 与现有 canonical key 合同冲突；
- finding 去重键过度依赖自由文本 security invariant，可同时造成过度合并和漏合并；
- 项目状态缺少单 writer、原子提交、并发和幂等合同；
- evidence ref 的相对路径基准未定义。

## 2. Reporting/运行指纹必改项

### R1. 空输入必须默认 fail closed

- 无 canonical finding 默认返回 `empty_input`，exit 2。
- 只有显式 `--allow-empty`，且 session gate=pass、Coverage 完整、无 proof-ready candidate、无 ingestion error 时可 exit 0。
- summary 是派生视图，不能授权空集成功。
- 退出码固定为 `0=成功 / 1=验证拒绝或冲突 / 2=空输入或前置合同缺失 / 3=IO/内部错误`。

### R2. Reporting 必须只有一条事实链

```text
finding.json + raw evidence + run_manifest
  -> Validator + Guardian
  -> finding_validation.json
  -> summary/final_report/benchmark
```

`finding_validation.json` 必须包含 manifest digest、规则/工具版本、discovery layout、finding/evidence SHA-256、拒绝原因、normalized root finding 和自身 digest。

- summary 只做确定性投影并引用 validation digest。
- benchmark 从 validation artifact 读取，或完整离线重验；不信任 summary 内可手写的 accepted/proof flags。
- Coverage 只做一致性核对，不得为 Finding 补 method/param/evidence。

### R3. 引入 ValidationContext，保留相对 target 合同

- `ValidationContext.primary_target` 是相对 `METHOD /path` 的唯一 base。
- `authorized_scopes` 只扩展授权范围，不可当 base。
- 第二授权域的相对 target 必须有 `scope_ref`，否则使用绝对 URL。
- 缺 manifest/primary target 时，相对 target fail closed。
- 同时校验 proof packet URL/Host 与 target/apis method/path 一致性。

### R4. Manifest 与 Receipt 分离

Manifest 覆盖 Engine/Skill Mode，记录：

- primary target 与 normalized scopes；
- authz hash（不存凭据）；
- git commit/dirty/tree digest；
- **实际注入**的 core skill/root AGENTS/prompt 路径和 hash；
- reporting/validator/fingerprint schema 版本；
- mode/model/session/project。

Receipt 在结束时绑定 validation/summary/coverage/registry delta hash。二者都是本地 provenance，不声称是密码学签名。

### R5. Codex 加载不能依赖外部 `/src`

- 项目根生成 `AGENTS.md`，`codex/AGENTS.md` 仅作兼容副本。
- `regen_agents.sh --check` 校验源文件、root AGENTS 和兼容副本无漂移。
- 提供只读 `doctor --json`，检测空 AGENTS、foreign `/src` symlink 和版本冲突。
- 遇到其他独立项目的 `/src` 只报 collision，绝不覆盖。
- Engine Mode manifest 只声明 `assemble_prompt()` 实际注入的规则。

### R6. Collector 有界扫描

- 精确扫 canonical 与两种已知 legacy 布局。
- suspicious scan 仅限 `findings/` 和 `evidence/` 的有界深度，不跟随 symlink。
- 路径必须位于 run root。
- 同 ID 异内容全部拒绝为 `duplicate_id_conflict`。
- malformed/oversize/unknown layout 进 ingestion error。
- canonical 和完全相同 legacy 副本由 canonical 胜出，但记录 shadow warning。

## 3. Cross-Run/数据合同必改项

### C1. 分离权威状态，不升级为单体 Blackboard 3.0

Blackboard 保持 2.x，只管 Facts/Intents/negative/dead-end 导航数据。新增独立文件：

```text
project_state/inventory.json
project_state/cell_registry.json
project_state/finding_registry.json
run_history/<sid>.json
```

`surface_index` 和 `domains_covered` 变为 cell registry 的派生视图，不再是第二真值源。

### C2. 项目状态只允许单 writer 原子提交

- 引入 project-state transaction helper：同一 project lock、读入后 merge、temp file + fsync + replace。
- 保留未知字段，支持幂等 run ID。
- 并发竞争或部分提交失败必须进 persistence error，不静默成功。

### C3. Cell 身份必须包含 role scope

项目级 cell 身份为：

```text
METHOD /path :: param @ role_scope × normalized_vuln_class
```

- `role_scope=unknown` 的 legacy 阴性不可闭合具体角色格。
- 多角色记录拆为精确 role cell，不用一个排序后列表当安全边界。
- confirmed 只 supersede 同身份 negative。

### C4. 无 method endpoint 进 unresolved inventory

- 不生成 `UNKNOWN` canonical key，也不伪造 GET。
- 保存到 `unresolved` 列表，生成 method-discovery Intent。
- 只有物理请求、HAR/源码 method 证据或用户输入确认后进 canonical inventory/Coverage。

### C5. Finding 去重保守化

`exact_fingerprint_v1` 包含：

- scope；
- normalized vuln family；
- METHOD + templated path；
- risk param/location；
- access boundary/affected role；
- stable claim profile。

不同 endpoint/param 默认不自动合并；只有显式 `root_cause_ref` 或人工 merge 才跨位置聚合。自由文本 invariant 只参与 possible-duplicate 提示，不作唯一键。

### C6. Evidence ref 和指纹基准

- session 证据在 registry 中使用 `session:<sid>/<relative-path>`。
- project artifact 使用 `project:<relative-path>`。
- 不保存机器绝对路径作为去重真值。
- 每个 ref 保存 SHA-256，路径只用于定位。

### C7. Confirmed 过期和冲突

- confirmed 默认保持已知阳性，不因普通 response body 波动自动 stale。
- 只有强 target/deployment fingerprint 变化或显式 revalidation Intent 可转 stale。
- 后续 refutation 形成 conflict 和复验 Intent，不静默删除历史阳性。

## 4. 修订后的范围决议

1. Blackboard 保持 2.x；不在 v8.8 强行统一为 3.0。
2. Skill Mode 本版本只 fail closed、doctor 和迁移提示，不自动改写 legacy finding。
3. 项目级 registries 与 Reporting 真值链是 P0；目标 fingerprint 自动过期只实现显式接口，不在本版本猜测部署版本。
4. `/src` 兼容别名和外部安装不是运行正确性前提。
5. 历史 summary 仅是视图，不反向填充 Facts/Cells/Findings。

## 5. 新增发布门

- 空目录默认 exit 2；只有完整门控的 `--allow-empty` 可 exit 0。
- 商城 12 个旧布局与 linglong 平铺 JSON 均能被发现为 legacy，不再是 0/0。
- relative target + manifest primary target 通过；只有 `--allow` 或缺 manifest 时失败。
- duplicate ID conflict、symlink escape、malformed/oversize finding 失败。
- validation 后修改 finding/evidence，digest 校验失败。
- 手写 summary accepted 不能增加 benchmark 分数。
- manifest 在 fake adapter 首次 `run()` 前存在，receipt 绑定结束产物。
- root AGENTS 漂移和 foreign `/src` 由 doctor 稳定报告，不覆盖用户环境。
- Run 1 动态 endpoint 在 Run 2 无 recon 可恢复。
- 同 method/path/param/vuln/role 的 confirmed 覆盖旧 negative，其他 role 不受影响。
- 已闭 endpoint 不抢占新 endpoint 的有限 budget。
- 同对象 ID 变体指纹一致，不同 param/method/role 不误合并。
- project state 重复提交幂等；并发两轮不丢 inventory/cell/finding observation。
- 本轮 Intent 产生 Fact 后在 conclude 为 completed。
- 全量 pytest、self-check、`regen_agents.sh --check` 通过。

## 6. 复审结论

上述必改项全部回写主方案、并将验收门转为自动化测试后，方案可从“有条件驳回”转为“批准实施”。
