# Atoolkit v8.13.0 动态执行闭环修复验收报告

日期：2026-07-17

## 1. 验收结论

v8.13.0 的代码、协议、Direct/Wrapper/Engine 三条运行链路及最终验证门已完成实现和离线回归，
达到“冻结威胁分母不变、实验过程动态收敛、文本声明不能闭格、阴性证据不足自动回退、发现进入
下一 Run”的设计目标。方案反向审核指出的动态分母、Event 自证、Finding/negative 旁路、证据复用、
Direct 越权升级、双状态真值和旧 Run 兼容问题均已处理。

本验收不宣称 v8.13.0 已在真实 SRC 上提升发现率：用户本轮未执行 SRC 任务，且 v8.12 靶场 Run
属于 `direct_skill_diagnostic`，没有运行 manifest、冻结 plan、威胁模型和权威 coverage ledger，不能
当作 v8.12 Engine 架构的端到端验收。v8.13 的真实增益仍需下一轮同靶场 A/B 与授权 SRC 实跑验证。

## 2. v8.12 靶场暴露的问题

- 严格得分 1720/4800（35.8%），命中 9/28；宽松口径 11/28（42.7%）。
- 4 个假阴性：用户枚举只看消息、reset token 只测假值/空值/缺失、balance records 空数据未预备、
  `use_points` 只测布尔值。
- 11 个未测面共 1940 分，包括注册验证码、找回密码、管理员登录、图片删除、发货、订单时间、
  删除商品 SQLi、3 个 XSS 和支付跳转。
- 运行后发现但未报告 merchant 页面授权与 `config.js` AK/SK；7/11 未测面其实已在侦察清单，说明
  问题不只是 discovery，而是“清单存在但执行器没有确定性消费”。另有 4 个动态出现端点未同步初始
  inventory，说明当前 Run 与下一 Run discovery 缺少明确边界。
- 10/10 `finding.json` 均未通过现行 proof validator，且报告触发 2 处 secret redaction；结论与可交付
  Finding 发生分叉。

## 3. 已实现的修复闭环

1. 每个 frozen threat cell 编译 `execution-contracts.json`，保留模型 `evidence_required`，并加入 Host
   最低实验义务。IDOR/BOLA 无条件强制双身份与 ownership marker。
2. 模型只能提交 exact-cell `EXECUTION_EVENT`；Host 校验 obligation、barrier、范围和物理证据，固化
   evidence SHA-256，再写 create-only Run event 与 authority hash chain。
3. Host reducer 每轮重算 progress/queue/backlog；缺义务的 `not_vulnerable` 回退为
   `shallow_negative`，confirmed Finding 不由 Event 建立也不被 Event 推翻。
4. 空数据、对象、会话、格式、身份挑战和 WAF 产生 recovery obligations；历史 barrier 只有在全部
   对应恢复义务有证据时才退出 active 集合。
5. rejected Finding 映射回 exact threat cell 并以 `proof_repair` 最高优先级重排；动态发现仅进入
   `next_run_required` backlog，不污染当前 frozen denominator。
6. Direct 新增 pre-network preflight，解决 fresh black-box “先网络侦察还是先 init”的矛盾；Direct
   继续保持 untrusted diagnostic。Wrapped Skill 在 agent 静默后由 Host 重新校验并晋升事件。
7. final validator 从 authority chain 重放事件、重算证据哈希和四个投影；存在 open contract、证据被
   替换、投影被改写或 ledger/plan 不一致时拒绝交付。旧版本 metadata 不启用 v8.13 gate。

## 4. 实现复核中追加修复

- 修复 barrier 只增不减导致恢复后永久无法闭格。
- 修复事件只绑定路径、不绑定证据字节导致接受后可替换文件。
- 修复 Host 依赖模型 identity mode，导致 IDOR 可能错误使用单身份最低深度。
- 修复 XSS/SSRF/文件义务使用“必须利用成功”的阳性措辞，导致安全端点理论上无法完成阴性实验。

- 全量自动化测试：`400 passed, 1 skipped`。
- 动态闭环定向测试：`34 passed`，覆盖假阴性模式、障碍恢复、证据篡改、事件幂等/冲突、authority
  哈希链、discovery backlog、proof repair、Direct preflight、Engine 阴性回退及最终投影重算。

- `python3 run.py --doctor`：仓库一致性与版本一致性通过，版本为 8.13.0；仅检测到仓库外既存的

- `bash codex/regen_agents.sh --check`、`compileall`、`git diff --check`：全部通过。

## 6. 下一轮实跑验收标准

同一靶场应使用可信 Engine Threat Mode 运行，并与 v8.12 结果逐项比较：4 个假阴性是否转为继续
执行或可靠阴性、11 个高价值未测面是否进入动态队列、动态 endpoint 是否全部进入 next-run backlog、
Finding 是否通过 Canonical validator。随后再做授权 SRC 任务，主要验收“垃圾风险不入报告、可提交
Finding 的 proof-ready 比率、每个高价值 exact cell 的 open 原因可解释”。
