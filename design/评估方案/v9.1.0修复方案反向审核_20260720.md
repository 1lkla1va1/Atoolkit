# Atoolkit v9.1.0 修复方案反向审核（2026-07-20）

## 1. 审核结论

**有条件批准。**批准 v9.1.0 的代码内闭环修复；拒绝将当前 backend/wrapper 标为 authority 完全体。发布表述只能是“跨 Run 诊断续航与安全门完整”，不能是“可信执行环境完成”。

## 2. 逐项反证

### P0-1

将 `_process_containment_verified()` 或 adapter 常量改为 true 会让 Agent 可逃逸的 POSIX process group 冒充强隔离，属于严重伪修复，拒绝。显式 `--continue-from-run` 只解决实际续航断链：它必须绑定输入、限制 scope、不得提交 ProjectState，并在输出中标 diagnostic。

### P0-2

完全保存任意原字符串会造成 `Stored XSS`、`stored-xss`、`存储型XSS` 三个重复格。批准“exact alias + semantic family”双轨；拒绝继续用 `norm_vc()` 作为 key，也拒绝无迁移地重解释旧 key。

### P1-3/P1-4

仅在 prompt 增加一句“已探索/换编码”不可验证。必须有机器结构、物理 evidence 和 closure/finalizer 双门。跨阶段规则不能只靠 scheduler 提示，final validator 必须重算向量族差异。

### P1-5

独立检测器必须在实现上不调用 `redact_text()`，否则仍是同源盲区。检测器可以比 redactor 更宽，误报方向应 fail closed。

### P1-6

审核原结论与代码不符：accepted proof 文件已进入 `artifact_hashes`，最终验证会重算。要求补“验证后替换 proof → artifact verification fail”测试；不要求 Finding 自报 hash，因为同一 Agent 可同时改 JSON 与文件，自报值不增加 authority。

### P1-7/P2

captcha 必须先 recoverable，耗尽证据后再 NEED_INPUT。finalizer 默认 true、无 scope 放行和 unknown→GET 均应直接改为 fail closed；兼容工作放在显式迁移层，不保留隐式默认。

## 3. 回归风险控制

- ProjectState schema 3 迁移必须保留 schema 2 原记录和 evidence hashes，只把其 closure 状态标 stale。
- `canonical_surface_key()` 变更会影响多模块；要求以 unknown-method regression 覆盖 surface_key、cell_identity、ledger、scheduler 和 orchestrator。
- intuition gate 只约束 LOW_ROI，不能让 confirmed Finding 因缺探索文件失效。
- diagnostic agenda 只改变本 Run 排序与分母候选，不能成为 confirmed/negative truth。
- 所有新 artifact 必须 no-follow、bounded、hash-bound。

## 4. 批准范围

批准实施设计 3.1–3.7；若无法在不放宽现有 authority/proof 门的前提下完成某项，则保持 fail closed 并在验收报告列为 external blocker，不得为“COMPLETE”修改安全定义。
