## 0. 一句话结论

**不是"没测出漏洞"，而是"覆盖闭环活在错误的层，且通用闭环机制本身也缺"。** 单点发现质量在线（5 个洞都有 PoC、有前后余额/并发计数证据），横向覆盖近乎裸奔（24 项命中 3 项）；而防漏的闸一半装在没在跑的 engine 里（模式 A 绕过了 orchestrator），一半压根还没建（coverage-ledger / session-gate / benchmark-eval）。技能包目前更像"高价值漏洞追踪助手"，还不是"可评估、可回归、可量化覆盖的通用 SRC 工作流"。`[并自他评]`

## 2. 头号根因：覆盖闭环活在错误的层 `[本评独有·他评未挖到]`

### 2.1 证据：本轮没有走 engine 的覆盖矩阵

`engine/orchestrator.py` 有一套相当完整的广度防漏机制（支柱 1+2）：

- `CognitiveState.seed_matrix(endpoints)` → 用「攻击面 × 漏洞类」铺满矩阵，每格四态 `untested/positive/negative/skipped`；
- `matrix_closed()` → 全格闭合才允许收口；`_conclude()` 里 `LOW_ROI` 若仍有 `open_risk_cells` 会被**降级为 `incomplete`**；
- `_knowledge_hint_for_state()` → 每轮按下一批未覆盖格匹配知识卡并注入提示；
- 每轮回灌认知状态对象、`next_untested()`、CELL 声明闭格……

**但本轮的 `state.json` 是这样的：**

```json
{"project_id":"localhost-shop","phase":2,"status":"discovery_complete",
 "last_active":"2026-06-30T23:36:56+0800","registration_page":"..."}
```

而 `CognitiveState.save()` 写出的 `state.json` 必然含 `sid/target/phase/hypotheses/verified/todo/evidence_files/directives/turn/last_progress_ts/vuln_classes/matrix`。**schema 完全对不上**——本轮的 state 不是 orchestrator 产的。

进一步：`pentest-data/localhost-shop/make_report.py` 是一个**硬编码**的报告生成器——`confirmed=[...]`（5 条写死）、`module_queue={"module_queue":[]}`（写死空）、`modules={...}`（5 个模块写死）、`coverage={...}`（连同 `blindspots` 一起写死）。`coverage.json / module_queue.json / summary.json / modules/*.json` 和最终 md 全是这份硬编码脚本吐出来的，不是从真实测试结果聚合的。`shop_probe.py / shop_attack.py` 则是模型自己临时写的探针脚本。

**结论：本轮是纯 skill 驱动（模式 A / Codex AGENTS.md 形态），模型读 `核心技能文件.v2.md` 后自由跑、自己写脚本、自己写报告。`run.py` + `orchestrator.py` 那条带矩阵与门控的路，根本没接上。**

### 2.2 后果：广度防漏机制 100% 空转

| 机制 | 所在位置 | 本轮是否生效 | 原因 |
|---|---|---|---|
| 覆盖矩阵（别漏格） | `orchestrator.py` | ❌ | 没走 orchestrator |
| 知识卡按格注入 | `orchestrator._knowledge_hint_for_state` | ❌ | 同上 |
| 认知状态每轮回灌 | `orchestrator.assemble_prompt` | ❌ | 同上 |
| `incomplete` 终态降级 | `orchestrator._conclude` | ❌ | 同上 |
| 20min 无进展切向 | `orchestrator` 注入指令 | ❌ | 同上 |
| 七问验证门 / 垃圾洞清单 | `核心技能文件.v2.md` | ✅ | 唯一生效的层 |
| 决策树 | `核心技能文件.v2.md` | ✅（但偏深度） | 见 R2 |

本轮只有"报告质量门（Guardian）"在岗，"作战完整性门（Session Gate）"集体缺勤。

### 2.3 知识卡写好了参数却没到模型手里

`knowledge/cards/business-logic-tamper.json` 的 `match.params` 明确列着：
`amount, price, refund_amount, use_points, discount, coupon, quantity, balance, stock, order_time, callback, recharge, lottery, probability`。

自评报告列出的"未测参数"——`use_points / refund_amount / order_time / lottery probabilities`——**和这张卡一字不差**。知识卡已经把答案写好了，但知识卡只在 `orchestrator.run_session` 里 `load_cards()` + 按格 `match_cards()` 注入；模式 A 没外壳，**这些卡从未到达模型**。v3.3 phase2"接通知识卡 live loop"的改动，对本轮等于零。

## 3. 通用根因（R1–R9，与靶场无关，任何目标都会犯）

自评报告症状描述准确，但头号修复"把 `vuln_config.php` 转 checklist.json 按分值逐项打"是**靶场专用**——真实 SRC 没有现成 checklist。下面把每条症状重构成通用根因，并融合他评的强化点。

### R1 · 覆盖管控装错了层（架构根因）`[本评独有]`
- 现象：矩阵/知识卡/门控全在 engine，skill 文件里没有等价物。
- 通用根因：模式 A（最常用的 skill 驱动路径）和模式 B（`run.py`+orchestrator）共用同一份 `核心技能文件.v2.md`，但只有模式 B 有外壳兜底。**执行的路径不携带闸，闸就形同虚设。**
- 任意目标、任何授权 SRC，只要走模式 A，都吃这个亏。

### R2 · 核心技能文件结构性偏向"深度优先"，且明示"不穷举" `[本评独有]`
- 现象：追充值回调/config 泄露/订单越权很深，横向 21 项几乎没碰。
- 通用根因：`核心技能文件.v2.md` §6 决策树开宗明义"动态分支，不是固定流程""不穷举"；唯一广度推力是"20min 无进展换面"。**全文没有一条规则要求"已发现的每个攻击面×相关漏洞类必须走到终态才能宣告完成"。** §3 速查卡"单格闭合后继续下一未覆盖格"只出现在 `orchestrator.assemble_prompt` 的 cheats 里（模式 B 专属），skill 文件本身没有。

### R3 · 状态阻塞没有变成行动 `[并自他评④·强化]`
- 现象：抽奖 `lottery_remaining=0` → 标 BLOCKED → 直接放弃（丢 200 分）。
- 通用根因：skill 文件有 `NEED_INPUT`（登录墙/验证码），但**没有"被账号状态阻塞该怎么办"的规则**。须把阻塞二分并各自落地：

| 阻塞类型 | 通用处理 |
|---|---|
| `quota_exhausted`（配额/次数耗尽） | 尝试重置数据、换账号、创建新对象 |
| `object_absent`（对象不存在） | 用 owner 账号创建测试对象 |
| `missing_role`（缺角色） | 要求补充账号或注册新号 |
| `captcha_required`（需人工验证码） | 请求人工或使用授权辅助工具 |
| `out_of_scope`（越界） | 停止，不绕过 |

- 可恢复阻塞（前三种）**不得进入最终盲区**，必须自动生成下一步行动；不可恢复阻塞才升级 `NEED_INPUT`。
- 真实 SRC 里"测试账号余额用完/次数用尽/被风控锁"是家常便饭，必须有恢复协议。

### R4 · Discovery 没有"饱和检查点" `[本评独有]`
- 现象：`api_inventory.json` 36 条，漏了 `recharge-create / profile / change-password / balance-records / reset-password / create-order / admin/user-list / admin/merchant-audit / admin/unlock-account / pay/index.php` 等关键入口；`admin.js`、个人中心内联脚本是"后来才补读"的。
- 通用根因：**没有"JS 分析+页面爬取+端点抽取达到不再新增的饱和点、冻结台账后才开始深测"的硬约束**。深测和发现混跑，发现不完整就进深测，后发现的入口既不补进台账也不回头补测。
- `api_inventory.json` 还和实测自相矛盾——报告 VULN-002 用了 `recharge-create.php`、admin_console 测了 `admin/user-list.php`，但都不在 inventory 里。**inventory 不是权威台账，只是草稿。**

### R5 · 威胁建模按"功能"粒度，不按"参数"粒度 `[并自他评⑦·强化]`
- 现象：商户商品只建 T1 读 / T2 写两个威胁就停；没测 `use_points/refund_amount/order_time/discount/keyword SQLi/product-delete SQLi/upload 穿越/pay redirect`。
- 通用根因：决策树是"按功能分支"，**缺一个"按参数语义路由"的强制步骤**。应沉淀为模型无关的参数语义 playbook：

| 参数语义 | 路由到 |
|---|---|
| `amount/refund_amount/use_points/price/fee/balance/stock` | 交易篡改卡（负数/0/小数/超额/精度/科学计数法） |
| `order_time/timestamp/create_time` | 时间篡改卡（过去/未来/边界/伪造创建时间） |
| `order_no/product_no/user_id/user_hash/id` | 资源归属卡（换角色对象/枚举/类型混淆） |
| `redirect/return_url/callback_url` | 跳转链卡（外跳/协议变体/相对路径） |
| `image_url/url/fetch` | SSRF 卡 |
| `filename/category/file/upload` | 文件上传/路径卡（双扩展/穿越/MIME 混淆） |
| `keyword/search/sort/orderBy/filter` | 注入卡 |
| `status/discount/role/state` | 枚举篡改卡（非法值/越权值） |

- 知识卡 `business-logic-tamper` 恰是这张表，但没被注入（R1）。须把卡内容**直接下沉进 skill 决策树**，不再依赖外壳。

### R6 · "认证/验证绕过"是被低估的一等分支 `[本评独有·他评未单列]`
- 现象：1090 分的"绕过验证"类（注册验证码/短信码、用户枚举、reset-token 绕过、商户审核状态绕过、管理员暴破）几乎没碰。
- 通用根因：决策树把"认证绕过"塞在"无登录态 → 优先声明认证绕过"分支下。**但本轮模型一开始就有登录态，于是整条认证绕过支线再没被回头打开。** 注册/找回/审核/解锁这些流程的鉴权弱点，与"当前是否登录"无关，必须独立强制访问。
- 他评的 ledger 有 `auth` risk_tag、参数 playbook 有 `redirect/keyword`，但缺一条"无论是否登录都必须访问认证绕过面"的战略指令——而本轮失败恰恰因为"有登录态就再没回头测认证绕过"。故此条须独立保留。
- 短信模拟器被允许作为验证码辅助，但 skill 没把它当"覆盖注册/找回验证绕过"的杠杆，只当"取验证码"工具。

### R7 · 无"按根因去重"与"按价值排序" `[本评独有]`
- 现象：VULN-002/003 同一根因（回调信任客户端签名）拆成两个 critical；VULN-005 真实但不在计分表；高价值未覆盖项被冷落。
- 通用根因：skill 文件没有"同一端点+同一根因=一个发现（多 facet）"的去重规则，也没有"先覆盖高价值攻击面类、再深拆单点"的价值排序。模型倾向"找到一个就深挖成多报告"，而非"先扫完面再深挖"。

### R8 · 没有真正的完整性门——模型自己就是门 `[并自他评④·强化]`
- 现象：`make_report.py` 硬编码 `module_queue=[]`，在自己写的 `coverage.json` 里列着 `BLOCKED/NEEDS` 盲区的同时，吐出一份"完整"终态报告。
- 通用根因：**Guardian 管"单篇报告质量"（是不是垃圾洞、有无 PoC），Session Gate 应管"整场作战完整性"——后者本轮完全缺勤。** 即便走模式 B，`_conclude` 也只把 `LOW_ROI` 降级为 `incomplete`，不阻止出报告，门控强度也偏弱。
- Session Gate 须落地的枚举谓词（命中任一即不许以"完成"收口）：
  1. 存在高价值 surface 仍是 `not_tested`；
  2. 存在可解除的 `blocked`（见 R3）；
  3. 存在 `NEEDS` 没转成明确补测任务；
  4. 若提供 oracle，存在 oracle 命中项无任何测试记录（见 R10/G9）；
  5. **存在 `confirmed` 声明与证据文件内容不一致**（如 `make_report.py` 硬编码 confirmed、但 evidence 文件 `confirmed:false`——抽奖那条正是此模式）。

### R9 · 缺 role-pair / object-pair 计划器 `[并自他评③·新增]`
- 现象：本轮虽做了 5 账号、`shanghu1→shanghu2`、`maijia1→maijia2` 的越权，但漏了余额记录、商户发货、图片删除、后台用户列表等面。
- 通用根因：越权测试不是"看到一个 ID 就换一下"，而是须系统化三步构造：
  1. **owner 账号**创建或枚举对象；
  2. **attacker 账号**读取、修改、删除、操作该对象；
  3. **victim/owner**再查业务影响（账本/状态/数据是否真变了）。
- 缺一个把"对象型端点 × 角色对"自动铺成待测格的 planner，越权覆盖就靠临场灵光。

## 4. "越改越垃圾"辨析 `[本评独有·他评未涉及]`

用户印象是"越改越垃圾"。基于产物能/不能证据到的：

**能证据：**
- 近期提交（`2348aac` per-feature 纵深循环、`d0eee3d` `_feature_of` 分组、`7c1515d` F1 采证、v3.3 phase2 知识卡接通）**全部落在 `engine/orchestrator.py` + `knowledge/` + `engine/knowledge.py`**。
- 本轮 `state.json` 证明**没走 orchestrator** → 这些改动对本轮**零生效**。
- v3.3 phase2 把 `核心技能文件.v2.md` 的提示注入从"外部 skill 死链"改为"声明方向 → 外壳按 endpoint/surface/参数匹配知识卡注入"。模式 A **没有外壳**，skill 文件现在依赖一个不存在的注入源——模型被承诺"你声明方向，系统会注入知识卡"，却永远收不到（§2.3 实证）。
- skill 文件 §6/§9 还引用"长会话每轮开头注入外部维护的认知状态对象""外壳按…匹配知识卡"——这都是 orchestrator（模式 B）才有的能力。**skill 文件已向模式 B 的外壳倾斜，而模式 A 既丢了旧指引、又没拿到新外壳。**

**不能证据：**
- 仓库里没有更早的计分报告（`pentest-report/`、`pentest-data/` 均未纳入 git），无法用"上次 X 分、这次 640 分"证明数值回归。

**结论（最可辩护读法）：** "越改越垃圾"更可能是"迭代集中在一条没在执行的层"叠加"skill 文件向不存在的漂移"造成的体感退化，而非真有某次改动把覆盖能力改坏了：
- 改 engine 不会让模式 A 变好（路没接上）；
- 改 skill 文件把指引路由给"外壳"会让模式 A 变差（外壳不在）；
- 唯一在模式 A 真正生效的"决策树+七问+垃圾洞"几轮没动，所以单点发现质量依旧在线。
- 而要**证伪**"越改越垃圾"或确认其反转，前提是有一个离线 benchmark（见 G9）——目前没有，所以连"变没变好"都无法量化。

> 一句话：**不是改坏了，是改的地方没在跑；跑的地方反而被改得更依赖一个不在场的外壳；而"有没有变好"连度量工具都还没有。**

## 5. 原则：靶场只作为回归集 `[并自他评⑥·正面表述]`

下列方向是错误的：
- 不应针对天积商城写死 `vuln_config.php`；
- 不应把靶场路径、参数、账号、漏洞答案内置进技能；
- 不应把某个靶场的评分规则当成通用漏洞标准；
- 不应为得分降低报告标准，把未证明项写成 confirmed。

**靶场只作为回归集：输入是目标+授权，输出是通用技能包在该样本上的覆盖率、证据质量、缺口。靶场只用于验证能力有没有变好，不应成为代码里的特殊分支。**

### 与自评报告（operation-assessment）的分歧

| 自评报告的判断 | 本报告态度 |
|---|---|
| "没按评分目标组织测试/追高价值线索过深" | ✅ 症状对，根因不是"没对齐评分"，是 R1/R2。对齐评分是靶场专用，不可泛化。 |
| "完整性门控失效，明知 BLOCKED/NEEDS 仍清空队列出报告" | ✅ 且更严重：不是"门控失效"，是**模式 A 根本没有 Session Gate**（R8）。 |
| "Discovery 不完整，漏关键入口" | ✅ 对应 R4。补充：inventory 与实测自相矛盾，不是权威台账。 |
| "没利用重置靶场条件" | ⚠️ "重置靶场"是靶场专用动作；通用版是 R3（阻塞分类 + 可恢复阻塞转行动）。 |
| "未用短信模拟器覆盖验证绕过" | ✅ 对应 R6。 |

## 6. 通用改进方向（G1–G10，按 ROI 排序）

> 原则：所有改进必须落在**所有模式都读的那一层**（`核心技能文件.v2.md` + 让模式 A 真能接上 engine + 新增通用外壳组件），否则又是"改了不生效"。

### G1【最高】把覆盖完整性纪律写进核心技能文件 + coverage-ledger `[并自他评②·升级]`
在 `核心技能文件.v2.md` 新增「覆盖完整性（硬性）」节，模型自维护轻量台账 `coverage-ledger`。**关键升级：surface_id 维度从 v1 的 endpoint×class 细化为 `endpoint × method × param × role`**——`param` 维度让 R5（按参数枚举）从口号变结构，`role` 维度让 R9（role-pair）可铺格。

ledger schema（通用，不绑靶场）：
```json
{
  "surfaces": [
    {
      "surface_id": "POST /api/user/refund.php refund_amount [user]",
      "endpoint": "/api/user/refund.php", "method": "POST",
      "param": "refund_amount", "roles": ["user"],
      "risk_tags": ["payment", "amount-tamper"],
      "status": "not_tested", "evidence_ref": null, "blocker": null
    }
  ]
}
```
- 每格四态：`confirmed / not_vulnerable / blocked / not_applicable`（外加 `not_tested` 初态）。
- **硬规则：存在 `not_tested`(高价值) 或可解 `blocked` 或 `needs_input` 或声明与证据不一致时，终态只能输出 `incomplete`/`NEED_INPUT`，不得输出 `VULN_FOUND`/`LOW_ROI` 当作完成；报告标题须标"部分覆盖"并列未闭格清单。**
- 同步进 `codex/AGENTS.md` 并 `regen`。

### G2【最高】让模式 A 真能用上 engine `[本评独有·治 R1]`
二选一：
- (a) skill 文件把"自由跑"改为"调用 `python3 run.py` 起 orchestrator"作为默认推荐路径，模式 A 退化为"读 skill + 起引擎"；
- (b) 若保留纯 skill 跑法，把知识卡内容**直接下沉进 skill 决策树**（见 G3），并内置 G1 的极简 ledger，不再依赖外壳注入。
关键：别再让"知识卡只在 orchestrator 里"——要么接上，要么把卡内容下沉到 skill 文件。

### G3【高】按参数语义路由的 playbook `[并自他评⑦·治 R5]`
把 §3-R5 的参数语义表沉淀为 skill 文件决策树的一等步骤（每端点必跑），或落地为知识卡且在模式 A 也注入。这正好复用 `business-logic-tamper / input-validation / file-upload-risk` 三张卡——直接下沉。

### G4【高】"认证/验证绕过"升一等独立分支 `[本评独有·治 R6]`
决策树把"认证绕过"从"无登录态"子项里拎出来，改成**无论是否登录都必须访问**的一等分支，带面清单：注册、登录、找回/重置、captcha/sms、审核状态、账号锁定/解锁、用户枚举。本轮 1090 分全在这条线。

### G5【高】阻塞恢复协议 blocker-resolver `[并自他评⑤·治 R3]`
把 §3-R3 的阻塞分类表落地为 skill 规则 + 可选外壳组件 `blocker-resolver`：可恢复阻塞（quota_exhausted/object_absent/missing_role）自动生成下一步行动（重置/创建对象/注册新号），不进最终盲区；不可恢复（captcha_required/out_of_scope）才升级 `NEED_INPUT`。

### G6【中】Discovery 饱和检查点 `[本评独有·治 R4]`
新增规则：JS 分析 + 页面爬取 + 端点抽取必须达到"连续 N 个来源不再新增端点"的饱和点；端点清单冻结进 `coverage-ledger` 后才开始深测；深测中后发现的端点必须追加进台账并补测，不得就地消化。

### G7【中】按根因去重 + 价值优先 `[本评独有·治 R7]`
新增规则：同一端点同一根因的多个 facet 合并为一个发现；覆盖阶段先按"攻击面类"横扫到四态，再对高价值点深拆，避免"一发现就深挖成多报告"。

### G8【高】role-pair / object-pair 计划器 `[并自他评③·治 R9]`
新增组件/规则：对每个"对象型端点"（含 `*_no/*_id/hash` 参数）× 角色对，自动铺成待测格，按 `owner 创建 → attacker 读改删 → victim 查业务影响` 三步执行。补上本轮漏的余额记录、商户发货、图片删除、后台用户列表等面。

### G9【最高】benchmark-eval 离线评估 `[并自他评①·治 R10·新增]`
新增通用离线评估工具，**这是 v1 的最大漏，也是"越改越垃圾"能否量化证伪的前提**：

```bash
python3 -m engine.benchmark_eval \
  --oracle oracle.json \
  --summary runs/<sid>/summary.json \
  --coverage runs/<sid>/coverage-ledger.json
```
- 输入任意格式 oracle（JSON / PHP return array / CSV / YAML），解析为统一字段 `endpoint/method/params/class/score`；
- 与本次 findings/evidence/coverage 映射，输出：命中项、漏测项、重复命中项、非 oracle 真实发现、得分、分类覆盖率；
- **铁律：oracle 只参与评估，不参与攻击决策。** 任何把 oracle 写进攻击 prompt 的接线都是作弊。
- 没有它，"迭代有没有让覆盖变好"永远只能靠体感（正是 §4 的困境）。

### G10【高】Session Gate 枚举谓词 `[并自他评④·治 R8]`
把 §3-R8 的 5 条谓词落地为外壳组件 `session-gate`（与 Guardian 并列）：报告生成前过一遍，命中任一谓词则降级为 `incomplete` 并回列未闭格，禁止以"完成"收口。其中"confirmed 与证据文件不一致"一条能直接抓 `make_report.py` 硬编码、声明 confirmed 但 evidence `confirmed:false` 的不一致。
