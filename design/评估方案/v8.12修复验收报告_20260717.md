# Atoolkit v8.12 修复验收报告（2026-07-17）

验收对象：v8.12 Engine 两阶段威胁规划与 SRC 运行真实性修复。

验收结论：**PASS，允许提交发布**。

## 1. 历史问题与验收结果

本轮离线检查了 Xiaomi 报告问题和 ByteDance 历史 Run。ByteDance 目录仅有 inventory、旧报告、
hint/authz，缺少 manifest、run plan、coverage ledger、Finding 包和原始证明；同时存在共享浏览器身份、
API 名称代替 METHOD/path/param、跨模块 scope 含混、漏测业务功能、现象型 CSRF、原始秘密/PII 进入
产物和模型 provenance 自述不可靠等问题。检查过程未向任何历史目标发包，也未把原值复制进本仓库。

v8.12 已用通用合同而非厂商特判闭合这些根因：

- sibling Planning/Attack，各自在模型动作前创建 authority manifest/run plan；
- 有界、no-follow、稳定脱敏 Recon snapshot；
- discovery adequacy、frozen path scope、Feature/Threat schema 与证据引用校验；
- threat compiler 精确建格，不再做 endpoint × 默认漏洞类笛卡尔积；
- credential hash 去重、identity requirement、受限 `identities.json` 和 identity-aware closure；
- Attack 逐文件绑定晋升 Recon，并递归验证 Planning parent artifact；
- CSRF 必须有 before/cross-site/after、受害者会话、不同来源、可执行浏览器发起载体和状态变化；
- Canonical report 对 Finding、HTTP 和 PoC 片段统一稳定脱敏；
- finalization 后 sid 不可继续修改，崩溃恢复与新一轮测试边界明确。

## 2. 反向审核追加项

施工后复核额外发现并消除了四个缝隙：

1. Host 曾可判定双身份 ready，但 Attack 模型拿不到 `--identity` 原始 header；现已延后到 Planning
   完成后按 label 写入权限收紧的 Attack 文件。
2. 只绑定 discovery index 不足以证明每个晋升 snapshot；现已逐文件进入 Attack manifest。
3. 手工 HTTP 可伪造 Origin/Cookie；现增加物理跨站发起载体，且 Origin/Referer 均可作为来源证据。
4. 已 finalization 的同 sid Resume 会让新状态落不到原不可变快照；现明确拒绝，继续测试必须新开 sid。

并发 manifest 测试还触发了合法 create-only 发布窗口与严格 hardlink 检查的竞态。修复仅对瞬态
non-single link count 做 100ms 有界重试；持续硬链接仍被拒绝。并发用例连续 20 次通过。

## 4. 明确保留的边界

当前 Codex backend 仍不能证明同一 OS 用户下的全局只读隔离，也不能证明 live shell 的 pre-exec
host/path 出站硬约束。v8.12 的控制是 Planning 禁目标网络、新凭据延后落盘、输入输出高置信秘密扫描、
live 默认拒绝且 unrestricted 必须显式接受；这些不能表述成容器/VM 级隔离。进程 containment 未被
后端证明时，authority 产物仍是 diagnostic，不自动提升为跨 Run 项目真值。
