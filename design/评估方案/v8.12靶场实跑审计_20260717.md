# Atoolkit v8.12 靶场实跑审计（2026-07-17）

审计对象：`runs/shop_20260717` 的离线运行产物。审计期间未向靶场发送请求，也未在本文复制
Cookie、Token、Key 或个人信息原值。

## 1. 结论

本轮证明了模型具备找到高价值业务漏洞的能力，但没有证明 Atoolkit v8.12 的两阶段威胁计划和
Canonical 交付闭环已经被实跑验证。

- 靶场严格得分为 `1720/4800 = 35.8%`，正式 finding 对应 9/28 个漏洞点；宽松口径为
  `2050/4800 = 42.7%`。
- 命中包括免费充值、超额退款、客户端控制抽奖概率、订单 IDOR、垂直越权、SQL 注入、上传绕过、
  SSRF 和审核状态越权，说明“从业务结果出发”方向是有效的。
- 同时存在 4 个假阴性、11 个完全未测、2 个发现未报；XSS 为 0/3，输入验证仅 3/10。
- 10/10 个 `finding.json` 均无法通过当前仓库的 Finding proof validator。
- 本轮声明为 `direct_skill_diagnostic`，目录中没有 `run_manifest.json`、`run_plan.json`、
  `feature-graph.json`、`threat-model.json`、`inventory.json`、`coverage-ledger.json` 或
  `finding_validation.json`。因此它是一次模型手工作战记录，不是 v8.12 Engine 两阶段路径验收。
- `final_report.md` 仍命中 2 处高置信 secret 脱敏规则，不能作为可直接对外提交的 Canonical 报告。

## 2. 靶场命中与漏失

| 类型 | 数量/得分 | 主要表现 |
|---|---:|---|
| 正式命中 | 9/28，1720 分 | 2 个 P1 业务结果命中，交易面相对最好 |
| 假阴性 | 4 个，700 分 | 单通道、单值或无测试数据时过早判安全 |
| 完全未测 | 11 个，1940 分 | 已知端点没有进入强制执行队列，或动态发现未同步清单 |
| 发现未报 | 2 个，330 分 | 信号没有进入验证/修复队列，最终报告靠模型自行取舍 |

四个假阴性的根因并不是“模型不会某个 payload”，而是实验义务没有成为机器状态：

1. 用户枚举只比较消息，未执行时间和行为通道；
2. reset token 只测假值、空值和缺字段，却被标为 full/not_vulnerable；
3. balance records 在数据为空时直接判阴，没有先创建可区分的 owner/peer 记录；
4. `use_points` 只测布尔主流程，没有完成负数、超大值、类型边界和状态前后差异。

11 个未测中，`register`、`forgot-password`、`admin/login`、`product-image-delete`、`ship-order`、
`product-delete`、`submit-audit`、`product-add` 等 7 个方向已经出现在侦察清单中，却没有进入不可跳过的
执行队列；`create-order`、`pay/index`、`balance-records`、`reset-password` 等 4 个在初始清单中缺失，
其中后两者虽在后续证据中出现，却没有回写为 authoritative inventory。这同时暴露了“已知未执行”和
“动态发现不同步”两种问题。

## 3. Finding 可信度审计

10 个 Finding 都是旧式精简结构。共同缺口包括：

- 缺少 `apis`、`proof_packets`、`verification.assertions`、修复建议和 Feature/Threat 绑定；
- `claim.proof_packet_ids` 引用不存在的 packet ID；
- `impact_claims.proof_refs` 使用了相对 Finding 目录错误的路径，validator 无法读取；
- `poc.sh` 不满足可解析 curl/raw HTTP 合同；
- 交易类缺 before/exploit/after，授权类缺 owner/attacker/object differential，SSRF 缺目标控制或
  唯一回连 marker，文件类缺写入后读取/位置证明；
- 报告中的相对 `POST /api/...` 被写入 `target` 字段，无法绑定 manifest primary target。

这些问题不等价于 10 个漏洞都是误报。它们表示：模型叙述与部分响应具有价值，但 Atoolkit 不能据此
自动把它们注册为 accepted、confirmed root finding，也不能把对应 coverage cell 关闭。

## 4. 对 v8.12 的准确裁定

v8.12 的代码目标是 Planning/Attack 两阶段、冻结 threat cells、身份 readiness 和 Canonical
交付。本轮没有进入这些入口，因此不能用本轮低分直接否定 v8.12 设计，也不能用 9 个命中证明设计
有效。能够确定的是产品接缝仍有缺口：Direct 使用者可以绕过已经存在的 `skill_runtime`，回到
Markdown 清单、手工证据文件和模型自行收口。

这次结果应被归因为：

```text
discovery/inventory 同步缺口
  + execution queue 非强制
  + 实验前置条件/深度不是逐项状态
  + finding proof 失败后没有自动返工
  + Direct 运行真实性门未阻止“手工完成”叙事
```

## 5. v8.13 修复输入

v8.13 不应增加一份更长漏洞清单。它需要补上 Feature → Threat → Cell 之后缺失的
Experiment 层：

1. 把 threat 的 `evidence_required` 和确定性最低深度编译为 experiment obligations；
2. 每次真实实验必须提交 exact surface、已完成义务、barrier 和物理 evidence refs；
3. Host 归并后动态生成下一队列，空数据、过期会话、WAF、格式错误先进入恢复动作；
4. 未完成义务的阴性保持 `shallow_negative/exploring`，不能被“full”文字关闭；
5. Finding 被 validator 拒绝后自动进入 proof-repair 队列；
6. 新 endpoint 记录为下一 Run discovery backlog，不在同一 Run 偷改 frozen denominator；
7. Direct/Skill 若没有 runtime status、queue、observation 和 checkpoint，只能明确输出 incomplete
   diagnostic，不得生成“完整测试”报告。
