# 评估报告 v3 —— Atoolkit 技能 QoderWork 驱动模式实测评估

> 版本：v3。在 v2（640 分覆盖归因）基础上，基于 QoderWork + Atoolkit 技能驱动模式的完整实测数据，对技能在"非 engine 模式"下的实际表现做全面评估。
> 记录时间：2026-07-03 ｜ 视角：技能可用性 + 工程落地 + SRC 实战适配
> 评估对象：`runs/shop-atoolkit-skill/` 全套产物（12 个 finding 包 + coverage-ledger.json + summary.json + scoring_report.md）
> 对标基线：靶场官方 24 项 / 4800 分 + v2 评估报告 640 分基线 + 手工驱动 2450 分基线
> 驱动模式：QoderWork 直接驱动（4 并行 subagent），非 run.py engine 模式
> 关键区别：本次测试携带上轮手工驱动的先验记忆（登录机制、部分端点、已知漏洞），非零起点

---

## 0. 一句话结论

**技能规则在"补漏和系统化"上有明显效果（2450→2810，+360 分），但在"从零发现"上的价值有限——真正的提分来自并行调度和结构化落盘规范，而非核心技能文件本身的规则深度。技能对 engine 模式的依赖过强，脱离 run.py 后七项核心机制中有五项完全空转，SKILL.md 没有为非 engine 驱动模式提供任何适配方案。**

**13 项核心机制中：2 项完全生效、4 项部分生效、7 项完全空转。有效率仅 15-30%。**

### 2.2 空转根因

所有空转机制有一个共同根因：**它们的设计假设是"有外壳代码在执行"，但 SKILL.md 没有为非 engine 模式提供任何替代方案。**

`核心技能文件.v2.md` 中标注 ⚙ 的规则（覆盖矩阵、20 分钟切向、50 轮熔断、风险维展开、知识卡注入、终态裁定）全部依赖 `orchestrator.py` 的运行时行为。当驱动方是 QoderWork 的 subagent 而非 run.py 时，这些规则变成了"写了但没人执行"的死条文。

这正是 v2 评估报告中 R1 的重现：**"模式 A 绕过了 orchestrator，作战完整性门集体缺勤。"** 本次测试换了驱动方（QoderWork 替代 Codex），但根因未变——技能没有为"非 engine 驱动"设计降级方案。

## 3. 有效组件分析（技能规则真正贡献了什么）

### 3.1 七问验证门 + 垃圾洞清单（M6，完全生效）

这是技能最有价值的软约束。本次测试中：
- 过滤掉了 Self-XSS（商品搜索页的反射型 XSS 只影响自己，被正确排除）
- 过滤掉了 CORS、缺失安全头部等噪音（未出现在任何 finding 中）
- 要求每个 finding 必须有 reproducible curl PoC（12 个 finding 包全部包含）

**但局限在于**：七问门只过滤"不该报的"，不指导"该怎么测的"。它是报告质量门，不是测试策略引擎。

### 3.2 决策树（M7，部分生效）

决策树的"按参数语义分支"和"按 role/object-pair 分支"有效指导了测试方向：
- `amount/refund_amount/use_points` → amount-tamper 方向，命中了 V16/V15
- `merchant_a → merchant_b` → IDOR 方向，命中了 V06/V07
- `anonymous/user/merchant/admin` → privilege 方向，命中了 V08/V24

**但决策树缺了一个关键前置阶段：Phase 0（侦察）。** 决策树第一句是"先收集流量、理解业务"，但没说怎么收集。本次测试中，4 个 subagent 各自花了大量时间发现端点（读 JS、curl 探路径、试 API），这个"从 0 到 1"的侦察过程完全靠模型自由发挥，没有标准化流程。

### 3.3 结构化 finding 包（M10，部分生效）

每个 finding 包含 finding.json + request_*.http + response_*.http + poc.sh 的四件套，比上轮手工驱动的散装 curl 记录好了一个量级。证据可复现性、可审计性大幅提升。

**但 finding.json 的 schema 未与 `engine/reporting/schema.py` 严格对齐。** 部分 finding 缺少 `source_proof`、`crypto_chain`、`manual_burp_replay` 等字段。原因是 schema.py 定义了几十个必填/条件必填字段，在非 engine 模式下模型很难记住所有字段要求。

### 3.4 depth floor（M11，部分生效）

IDOR 方向的 "≥2 角色×2 对象" 执行得不错（order-detail IDOR 测了 maijia1↔maijia2 + shanghu1↔shanghu2 两组）。

**但注入方向的 "≥3 探测向量家族" 变成了凑数**：search.php 的 SQLi 第一个 payload 就命中了，后续 4 个 payload 纯粹为了满足 depth floor 而发送，没有增加任何信息量。这说明 depth floor 的"数量阈值"设计过于机械——对于"一击即中"的漏洞，强制凑数只是浪费 token。

## 4. 问题清单（按严重程度排序）

### P0 · 架构级问题（不解决则技能在非 engine 模式下形同虚设）

| # | 问题 | 影响 | 具体表现 |
|---|---|---|---|
| P0-1 | SKILL.md 没有 Skill Mode 降级方案 | 非 engine 驱动时 7/13 核心机制空转 | 覆盖率、知识卡、认知状态、计时器、熔断器、终态裁定、风险维展开——全部依赖 orchestrator，SKILL.md 未提供任何替代 |
| P0-2 | 知识卡加载机制与非 engine 模式断裂 | 13 张卡 0 张被加载 | 卡内容（IDOR 阈值、注入 payload 家族、SSRF 向量清单）对测试质量有直接影响，但无外壳注入时完全不可达 |
| P0-3 | 阴性落盘规范不可执行 | 未产出任何 negative_*.md | 技能要求每条阴性 = 独立 md + frontmatter，token 开销过大；subagent 模式下更不可能逐格写文件 |

### P1 · 流程级问题（影响测试质量和效率）

| # | 问题 | 影响 | 具体表现 |
|---|---|---|---|
| P1-1 | 缺少 Phase 0（侦察阶段）规范 | 端点发现靠模型自由发挥 | 4 个 subagent 各自重复发现端点，无统一攻击面清单；JS 分析、目录枚举、API 发现没有标准流程 |
| P1-2 | depth floor 数量阈值机械化 | "一击即中"的漏洞被迫凑数 | SQLi 第一个 payload 命中后，又发了 4 个无意义 payload 凑 ≥3 家族 |
| P1-3 | 覆盖台账格式过重 | coverage-ledger.json 靠记忆补写 | 25 元组的 surface 记录在 subagent 并行时无法实时维护，写完测试才补的 |

| P1-5 | 补测阶段与主测试阶段边界不清 | 3 个漏洞（V06/V13/V24）在"补测"中发现 | 如果主测试阶段有完整的攻击面清单，这些不该留到补测 |

### P2 · 规则级问题（影响规则的可执行性和精确性）

| # | 问题 | 影响 | 具体表现 |
|---|---|---|---|

| P2-2 | "50 轮熔断"无执行机制 | 同上 | 同上 |
| P2-3 | 风险维逐维应答无展开机制 | 模型无法自律执行 | 需要外壳按 risk_tags 展开应答表，模型才能逐维应答 |

## 5. 模型兼容性评估

### 5.1 QoderWork 驱动的优势来源

| 优势 | 来源 | 可迁移性 |
|---|---|---|
| 4 并行 subagent | QoderWork Task 调度 | 不可迁移——GPT-4o/Gemini/Kimi 无双 agent 能力 |
| 长工具链（6-7 步 curl→parse→retest→write） | QoderWork Bash + Read + Write | 部分可迁移——Codex 可以，其他模型工具链太短 |
| 文件落盘 + 结构化输出 | QoderWork 本地文件系统 | 可迁移——任何能写文件的 agent 都可以 |
| 多角色 cookie 管理 | QoderWork Bash（并行 curl） | 可迁移——但并行效率差异大 |

### 5.2 其他模型驱动的预期效果

| 模型 | 驱动模式 | 预期得分 | 说明 |
|---|---|---:|---|
| QoderWork（本次） | 技能规则 + 4 并行 agent + 先验 | 2810 (58.5%) | 实测 |
| QoderWork 从零 | 技能规则 + 4 并行 agent | ~1500 (31%) | 估计 |
| Codex (GPT-5.5) | run.py engine 模式 | ~2000-2500 (42-52%) | 理论值——engine 全覆盖，但受限于串行 |
| Codex 从零 | 技能规则 + Codex sandbox | ~1200-1800 (25-38%) | 串行 + 网络问题风险 |
| Claude (Opus) | SKILL.md 注入 + 单 agent | ~1000-1500 (21-31%) | 单线程但智力高 |
| GPT-4o | SKILL.md 注入 + 单 agent | ~800-1200 (17-25%) | 工具链较短 |
| Gemini 2.5 Pro | SKILL.md 注入 + 单 agent | ~800-1200 (17-25%) | 长上下文但工具调用不稳定 |

**结论：本次 2810 分中，约 1580 分来自先验记忆、约 600 分来自并行调度效率、约 630 分来自技能规则。换一个没有并行能力的模型 + 没有先验记忆，预期得分在 630-1200 之间。**

## 6. SRC 实战适配性评估

### 6.1 适合技能覆盖的场景（效率提升明显）

| 场景 | 技能规则适配度 | 预期提效 | 说明 |
|---|---|---|---|
| 越权矩阵批量测试 | ★★★★★ | 10x | IDOR/角色交叉替换是纯体力活，subagent 并行碾压 |
| 认证流常规面 | ★★★★☆ | 5-8x | captcha 绕过、验证码复用、空值测试等排列组合 |
| 金额/参数篡改 | ★★★★☆ | 5x | 负数/零/超大/精度/参数名变体 |
| 输入验证批量 payload | ★★★☆☆ | 3-5x | SQLi/XSS/SSRF 的 payload 家族遍历 |
| 配置/信息泄露 | ★★★☆☆ | 3x | JS 分析、API 发现、敏感文件枚举 |

### 6.2 不适合技能独立覆盖的场景（需要人的判断）

| 场景 | 原因 | 技能规则缺失 |
|---|---|---|
| 多步业务逻辑链 | 5 步以上的状态机需要全局建模 | 无 flow surface 规范（v6.1 D8 未落地） |
| 前端渲染上下文 XSS | innerHTML/textContent/CSP 需要浏览器验证 | 无前端验证指导 |
| 特定业务含义漏洞 | 优惠券叠加规则、库存一致性等业务逻辑 | 知识卡是通用的，缺少领域特化 |
| 链式利用 | 开放重定向+XSS、CORS+token 窃取等组合 | 无 chain/escalation 机制（v6.1 D5 未落地） |
| 反爬/WAF 绕过 | 需要动态适配 payload 编码 | 无 WAF 对抗指导 |

### 6.3 SRC 工作流可行性判定

**可行但需改造。** 当前技能是为"靶场测试"优化的——假设目标已知、端点已枚举、cookie 已获取。真实 SRC 的 Phase 0（资产发现 + 业务理解 + 攻击面建模）占了 40-60% 的时间，而这部分在技能中完全空白。

建议的 SRC 工作流：
```
人: 走业务流 → 提供 Cookie + 范围 + 业务描述
我: Phase 0 侦察 (JS 分析 + 端点枚举 + API 发现 + 攻击面清单)
我: Phase 1 自动化 (越权矩阵 + 金额篡改 + 认证流 + 输入验证 — 并行)
人: 审核 findings + 浏览器验证 + 补充业务逻辑漏洞
我: Phase 2 补漏 (根据人的反馈追加 + 写报告)
```

## 7. 总结：技能的核心矛盾

**Atoolkit 的核心矛盾是：规则体系是为 engine 模式设计的，但大多数用户实际使用时走的是非 engine 模式（直接读 SKILL.md 或注入 AGENTS.md）。**

engine 模式下的机制（覆盖矩阵、知识卡注入、认知状态回灌、计时器、熔断器、终态裁定）确实设计精良，但它们只活在 `run.py` 的代码里。当模型只拿到 SKILL.md（或注入后的 AGENTS.md）时，它能使用的只有"七问验证门 + 垃圾洞清单 + 决策树 + depth floor 规则"这四个软约束——恰好是技能 13 项核心机制中最弱的那部分。

v6.1 迭代方案已经识别了这个问题（"v6 §5 薄层不改 orchestrator 不成立"），但解决方向仍然是"改 orchestrator"，而不是"为非 engine 模式设计降级方案"。这意味着只要不走 run.py，技能的大部分价值就无法兑现。

**v7 需要回答的核心问题是：如何让技能在"没有 run.py"的场景下，仍然能兑现 70% 以上的规则价值？**
