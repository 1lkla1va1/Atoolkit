# Atoolkit 实跑低分与 Finding 可信度审计

> 日期：2026-07-12  
> 范围：项目代码、Codex 任务历史、`runs/shop_target` 两轮产物。  
> 原则：只评价覆盖、证据和产品合同，不依据靶场答案反向设计漏洞打法。

## 1. 结论

低分不是因为目标“只有几个漏洞”，也不是因为高危必须出现 RCE。主要原因是三层同时失真：

1. 有效作战轮次不足：一次正式评分仅覆盖少量高价值面，后续重跑多次被模型不可用、额度、浅阴性循环阻塞。
2. 调度把 endpoint 与多个参数合并，并优先消费旧 Finding 的 chain/spread，遗漏新的 root cause。
3. Skill Mode 的精简 Finding 只校验字段/文件存在，候选、基础漏洞、影响升级和利用链被混成 confirmed。

因此旧报告的 Finding 数量不能直接等价为真实漏洞数，Coverage Cell 数也不能等价为已测 endpoint 数。

## 3. `runs/shop_target` 的 14 个 Finding

严格按“现有磁盘证据能否直接证明报告结论”分级：

- A（基础漏洞证据可成立）：1 项，充值回调凭据暴露并可直接调用回调；但“无限资金”仍未证明。
- B（高可信候选，缺关键对照/回查/原始 trace）：12 项。
- C（当前不应作为独立漏洞）：1 项，跨商户读取商品价格/库存/折扣，尚未证明这些字段存在隔离边界。

主要证据缺口：

| 类别 | 当前缺口 |
|---|---|
| Admin/IDOR | 缺 owner/attacker 身份、对象归属、匿名/失败对照；写操作缺 owner 回查。 |
| XSS | 只有写入响应和前端 sink 推断，没有受害者浏览器执行 trace/唯一 marker。 |
| SSRF | 响应被截断，缺受控目标 callback 或目标唯一内容标记。 |
| 路径穿越 | 只返回含 `../` 的路径，缺实际读回/位置证明/内容哈希。 |
| Race | “20 并发、9 成功”是人工摘要，不是编号原始响应和 before/after 状态。 |
| 交易 | 多项缺合法 control、状态 before、攻击动作、持久状态 after。 |

重复/聚合关系：

- 两个 Admin 读取接口可能同属统一 RBAC 缺失，应按根因聚合。
- 商品详情读取与商品图片删除可能同属 merchant ownership，但读取项尚未证明安全边界。
- 两个 XSS 的 source/sink/受害角色不同，可作为同一修复专题下的两个受影响链路。
- 回调伪造与回调竞态是两个根因，但竞态利用依赖前者的条件，不能分别夸大成两个“无限资金”。
- 第二轮已否定“管理员信息读取 → 密码重置 → 账户接管”，不得再计 ATO。

## 4. 为什么没有 RCE 或“大量数据泄漏”

RCE、账户接管、大量数据泄漏是结果，不是每轮必须完成的配额。目标是否存在这些根因未知；没有证据时强行追求只会提高误报。

正确的深度定义是：

```text
root finding（安全不变量已被破坏）
  → impact proof（下游结果已观察，不另计漏洞）
  → chain hypothesis（待验证 Intent，不进 accepted/评分）
  → proven chain（每一步有独立证据，才允许提升最终影响）
```

P1 交易漏洞可以是真高危而不是 RCE；XSS 只有浏览器脚本执行证据时是 XSS，只有实际目标身份/会话证明时才能进一步声称 ATO。

## 5. 已落地修复

- Coverage identity 改为 METHOD/path/param/vuln class，GET/POST 和多参数不再串格。
- `surface_budget` 覆盖被准入 endpoint 的全部参数 Cell；未授权 Cell 不能偷偷闭合。
- 浅阴性必须有向量和响应深度；跨轮 legacy negative 默认重新验证。
- Skill Mode legacy confirmed facts 改为 `legacy_unvalidated`，生成高优复验 Intent，不再直接继承为真。
- Finding 统一 root/impact/chain 合同；链假设不能进入 proven impact、accepted 或 benchmark。
- IDOR/XSS/SSRF/Race/File/Business 按类别要求 control、raw packet、marker/assertion 与状态证据。
- 未授权/IDOR 新增公开性与访问预期证明；公众列表、公开商品/资料、匿名 200 默认不是漏洞。
- Guardian 在 Coverage 闭格前执行；最终 verifier 拒绝时精确回滚 provisional positive Cell。
- Benchmark 只消费 accepted + proof_confirmed + root_finding，缺 method/role/evidence 不再通配；Ledger 不能合成得分。
- 新高价值 root cause 排在旧 spread 前；纯文本 SPREAD 不再直接得到 depth 4；无进展 chain Intent 会让出队列。
- 授权 host 使用解析后的 scheme/host/port/path 精确范围；会话路径、文件权限、重定向复验和进程清理已收紧。

## 6. 后续迭代

P0：把类级 proof assertion 独立成 `ProofDecision` 模块，增加浏览器 trace、callback log、并发 manifest、JSONPath/numeric delta 等解析器。  
已落地：用 risk tags 只生成适用漏洞类，并限制 chain/spread 抢占高价值 root 队列。  
P1：把公开性基线、同接口拒绝对照和敏感字段语义进一步抽成独立 `AuthorizationDecision`。  
P1：做一次不读 oracle 的 A/B 黑盒回归，比较 root recall、proof-confirmed precision、每 10 轮新 root 数、开放高价值参数数。  
P2：网络安全边界仍需受控请求 broker/强制代理；当前命令流审计是事后熔断，不能替代执行前 egress 限制。
