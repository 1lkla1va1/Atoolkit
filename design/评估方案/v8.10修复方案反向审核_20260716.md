# Atoolkit v8.10 修复方案反向审核（2026-07-16）

审核对象：`design/迭代方案/迭代方案v8.10_实测反馈驱动的执行闭环.md`

说明：这是同一实施会话内的反向设计审核，不冒充独立第三方审计。审核目标是先寻找会导致误报能力、破坏安全边界或只对靶场过拟合的方案缺陷，再决定是否施工。

## 第一轮结论：REJECT

初始构想有四个阻断问题：

1. 试图让 Direct/Qoder runtime 自建 manifest 并复用 canonical finalizer，会形成 self-authority，与 v8.9 的核心安全结论冲突。
2. 计划按 Qwen 漏项给具体端点增加规则，属于 oracle 过拟合，无法迁移到真实 SRC。
3. 计划让“最后一条 observation 胜出”，无法处理 `ship-order` 先错误阴性、后真实对象确认阳性的情况，也会让并发 agent 产生非确定结果。
4. 计划让 checkpoint 直接生成 final report，会重新制造多份报告真值，复现本轮 `findings_summary` 与 `score_comparison` 分叉。

因此初稿不准实施。

## 修订后的逐项审核

### 1. Authority 边界

修订：Direct runtime 明确只生成 diagnostic ledger/queue/status，不创建可信 authority，不改 ProjectState，不产生 delivery complete。可信交付仍只允许外部 attested wrapper。

结论：通过。

### 2. 泛化性

修订：不编码 `product-delete.php`、`balance-records.php` 或任何靶场路径；改为通用参数风险并集、实验前置条件和 barrier signal。

结论：通过。

### 3. 安全边界

修订：WAF 卡只影响状态和知识路由，不自动执行攻击；验证码规则禁止 OCR/代答凭据，安全测试流程绕过失败后请求人工输入；不为靶场暴力破解分数放宽 DoS/爆破限制。

结论：通过。

### 4. 多 agent 合并

修订：observation append-only；reducer 按 proof 强度和实验有效性归并。无 proof 的阳性/阴性冲突进入 `exploring`；proof-valid canonical Finding 可关闭为 confirmed，同时保留 conflict audit。

结论：通过。

### 5. 阴性可靠性

修订：向量数量只是必要条件，不再是充分条件。WAF、会话失效、对象不存在、空数据、缺角色、请求格式未解析都会阻止 `not_vulnerable`；IDOR 还需真实归属证据。

结论：通过。

### 6. 状态兼容性

风险：把 shallow 从 `not_tested + marker` 改为一等状态会影响统计与旧测试。

修订：旧旁路格式继续可读；closed set 不变；session gate、stats、next queue 和 reporting terminal status 同步修改，并用 round-trip 回归覆盖。

结论：通过，但实施时必须跑完整测试，不允许只跑新增测试。

### 7. 报告单一真值

修订：checkpoint 不渲染漏洞报告，只产生诊断状态和 queue；最终报告仍由 canonical validation projection 生成。人类 Markdown 不可反向覆盖 ledger。

结论：通过。

### 8. 指令长度

修订：不冒险大删现有安全边界；新增短 hot path 作为 Qoder/Direct 首读入口，完整核心保留为兼容与自动注入源，知识卡由机器按当前 surface 选择。

结论：通过。它减少 Direct 模式主动加载量，但不会减少 Codex 自动注入 AGENTS 的固定长度；后者属于后续可单独基准评估的优化，不在本轮冒险重写。

## 最终结论

**APPROVE FOR IMPLEMENTATION**。

批准范围仅限：风险推导/知识路由、barrier-aware negative gate、七态 ledger、Direct diagnostic runtime、hot path 与对应测试文档。以下内容明确不批准：伪造可信 Direct authority、靶场端点特判、自动爆破/OCR、checkpoint 自行生成最终报告、跳过全量回归直接 push。
