# Atoolkit v8.8 修复版本验收报告（2026-07-13）

## 1. 验收结论

Atoolkit 已按“设计修复方案 → 独立审核 → 迭代修复”完成 v8.8.0 实现。修复范围只在本项目内，没有按 `ai-src-toolkit` 改造，也没有覆盖外部 `/src` 或用户全局 Codex 配置。

v8.8 已把“多次 Run 的信息累积”从提示词约定改成可校验的系统合同：后续 Run 可以继承同一项目的已发现攻击面、未决 method、精确 coverage cell、proof-confirmed 根因事实、未完成 Intent、充分阴性和有效 dead-end；不再把手写 summary、普通 SKIP、legacy finding 或缺证据的 confirmed 当成项目真值。

这意味着它已经具备用于同一 SRC 业务连续 Run、减少重复测试并扩展项目信息的工程基础。但“每次 Run 都新增漏洞”不能保证；漏洞增量仍取决于目标剩余攻击面、账号/状态、模型探索质量和证据可获取性。是否提高真实发现率，需要用全新的 v8.8 Run 做边际指标评估，不能拿旧版手写汇总直接证明。

## 2. 旧靶场结果复核

### 2.1 Linglongsec

- 旧 `summary.json` 自报 24 个 finding、38 个 surface 中 20 个 confirmed。
- 磁盘只有 7 个平铺 legacy finding JSON，没有 v8.8 canonical finding package 和 pre-network manifest。
- v8.8 严格口径：7 个 legacy 输入、0 个 canonical、0 个 proof-confirmed；缺 manifest 时终态为 `precondition_missing`、exit 2。

结论：旧结果能说明探索方向较多，但不能证明 24 个均达到 SRC 可提交证据标准，也不能作为跨 Run 的 confirmed 项目真值。

### 2.2 Shop Run 1（2026-07-12）

- 自主报告命中 12/28（42.9%），加权 2040/4800（42.5%）。
- 对照靶场配置后补测达到 20/28、3440/4800；这部分属于 oracle/config 辅助，不能计入自主发现率。
- 12 个 finding 全是旧 `evidence/finding_*` 布局。按严格 Proof Contract：0 accepted；证据可挽救复核为 11 个待补证候选、1 个被物理响应直接反驳。
- Coverage、negative、Fact-Intent 和 summary 之间存在明显不一致，无法可靠判断精确 `method × param × role × class` 是否已经闭合。

### 2.3 Shop Run 2（2026-07-13）

- 汇总报告 10 个漏洞主张；靶场对照命中 9/28，1420/4800（29.6%），另有 1 个配置外主张。
- 会话保存了大量响应证据，但没有 canonical `findings/finding_<id>/finding.json`、run manifest 或 validator receipt，因此 v8.8 口径仍是 0 个可进入项目 registry 的 proof-confirmed root finding。
- 与 Run 1 相比，大部分为重复方向，仅出现少量新主张，同时丢失了上一轮部分方向。旧架构没有可靠 carry-over，不能据此认定第二次 Run 带来了稳定、单调的信息扩展或漏洞增长。

### 2.4 对旧两轮的最终判断

旧版多次 Run 有“随机补洞”的现象，但没有形成可量化、可复验的累计提升闭环。原始命中数可以看作探索能力信号，不能看作交付真值，也不能证明多 Run 已提高真实 SRC 漏洞发现率。

## 3. v8.8 已完成的修复

### 3.1 跨 Run 项目真值

- 新增单 writer、加锁、revision、原子落盘的 `project_state.json`。
- Inventory 区分已解析 METHOD surface 与 unresolved path；未知 method 不再伪造成 GET，并生成 method-resolution Intent。
- Cell 主键固定为精确 `asset + method + templated path + param + role + normalized class`。
- Confirmed 只覆盖同一精确角色单元的 negative；多角色必须全部有独立证据才能整体继承。
- 证据统一保存为 `session:<sid>/<path>` 或 `project:<path>`，并绑定 SHA-256；证据删除或篡改会让历史 cell 重新打开。
- Finding 用保守 fingerprint 去重，不按自由文本或相似 endpoint 过度合并。
- 历史 proof-confirmed Fact 以有界、data-only 块注入后续 Prompt，包含 canonical finding ID、asset、method、endpoint、根因、角色、摘要和证据引用；legacy pending Fact 不会冒充已确认事实。
- 有效 dead-end 必须提供精确单元、枚举 reason code、refutation 和物理证据；普通 SKIP、预算不足和缺账号不会跨 Run 关闭单元。

### 3.2 Reporting 与交付真值

- Canonical、已知 legacy 和 suspicious finding 布局统一做有界发现；拒绝 symlink escape、越界路径、重复 ID 冲突、malformed 和 oversize 输入。
- `run_manifest.json` 在首次模型调用前生成，并与会话外 authority copy 一致；禁止 self-authority。
- Validator 是 finding registry、summary 和 benchmark 的唯一 accepted/proof-confirmed 来源。
- 空 finding 只有在 inventory、coverage、candidate ledger、session gate 全部物理闭合时才能 `allow-empty`；自报 confirmed 不再放行。
- 历史 confirmed 空结果例外绑定当前项目 `project_state.json`、当前 manifest 主目标和项目身份，不能跨项目复用。
- Validation 异常、artifact digest 不一致、project-state/receipt/summary 持久化失败全部 fail closed。
- `no_work` 只有在项目闭合且本轮 run history 成功提交时才是成功终态。
- 最终 `run_receipt.json` 绑定 manifest、validation、coverage、summary 和 project-state delta；交付状态单独落盘。

### 3.3 独立项目边界

- 根目录 `AGENTS.md` 由本项目 v3 核心技能文件生成，`codex/AGENTS.md` 只是兼容副本。
- `doctor` 只报告全局 `/src` 指向外部 `ai-src-toolkit` 为 `foreign`，不覆盖、不接管。
- Atoolkit 版本在 engine、SKILL 和 CHANGELOG 中统一为 8.8.0。

## 5. 新版多 Run 预期与评估方法

建议对同一个授权 SRC 业务连续跑 3–5 个全新 v8.8 Run，并固定授权范围、账号角色和版本窗口。每轮至少记录：

1. 新增 canonical inventory surface 数、unresolved method 减少数；
2. 新关闭的精确 cell 数，以及 confirmed / not_vulnerable / not_applicable 分布；
3. 新增 proof-confirmed root fingerprint 数、重复 finding 数、候选转 confirmed 比例；
4. 重测已闭 cell 的比例和被证据失效重新打开的比例；
5. 靶场 weighted recall 或真实 SRC 最终 accepted 数；
6. 每轮边际新漏洞数和每个新漏洞的有效测试成本。

判定“多 Run 有效”的标准不是总报告数持续上涨，而是：项目 inventory 和精确 coverage 稳定扩展、重复测试率下降、proof-confirmed 唯一根因累计增加、旧证据失效能够自动重开，且 benchmark/平台 accepted 指标优于单次 Run 基线。

## 6. 剩余限制

- v8.8 不自动把旧 legacy 报告升级为 confirmed；旧数据必须重新跑或按新 Proof Contract 补证。
- 目标部署版本的自动 fingerprint/stale 判定尚未实现；当前以显式复验和证据完整性为准。
- Business Graph 仍是项目真值的派生视图，不应单独作为“已测完”依据。
- 多 Run 能提高覆盖利用效率和发现机会，但无法保证漏洞数量单调增加。
