# Atoolkit v8.13 动态执行闭环反向审核（2026-07-17）

审核对象：`design/迭代方案/迭代方案v8.13_威胁驱动的动态执行闭环.md`

## 第一轮裁定：REJECT

初始构想若直接实现，存在八个阻断风险：

1. 根据每轮新发现原地追加 threat cell，会破坏 v8.12 frozen denominator；
2. Execution Event 若只靠模型文本声明，会成为新的“说做了等于做了”；
3. Host 若把义务完成直接当成安全结论，会绕过 negative/proof validator；
4. 一个 evidence 文件可被滥用来声称完成全部义务；
5. 通用漏洞 checklist 若重新生成 cell，会退回 v8.10 之前的笛卡尔积；
6. Direct runtime 增强可能被误称为可信 authority；
7. 新状态机与 CognitiveState/coverage ledger 双写，可能产生新的投影分叉；
8. 历史 Run 没有 event，若无版本门会被批量回退并破坏兼容性。

因此初稿不得施工。

## 修订与复核

### 1. 不动态改写分母

修订后，新 endpoint/param 只进入 evidence-bound backlog 和下一 Run Planning，不直接成为当前 Run
cell。Experiment 动态发生在 frozen threat 内部，只改变执行队列和义务状态。

结论：通过。

### 2. 事件必须绑定物理证据

Event 的 surface/obligation/evidence ref 由 Host 校验，引用必须位于 Run 内且真实存在；Host 在事件
接受时写入 evidence SHA-256，final validator 重新计算后必须与 authority event 一致；create-only ID
防止后写覆盖。没有物理证据的“完成”不进入 progress，接受后替换证据也会使交付失败。

剩余风险：Host 无法通用理解每个响应的全部业务语义。因此 Event 只调度，不独立确立阴性或漏洞；
最终结果仍由 canonical negative/Finding assertions 验证。

结论：通过。

### 2.1 Barrier 恢复可达性

实现复核发现，如果 reducer 只对历史 barrier 做集合并集，空数据/会话等障碍即使已恢复也会永久
保持 active。修复后事件仍 append-only，但 Host 会在全部对应 recovery obligation 已绑定证据时
确定性消除 active barrier。未完成恢复义务时仍不能闭格。

结论：修复后通过。

### 3. 结果门保持唯一

accepted Finding 可以关闭对应 exact cell；`not_vulnerable` 必须同时通过既有 negative depth gate 和
execution obligations；Event 不能直接写 terminal ledger 状态。

结论：通过。

### 4. 证据复用

同一个 before/after 包可能合法满足多个相关义务，不能机械要求每义务一个文件。修订保留
`obligation → evidence_refs` 映射，并把语义真实性留给 packet assertion/negative validator；事件层
只拒绝零证据、越界和空文件。finalizer 输出映射供人工复核。

结论：通过，接受为显式剩余风险。

### 5. 不重新生成漏洞矩阵

确定性层只给当前已冻结 threat 增加实验深度义务，不增加 vuln class、Finding 或 sibling cell。
模型的 `evidence_required` 原文仍是主语义，Host floor 只防单请求、单身份、空数据等无效阴性。
IDOR/BOLA 的双身份 floor 不信任模型的 `identity_requirement=single`；XSS/SSRF/文件 floor 使用
“执行并记录成功或拒绝结果”的中性义务，避免安全端点因无法成功利用而永远不能形成阴性。

结论：通过。

### 6. Direct 信任边界

Direct 所有 status 固定 `authority_trusted=false/delivery_eligible=false`；缺 preflight/checkpoint 的
历史手工 Run 只可离线审计，不能回填成可信运行。

结论：通过。

### 7. 单一投影源

Execution contracts/events 是 append-only 输入，`execution-progress.json` 和 queue 是 Host 每轮重算的
派生投影。Coverage ledger 仍是最终七态权威；execution gate 只能把证据不足的阴性重新打开，不能自行
写 confirmed。最终 validator 同时核对两者。

结论：通过。

### 8. 兼容与启用范围

强制 execution closure 仅用于 v8.13 初始化的 `execution_contract_version=1` Run。旧 manifest/ledger
保持可读并显式 `execution_telemetry=absent_legacy`，不会伪装成已验证，也不被静默改写。

结论：通过。

## 实跑问题映射

| v8.12 实跑问题 | v8.13 控制 |
|---|---|
| 7 个已知方向完全未执行 | exact contract queue，未产生 event 的 cell 始终 open |
| 4 个动态端点未同步初始清单 | evidence-bound next-run backlog |
| 用户枚举只看消息 | 三通道 execution floor |
| 空 balance 数据判安全 | object/data prerequisite blocker |
| use_points 只测主值 | transaction boundary/state-delta floor |
| XSS、redirect 全漏 | 前提是 Planning 已建立相应 threat；queue 保证建立后不能跳过 |
| 10 个 Finding 全部 proof-invalid | validator rejection 进入 proof-repair queue |
| 报告含 secret | v8.12 Canonical renderer 保持唯一交付源 |
| 手工 Direct Run 无运行真值 | runtime preflight/checkpoint 缺失即 incomplete diagnostic |

## 最终裁定

**APPROVE FOR IMPLEMENTATION**，但批准范围严格限定为：

- frozen threat 内的 Experiment Contract、evidence-bound Event、Host reducer 和动态 queue；
- Direct diagnostic 与 Engine Threat Mode 使用同一合同；
- 阴性回退、proof repair、barrier recovery 和 next-run discovery backlog；
- finalizer/validator 一致性门、测试、文档和版本升级。

不批准：

- 当前 Run 由模型任意新增/删除 threat 或 cell；
- Event 替代 Finding/negative 证明；
- 根据具体 8180 靶场 endpoint 写特判；
- Direct 因有 execution state 就提升为 trusted/delivery eligible；
- 跳过失败先行、全量回归、doctor/self-check 和 regen 校验直接发布。
