没有任何厂商在专门训练模型的网络安全能力，所有厂商训练的，都是通用 Agent 能力
网络安全能力，是这套训练方式在攻防场景下的间接投影
类比来说：就像练了十年钢琴的人，手指灵活度和节奏感让他学吉他也比别人快——不是因为他练过吉他，而是底层能力发生了迁移
那么能力究竟从哪来？
对齐各家训练路线后，可以归纳出一套共同的能力形成路径，分为四层
第一层：预训练—知道
模型从 GitHub、技术文档、安全社区中吸收漏洞原理、利用思路与工具用法
安全内容只占训练数据的一小部分，但足以让模型建立基础认知：SQL 注入是什么、缓冲区溢出怎么回事、PKI 体系如何运作
各家做法略有差异：
DeepSeek-V4
描述了"Specialist Training"阶段，在特定领域数据上进行初步微调以建立基础能力
GLM-5
披露总训练 token 量达 28.5T
Kimi K2.5
采用 joint text-vision pretraining，让文本与视觉模态在预训练阶段就互相增强
这一层解决的是"听说过"——模型知道这些概念存在，但还不会用
第二层  SFT 与专家蒸馏—"理解"
各家普遍的做法是：针对数学、编程、推理、指令遵循等不同领域，分别训练专家模型，再通过 on-policy distillation 合并进统一模型
DeepSeek-V4 原文描述如下：
"A separate expert model is trained independently... then unified via on-policy distillation."
具体流程是：基础模型先在高质量领域数据上做 SFT，再用 GRPO 进行强化学习，最终以统一模型为学生、专家模型为教师完成蒸馏。
模型开始真正理解代码逻辑、配置语法与系统调用关系。理解代码，是理解漏洞的前提
有趣的是 DeepSeek 的实践证明，即使统一模型跳过了安全专项 RL，只要编程与 Agent 领域的专家模型足够强，通过蒸馏就能迁移出相当的安全能力
注：【蒸馏迁移的是代码理解层面的安全能力（看懂漏洞），不是自主执行层面的安全能力（打出漏洞链），且蒸馏出来的安全能力有明确天花板)】
在全新环境中自主打出完整的攻击链，这仍然是当前模型都未解决的问题，不过安全能力的获取路径，或许比我们想象的更为廉价
第三层，可验证环境下的强化学习—"试错"
安全能力真正涌现的起点
代码能不能跑通、漏洞能不能复现，变成了奖励信号,模型在真实沙箱里反复尝试，观察结果，修正策略—不再靠"背答案"得分，而是在环境里学会"试错"
以
GLM-5
为例，其可验证环境的构建分为两类：
软件工程环境
：用 LLM 自动从 GitHub 仓库构建可验证环境，pipeline 包括环境分析、依赖安装、测试命令生成、日志解析函数生成，最终构建出：
"over 10k verifiable environments across thousands of repositories spanning 9 programming languages, including Python, Java, Go, C, CPP, JavaScript, TypeScript, PHP, and Ruby"
终端任务环境
：用 LLM 从种子数据批量生成可验证的终端任务草稿，经去重与验证后构建训练集。
从被动记忆转向主动探索
。网络安全本质上是一件"动手的事"，而不是知识问答，这一层的训练逻辑与此高度契合
直接的实践印证来自
GPT-5 System Card
的 CTF 评测设计：
"To evaluate the model against the CTFs we give it access to a headless Linux box with common offensive cybersecurity tools preinstalled as well as a harness which allows the model to call those tools or otherwise execute commands similar to as a human."
第四层是Agent调用可靠完成任务
模型不再写一段文字告诉你"应该怎么做"，而是直接调用 nmap、Burp Suite、exploit 框架，像人一样在终端里执行命令、观察反馈、调整下一步。
Kimi K2.5
的 Agent Swarm 框架在这一层走得更远
传统 Agent 按顺序执行工具调用，而 Kimi K2.5 引入了 PARL 框架，允许模型动态分解复杂任务、创建子 Agent、并行调度执行
意味着模型学的不是"如何当黑客"，而是"如何分配和管理一项复杂任务"，网络安全只是这种能力的一个应用场景

安全能力的真正涌现点在第三层之后。前两层给了模型"知道"和"理解"，后两层给了模型"试错"和"行动"的能力
网络安全需要的恰恰是后者，而模型厂商现在主要发力点也在这里
由此我们能得出什么？
1:安全能力是通用 Agent 能力的投影，没有人在专门训练它
厂商的投入都集中在 Agent 能力、可验证环境、长程推理上安全能力是这套训练方式照进安全场景的影子
在当前范式下，安全能力的天花板由通用能力的进度决定，而不是由安全领域的专项投入决定
代码越强、推理越长、Agent 越自主——安全能力的下限跟着水涨船高，无需单独驱动
安全场景要求进入一个从未见过的网络环境，不知道目标跑了什么服务、不知道配置逻辑
不知道防御机制，需要自主探索、假设、验证、失败后换路线，天然命中当前Agent训练范式
2:可验证环境的密度，比参数规模更关键
模型参数增大，能力提升是边际递减的  但给模型构建十万个有真实反馈的沙箱环境，驱动效果反而更显著。
当前各家厂商大量工程精力的真正去向不是让模型变得更大
而是让训练环境不崩溃,反馈信号不失真。环境越复杂，噪声越大，工程难度越高
3:当前模型的真实瓶颈不是知识，是长程自主执行
安全知识，模型早就够了。真正欠缺的是
在复杂环境中持续推进、遭遇失败后自主换路线的能力
拿网上流传的"模型秒杀 CTF 题"的演示视频，很多大概率是模型在复现训练数据中的已知解法
而非真正的独立推理CTF 题库公开已久，训练数据污染是大概率事件
判断模型真实安全能力，应该看 Cyber Range 的得分，看模型在全新场景下的因果链推进能力
而不是看它能不能解出一道已经在网上流传多年的题目。
随后我又想到 既然安全能力是通用能力的副产品，辨证思维来看，那防御能力同样如此
模型完全可以做代码审计、日志分析、漏洞修复，这些防御场景与进攻场景在技术底层几乎是对称的
但现实是，这些防御应用获得的工程投入和公众关注，远远低于"模型会不会成为黑客"的恐慌叙事，不禁莞尔一笑
