# 也许我们不需要Skill

注:本文为自我观点
关于Skill，一直是热门话题各种AI渗透一键化""代审自动化"Skill满天飞
仿佛给大模型挂上Skill就能实现数码宝贝超进化
颇有当年各种社区求神奇prompt的味道—拿回去指望模型服下立地成仙
真是如此吗？
我理解的Skill
Skill是一段在特定时刻被完整注入对话上下文的行为约束文本
当你输入
/skill-name
时，整个SKILL.md的内容被当作一段文本
追加进当前对话窗口 和系统提示、CLAUDE.md、Rules共处同一个上下文空间
Skill越长，挤占的有效上下文越多
多个Skill同时加载时，之间存在互相冲突性
Skill不是插件，不是外挂能力模块，更不是什么神秘的增强包 本质一段被拼进去的文字
模型读到这段文字后，补充知识，调整自己的输出倾向
测评基准
我选了三个基准测评    筛选原则：通用广泛，视角独特
SWE-Skills-Bench
—专注软件工程，测试开源社区流通的Skill在真实代码库中的效果
SkillsBench
—多机构联合发布的通用基准，设计86个跨领域任务覆盖11个领域
量化"有Skill" vs "无Skill" vs "模型自生成Skill"的三方对比
SkillReducer
—追问"为什么有些Skill看起来有用其实没用"，观察真正起作用的区域
无效是常态
SWE-Skills-Bench在真实软件工程场景里测试了
49个Skill
结果：39个—
80%
—对任务成功率零影响。另有3个因版本冲突反而拖后腿，性能最高下降
10%
换句话说，随机从marketplace装一个Skill，八成概率只是占了上下文的位置。还有小概率会坏事
内容空心化
SkillReducer分析了
55,315个公开Skill
，发现：
26.4%
缺少路由描述（模型不知道何时该调用这个Skill）
60%以上
的主体内容删掉后不影响输出质量
引用的外部文件每次调用注入无效token
系统性地移除冗余内容后，功能质量反而提升了2.8%。 少既是多
AI自己写的Skill更差
SkillsBench的三方对比显示：让模型挂载自己生成的Skill，
平均性能下降1.3个百分点
整体来看，AI自生成Skill的期望收益为负
在没有人工校准的情况下，模型倾向于生成冗余、泛化、缺乏针对性的Skill
恰好踩中了前面说的所有坑
为什么大部分Skill无效
为什么Skill没用？
从LLM的工作机制反推  模型接收到一段Skill文本后，能产生的作用有三种：
第一，路由—把模型的注意力集中到正确的知识区域
模型预训练时学过海量知识，但面对一个具体问题时
未必能自动激活最相关的那部分  好的Skill像一把钥匙，打开正确的知识抽屉
第二，补充—提供模型训练数据中没有的私有信息。
企业内部的SOP流程、团队的命名约定、项目特有的架构决策
这些东西模型没见过，需从外部注入
第三，约束—压缩输出空间，排除不符合要求的模式
告诉模型"不要做什么"，把无限的可能性收窄到你需要的范围内
满足任一维度，Skill有存在的价值  都不满足,那只是一段占位的噪音
无效Skill
大部分在重复模型已经知道的常识（不路由），
没有包含任何私有信息（不补充）
表述过于宽泛模糊、模型无法执行（不约束），自然无效。
什么样的Skill有效
案例一：负面约束型（作用于维度三：压缩输出空间）
Banned Patterns
禁止使用 "In today's rapidly evolving landscape"
禁止使用 "game-changer"、"cutting-edge"、"revolutionary"
禁止无证据的模糊断言
禁止在没有数据支撑时使用"显著提升"
不教语法，不教结构，不教修辞    只做一件事—列出禁止清单。
为什么有效？大模型对负面约束的遵循度远高于正面建议
"写得好"是开放命题，模型有一万种解释方式
"不要出现game-changer"是封闭命题，可以直接验证是否遵循
案例二：决策路由型（作用于维度一：精准路由）
观察到的现象            优先测试方向
用户输入反射到HTML/JS   XSS / SSTI
服务端主动访问URL       SSRF
接收XML/Office/SVG     XXE
API中存在大量对象ID     IDOR / BOLA
模型预训练数据里已经有XSS、SSRF、XXE的完整知识
安全测试最大的难点不是"怎么测"，而是"先测什么"
用可观察的现象作为钥匙，引导模型打开正确的知识抽屉  不教知识，只做路由
案例三：检查清单型（作用于维度三：约束 + 防遗忘）
Steps: Build → Type Check → Lint → Test → Security Scan → Diff Review
Output format:
Build:   [PASS/FAIL]
Types:   [PASS/FAIL] (X errors)
Lint:    [PASS/FAIL] (X warnings)
Tests:   [PASS/FAIL] (X/Y passed)
Security:[PASS/FAIL]
模型在多步骤任务中有遗忘倾向—执行到第四步时可能忘了第二步还没做完
检查清单在上下文中持久化一个待办列表
让模型能回头核对  输出格式模板进一步收窄行为空间
怎么组织一个Skill
知道什么内容有效，还要知道怎么编排 混乱的堆砌会让模型在加载时找不到关键指令
推荐结构：
name: your-skill
description: What it does. Use when user asks to [specific trigger phrases].
Instructions
Step 1: [First Major Step]
Step 2: [Second Major Step]
Examples
Example 1: [common scenario]
Constraints
[Hard rule 1]
[Hard rule 2]
Troubleshooting
Error: [Common error message] → Solution: [fix]
原则：
重要指令置顶
模型对上下文开头和结尾位置的信息回忆率最高，中间位置最容易被忽略
不要把核心约束埋在中间地带
Skill陷阱
如果Skill符合以下任一描述，大概率无效甚至有害
陷阱一：常识百科型
把KISS、DRY、YAGNI、"变量名要有意义"写进Skill。
这些是编程常识，在模型预训练数据里出现了成千上万次
重复写入不会让模型"更加遵守"，只会浪费上下文空间
验证方法：
直接向模型提问相关内容
如果它回答的质量与你Skill里写的相当，那这个Skill就没有存在必要
陷阱二：触发过宽型
Skill的description写成"处理数据相关任务"或"AI开发相关"—几乎所有请求都能匹配。
结果：不相关的场景也会加载这个Skill，导致指令冲突
模型被迫在互相矛盾的指令间做取舍，输出质量反而下降
陷阱三：单文件大全型
把CTF解题、漏洞分析、渗透测试、代码审计、应急响应一股脑塞进一个600行的SKILL.md。
前面提到的"Lost in the Middle"效应在这里生效：
上下文过长时，中间位置的内容更容易被模型忽略
精心写关键指令，可能从未被模型认真对待
陷阱四：空壳包装型
My Awesome Skill
Read and follow instructions in ./real_instructions.md
内容只有"去读另一个文件"。多一层无意义的间接引用
增加延迟，浪费token。用户完全可以直接引用目标文件
陷阱五：互相冲突型
Skill不会单独运行。模型可能同时加载2-3个Skill
如果Skill A说"代码要详细注释"，Skill B说"代码要极简无注释"，结果会比什么都不用更糟。
避免冲突的原则：
命名空间隔离：
用前缀消除歧义，如
github_list_prs
vs
slack_send_message
职责单一：
一个Skill只做一件事
不重复声明：
如果Rules或CLAUDE.md已经定义了某条策略
Skill里不要再写一遍（重复声明可能产生微妙的表述差异，反而引发冲突）
自检问题
无效示例
有效示例
1
这个信息模型本来就知道吗？
"写代码要用有意义的变量名"
"我们项目API响应必须用
{success, data, error}
信封格式"
2
能用约束而非建议表达吗？
"你应该考虑类型安全"
"函数超过50行必须拆分"

触发条件足够精确吗？
"处理数据" / "AI相关"
"Use when setting up Docker Compose for microservices"

会和用户请求冲突吗？
Skill说"写小说"，用户说"解释Transformer"
Skill说"按CTF Writeup模板输出"，用户说"解这道题"

600行覆盖6个不相关领域
100行路由 + 一个300行专题Skill
验证
写完Skill后，做三轮测试：
触发测试：
跑10-20个用例，确认Skill在应该触发时加载、不该触发时静默。
功能测试：
同一请求跑3-5次，观察输出结构是否一致、核心约束是否被遵循。
对比测试：
同一任务，分别在有Skill和无Skill的条件下执行，对比token消耗、工具调用次数、输出质量

我的思考
日常保持
5-8个
精选Skill持续打磨，效果最优
超过
10个
同时加载，出现明显边际递减
单个Skill建议控制在书写量，少既是多，约束>期望
模型不同领域的预训练数据深度不同，Skill效果差异极大
不要在模型原生能力已强的领域过度投资
（如软件工程）
当前市面Skill大致分三类，我认为命运各不相同：
类型
定义
举例
趋势
常识重复型
教模型已经会的东西
"遵循SOLID原则"
模型越强越无用，逐渐消亡
激活引导型
用少量上下文唤醒模型沉睡的领域知识
安全测试决策矩阵
长期保留——模型学到的知识
在目前比它默认调用的多
约束对齐型
告诉模型"不要做什么"，补充私有规则
团队代码规范、合规红线
模型越强约束越重要，持续强化
演化方向：
常识重复型消亡
—模型自己会了，外部不需要再教
激活引导型保留
—模型学到的知识永远比它默认调用的多，总需要一把钥匙打开正确的抽屉
约束对齐型强化
—团队约定、合规要求、行为红线无法被通用模型内化，反而成为差异化竞争力的来源
私有知识型迁移
——高价值的私有知识会逐步被新的检索方案接管，不再依赖Skill做载体
Skill的未来形态会从操作手册变成法则—数量更少，抽象层级更高，影响范围更大
"函数超过50行必须拆分"是操作手册。"任何代码变更必须通过安全审查后才能合并"是法则。前者会被模型内化，后者是需要外部声明
Skill是杠杆，不是咒语  用对了放大能力，用错了放大噪音
与其写10个平庸的Skill，不如维护2个精准的。与其追问"怎么写出更好的Skill"，不如先停下来问一句
这个Skill做的事，模型自己不能做吗？
如果答案是"能"，那你不需要这个Skill
我们不需要重复教AI做事的Skill。我们需要的，是激活它沉睡能力的钥匙，和让它不越界的红线
