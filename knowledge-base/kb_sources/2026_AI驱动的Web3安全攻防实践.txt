AI 驱动的代码审计
如何信任 AI 的审计结论？

AI 审计的四大难点
1
上下文窗口是有限资源
审计是高密度信息任务，长度 ≠ 质量
简单堆叠代码会导致注意力稀释
2
LLM 单次生成范式下的审计惰性
生成机制倾向报告显眼问题
经常忽略需要复杂状态推导的深层逻辑缺陷
3
RAG 召回质量决定参考价值
注入什么比注入多少更重要
召回错误的安全模式会导致垃圾进，垃圾出
4
单模型覆盖缺口与概率性遗漏
单模型存在结构性的漏洞识别盲区
且单次生成的结论缺乏外部交叉验证机制

难点一：上下文窗口是有限资源
认知陷阱：
上下文长度与审计质量不成正比
核心瓶颈：
有效注意力的分布问题，而非窗口大小
精确带入其调用链 + 相关状态变量
从“喂更多”到“喂更准”
工程解法：

难点二：LLM 单次生成范式下的"审计惰性"
行为倾向：
单次生成模式下，优先报告最显眼的、
最有把握的问题。
LLM 生成本质是概率采样：
大模型在回答时，做的是"最可能正确的生成"，
而非"穷尽所有可能的检查"。

难点三：RAG 召回质量决定参考案例价值
1
召回什么？
纯语义检索有盲点，BM25 有漏洞；检索策略错误则质量从源头打折
1
召回多少？
Top-K 的 K 是真实权衡——案例太少覆盖不足，太多引入注意力稀释
1
知识库质量
召回上限由知识库决定，描述模糊 / 标注错误 → 垃圾进，垃圾出(GIGO)

难点四：单模型结构性缺口 + 概率性遗漏 + 无法自验
1. 结构性覆盖缺口
非随机的系统性盲区：
不同模型在漏洞空间上存在稳
定的能力分布偏差
对特定漏洞类型可能“选择性
不敏感”
2. 概率性遗漏
生成的不确定性：
单次生成是高概率路径的一次
采样，而非完整搜索
同一输入可能得到不同结果
3. 无法自我验证
闭环缺失的致命伤：
在单模型单次生成下，缺乏外
部参照
无法判断“没发现”是“没有”
还是“没看到”

系统设计：四层架构简览
核心设计原则：每一层都在减少搜索空间、提升信号密度，使最终结论更接近“可验证的审计判断”
Layer1 AST 解析 + 调用链分析
精确提取结构信息，控制输入，限定 LLM 的关注范围，减少注意力稀释
Layer2 RAG 检索历史参考案例
真实漏洞知识注入，提供可复用的经验，而非仅语义相关信息
Layer3 多模型并行审计 + Memory
多视角独立分析，控制覆盖，历史记忆沉淀为可复用审计经验
Layer4 LLM-as-Judge 汇总 + 反思
语义去重 + 置信度分层，控制输出质量，将多路径输出转化为可决策的信号

第一层：AST 解析 + 调用链分析
核心逻辑：先看结构，不让 LLM 直接面对原始文本
提取的核心内容
函数信息
包括函数签名、可见性、修饰符（modifier）等
调用链
揭示函数间的依赖和影响传播路径，避免孤立分析
状态变量读写关系
定位状态突变的源头，是判断重入等漏洞的核心
线索
外部交互关系
分析与外部合约的交互，以及调用发生在状态更新
之前还是之后

第二层：RAG 检索历史参考案例
知识库四大来源：构建高价值的领域专业上下文
来源渠道
提供价值
公开审计报告
来自主流安全团队的审计报告，覆盖真实项目中的漏洞模式与修复方案
漏洞赏金报告
来自各大 Bug Bounty 平台的公开披露报告，提取真实攻击场景与利用逻辑
内部审计知识库
SlowMist 多年审计过程中沉淀的案例库、漏洞模式归纳与审计方法论
真实链上攻击事件
SlowMist 多年积累的历史真实攻击事件，结构化为可检索的攻击模式样本
核心：模型拿到的不是通用安全知识，而是真实发生过、被验证过的漏洞参考

数据清洗与 Embedding 策略
核心原则：code-to-code semantic retrieval + structured metadata augmentation
早期问题：整份报告 Embedding
噪音淹没信号：
审计报告中包含大量叙述性内容，关键漏
洞代码信号在向量空间中被弱化
语义偏移：
如果将整段文字一起进行向量化，导致
检索结果“相关但不可用”
改进方案：代码剥离与元数据分离
提纯代码特征：
从审计报告中抽取漏洞相关代码片段，剥离
背景描述与分析性文本，仅对纯代码片段进
行 embedding，将表示空间收敛到“代码
级语义”
元数据后置注入：
漏洞类型 / 严重程度 / 影响范围 / 修复建
议等信息以 metadata 形式独立存储，在召
回后进行结构化注入

混合检索策略 (Hybrid Search)
RRF：融合 Dense 与 Sparse 的排序结果，使多路检索信号共同决定最终排名。
Query: 代码语义 Embedding + AST 结构特征
Dense (语义向量)
语义相似漏洞召回（semantic similarity）
Sparse (BM25)
精确命中高特异性危险 (lexical matching)
↓ RRF 融合排序
Top-K 参考案例（漏洞代码 + 类型 + 描述 + 修复建议）

第三层：多模型并行审计 + Memory 系统
多模型并行
独立视角分析：
可选 20+ 类模型并行独立分析
代码，并各自输出结构化审计
结果，从不同模型视角进行交
叉验证
共识验证机制
多模型同时发现：
若多个模型同时识别出相同漏
洞模式 → 提升置信度
若仅单一模型识别 → 保留结
果但降低置信权重
Memory 系统
审计经验沉淀：
每次审计完成后，进行结构化
沉淀，并写入长期记忆库，在
后续相似项目审计中，这些历
史记忆将作为额外上下文注入 
RAG 流程

第四层：LLM-as-Judge 汇总 + 反思
核心目标：对多模型结果进行聚合与裁决，输出可解释、可追溯的审计结论
核心任务
语义归一
识别不同模型对同一漏洞的不同表述，合
并为统一问题实例
结果聚合
将多模型输出的候选漏洞进行分组与整合
输出成果
去重后的漏洞实例集合
基于结构化表示进行语义归一，将多模型对
同一问题的不同描述合并为统一漏洞实例
结构化审计报告
每个漏洞包含：代码片段、漏洞类型、影响
描述、修复建议及置信度信息，确保结果可
解释、可追溯
置信度评估
基于多模型一致性与证据强度，对每个漏
洞进行置信度排序
冲突处理
当不同模型结论不一致时进行裁决或标记
置信度分层与 Consensus 标注
基于多模型一致性对漏洞进行分级

系统设计的核心逻辑
本质：用领域数据与审计经验来约束和引导模型行为，而不是依赖模型的自由生成。
架构层级
控制对象
AST + 调用链
输入约束
RAG 案例
知识增强
审计指南
策略约束
多模型 + Judge
决策与验证
在安全审计场景中，可验证的推理路径比单次正确答案更重要。
实现手段与目的

AI 驱动的链上攻击分析
AI 能否在攻击发生后独立完
成根因分析？

链上攻击分析的目标
一笔攻击交易发生后，系统必须能够独立回答以下三个问题：
1
攻击者如何获利
资金流向分析：
资金从哪个合约流出，通过什
么操作路径，最终拿走了多少
2
漏洞根本原因
精准定位缺陷：
哪个合约的哪个函数存在什么
具体缺陷
3
完整攻击步骤序列
还原攻击链条：
从第一个调用到最终获利
的完整链条

分析的现状与工具局限
所有定位和推断仍然高度依赖人工经验
庞大且复杂的调用树
现状：
一笔攻击交易背后的的调用树可能有成百
上千个节点
面对海量的内部调用，看到密密麻麻的调
用层级，每个节点都可能是关键路径，也
可能是噪声。根本原因的分析很大程度依
赖于人工经验
现有工具的局限
可视化 ≠ 根因分析：
Tenderly 等强大的工具确实提供了非常详
细的调用链可视化，但它们只是展现了“
发生了什么”
但工具不会主动告诉你“该看哪里”
也不会告诉你漏洞利用的“原因是什么”
这正是我们希望通过 AI 自动化解决的难点

链上攻击分析的三大难点
自动化分析的本质阻碍，不在于算力，而在于逻辑架构
1
信息分层与注意力稀缺
暴力注入的失效：
即便大模型支持百万Token，
将全量Trace数据暴力注入依
然会导致严重的注意力稀释。
信息必须按需加载。
2
假设驱动的非线性过程
不可预先规划：
分析不是流水线作业，而是"建
立假设->定向寻找->验证"的动
态非线性过程。传统的 Plan-
and-Execute框架会直接锁死分
析路径。
3
领域知识需精确路由
阶段感知的缺失：
单纯的语义相似度检索，无
法感知当前 Agent 处于分
析的哪个阶段，容易召回错
误知识造成推理干扰。

难点一：信息是分层的，注意力是稀缺的
分析过程本身必须是主动的信息请求序列，问题不在“信息不够”，而在“信息何时进入”
认知陷阱与根因
认知陷阱：长窗口依赖
即便当前数十万Token的上下文窗口已成标
配，但注意力 (Attention) 仍是有成本的稀
缺资源,且推理成本随内容长度呈线性甚至
指数增长。原始Trace数据如果不加过滤地
全量注入，会导致严重的"上下文漂移"，使
模型遗忘最初的分析约束。
工程解法方向
正确策略：
在对的时机，注入对的信息。
这意味着分析不能是一个"一把梭"的生成任务，
而必须被设计为一个主动的信息请求序列。模
型必须像人类专家一样，决定:
•
先看什么信息，再看什么信息?
•
何时深入具体的某个调用?
•
何时跳过无意义的调用?

难点二：分析是假设驱动的非线性过程
人工分析的真实逻辑：两条线交替推进，假设不断被验证或修正
真实分析路径推演
//资金流与调用路径的非线性交织
1.发现资金流异常/潜在获利点
  -> 建立假设: 可能是价格操纵或闪电贷
2.根据假设，定向寻找特定调用路径
  -> 发现可疑的关键函数调用
3.假设得到部分支持，继续深入
  -> 跟进受害合约内部状态变更逻辑
4.最终确认:逻辑闭环
  -> 锁定漏洞根因与完整步骤
本质区别与框架局限
双线交替推进：
分析并非单向线性，而是"资金流"与"调用路
径"两条线在假设的驱动下交替推进、互相印
证。
为什么不能用 Plan-and-Execute?
与传统的"按顺序执行预定工具调用"有本质区
别。无法预知下一步会发现什么，任何预先制
定的静态计划都会限制真实的探索路径。

难点三：领域知识需要精确路由
不是“语义检索问题”，而是“按攻击类型与分析阶段进行精确路由的问题”
不同攻击的特征信号
不同攻击类型的识别信号与验证路径完全
不同：
价格操纵 → 关注 交易行为模式
重入攻击 → 关注 调用结构特征
⇒ 需要 routing，而不是 semantic retrieval
语义检索的局限性
语义相似 ≠ 攻击类型相似
传统的语义相似度检索无法精确区分这些完全
不同领域的专业知识，很可能在一次检索中把
它们混在一起召回，给 Agent 造成严重的推
理干扰。
更根本的架构缺陷
传统的RAG系统是外部被动触发的，它完全感
知不到Agent当前处于分析的哪个阶段，也就
无法按需路由特定的知识。

超越 ReAct：为什么需要不同的推理策略
ReAct（Thought → Action → Observation → Thought → ...）的局限：
来源渠道
具体局限与影响
状态完全依赖上下文窗口
内容增多导致上下文漂移，早期关键信息被噪声淹没
无假设管理机制
每个 Thought 是即时响应，无法管理“当前假设是否在被验证”
Plan-and-Execute 不适用
攻击分析路径不可预先规划，预先规划会锁死分析路径
ReAct 解决的不是“怎么做复杂分析”，需要专门设计的推理架构，超越传统 ReAct 范式

假设驱动的定向探索
Search Strategy，而不是 Prompt Strategy
基于专家经验抽象的工程化路径
资金流异常
建立可证伪的具体假设
定向展开调用路径(每次展开必须有明确动机)
验证/修正/推翻假设
资金流+调用路径(两条线交替推进)
两条线交汇点 = 潜在漏洞利用点
核心流程解析
动态演进
假设在推进中不断被验证、修正或推翻
双线交织
资金流(宏观)与调用路径(微观)交替寻找两
条线的交汇点以确认根因

"资金流 + 调用路径"双线推理机制
核心机制：LLM 启发式搜索与工具层确定性验证解耦同步
资金流视角
概率性怀疑
提供：概率性信号（suspicion）
作用：定位潜在获利点
特点：不确定，但覆盖广
调用路径视角
逻辑性实锤
提供：确定性验证（proof）
作用：确认具体漏洞与利用路径
特点：精确，但依赖目标
类似符号执行（确定性）× 模糊测试（概率性）的联动逻辑，两者单独用各有局限，启发式指明方向，
确定性落实细节，联动才最高效。

不再信任 LLM 的工作记忆
我们不让模型“记住”，而是让系统“知道”。
问题现象
20 轮以上深度分析中，原生 Context Window 像漏斗——早期关键线索被噪声淹没
本质洞察
这不只是记忆问题，而是认知对齐问题：Agent 需要每轮都清楚“我处于分析的哪个阶段”
解决方案：认知状态架构（Cognitive State Architecture）
当前认知快照不存储在对话 Context 历史里，而是由外部程序维护
结构化状态对象 + 每轮全新对话 + 完整注入 Prompt → 强制每轮认知对齐

认知状态架构：六字段状态对象
用结构化状态对象，显式表示 Agent 在任意时刻的“认知快照”
完整认知表示
六字段构成了 Agent 任意时刻
认知状态的完整快照。
原生审计日志
这就是一份自带的、机器与人
类均可读的 Audit Trail。
全程可追溯
每个假设的建立、推翻、修正
全程清晰可查。

认知状态管理：两阶段输出（CoT Decoupling）
如何在释放模型推理能力的同时，保证输出结构严谨、可控、可解析？
第一阶段：自由推理
无格式要求
目的：LLM 充分展开假设-验证的推理链，不受 
JSON 语法的拘束。
• 无格式约束
• 允许充分展开假设 → 验证 → 修正
• 最大化推理深度（Reasoning Capacity）
第二阶段：结构化输出
严格机器可解析格式
目的：将第一阶段的推理结果，转化为对六字段
状态对象的更新。
• 严格 schema
• 机器可解析、工具可使用
• 更新认知状态（State）
两阶段分离，让"想清楚"（自由发散）和"说清楚"（结构更新）互不干扰。

意图触发式动态上下文注入
检索触发从“文本相似度匹配”升级为“基于推理状态的主动知识调度”。
全量放入 System Prompt：信息过载，容易让 Agent 失去重点。
语义 RAG 检索：检索系统只看文本相似度，感知不到当前分析阶段。
Long Context + Caching：知识在 context 里，但模型不一定在对的时候关注对的部分。
意图触发式动态上下文注入 (Intent-Triggered Context Injection)
机制：Agent 在每轮状态更新中主动声明知识需求（由推理逻辑决定，非语义相似度猜测）
核心区别：检索触发的依据不是文本相似度，而是Agent 的逻辑推理意图。
